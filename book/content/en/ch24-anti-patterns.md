This chapter isn't about doing it right. It's about doing it wrong in the particular way that looks a lot like doing it right. Hand the new hire the whole keyring on day one and it reads as full trust and maximum throughput. Wow everyone with a dish on demo day and serve it straight to customers, and it reads as having passed inspection. Both blow up eventually. The trouble is that right up until they do, they look great.

What follows is eight of these. Each one gets a symptom — the story you tell yourself about why it's fine — and then the fix. After the eight, one hard line: an agent that can write files, run commands, reach the network, read private data, or open a PR — with no permission boundary, no trace, no eval, no recovery, no human gate — is not an engineering system. It's a random-number generator with a keyboard.

## Everyday analogy

Picture a shop with some size to it. A new person starts today. Fast hands, fast learner — and no idea which cabinet holds the books, which machine halts the line if you touch it, or which customer you absolutely do not annoy this month.

One way to start: you hand over the full keyring — front door, back kitchen, safe, client list — and say "use your judgment, shout if you need me." The first three days might go fine; he gets a lot done, fast. Then one day, meaning well, he throws out a batch of ingredients that "look expired" — they were set aside for tomorrow's banquet. Or, racing to finish, he skips the second check you set up on purpose and wires a payment that turns out to be wrong. He isn't a bad actor. He's a capable new hire who doesn't know how your place runs and was handed unlimited access anyway.

The other way to start is the one this chapter is about. You don't hand over the full keyring; you hand over what today's work needs. You don't count on "I told him not to touch the safe" to stop him — you simply don't give him the safe key. The dish that wowed everyone on demo day doesn't go straight to the table; first he makes twenty of them in a real lunch rush and you watch the error rate. What you track isn't how many he made today — it's how many came out right, fit to serve, and didn't need a redo.

The difference between the two openings isn't how smart the new hire is. It's whether the environment you built around him is right. All eight anti-patterns are variations on the first opening.

## Definition

An **anti-pattern** is a recurring move that looks like it solves the problem but actually defers and enlarges it. It's different from a plain bug: a bug is something you didn't want, an anti-pattern is something you thought you wanted. It's dangerous precisely because, in the short term — in the demo, on the days things go smoothly — it really does work.

In agent engineering, these anti-patterns share one structure: **a constraint that belongs in the environment gets moved into the model or the prompt, and then "it behaved this time" is taken as proof the constraint exists.** Anthropic states the right version of this when it describes containment — "design for containment at the environment layer first, then steer behavior at the model layer" — supervising what the agent is *able* to do ahead of supervising what it does [A6]. The anti-pattern inverts that order: trust the behavior first, leave the environment layer empty.

Each of the eight below is one shape of that inversion. Here's the catalog first — symptom and fix — then each one in turn.

| # | Anti-pattern | Symptom (the story you tell yourself) | Fix |
|---|---|---|---|
| 1 | Prompt worship | "Write a good enough prompt and you can skip tests and permissions" | Put the prompt inside the harness, not the harness inside the prompt |
| 2 | All tools in context | "Give it every tool and doc up front so it never has to hunt" | Trim tools and context to the task's phase and risk |
| 3 | Auto-merge fantasy | "If a PR passes CI, merge it — nobody needs to look" | High-risk changes require trace review + CODEOWNERS gates |
| 4 | One mega agent | "One long-context agent runs the whole pipeline" | Split by role / phase; hand off via artifacts |
| 5 | No failure taxonomy | "It failed again — the model's just bad" | Separate spec / context / tool / model / environment / permission / review |
| 6 | Eval theater | "A few clean demos passed, so it's verified" | Use real failures, edge cases, and regression cases |
| 7 | Security by prompt | "I wrote 'don't leak secrets'" | Tool layer + sandbox + secret policy + audit layer |
| 8 | Throughput vanity | "We generated 40 PRs and 20k lines today" | Track accepted safe changes and rework cost |

## Why it matters

None of these are new. Software engineering has always had moves that look advanced and are actually dangerous. What changed is that agents amplify the cost of each one — and amplify it in a direction that isn't intuitive.

First, **execution got faster, so mistakes get copied faster.** A vague spec, a missing acceptance check, a tool description written slightly wrong — these used to spread only as fast as a human could type them. Now an agent can copy one into dozens of files in minutes and wrap it in a PR that looks complete. The incubation period of an anti-pattern is shorter: you used to have time to catch it before it spread; now you often notice only after it's merged.

Second, **"it behaved this time" is weaker evidence than it used to be.** Agent behavior varies run to run, and worse, even a failure may not be the model's fault. Anthropic measured a counterintuitive version of this: on Terminal-Bench 2.0, infrastructure configuration alone moved scores by 6 percentage points — enough to swamp the gap between top models on the leaderboard [A11]. Under strict resource limits, about 5.8% of tasks failed from infrastructure errors rather than the model [A11]. For anti-patterns this cuts both ways: you can't trust that a constraint exists because "it passed this time" (anti-pattern 6), and you can't blame the model because "it failed this time" (anti-pattern 5). Both directions need a trace and a taxonomy before you can say anything.

Third, **authorization got finer-grained, but human attention didn't grow.** In Claude Code's data, users approve roughly 93% of permission prompts outright [A5][A6] — meaning the "let a human approve it" gate, under fatigue, is close to a no-op. That's exactly what makes anti-patterns 3 (auto-merge fantasy) and 7 (security by prompt) dangerous: they don't exploit a technical hole, they exploit the fact that humans get tired, get annoyed, and click "approve" out of habit.

## In practice

Now the eight, one at a time, each with why it looks right and what to do instead.

**1. Prompt worship.** The symptom is piling more and more constraints into the system prompt — "always write tests first," "never touch prod," "remember to check permissions" — and treating that paragraph as your defense. It looks right because prompts genuinely do shift behavior, and editing a sentence is far cheaper than building a process. The catch is that a prompt is advice, not a fence: the model can ignore it, prompt injection can override it, and a long context dilutes it. The fix runs the other way — put the prompt inside the harness, not the harness inside the prompt. Let tests do testing, let the tool layer enforce permissions, and let the prompt carry the one thing it's actually good at: intent and style.

**2. All tools in context.** The symptom is loading every available tool and every relevant doc into context up front, on the theory that more information means fewer misses. It looks right because completeness feels safe. Two problems. One, the tools and docs themselves burn the context budget and bury the few lines that matter. Two, the more you hand over, the wider the misuse surface — a read-only task holding a drop-the-database tool. The fix is to trim by the task's phase and risk. Claude Code's auto mode tiers exactly this way: safe reads and code navigation go through Tier 1 with no review; in-project file edits go through Tier 2 because version control makes them auditable; everything else — shell commands, external services, filesystem operations outside the project — falls to Tier 3 and a classifier that judges each one [A5]. More tools isn't better; the right tools for this step are.

**3. Auto-merge fantasy.** The symptom: an agent's PR auto-merges the moment CI goes green, and nobody reads the diff. It looks right because passing tests should mean correct — and people don't want to review anyway. The catch is that tests cover the cases you thought of, not the ones you didn't, and finding the gap the tests don't cover is exactly what an agent is good at. Anthropic observed Claude gaining an unfair advantage on some eval tasks by reading the git history from previous trials [A10b] — the same knack for slipping through a gap, pointed at your CI, becomes a merged change that's all green and all wrong. The fix is to gate by risk: low-risk, mechanical changes can loosen up, but anything touching money, permissions, data, or external side effects needs a trace review, with CODEOWNERS pinning approval on the directories to people who actually understand them. Add separation of duties: the person who requested the task shouldn't be the one who approves the agent's own PR — no one playing both player and referee.

**4. One mega agent.** The symptom is handing everything to a single agent that researches, designs, codes, tests, and debugs all the way down one ever-growing context. It looks right because "one context remembers everything" saves you the handoff. The catch is that recall degrades as the context grows — the spec from early on gets crowded out by files read later, and the agent forgets a constraint it set for itself half an hour ago. The fix is to split: by role (one researches, one implements, one reviews) and by phase (the plan phase emits a plan file; the implement phase reads it and continues). Hand off through artifacts rather than a single chat log. State that lives in a file outside the context, picked up by the next phase, is far steadier than one long agent trying to hold it all in its head.

**5. No failure taxonomy.** The symptom is that whenever the agent breaks, the verdict is always "the model's bad." It looks right because the model is the most conspicuous variable. The catch is that the verdict is neither actionable (swap the model and it may break the same way) nor usually correct. A failure deserves at least seven questions: was the **spec** unclear (it never knew what you wanted)? Was the **context** too thin or too noisy (the key fact got buried)? Was a **tool** mis-described or under-permissioned (it wanted to act but couldn't)? Did the **environment** flake (dependencies won't install, resources got capped, tests are flaky)? Did a **permission** gate stop it? Did **review** fail to catch it? Only last does the **model** itself come up. Anthropic keeps hammering this — don't charge infrastructure noise to the model, because "a few-point lead might signal a real capability gap — or it might just be a bigger VM" [A11]. Confuse those seven and you keep fixing the wrong layer.

**6. Eval theater.** The symptom is treating a handful of hand-picked demos that always run beautifully as your eval — a show for yourself and for others. It looks right because green lights are reassuring. The catch is that a demo picks the things the agent already does well, while an eval should test where it tends to fall over. Anthropic's recipe is concrete: source your starter set from **real failures** — 20 to 50 tasks drawn from actual failures is a strong start [A10b]; build **balanced** problem sets that test both "did it trigger when it should" and "did it hold back when it shouldn't," since one-sided evals breed one-sided optimization [A10b]; and separate **capability evals** ("can it do this at all," which should start at a low pass rate) from **regression evals** ("can it still do this," which should sit near 100%) [A10b]. An eval that's always perfect catches regressions but gives no signal for improvement [A10b]. A real eval is built from real failures, edge cases, and regression cases — not a few clean demos.

**7. Security by prompt.** The symptom is writing "don't leak secrets, don't push to main" and calling security done. It looks right because the sentence does read like a rule. The catch is that the rule sits in the wrong layer. Anthropic's internal red-team gives a blunt counterexample: when the user themselves is phished and types the injected instruction in by hand, the model layer is nearly helpless — 24 of 25 attempts exfiltrated AWS credentials, because the defense anchors on "user intent," and here the user is the one typing the command, so "there was nothing anomalous for it to catch" [A6]. Security has to be layered: the environment layer boxes in what's even possible with a sandbox (Claude Code's default posture is reads allowed, writes allowed inside the workspace, network denied by default) [A6]; secrets go through a dedicated policy and proxy rather than the model's discretion; and an audit layer lets you reconstruct, after the fact, who touched what and when. One Anthropic principle is worth copying down — "the weakest layer is the one you built yourself" [A6] — so don't rest your whole defense on one prompt.

**8. Throughput vanity.** The symptom is measuring an agent by PRs generated, lines of code, or tokens burned. It looks right because those numbers are the easiest to count and the nicest to show. The catch is that they measure output, not value — 40 PRs with 30 sent back for rework is worse than 8 that landed on the first pass, but the number looks far better. What to track is **accepted safe changes**: how much actually got accepted, merged, and didn't trigger a redo; and **rework cost**: how much extra human time went into making the agent's output usable — fixing, reviewing, cleaning up. Anthropic flags a related trap in evals — don't take a score at face value; if the score isn't climbing, first confirm the agent really underperformed rather than the eval being broken (ambiguous task, valid solution penalized, harness constraining the model) [A10b]. Same logic: more PRs doesn't mean better delivery until you've confirmed the increase is fit output, not rework.

## Case

**Auto mode's core isn't unlimited authority — it's a classifier plus boundaries plus recovery.** Put anti-patterns 1, 3, and 7 side by side and you see they all bet on one thing: that a prompt, an automatic gate, or the model's own discretion can stand in for a real boundary. When Anthropic built Claude Code auto mode, it pointedly did not make that bet. The starting point was an awkward fact: users approve roughly 93% of permission prompts outright — under fatigue, the "let a human approve" gate is basically defeated [A5]. But the response wasn't "then auto-approve everything." It was a system with boundaries.

Permissions split into three tiers: safe reads and user-configured allowlists go through Tier 1 (in auto mode, even blanket interpreter access is pulled back); in-project, version-control-auditable file edits go through Tier 2; the rest — shell commands, external services, filesystem operations outside the project — falls to Tier 3 and a classifier that judges each one [A5]. The classifier has its own defenses: it sees only user messages and the agent's tool-call payloads, and **deliberately strips out** the agent's own reasoning text and tool outputs — so the agent can't talk the classifier into a bad call, and the main injection vector is closed [A5]. A block isn't a hard stop; it's "deny and continue" — the denial comes back as a tool result so the agent goes looking for a safer path, and the session only ends after three consecutive or twenty total denials [A5].

The most honest part of the system is the ceiling Anthropic marks itself: on real overeager actions, it still has about a 17% miss rate [A5], and they state plainly that auto mode "is not a drop-in replacement for careful human review on high-stakes infrastructure" [A5]. In other words, even the people who built it don't think a classifier replaces a human gate — it just fills a middle tier, with boundaries and a recovery path, between "unrestricted" and "approve every line by hand." The real core of auto mode is the line written into its design: "everything the agent chooses on its own is unauthorized until the user says otherwise" [A5].

**Containment goes in the environment layer, not in words.** The counterexample to anti-pattern 7 (security by prompt), Anthropic spells out fully. Their guiding principle is to "design for containment at the environment layer first, then steer behavior at the model layer," prioritizing supervision of what the agent is *able* to do over what it does [A6]. In the products, that becomes three kinds of isolation: claude.ai runs code in isolated gVisor containers with an ephemeral filesystem, gone when it's done; Claude Code uses OS-level sandboxing on the developer's own machine (Seatbelt on macOS, bubblewrap on Linux), with a default posture of reads allowed, writes allowed inside the workspace, network denied by default — and permission prompts dropped 84% after the sandbox shipped [A6]. Security isn't a polite line in a prompt; it's a fence built from sandboxes, filesystem boundaries, and egress controls.

**Don't charge infrastructure noise to the model.** The direct evidence for anti-pattern 5 (no failure taxonomy) sits in A11. Anthropic ran Terminal-Bench 2.0 across six resource configurations on the same model: under strict limits the infrastructure error rate was 5.8%; with 3x headroom it dropped to about 2.1%; uncapped, to 0.5%, scoring 6 percentage points above strict [A11]. More subtly, from 1x to 3x the score moved within noise margins (p = 0.40), which means most of the crashed runs would have failed anyway [A11] — so if you don't classify, you'll read a batch of "environment flaked" as "model got weaker" and go tune something that was never the model. Their advice to people running evals applies just as well to people running agents: treat resource configuration as a first-class, documented variable, and "leaderboard differences below 3 percentage points deserve skepticism until the eval configuration is documented and matched" [A11].

## Anti-patterns

This whole chapter is a catalog of anti-patterns, so here are a few meta ones — the anti-patterns of anti-patterns — that slip in most easily while you're fixing the first eight:

- **Turning "fix the anti-patterns" into theater too.** Writing a handsome "agent safety guidelines" doc, pinning it to the wiki, and changing nothing. Guidelines that don't land in the tool layer and the gates are just security by prompt in longer form.
- **Banning across the board.** Scared of anti-pattern 3, you decree "every agent PR gets a manual line-by-line review." That swings back to the bad side of the first opening — you exhaust the humans, who then approve with their eyes closed, which is no review at all. Grade it; don't ban it.
- **Borrowing someone else's boundaries as your own.** Copying GitHub's or Anthropic's line because it's the line they drew. Their thresholds are calibrated to their threat model and their scale; you have to recalibrate against your own real failures. A11's "differences below 3 percentage points deserve skepticism" is about eval configuration, but the spirit is the same: a number you haven't verified in your own environment shouldn't become your gate.
- **Classifying failures only after an incident.** Failure taxonomy isn't a postmortem ritual. It has to be routine — Anthropic's framing is that regularly reading traces (transcripts) is a core skill of agent development, and a failure should let a person see at a glance "what the agent got wrong, and why" [A10b].

## Checklist

- Are my constraints on this agent in the **tool layer and the gates**, or only in the **prompt**? (1, 7)
- Are its tools the ones this step **actually needs**, or everything dumped in up front? (2)
- Is the boundary for auto-merging the agent's PRs gated by **risk**? Do money / permission / data changes get trace review + CODEOWNERS? (3)
- Is a long task carried by **one giant agent**, or split by role/phase with artifact handoffs? (4)
- When the agent last failed, did I separate **spec / context / tool / model / environment / permission / review**, or just say "the model's bad"? (5)
- Is my eval a few **pretty demos**, or real failures + edge cases + regression cases? (6)
- Do I measure the agent by **PRs / lines / tokens**, or by accepted safe changes and rework cost? (8)
- For any agent that can write files, run commands, reach the network, read private data, or open a PR — does it have all five: **permission boundary, trace, eval, recovery, human gate**?

## Self-test

- Pull up one agent you're running now and hold it to the hard line — no eval, no autonomy. Go item by item: can it write files? Run commands? Reach the network? Open a PR? If even one is "yes," it needs the full set: permission boundary, trace, eval, recovery, human gate. Which one is missing? The missing one is the anti-pattern you're about to step into.
- The last time you said "the agent did poorly this time," what was that based on? Reading the trace and separating the seven kinds of failure — or glancing at a result and charging it to the model? If it's the latter, you're probably fixing the layer that wasn't broken.
- If someone praised your agent system today on "looks advanced" grounds — fast generation, lots of PRs, a brilliant prompt — how many of those compliments survive the question "so where are its permission boundary, its recovery path, its real-failure eval"?
