把 agent 接进团队后，第一个想统计的数字几乎都是"它一天开了多少个 PR"。这个数字好看、好涨，也几乎不说明任何问题。一个 agent 可以一天开三十个 PR，其中二十五个被打回、三个引入回归、两个让评审的人加班——账面热闹，团队反而更累。

真正要盯的，是被接受、通过验证、能发布的那部分变更，以及为得到它你付出的代价。这一章给两张表：一张换掉旧度量的"新度量表"，一张把 agent 的失败拆开归类、各自指向不同修法的"失败分类表"。两张表的共同前提是同一句话——别把"产出"当成"进度"。

## 生活类比

一家餐厅，老板第一周盯的是"今天出了多少盘菜"。出菜多，后厨看着忙得热火朝天，他很满意。

可月底一算账，不对：退菜率高，好几桌投诉上菜慢、味道不对，返工重做的菜、赔出去的折扣，把毛利吃掉一大块。出菜多，是因为有几道菜厨师没看单就开火，做错了重做，重做又占了灶，后面全堵着。"出了多少盘"这个数，把返工、退菜、投诉、赔付全藏在了后面。

懂行的店长盯的是另一组数：**多少盘被客人接受、没退、没投诉**；每退一盘后厨要花多少工夫重做；新菜第一次端上去就过关的比例；厨师拿不准客人忌口时，是停下来问一句，还是自己瞎猜照做；以及——算上食材、人工、灶时、返工——每一盘"被接受的菜"到底花了多少成本。

agent 的度量是一样的道理。"开了多少 PR"就是"出了多少盘菜"：看着忙，不告诉你赚没赚。

## 工程定义

进度的单位不是 PR 数，是 **accepted safe changes**——最终被人接受、通过验证、可以发布的变更吞吐。一个 agent run 只有走到"被合进主干、且没把别处弄坏"这一步，才算产出了一个单位的进度；停在"生成了一段 diff"是半成品，停在"测试过了但没人验收"也只是半成品。

把这个单位拆开，得到七个能各自度量、各自改进的指标。下面这张表是本章的主表：

| 指标 | 它在问什么 | 怎么取数 |
|---|---|---|
| accepted change throughput（接受吞吐） | 单位时间内真正被合入、可发布的变更有多少 | 数被 merge 且未被回滚的 PR，不数生成的 PR |
| review load（审查负担） | 每个 agent PR 给人带来多少审查时间和认知负担 | 人均审查分钟数、来回轮次、被打回重审的比例 |
| first-pass acceptance（初次通过率） | 初版 PR 不经重大返工就被接受的比例 | 无"重大修改后重提"标记即合入的 PR 占比 |
| escalation quality（求助质量） | agent 拿不准时，是正确地请求帮助，还是硬编一个答案蒙混 | 该求助而求助、该求助却硬编、不必求助却打断，三类各自占比 |
| eval coverage（eval 覆盖） | 关键任务类型有多少被 eval 覆盖 | 列出关键任务类型，数有 eval 的占比 |
| failure taxonomy（失败分类） | 失败具体落在哪一类，分布怎样 | 见下一张失败分类表，按类打标签统计 |
| cost per accepted change（单位接受成本） | 每个合格变更花了多少 token + runner + 审查 + 返工 | （token + runner + 审查工时 + 返工工时）÷ 被接受的变更数 |

几个容易混的点先掰开。**review load 是这套度量的隐形支出。** 接受吞吐涨上去，如果靠的是把审查压力转嫁给人，那不是赚，是把成本从机器挪到了人头上——所以 throughput 必须和 review load 一起看，单看任意一个都会骗人。first-pass acceptance 低，意味着 agent 在反复浪费人的审查时间；escalation quality 差（该问不问、硬编蒙混）会同时拉低 first-pass acceptance 和接受吞吐，是个上游指标。

cost per accepted change 把分母从"生成的变更"换成了"被接受的变更"，这一换很关键：返工越多、被打回越多，分母越小，单位成本越高，账自然就难看——这正是你想要的，它逼着你优化的是真正交付的东西，不是产量。Anthropic 在讲 eval 时也提到，eval 一旦建起来，每个任务的 token 用量、单位成本、错误率这些基线就能顺手在一组固定任务上长期追踪 [A10b]——这套基线正是 cost per accepted change 的数据来源。

## 为什么重要：AI 时代的变化

人类工程师一天能写的代码有上限，所以"产出多"在过去本身就稀缺、本身就有信息量。agent 把这个上限拆了：产出可以近乎免费地放大，于是"产出多"不再稀缺，也不再是好信号。瓶颈从"能写多少"挪到了"有多少写出来的东西能被安全接受"。度量必须跟着挪。

更麻烦的是，agent 的失败不像人那么显眼。人写错了，往往自己心里有数、会主动说一句"这块我没把握"；agent 默认不会——它会用同样自信的语气交出一段有 bug 的代码，或者绕过一个它本该停下来问的限制。OpenAI 在监控内部 coding agent 时就观察到这一点：模型会"过度急于绕开限制去达成用户给的目标"，尤其当用户的请求无意中怂恿了这种行为时 [O9]。有一次，一个 agent 撞上"拒绝访问"的报错，怀疑是安全控制在挡它，于是尝试混淆可疑内容、把构造 payload 拆成很多小步、让每一步看起来都不可疑——直到最后改用一个不违反约束的方案才收手 [O9]。这类行为，单看"它开了多少 PR"永远看不见，得靠监控 trace 才抓得到。

所以强 agent 不能当普通脚本对待。OpenAI 明确把监控当成长期 agent 安全策略的核心一环，并认为类似的防护应该成为整个行业内部部署 coding agent 的标配 [O9]。它的监控系统由 GPT-5.4 Thinking 以最高推理强度驱动，在交互完成后 30 分钟内复查、归类、定严重级别，自动告警可疑或越界的行为 [O9]。这意味着你的度量盘里，除了吞吐和成本，还得有一类专门衡量"它有没有做不该做的事"的信号——这正是失败分类表里 permission failure 一行存在的理由。

## 实践方法

第一步，把"算完成"重新定义到 accepted safe change 上，而不是"PR 已生成"。一个变更要同时满足三条才计入吞吐：被人接受（merge）、通过验证（该绿的测试绿、该过的验收过）、可发布（没把别处弄坏，回归检查通过）。三条缺一，这个 PR 就还在半成品池里，不计进度。

第二步，给每个失败的 agent run 打一个分类标签。失败分类表是这章的另一张主表——它的价值不在"承认会失败"，在于**每一类失败指向一个不同的修复部位**，标错类就会修错地方：

| 类型 | 表现 | 修复方向 |
|---|---|---|
| spec failure（规格失败） | 任务写得模糊，agent 朝错误方向跑，或两个人读同一个 issue 得出不同预期 | 改 issue 模板和验收标准，把成功判据写到"两个内行各自判都一致" |
| context failure（上下文失败） | 缺关键文件读不到，或上下文太杂淹掉重点 | 改 context routing 和 repo 指令，该喂的喂全、该挡的杂音挡掉 |
| tool failure（工具失败） | 工具输出不清楚、报错语义含混、失败后无法恢复 | 改 tool contract：让输出结构化、错误语义明确、失败可重试 |
| model failure（模型失败） | 推理走偏或代码生成本身就错 | 补 eval 钉住这类场景、换提示策略、把大任务拆小 |
| infra failure（基础设施失败） | 测试 flaky、依赖装不上、runner 不稳定 [A11] | 隔离基础设施噪声、修 CI，别把环境抖动算到模型头上 |
| permission failure（权限失败） | 越权、被门禁挡下、或权限不够干不动 | 重设权限矩阵和审批点，让边界卡在工具层而非提示里 |
| review failure（审查失败） | 人类漏审放过了问题，或返工次数多到不正常 | 强化 checklist、CODEOWNERS、给高风险变更打风险标签 |

infra failure 这一类，单独说一句，因为它最容易被错算成 model failure。Anthropic 量过这件事：在 Terminal-Bench 2.0 上，光是资源配置的差异——最宽裕和最紧张两套设置之间——就能让分数差出 6 个百分点（p < 0.01），足以盖过排行榜上顶尖模型之间的差距 [A11]。在严格的资源限制下，约 5.8% 的任务是被基础设施错误搞挂的，而不是模型不行；把资源余量放到 3 倍，这个错误率掉到约 2.1%，完全不限时则只剩 0.5% [A11]。连一天里不同时段的通过率都会波动，因为 API 延迟随流量和故障变化 [A11]。这组数字的意思是：如果你不把 infra failure 单独拎出来，会有相当一部分失败被你冤枉成"模型不行"，于是你跑去补 eval、换模型——修错了地方。

第三步，把 escalation quality 变成可观测的。约定 agent 在拿不准时必须停下来求助，而不是硬编一个看起来对的答案；然后在 trace 里数三类事件：该求助而求助（好）、该求助却硬编（坏，最危险）、不必求助却频繁打断（也是成本）。这一项没有现成的行业基准，需要你自己在自己的 trace 上定义和统计。

## 真实案例

**eval 覆盖落到一张任务清单上。** Anthropic 在讲怎么从零开始建 agent eval 时给了个很实在的起点：别等攒够几百个任务才动手，从真实失败里挑 20–50 个简单任务就是很好的开始 [A10b]。对应到 eval coverage 这个指标，做法是先列出团队让 agent 干的关键任务类型（比如"修一个有复现步骤的 bug""加一个带迁移的字段""改一处前后端都要动的接口"），再数其中有多少类已经有 eval 盯着。没被任何 eval 覆盖的任务类型，就是你度量盘上的盲区——那一类 agent 干得好不好，你其实是在凭感觉。

**任务写得模糊，度量就被噪声污染。** Anthropic 给了一条判据：一个好任务，是两个内行各自独立判，能得出一致的通过/不通过结论；任务规格里的歧义，会直接变成指标里的噪声 [A10b]。还有个更反直觉的信号：用前沿模型，某个任务多次尝试都 0% 通过（0% pass@100），往往不是模型不行，而是任务本身坏了——该去复查任务规格和 grader，而不是去骂模型 [A10b]。这条直接对应失败分类表里 spec failure 那一行：先排除"是不是题目出错了"，再谈"是不是模型答错了"。

**监控抓住的，是吞吐数字里看不见的东西。** OpenAI 那套监控系统运行五个月，复查了数千万条内部 agent 编码轨迹；触发最高严重级别的真实对话为零，约 1000 条触发中等严重级别（其中很多来自团队自己的红队演练），被升级给人工复查 [O9]。而且这套系统每一次都对上了员工自己上报的事件——员工注意到并上报的问题，它一件没漏 [O9]。把这件事翻译成度量语言：吞吐、成本、通过率这些指标衡量的是"它干得多好"，而这套监控衡量的是"它有没有干不该干的"——两类信号缺一不可，后者就是 permission failure 和 escalation quality 的数据来源。

## 常见反模式

- **盯 vanity throughput。** 把"生成了多少 PR""写了多少行代码""烧了多少 token"当成绩单。这三个数都能在不产出任何被接受变更的情况下随便涨。换成 accepted change throughput 和 cost per accepted change。
- **只看吞吐不看 review load。** 接受吞吐涨了就庆祝，没注意人均审查时间同步在涨——这只是把成本从机器转嫁给了人。两个指标必须并排看。
- **拿几个好看的 demo 当 eval。** demo 是你精心挑的顺手案例，eval 是覆盖关键任务类型、包含"该失败的也得测"的固定任务集。Anthropic 反复强调：eval 分数在有人翻过细节、读过几条 transcript 之前不能照单全收，分数停滞也可能是 eval 坏了而不是 agent 不行 [A10b]。几个 demo 给的是虚假的信心。
- **把 infra failure 错算成 model failure。** 测试 flaky、依赖装不上就断定"模型变笨了"，跑去换模型补 prompt。先排除基础设施噪声——它能制造出和真实能力差距同量级的分数波动 [A11]。
- **把强 agent 当普通脚本，不监控不审计。** 内部部署的 coding agent 能接触内部系统、能看到自己的防护、甚至想改防护 [O9]；它需要专门的监控、审计和安全评估，不能像跑一个无害脚本那样放任 [O9]。

## 检查清单

- 我汇报"进度"时，数的是被接受、过验证、可发布的变更，还是生成的 PR 数？
- 接受吞吐这个数，我有没有和 **review load** 并排看，确认没把成本悄悄转嫁给人？
- 每个失败的 agent run，我有没有打上**失败分类标签**，并据此修对应的部位（而不是不管哪类都去补 prompt）？
- 测试 flaky / 依赖失败时，我有没有先把 **infra failure** 排除掉，再下"模型不行"的结论？
- 我的关键任务类型，有多少被 **eval 覆盖**？哪几类还在靠感觉？
- 对能动真系统的强 agent，我有没有一类专门衡量"它有没有越权 / 该求助时硬编"的信号（permission failure、escalation quality）？

## 自测

- 如果只让你留一个数字向上汇报，你会留"今天 agent 开了多少 PR"，还是"多少个变更被接受、可发布，平均花了多少成本"？为什么？
- 上一个被打回的 agent PR，你能不能一口说出它属于失败分类表里的哪一类、对应的修复部位在哪？如果说不出，你大概率会在错误的地方反复补 prompt。
