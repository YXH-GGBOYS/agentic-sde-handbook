你写的每一行代码，至少有两个读者：三个月后的自己，和那个能力很强、但完全不懂你家规矩的新外包——agent。变量名叫 `u`、`t`、`x`，人要猜，agent 也要猜。区别是，人猜错了顶多卡半天回来问你；agent 猜错了，它不会停下来，它会照着自己的猜测往下改，把错的地方改得很顺手。

写代码这件事正在变快变便宜。但读代码、改代码、确认改对没改错——这几件没变快，反而因为代码量上来了变得更贵。这一章讲的就是这个错位：当产出端松了，理解端就成了新的瓶颈，可维护性因此不是更次要，是更重要。

## 生活类比

设想你雇了个新来的帮厨，手脚极快，但对你这间厨房一无所知。你写了张便条贴在台面上：「u 切一下，t 到 3 放 x」。你自己看得懂——`u` 是洋葱，`t` 是计时器到 3 分钟，`x` 是那罐没贴标签的盐。但帮厨不知道。他看见 `t` 到 3，可能以为是火候档位 3；看见 `x`，抓起旁边那罐糖就放了进去。

他不会停下来问。他手快，照着自己的理解一路做下去，等你回头，一锅菜已经齁甜了。

现在把便条改一下：「洋葱切丝，计时 3 分钟后加一小勺盐」。同样一句话，帮厨不用猜了，照着做就对。多写的那几个字，不是给你看的——你本来就懂——是给那个不懂你家规矩的人看的，让他少犯一次错。

代码里的命名和结构就是这张便条。可读性不是写给已经懂的人看的装饰，是写给不懂的人看的接口。过去那个"不懂的人"主要是新同事和三个月后的你；现在又多了一个读得飞快、改得也飞快、但一样会猜错的 agent。

## 工程定义

这一节先把三个基础概念讲清楚，不假设你已经会。后面所有讨论都建在这三个概念上。

**可读性**，是代码把"它想干什么"表达得足够清楚，让读的人不用猜。判据很朴素：另一个人（或者半年后的你）打开这个文件，不查别处、不问作者，能不能看懂这段在做什么、这个分支为什么存在、这个变量装的是什么。看不懂就要猜，一猜就可能猜错。可读性差不是"风格问题"，是把理解成本从写的人转嫁给了每一个后来读的人——而读的次数远多于写的次数。

可读性主要落在两件事上。一是命名：变量、函数、类的名字应该说出它是什么、要干什么。`isEligible` 比 `flag` 多告诉你一件事——这个布尔值代表"有没有资格"，不是随便一个开关。二是结构：一个函数只做一件能用一句话说清的事，分支和嵌套不要深到读的人得在脑子里压栈。一段三百行、里面套了五层 `if`、混着网络请求和金额计算的函数，不是"功能强"，是没人敢动。

**重构 vs 重写**，是两条改善代码的路，差别在风险。重构（refactoring）是在不改变外部行为的前提下，改善内部结构——函数对外的输入输出、对调用方的契约都不动，动的只是里面怎么实现。把一个大函数拆成几个小函数、给一个含糊的变量改个名、把重复的三段合成一个，都是重构。它的安全网是测试：重构前测试是绿的，重构后还是绿的，就说明外部行为没变。重写（rewrite）是推翻重来，外部行为可能也跟着变，风险大得多，需要重新验证。日常该优先重构；只有当结构烂到重构的成本超过重写时，才考虑重写——而这个判断很容易高估自己、低估旧代码里藏着的边界情况。

**技术债**，是为了眼下的速度，向未来的维护成本借的一笔账。它本身不是坏事——为了赶一个 deadline 先用临时方案、跳过一层抽象、硬编码一个值，都是合理的借贷。坏的是借了不记、不还。债务的危险不在"欠"，在"忘了欠"：三个月后没人记得这里为什么这么写、当时绕过了什么、什么条件下会爆。好的做法是借的时候留一条记录——为什么临时这样做、风险是什么、什么时候、什么条件下该还。把隐性的债变成显性的、写下来的债。

这三个概念串起来是一条线：可读性决定别人（和 agent）读你的代码要花多少力气；重构是在不破坏行为的前提下持续降低这个力气；技术债是你主动欠下、并承诺偿还的那部分理解成本。三者共同的货币，都是"理解的成本"。

## 为什么重要：AI 时代的变化

过去有一种隐含的算账方式：写代码慢、贵，所以可读性、重构这些"为了将来好维护"的投入，常常排在"先把功能做出来"后面。反正以后真要改，再花时间读懂也就是了——读懂的成本，和写的成本比，不算什么。

agent 把这笔账的两端都改了。写这一端大幅变便宜：一个 agent 能在几分钟里产出能跑的代码。可读、改、验收这一端没跟着变便宜——而且因为产量上来了，要读要改的代码总量反而多了。当写的成本逼近零，瓶颈整个移到了理解这一端。可维护性因此被重新定价：它从"将来才需要兑现的投入"，变成"现在就在持续支付的开销"。

更关键的一点变化是：可读性现在是给人、也是给 agent 的接口。这话不是修辞。Anthropic 在讲 Claude Code 的用法时反复强调一件事：你给 agent 的指令越精确，需要回头纠正的次数越少——含糊的 "fix the login bug" 远不如点明症状、可能的位置、和"修好了长什么样"的版本来得管用 [A1]。同一个道理对代码本身成立：清楚的命名、明确的约定，就是 agent 读你这个仓库时的输入。一个叫 `user.isPremiumMember` 的字段，agent 一眼知道它代表什么、什么时候该用；一个叫 `u.t` 的字段，agent 只能从上下文猜——猜错了，它不会停，它会照着错的理解改下去。变量名差，对 agent 不是"不好看"，是直接喂给它错误的上下文。

agent 时代还多了一种专门写给 agent 看的"家规便条"——`AGENTS.md` / `CLAUDE.md`。它是 agent 每次开工都会读的一份持久上下文，写明这个仓库该用哪些命令、哪些目录别碰、什么算完成、有哪些不显眼的坑 [A1]。它的纪律和可读代码完全一致：要短、要人能读、每一行都得能回答"删掉它会不会让 agent 犯错"——答不上来就删，因为冗长的家规便条会让 agent 把真正要紧的规则淹没掉、读漏掉 [A1]。可读性、约定、家规便条，本质是同一件事的三个层次：把你脑子里默认的规矩，写到 agent 能读到的地方。

技术债记录在这里也升了值。过去那条"为什么临时这么写"的注释，主要是写给将来的人看的；现在它可以直接喂给 agent 当上下文。你让 agent 去碰一段有债的代码，把那条债务记录贴给它，它就知道这里为什么长这样、哪个边界不能动——而不是把临时方案当成正式设计，照着它往外扩。没有这条记录，agent 只会看到当前代码长什么样，看不到它"本不该长这样"。

## 实践方法

**命名说出意图。** 把"机器看得懂但人要解码"的表达，换成"读一遍就知道在干嘛"的表达。一个具体例子：

```python
# 改之前：每个符号都要猜
if u.t == 3 and x > 7:
    process(u)

# 改之后：读一遍就懂
if user.is_premium_member and order.total > FREE_SHIPPING_THRESHOLD:
    grant_free_shipping(order)
```

右边没变快、没变短，但它不再需要读的人（或 agent）去猜 `t == 3` 是什么、`7` 是哪来的魔法数字。这正是 A8 那条原则在代码里的对应：agent 处理有意义的自然语言名字，远比处理 `uuid`、`256px_image_url` 这种隐晦标识符顺手；把含糊的标识换成语义清楚的名字，能实打实减少它的误判 [A8]。对人成立的，对 agent 也成立。

**把大函数拆成有职责说明的小函数。** 一个三百行、混着校验、计算、写库、发通知的函数，拆成 `validate_order`、`calculate_total`、`persist_order`、`notify_user` 四个。每个函数名就是它的职责说明，读的人不用读完整个函数体才知道它在干嘛。这是纯重构——对外行为不变，测试该绿还是绿——但它把"理解这段代码"的成本砍到了原来的几分之一。A8 给写工具的建议是"像跟一个新来的同事介绍这个工具那样"去命名和描述 [A8]；写函数也是同一条标准：假设读的人对这块一无所知，名字得替你把话说清楚。

**写一条技术债记录，而不是只留个 `# TODO`。** 一条有用的债务记录回答三个问题：为什么临时这样做、风险是什么、什么时候/什么条件下该还。

```python
# 技术债（2026-06-13，作者 dsh）：
# 这里硬编码了汇率 7.2，因为汇率服务上线前要先发版。
# 风险：汇率波动期间金额会算错，仅影响展示、不影响落库。
# 偿还条件：汇率服务上线后，改为调 get_rate()，删掉这个常量。
```

光一个 `# TODO: fix rate` 三个月后没人知道说的是什么、能不能动、动了会不会出事。写全的记录，既是给将来的人看的，也是你日后让 agent 来还这笔债时，直接贴给它的上下文。

**让工具的输出短、结构化、带路径行号。** 这条既是为了人，也是为了 agent。一个好的报错或检查输出，应该直接告诉你"哪个文件、第几行、什么问题"，而不是甩一大段 traceback 让你自己捞。A8 把这条讲得很具体：工具应该只返回高信号的信息，别让 agent 把有限的上下文浪费在无关内容上；Claude Code 默认就把单次工具输出限制在 25,000 token 以内，靠分页、过滤、截断来控量 [A8]。与其做一个返回全部日志的 `read_logs`，不如做一个 `search_logs`，只返回相关的几行加上下文 [A8]。你自己写的 lint 脚本、测试报告、CI 输出也照这个标准来——短、带路径行号、结构化——人读得快，agent 也读得准。

## 真实案例

**一段难懂逻辑的重构前后。** 下面这段是真实代码里常见的样子——能跑，但没人敢动：

```python
def chk(d, u):
    r = []
    for i in d:
        if i[2] == 1 and (u[3] == 'p' or i[4] > 100):
            if i[5] not in [x[0] for x in r]:
                r.append((i[5], i[1]))
    return r
```

`d` 是什么？`u[3] == 'p'` 是什么意思？`i[4] > 100` 那个 100 是什么？全得猜。一个人读懂它要十分钟，agent 拿到它，要么猜错语义改坏，要么干脆把它整段重写——而重写一段你看不懂的代码，风险极大，因为你根本不知道它原本处理了哪些边界情况。重构后：

```python
def find_eligible_items(items, user):
    """挑出对该用户有资格免邮的商品，按商品去重。"""
    eligible = []
    seen_skus = set()
    for item in items:
        is_in_stock = item.status == IN_STOCK
        qualifies = user.is_premium_member or item.price > FREE_SHIPPING_THRESHOLD
        if is_in_stock and qualifies and item.sku not in seen_skus:
            seen_skus.add(item.sku)
            eligible.append((item.sku, item.name))
    return eligible
```

行为没变，测试照绿。但现在 `is_in_stock`、`qualifies` 这些名字替你把判断条件说清楚了；`seen_skus` 用 `set` 替代了原来 `x[0] for x in r` 那个 O(n²) 的去重，顺带连性能问题一起暴露出来了。更重要的是：现在 agent 来碰这段，它读得懂、能在不破坏语义的前提下精确地改，而不是面对一团乱码靠猜。

**为什么 agent 写的代码尤其要审可维护性。** agent 有一个很具体的特点：它能很快产出"能跑但难懂"的代码。它不会累，不会因为函数太长而烦躁，所以它很容易写出三百行的大函数、嵌套很深的分支、为了过测试而堆的特判——这些代码当下能跑，验收（如果验收只看"测试绿不绿"）也能过。Anthropic 把这个坑起了个名字叫"信任但没核实的缺口"：agent 产出一个看着很合理的实现，实际没处理好边界情况 [A1]。可读性差的代码把这个缺口放大了——因为它本来就难读，你审的时候更不容易发现里面藏着的问题。

这给出一条具体的审查标准：审 agent 的产出，不能只看"它跑过了吗"，还得看"三个月后我（或下一个 agent）改得动吗"。命名是否说清了意图、函数职责是否单一、有没有为了过测试堆出来的特判——这些不是风格洁癖，是在控制下一次有人（或 agent）来改这段代码时会不会踩雷。A1 给的解药很直接：始终给 agent 一个它能自己跑的验证手段（测试、脚本、截图），验不了的就别合 [A1]——而能验，前提是代码本身读得懂、查得明白。

## 常见反模式

- **靠注释救可读性。** 用一长段注释解释一个名字起得很烂的函数在干嘛。注释会过期、会和代码漂移，而一个好名字不会。能用命名说清的，别推给注释。
- **借债不记账。** 临时方案直接上，不留任何记录。三个月后这段代码成了"谁也不敢动的黑箱"——不是因为它复杂，是因为没人知道它当时绕过了什么、动了会不会爆。
- **重构混进功能改动。** 一个 PR 里既改了行为又重排了结构，审的人分不清哪些 diff 是"行为变了"、哪些是"只是搬了位置"。重构要单独成一个 PR，让"测试还绿"能真正证明"行为没变"。
- **agent 代码只验"能跑"，不验"能维护"。** 测试绿就合并，不看这段三个月后改不改得动。能跑但难懂的代码，是 agent 最擅长批量生产的东西——验收只卡"绿"，等于把可维护性这道关整个撤了。
- **家规便条越写越长。** 把所有能想到的规矩都堆进 `CLAUDE.md`，结果真正要紧的规则被淹没，agent 反而读漏。A1 的判据是：每一行都问"删掉它会不会让 agent 犯错"，答不上来就删 [A1]。

## 检查清单

- 这段代码，另一个人不查别处、不问我，读一遍能看懂在干嘛、分支为什么存在吗？
- 关键变量和函数的名字，说出了它"是什么 / 要干什么"，还是要读的人去解码（`u.t`、`flag`、`x`）？
- 我刚做的是重构还是重写？如果是重构，重构前后测试都绿了吗？它是不是单独一个 PR、没混进行为改动？
- 我借的每一笔技术债，有没有一条记录写清"为什么、风险、什么时候还"？还是只留了个 `# TODO`？
- 我让 agent 读的这个仓库，有没有一份短的、人能读的 `AGENTS.md`，把命令、禁区、完成标准说清楚？
- 我审 agent 产出时，除了"测试绿不绿"，有没有看"三个月后改不改得动"？
- 我写的工具/脚本，输出是短的、带路径行号的、结构化的，还是甩一大段让人和 agent 自己捞？

## 自测

- 翻开你手上一段三个月没动过的代码。不查 git、不问任何人，你能在一分钟内说清它在干嘛、每个分支为什么存在吗？如果不能，问题在当初的命名和结构，不在你的记性。
- 把上面那段 `chk(d, u)` 拿给一个不知道它做什么的人（或一个 agent）。他要花多久才敢动它？这个时间，就是可维护性的体温计——它量的不是代码好不好看，是下一次有人来改它，会不会改坏。
- 你上一次让 agent 改一段有技术债的代码，有没有把"这里为什么临时这么写"的记录一起喂给它？如果没有，它是把临时方案当成正式设计照着扩了，还是侥幸没扩——你怎么知道？
