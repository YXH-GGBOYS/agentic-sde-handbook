需求这件事，过去常被当成"软技能"——会沟通、能揣摩老板意图的人额外加分，说不清楚也不致命，因为执行慢，做错了方向还有时间在中途掰回来。agent 把执行那一段大幅压缩之后，这个缓冲没了。一个含糊的任务，会被 agent 又快又自信地朝错误方向做完。于是"把需求说清楚"从一项加分的软技能，变成了一项决定成败的硬工程能力。这一章先讲清楚什么是需求、什么是好的问题定义，再讲为什么在 agent 时代它的价钱涨了。

## 生活类比

你饿了，跟楼下店家说"来份炒饭"。

来的是一个完全不懂你的新外包，他听到的就是字面四个字：炒饭。于是他端来菜单上最便宜、最敷衍的那一份——一个鸡蛋、几粒葱花、米饭还有点夹生，分量一个人都吃不太饱，里面撒了香菜，二十分钟后才慢悠悠送到，这会儿你已经饿过劲了。

他没做错。"炒饭"他确实给了。错的是你心里其实有一整套没说出口的条件：要够两个人吃、不要辣、不要香菜、用隔夜饭别用生米、最好二十分钟内到。这些你都知道，只是觉得"不用说"——可对一个不懂你家口味的人，"不用说"等于"他不知道"。你以为你提了个需求，其实你只提了半句。

这里要分清两层东西。你嘴上说的"炒饭"是一个**解决方案**——一道具体的菜。你心里真正的**需求**是"今晚两个人吃饱，不踩我讨厌的那几样"。同一个需求可以有很多种解法：炒饭可以，盖浇饭可以，两碗面也可以。你一上来就把解决方案当需求报了出去，外包就只能照着这个解法里最省事的那一档来做，因为别的他一概不知道。

换成一个老熟人的店，老板会反问你："两位还是一位？要不要辣？香菜放不放？"——他在帮你把没说出口的条件补全。新外包不会反问，他只会照字面执行。agent 就是这个新外包：能力可能很强，但你说半句，他就照半句干，而且干得飞快。

## 工程定义

把上面的直觉翻译成工程语言，需求这件事可以拆成几个分得很清的概念。新人最该先把它们分清楚。

**需求，是要解决的问题，不是解决问题的办法。** 需求描述的是"谁、在什么场景下、遇到了什么麻烦、为什么现在得解决"，它不预先指定怎么实现。"两个人今晚要吃饱、不踩雷"是需求；"做一份炒饭"是一个解法。工程里最常见的混淆，就是需求总爱穿着解决方案的外衣出现：有人冲进来说"首页加个 AI 推荐模块"，听起来像需求，其实是一个已经选好的解法。真正的需求藏在它后面——可能是"新用户进来不知道买什么，第一屏就走了"。把解法当需求接下来，你会精确地实现一个也许根本不该做的东西。

辨别的办法很朴素：对着任何一句"要做 X"，追问一句"为了解决谁的什么问题"。答得上来，X 后面站着一个真问题；答不上来，X 只是一个还没找到主人的解法。

**好的问题定义，是把这个真问题说到可被验证。** 它至少回答四件事：谁是受影响的人（哪一类用户）、在什么场景下遇到（具体到一次真实操作）、痛在哪（可观察的现象，不是"体验不好"这种感受词）、为什么是现在（不解决会怎样）。四件事齐了，这个问题才从一团模糊的不满，变成一个别人接手也能照着干的东西。

**成功指标，是事先约定好的、可观察可量化的判断信号。** 它回答的是"做到什么程度算成功"。关键在"可观察"：转化率从 3% 到 5%、首屏跳出率下降、某类工单的人工处理时长从十分钟降到三分钟——这些都能被外人独立核对。"用户更满意了""体验更顺了"不是成功指标，因为没有人能站出来说"这条达没达到"。把指标定下来，就是在动手之前先约定好验收的尺子。

**验收标准，是成功指标落到这一次具体交付上的检查项。** 指标可能是个长期目标（转化率提升），但这一次 PR 交上来，你得有一组此刻就能勾的检查项：新模块在登录态和未登录态都能正常渲染、接口超时有兜底、加它之后首屏加载时间不超过某个值。验收标准越接近"能跑一条命令、能看一眼就判定真假"，它就越有用。

这四样——需求、问题定义、成功指标、验收标准——是一条链：从"谁有什么麻烦"，到"做到什么算解决了"，到"这一版怎么验"。链子断在哪一环，后面就开始靠猜。

## 为什么重要：AI 时代的变化

需求说不清，从来都有代价。变的是这个代价的大小，以及它砸下来的速度。

在 agent 之前，执行是慢的，这层慢反而是个缓冲。你给同事一个含糊的任务，他做着做着会卡壳、会来问你、会在某个下午突然意识到"这地方好像不对"，然后回来跟你对一下。模糊需求在执行过程里有大把机会被纠偏。说不清需求，顶多是多绕几圈。

agent 把执行那一段压扁了。它不卡壳、不犯怵、也基本不回头反问——它会把你给的那半句话，又快又自信地补全成一个完整的、自洽的、而且很可能是错的方向，然后一口气做完。它不会在某个下午醒悟"这地方不对"，因为它从一开始就没察觉哪里不对：你说的就是它知道的全部。等你看到结果，错误方向已经铺进了好几个文件。模糊需求的代价，从"多绕几圈"变成了"又快又彻底地做错"。

这就是"重定价"的确切含义：执行变便宜，把需求说清楚这件事相对就变贵了。在一个执行是瓶颈的世界里，需求糙一点没关系，反正实现慢，有时间修。在一个执行几乎免费的世界里，瓶颈整个挪到了上游——挪到了"你到底要什么、做到什么算成功"这件事上。一个含糊的 issue 不再是省下来的几分钟，它是后面一整轮被自信地做错的工作。

所以"把需求说清"这件事，从软技能升级成了硬工程能力。它不再是"会沟通的人额外加分"，而是"说不清的人，agent 就替你把错误放大"。GitHub 那套给 coding agent 写任务的方法，第一条就是把 issue 当成"写给一个对这个代码库完全陌生的新人"来写 [G3]——这句话的分量，正在于 agent 真的就是那个新人，而且它不会因为你写得糊涂就停下来问你。

还有一层连带的好处。当你被逼着把成功指标写成可观察、可量化的东西，这些指标就不只是文档里的话——它们可以直接长成验收。Anthropic 在做一个长任务 agent 时，让它先把一个 web app 拆成两百多条端到端的特性需求，每一条都写成"用户能打开新对话、输入一句话、回车、看到 AI 回复"这种具体到可执行的描述，初始全部标成"未通过"，agent 只有真测过才能把状态改成"通过" [A2]。这份清单不是事后写的验收文档，它就是需求本身——只不过需求被写得足够具体，于是顺手变成了验收。需求说清的尽头，是需求可执行。

## 实践方法

把需求说清不靠灵感，靠把几件事按顺序问完、写下来。

**先把解法翻回问题。** 拿到任何一个穿着解决方案外衣的需求（"加个 X""改一下 Y"），先别接，追问一句"这是为了解决谁的什么问题"。问到一个具体的人、一个具体的场景、一个可观察的痛点为止。问不出来，说明现在还不该动手——该动手的是把问题搞清楚。

**写一张 agent-ready 的 issue。** 一张能直接交给 agent（也能交给一个陌生同事）的任务卡，建议写全这几栏：

- **背景**：这是什么、现在长什么样、为什么有人提这个需求。GitHub 的说法是，按"写给一个对代码库完全陌生的人"的标准来写，把一个新人需要的上下文补上 [G3]。
- **目标**：这次要达成什么。用问题的语言，不是解法的语言。
- **非目标（non-goals）**：这次明确不做什么。这一栏后面单讲，它专治 agent 的过度发挥。
- **验收标准**：怎么算完成。尽量写成可观察、可勾的检查项。
- **验证命令**：用什么命令能跑出结果（哪个测试、哪个脚本、访问哪个页面看什么）。
- **预期证据**：跑完应该看到什么（哪个测试绿、截图里该有什么、日志里该出现什么）。
- **风险**：这次改动可能碰到哪些边界、哪些地方容易出岔子。

不是每个任务都要凑满七栏，但背景、目标、非目标、验收这四栏，糊掉任何一栏，agent 都会拿那一栏当自由发挥的入口。

**给 agent 写任务，记 GitHub 的 WRAP 四步 [G3]：**

- **W（Write effective issues）写清楚问题**：标题要说清"在哪儿动手"，正文补全新人需要的上下文，最好附一个你想要的实现样子的例子。它给的对照很直白——别写"把整个仓库都改成 async/await"，要写"把认证中间件改成新的 async/await 写法……并补上单元测试" [G3]。前者是一团没有边界的祈使句，后者圈出了范围、给了验收。
- **R（Refine your instructions）细化指令**：把仓库级、组织级的偏好和模式沉淀成长期生效的指令，别每次重新口头交代 [G3]。
- **A（Atomic tasks）原子任务**：把大问题拆成小而独立的任务，别甩一个巨型范围。它的例子是——别写"把三百万行 Java 重写成 Golang"，而是给认证、数据校验这些模块各开一个 issue [G3]。任务越原子，验收标准越容易写清，agent 越不容易跑偏。
- **P（Pair with the coding agent）与 agent 配对**：用各自的长处。人擅长理解上下文、处理模糊、跨系统思考；agent 擅长不知疲倦地执行、做重复活、把多种路子都试一遍 [G3]。需求定义、判断方向，是人这一侧的活；agent 不接管这部分，它接管的是定义清楚之后的实现。

**用非目标给 agent 划边界。** "非目标"专门防一种事：agent 看到一个模糊的目标，会自作主张地"顺手"多做。你说"加个推荐模块"，它可能顺手改了你的推荐排序逻辑、动了首页布局、引了个新依赖——每一样单看都像"为你好"，合起来就是一次你没要的大改动。把"这次不动排序算法、不改首页其他模块、不引新依赖"明明白白写进非目标，等于给它划出操作间的墙。墙立起来，它的"过度发挥"就没了去处。

**让成功指标顺手长成验收。** 定指标的时候，多想半步：这条指标能不能写成一个此刻就能跑、能判真假的检查？能，就把它写成验证命令和预期证据，像 A2 那份被一条条核掉的特性清单那样 [A2]。需求写到这个份上，验收就不用另起炉灶——需求本身就是验收的种子。

## 真实案例

**把"首页加一个 AI 推荐模块"翻成一张问题定义卡。**

有人冲进来提："首页加一个 AI 推荐模块。" 这是一句穿着解决方案外衣的话。按上面的办法翻回问题，一栏栏填：

- **背景**：新注册用户落到首页后，面对一整屏商品不知道从哪看起，相当一部分人没点进任何商品就走了。
- **目标用户**：首次访问、还没有任何浏览或购买历史的新用户。
- **痛点**：首页对新用户是"冷启动"的——没有历史，现有的"猜你喜欢"对他们是空的或随机的，于是第一屏就劝退。
- **成功指标**：新用户首屏跳出率下降；新用户从首页点进至少一个商品详情页的比例上升。（具体目标值由业务定，但形式必须是这种可观察、可量化的。）
- **非目标**：这次不改老用户的推荐逻辑、不动首页其他模块的布局、不引入新的推荐算法服务——先用现成的能力验证这个方向对不对。

注意翻完之后发生了什么：原来那句"加个 AI 推荐模块"成了诸多解法里的一个候选，而且未必是最优的。也许对"冷启动新用户"这个真问题，热门榜单、新人专区、一个简单的引导流程都能解，根本不需要上 AI。把问题定义清楚，最大的好处往往不是把方案做得更好，而是让你看清这个方案该不该做。

**坏 issue vs 好 issue（agent-ready）。**

坏 issue，标题就一句："修一下推荐。" 正文空着，或者只有"推荐不太准，优化下"。把它丢给 agent，等于把那句"来份炒饭"丢给新外包：它不知道谁觉得不准、哪个页面的推荐、什么叫"准"、改到什么程度算完。于是它会自己脑补一个"不准"的定义，挑一个它觉得合理的方向，又快又自信地动一大片代码——可能动了排序权重、可能换了召回逻辑、可能顺手重构了一个它看不顺眼的函数。你拿到一个跑得通、却完全不知道对不对、也无从验证的 PR。

好 issue 长这样：

> **标题**：新用户首页推荐冷启动——首屏增加面向无历史用户的推荐位
>
> **背景**：新注册、无浏览/购买历史的用户落到首页，现有"猜你喜欢"对他们为空，第一屏跳出率偏高（见非目标，本次不改老用户逻辑）。相关代码在 `home/recommend/` 模块。
>
> **目标**：为无历史的新用户在首屏展示 6 个相关推荐位（示例值，具体由业务定），降低其首屏跳出、提升点入商品详情的比例。
>
> **非目标**：不改动老用户（有历史）的推荐逻辑；不调整首页其他模块布局；不引入新的外部推荐服务。
>
> **验收标准**：(1) 无历史用户访问首页时，推荐位渲染出非空结果；(2) 有历史用户的首页行为完全不变；(3) 推荐接口超时或失败时，首屏有兜底内容、不空白、不报错；(4) 加入该模块后首屏加载时间不超过现状 + 200ms。
>
> **验证命令**：跑 `home/recommend/` 下的测试套件；用一个全新（无 cookie/无历史）会话访问首页，确认推荐位非空；用一个有历史的账号访问，确认行为不变；模拟推荐接口超时，确认兜底生效。
>
> **预期证据**：测试全绿；两类会话各一张首屏截图；接口超时场景下首屏仍正常的截图。
>
> **风险**：推荐接口在高延迟下可能拖慢首屏，验收标准 (3)(4) 专门盯这个；注意别让新逻辑误命中有历史用户的分支。

两张 issue 喂给同一个 agent、同一个模型，产出的东西天差地别——差别不在模型，在你给的那张卡。坏 issue 把"定义问题"这步推给了 agent，而这恰恰是它最不该替你做、也最容易自信做错的一步。好 issue 自己把问题定义死了，只把"在已定义的边界里实现"留给 agent。这正是 WRAP 里"人处理模糊、agent 处理执行"的分工落到一张卡上的样子 [G3]。

## 常见反模式

- **把解法当需求报上去。** "加个 X""上个 Y"直接进 issue，没人追问"为了解决谁的什么问题"。结果是精确地实现了一个也许根本不该做的东西。解药是对每一个"要做 X"先翻回问题。
- **成功指标写成感受词。** "更好用""更顺畅""体验提升"——没有人能独立判定这条达没达到，于是它既验不了收，也长不成 eval。指标必须可观察、可量化。
- **省掉非目标。** 只写"要做什么"，不写"明确不做什么"。agent 会把这片空白当成自由发挥的许可，顺手做一堆你没要的改动。非目标不是客套，是墙。
- **issue 写给"懂行的自己"，不写给陌生人。** 默认 agent（或下一个接手的人）知道那些"不用说"的背景。它不知道——你说半句，它就干半句。GitHub 的标准是按"写给完全陌生的新人"来写 [G3]。
- **任务一个巨型范围。** "重写整个模块""全站改成新框架"。范围越大，验收越没法写清，agent 越容易在中途迷路、还自以为做完了。Anthropic 观察到的一个典型失败就是 agent 想一口气把整个 app 做完，结果做崩 [A2]。拆成原子任务 [G3]。
- **不定验收就开跑。** 没想清楚"算完成"的判据就把活派出去，等 agent 交回来再用眼睛扫一遍"看着对"。Anthropic 记录的另一个失败正是 agent 在没有真正端到端测试的情况下就把特性标成"完成" [A2]。验收要事先约定，且尽量可执行。

## 检查清单

- 我手上这个需求，是一个**问题**，还是一个已经选好的**解法**？我能说出它背后"谁的什么痛点"吗？
- 问题定义齐不齐：**谁、在什么场景、痛在哪、为什么现在**？
- 成功指标是不是**可观察、可量化**的，而不是"更好用"这类感受词？
- 我写了**非目标**吗？agent 能从哪几个方向过度发挥，我堵上了吗？
- 这张 issue 是按"**写给一个对代码库完全陌生的新人**"的标准写的吗 [G3]？
- 验收标准能不能配上**验证命令**和**预期证据**，让"算完成"可执行、可勾？
- 这个任务是**原子**的吗，还是一个会让 agent 迷路的巨型范围 [G3]？

## 自测

- 拿你最近派给 agent（或同事）的一个任务，问自己：我当时给的是问题，还是一个我已经选好的解法？如果是解法，它背后真正的问题是什么？这个解法是不是最该做的那个？
- 把那个任务的"算完成"写成一句话。这句话，另一个人能不能照着独立判定真假？如果不能，你缺的不是更强的 agent，是一条说得清的验收标准。
