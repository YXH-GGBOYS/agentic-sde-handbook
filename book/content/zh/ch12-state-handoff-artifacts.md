一个跑了三小时的 agent，在第四个 context window 中途断了。下一个 agent 接手时，对前面发生的事一无所知——它会环顾四周，看到一些代码，然后要么从头重做，要么直接宣布"看起来已经完成了"。Anthropic 给长任务 agent 打过一个很准的比方：这就像一个排了班的工程团队，每个新来的人对上一班发生的事毫无记忆 [A2]。

要让这种团队接得上班，不能靠"回忆上一班聊过什么"。得靠上一班留下的几张纸：干到哪了、卡在哪、为什么这么决定、下一班先干哪个。这几张纸，就是这一章要讲的状态与交接工件。

## 生活类比

工地上，一个班组干完一班要交给下一班。管得好的工地不靠口头交代，靠两样东西：施工日志和交接班记录。

施工日志记的是事实：今天浇了哪几根柱子、混凝土标号多少、哪根因为下雨没浇完。交接班记录记的是状态和判断：这一班干到三层东侧、卡在预埋件尺寸对不上（已经报给甲方，等回复）、下一班先别动东侧、先把西侧的钢筋绑完。

关键在于，接班的人不在场，也没听过上一班的讨论。他走进工地，读完这两份东西，就能接着干——知道现在是什么状态、为什么之前那样决定、下一步先动哪里。如果上一班只在脑子里记着"那个预埋件有点问题"，没写下来，下一班就会照着错的尺寸往下做，等浇完才发现，砸了重来。

长任务 agent 的交接，就是这件事。每个新 session 都是一个没听过上一班讨论的接班人。它能不能接上，全看上一班在工地上留下了什么。

## 工程定义

**状态与交接工件**，是 agent 在一次 session 结束时留下的、让下一个 session（或下一个人）能不从零开始接手的那组文件。

它的核心问题来自一个硬约束：context window 是有限的，多数复杂任务一个 window 装不下，所以长任务 agent 必须分段工作，而每一段开始时对前面发生的事没有记忆 [A2]。这个"无记忆的换班"是结构性的，不是模型不够聪明能解决的。哪怕模型支持 compaction（把前面的对话就地压缩成摘要），Anthropic 也发现这不够——compaction 不总能把清楚的指令传给下一个 agent [A2]。所以真正可靠的接力，靠的是显式写到磁盘上的工件，不是靠内存里那点残留上下文。

这一章把交接工件拆成五件，各管一摊。这是本书的归纳，方便你检查"该留的有没有留"，不是某个厂商的固定格式：

- **PLAN**：目标、非目标、任务分解、风险、验证策略。这一长任务要做成什么样，哪些明确不做。
- **STATUS**：已完成、正在做、下一步、当前阻塞。接班人最先读这个。
- **DECISIONS**：重要取舍、为什么这么选、放弃了哪些替代方案。防止下一班把上一班想清楚的事又推翻重想一遍。
- **VALIDATION**：跑过哪些命令、结果如何、哪里失败了、哪些项还没验证。区分"我测过了"和"我以为它能行"。
- **HANDOFF**：给下一个人或 agent 的最小接手上下文——现在从哪读起、第一步先干什么。

不是每个任务都要五份独立文件。小任务可能一份进度文件就把 STATUS、VALIDATION、HANDOFF 三件事都写了。重点不是文件数量，是这五类信息别漏。

## 为什么重要：AI 时代的变化

人类团队换班，丢失的上下文可以补救——接班的人能打电话问上一班、能翻群里的聊天记录、能凭经验猜个八九不离十。agent 没有这些退路。它只读你显式留给它的东西；没写进工件的状态，对它等于不存在。

这件事在长任务上尤其要命，因为失败会顺着接力链放大。Anthropic 观察到两种典型崩法 [A2]。一种是 agent 想一口气把整个 app 做完，结果在实现到一半时 context 用尽，把一个"功能做了一半、又没写文档"的烂摊子留给下一班——下一班只能猜前面发生了什么，再花大把时间把基本功能重新跑通。另一种发生在项目后期：某个 agent 实例环顾四周，看到已经有不少进展，就直接宣布任务完成了。两种都不是模型"不够强"，是没有工件兜住状态，于是每一次换班都在赌运气。

还有个反直觉的点：跑得越快，越需要把状态写下来。一个 session 几分钟就能改一大片代码，如果它走的时候不留痕，下一个 session 要花更多时间去考古"这堆改动是怎么来的"。Anthropic 的说法是，留下进度文件和 git 历史能省掉 agent "猜测前面发生了什么、再把基本 app 跑通"的时间 [A2]。交接工件不是给长任务额外加的负担，它就是让长任务能跑下去的那块地基。

## 实践方法

Anthropic 在做 claude.ai 克隆版时的做法，是把"留交接物"这件事内建进 harness：第一个 session 用一个专门的 initializer agent 把环境搭好，之后每个 coding agent 都要做出增量进展、并在走之前留下结构化更新 [A2]。具体留三样：

1. 一个 **`init.sh` 脚本**，能把开发服务器跑起来。下一班不用自己摸索怎么启动这个项目。
2. 一个 **`claude-progress.txt` 进度文件**，记录历次 agent 都干了什么。
3. **git 提交历史**，每次改动配描述清楚的 commit message。这让 agent 能用 git 回退坏掉的改动、恢复到能跑的状态 [A2]。

接班的流程也固定下来。每个 coding agent 开工先跑一套"摸清现状"的步骤 [A2]：

- 跑 `pwd` 看自己在哪个目录工作（它只能改这个目录里的文件）。
- 读 git log 和进度文件，搞清楚最近在干什么。
- 读特性清单，挑一个优先级最高、还没做完的特性来做。

这套步骤省 token——agent 不用每个 session 都重新搞清楚怎么测这套代码 [A2]。它也直接对应工地的交接：先读交接班记录，再读施工日志，再决定今天先动哪里。

A3 把这套实践提炼成一条设计原则：长任务 harness 要做两件事——把构建分解成可处理的小块，用结构化工件在 session 之间交接上下文 [A3]。再加上可验证的产物和人为干预点，就是长任务应用开发 harness 的骨架。

这里有个判断点要单独说一句。A3 提到，随着模型变强（从 Sonnet 4.5 到 Opus 4.5、4.6），一部分脚手架可以拆掉——比如 Opus 4.5 基本不再有"context anxiety"（快到上下文上限时提前草草收尾），于是 context reset 这一层就能去掉，让 agent 一个 session 连续跑完整个构建 [A3]。但有一句话值得记牢：harness 里的每个组件，都编码了一个"模型自己做不到某件事"的假设，这些假设值得反复压力测试，因为它们可能本来就不对，也可能随着模型变强很快过期 [A3]。换句话说，交接工件这套东西要不要、要多重，得看你对着的模型在长任务上到底有多稳——不是越多越好。

## 真实案例

claude.ai 克隆版的状态工件，核心是一份 JSON 特性清单。initializer agent 把用户那句"做一个 claude.ai 克隆版"的高层 prompt 展开成一份详尽的特性需求文件，在这个例子里是**两百多条**端到端特性 [A2]，每条长这样：

```json
{
    "category": "functional",
    "description": "New chat button creates a fresh conversation",
    "steps": [
      "Navigate to main interface",
      "Click the 'New Chat' button",
      "Verify a new conversation is created",
      "Check that chat area shows welcome state",
      "Verify conversation appears in sidebar"
    ],
    "passes": false
  }
```

这份清单的设计，把"验收"和"可恢复状态"合成了一件东西。两百多条特性初始全部标成 `"passes": false`（未通过），给后面的 coding agent 一张"完整功能应该长什么样"的全景图 [A2]。coding agent 只被允许改一个字段——把 `passes` 从 false 翻成 true，而且只有在真正仔细测过之后才能翻 [A2]。这就同时解决了两个崩法：因为有这张全景图，agent 不会过早宣布整个项目完成；因为只有真测过才翻"通过"，agent 也不会把没真正跑通的特性标成完成。

两个细节值得抄。一个是为什么用 JSON 而不是 Markdown：Anthropic 试下来发现，模型乱改或覆盖 JSON 文件的概率比 Markdown 低，所以这种"既是验收标准又是状态"的文件用 JSON 更稳 [A2]。另一个是验收的口径——Anthropic 用了措辞很重的指令，类似"删除或修改测试是不可接受的，因为这可能导致功能缺失或带 bug" [A2]，把这份清单钉成不许 agent 为了"显得做完了"而偷偷动的硬约束。

A3 里的三 agent 架构（planner / generator / evaluator）把交接又往前推了一步：agent 之间的通信全走文件——一个 agent 写一个文件，另一个 agent 读它、再在文件里回应或新建一个文件让前一个 agent 读 [A3]。每个 sprint 开工前，generator 和 evaluator 先谈一份"sprint 契约"，在写任何代码之前先就"这块活算干完长什么样"达成一致 [A3]。这其实就是把 PLAN 和 VALIDATION 这两件交接工件，做成了两个 agent 之间可协商、可落盘的合同。

## 常见反模式

- **靠 compaction 当交接**：以为把前面对话压缩成摘要就够了。compaction 保留连续性，但不给下一个 agent 一张白纸，清楚的指令也不总能传过去 [A2]。可靠的接力靠显式写到磁盘的工件，不靠压缩后的残留上下文。
- **只记"做了什么"，不记"为什么这么决定"**：进度文件里写满"实现了 X、改了 Y"，却没有一条 DECISIONS。下一班看不到取舍背后的理由，很容易把上一班想清楚的事推翻重做，甚至踩回上一班已经避开的坑。
- **把"改过代码"当成"验证过"**：agent 改完代码、跑了单测或 curl，就把特性标成完成，却没从真实用户的视角端到端验过。Anthropic 明确观察到这种崩法——除非显式要求用浏览器自动化按真人那样测，否则 agent 会认不出"这个特性其实端到端走不通" [A2]。VALIDATION 工件要区分"真测过"和"以为能行"。
- **交接工件写给自己看，不写给陌生人看**：HANDOFF 假设了一堆"接班人本来就知道"的上下文，于是真正没记忆的下一个 session 读不懂。判据和工地交接班一样——一个没听过任何讨论的人，能不能只读这份工件就接着干。
- **状态文件用 agent 爱乱改的格式**：把"既是验收又是状态"的关键文件用 Markdown 存，agent 顺手就给改了。Anthropic 改用 JSON，正是因为模型乱动 JSON 的概率更低 [A2]。

## 检查清单

- 我的长任务有没有一份 **PLAN**：目标、非目标、分解、风险、验证策略都写了吗？
- 有没有一份 **STATUS**，让接班的 agent 一眼看清"已完成 / 正在做 / 下一步 / 当前阻塞"？
- 重要取舍有没有进 **DECISIONS**——为什么这么选、放弃了哪些替代方案？
- **VALIDATION** 区分了"真跑过、结果如何"和"还没验证"吗？标"完成"是真端到端测过，还是只改了代码？
- **HANDOFF** 是写给一个没听过任何讨论的陌生人的吗？他能只读工件就接着干吗？
- agent 走的时候，留没留下能让下一班接力的东西——进度文件、git 历史、启动脚本？
- 关键状态文件用的是 agent 不容易乱改的格式（比如 JSON）吗？

## 自测

- 如果你现在这次 agent run 中途断电，下一个 session（或一个同事）接手，他要花多久才能搞清楚"现在到哪了、为什么这么做、接下来先干什么"？答案越接近"几分钟读完几份文件"，你的交接工件越成熟。
- 你上一次让 agent 标一个特性"完成"，那个"完成"是它真按真人流程端到端测过，还是只改了代码、跑了单测就自己宣布的？
