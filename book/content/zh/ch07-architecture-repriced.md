架构是埋进系统里的"不可逆管线"。它平时看不见，一旦定了型，改一次就要砸墙。过去这件事的代价主要落在人身上——边界乱了，人改起来慢、容易踩雷。

现在多了一个新角色：一个改代码飞快、却完全不懂你家结构的 agent。它一次能横跨好几个文件下手。边界清楚，它的误伤被关在一个模块里；边界乱，它会在错误的墙上猛凿，一处改动牵连一大片，而且改得太快，你很难在中途拦下来。所以架构、模块化、契约这几件老手艺，在 agent 时代不是被淘汰了，是被重新标了价——而且大多是涨价。

## 生活类比

想想家里装修走水电。

水电管线是埋进墙里的。开工前师傅会先弹线、定走向：强电走哪、弱电走哪、给水排水从哪过、哪几路要预埋。这些线一旦埋进墙、批了腻子、贴了砖，就看不见了。后面你想挪一个插座、改一段水管，不是动一颗螺丝的事——得砸墙、凿槽、重新埋管、再补回去。一次小改动，墙面、瓷砖、防水、甚至楼下的天花板都可能跟着遭殃。

所以好师傅在埋管之前会做两件事。一是把走向规划清楚，让以后大概率要动的地方留好检修口、留好余量，别把什么都焊死在墙里。二是分区：强电和弱电分开走、给水和排水分开走，各走各的管道井，互不串线。这样哪天厨房水路出了问题，电路不用一起拆；客厅想加一路插座，卫生间的防水层不用被牵连。

agent 改代码，快得像一个不知疲倦的施工队。但施工队的快是好事还是坏事，全看墙里的管线规划得清不清楚。规划得清楚，它在厨房凿墙换水管，凿到的就只是厨房那一段；规划得乱——水电缠在一起、没有分区、哪根管通向哪儿全靠猜——它一锤子下去，可能把不相干的另一路也带断了。而且它砸墙的速度比你喊"停"的速度快。

## 工程定义

先把四个基础概念讲清楚，它们是一层套一层的。

**架构（architecture）**，是一个系统的骨架和边界。它回答的是大问题：系统分成哪几大块、每块负责什么、它们之间怎么连、数据往哪流、哪些决定是早期定死、后面很难再改的。给水排水的总走向属于架构；某个插座装在墙左还是墙右，不属于。架构决策的特点是"贵在可逆性差"——埋进墙里的，改起来要砸墙。

**模块化（modularity）**，是把系统拆成一块块职责清楚的小单元，每一块都能被单独理解、单独测试、单独替换。判断一个模块拆得好不好，有个朴素标准：你能不能不打开它，光看它对外的样子，就说清它是干嘛的？厨房的水路是一个模块——你不用知道墙里每根管怎么拐弯，只要知道"这是厨房进水、那是厨房排水"。

把模块拆好，靠的是两个互为表里的性质：

- **高内聚（high cohesion）**：一块该管的事都收在自己这块里，不散落到别处。处理订单的逻辑都在订单模块，不会有半截藏在用户模块里。
- **低耦合（low coupling）**：模块之间的牵连尽量少、尽量细。它们通过一个明确的接口打交道，而不是你伸手进我肚子里掏。低耦合的好处很实在：换掉一块，不连累其余的。

**契约 / 接口（contract / interface）**，是模块对外的那张"菜单"。点外卖时你不进后厨，只看菜单点餐、等送达——你认的是菜单（有什么、怎么点、多久到），不碰后厨怎么颠勺。模块的契约也是这张菜单，它要说清五件事：

- **输入**：调用方要给什么，类型、格式、必填项。
- **输出**：会返回什么，结构长什么样。
- **错误**：哪些情况会失败，失败了怎么告诉调用方。
- **权限**：谁能调、要不要授权。
- **性能**：大概多快、有没有调用频率上限。

契约的核心精神是：调用方只认这张菜单，不依赖后厨的实现细节。只要菜单不变，后厨今天用煤气灶、明天换电磁炉，点餐的人不需要知道、也不会被影响。A8 把这件事说得很本质：传统写软件，是在两个确定性系统之间立一份契约——`getWeather("NYC")` 每次都以同样的方式取纽约的天气 [A8]。契约就是这份"你给我什么、我还你什么"的约定，把它写清楚，模块之间才能各管各的。

## 为什么重要：AI 时代的变化

上面这套——架构、内聚、耦合、契约——在没有 agent 的年代也是对的。它过时了吗？没有。它被重定价了，而且大多是涨价。原因是引入了一个新的施工队，它的两个特性，把好边界和坏边界的差距放大了。

第一个特性：**爆炸半径（blast radius）变大了**。一个人改代码，通常一次盯着一两个文件，改之前会先看看周围牵连到谁。agent 不一样——它一次任务里能横跨十几个文件下手，读、改、跑命令一口气做完。这意味着同样一处"边界没划清"的毛病，在人手里可能只波及相邻几行，在 agent 手里会被它沿着混乱的依赖关系一路传导出去。边界清楚的系统，等于给每个模块砌了承重墙，把一次改动的误伤关在一个房间里；边界乱的系统，等于满屋子都是非承重的隔断，agent 一拆，半套房子跟着塌。

第二个特性：**它快到你来不及拦**。坏边界在人手里的危害是"慢慢渗"——你还有时间在 review 里发现"咦这改动怎么动到订单了"。agent 的危害是"瞬间扩散"——等你反应过来，它已经顺着错误的耦合把十个文件都改了。所以"能跑就行"这条老标准，在 agent 时代更不够用了。能跑，只证明这一次没塌；它没告诉你下一次 agent 顺着哪根没划清的管线，会把哪面不该动的墙凿穿。

反过来说，这也给出了一个朴素而有力的结论：清楚的模块边界，就是 agent 友好（agent-friendly）的边界。你为人类可维护性做的那套功夫——高内聚、低耦合、契约写清楚——恰好也是在给 agent 限定爆炸半径。这不是两件事，是同一件事。你不需要为了 agent 专门发明一套新架构原则，你只需要把老原则当真，因为现在违反它的代价被一个不知疲倦、改得飞快的施工队成倍放大了。

还有一层变化值得单独点出来：agent 不只是契约的"调用方"，它也越来越多地成为契约的"消费者"和"生产者"。当你给 agent 配工具（tool）时，那个工具本身就是一份契约——只不过这次契约的另一头不是确定性的代码，而是一个会犯错、会幻觉、会理解偏的非确定性 agent。A8 把这点讲得很直接：工具是一种新的软件，它反映的是确定性系统和非确定性 agent 之间的契约 [A8]。这意味着"把契约写清楚"这件事的重要性不降反升——因为读这份契约的，现在是个比人类更容易被模糊定义带偏的读者。

## 实践方法

把上面的判断落到日常，有几条具体做法。

**把模型 / AI 调用封装成一个统一接口，别散落各处。** 这是 agent 时代最高频、收益也最直接的一条。一个常见的烂摊子是：项目里几十个地方各自直接调某个模型供应商的 SDK，参数拼法各写各的，供应商的细节渗进了业务代码的每个角落。正确的做法是把它收进一个窄接口，比如一个 `generateAnswer()` 或一个 `LLMClient`，业务代码只认这个接口的菜单——输入提示、拿回答案、处理它定义的错误，不碰底层是哪家供应商、用的哪个模型、怎么重试。这样换一个模型、加一层缓存、接一个新供应商，改动都被关在这一个模块里。

**契约思维要落到 API 与数据契约上，而不是停在口头。** 接口的那五件事（输入、输出、错误、权限、性能）要写进代码能检查的地方：类型定义、schema、明确的错误类型。A8 强调输入参数要无歧义地命名、用严格的数据模型来约束——用 `user_id` 而不是含糊的 `user` [A8]。数据契约同理：跨服务、跨模块传的那个结构，字段名、类型、单位都要一致且明确，不能一头叫 `amount`（单位元）、另一头当成分。

**给 agent 用的工具 / 接口，要专门为它的可用性设计。** A8 给了几条很具体的原则，可以直接照做：

- **窄职责，边界清楚。** 每个工具要有一个清楚、互不重叠的目的；工具一多、职责一糊，agent 就会选错、用错 [A8]。A8 的办法之一是命名空间（namespacing）——用统一前缀把相关工具归组，划出清楚的功能边界 [A8]。
- **稳定、高信号的输出。** 工具只回必要的信息，别把一堆底层技术标识符（uuid、mime_type 之类）糊给 agent，它的上下文是有限的，塞满噪音就找不到重点 [A8]。
- **明确的错误。** 出错时给一句能照着改的话，而不是一坨不透明的错误码或 traceback [A8]。这跟好契约要"说清楚哪些情况会失败、怎么失败"是一回事。

A8 还给了一个好用的心法：写工具描述时，想象你在跟一个新来的同事交接——你脑子里那些没说出口的隐含背景（特殊的查询格式、术语定义、资源之间的关系），都要明明白白写出来 [A8]。这句话对 AGENTS.md、对模块文档、对任何一份"要交给不懂你家规矩的人"的契约，都适用。

**复杂度按需增加，别一上来就上重型架构。** A3 把这条原则提炼得很到位：harness 里的每一个组件，都编码了一条"模型自己做不了某件事"的假设——而这些假设值得反复压力测试，因为它们可能本来就不对，也可能随着模型变强很快过时 [A3]。它引用的更朴素的版本是：先找最简单能跑通的方案，需要时再加复杂度 [A3]。模块和接口同理：单次使用不要为了"将来可能扩展"先造三层抽象，等到真有第二个、第三个调用方，边界自然会浮现。

## 真实案例

**散落各处 vs 封装成一个服务。** 设想一个电商系统接了 AI 客服。第一版图快，谁要调模型谁就地写：订单页直接拼一段 prompt 调供应商 A，退款流程里又拼一段调供应商 A，商品推荐那边也是。三处的重试逻辑各写各的，超时设置不一样，错误处理也不一样。现在产品说"换个更便宜的供应商 B"——你得在三个不相干的业务模块里挨个改，每改一处都要重新测一遍那条业务流，因为模型调用和订单逻辑、退款逻辑缠死在了一起。更糟的是，如果你让 agent 去做这次替换，它得横跨订单、退款、推荐三个模块同时改，爆炸半径正好覆盖了你最不想出事的订单系统。

把它收进一个 `generateAnswer()` 服务之后，这次替换的形状完全变了：业务代码不动，只有那一个模块的内部实现从供应商 A 换成 B，业务侧的菜单（输入提示、拿回答案、处理错误）一个字没变。订单系统压根不知道供应商换了，也就不会被搅进来。好边界的价值，在"换一个模型"这种事上看得最清楚——它决定了这件事是动一面非承重隔断，还是砸订单系统的承重墙。

**契约写不清，agent 也会在边界上栽跟头。** A3 记录了一个很具体的例子。在那套 planner-generator-evaluator 的长任务 harness 里，生成 agent 写了一个重排动画帧的接口，把 `PUT /frames/reorder` 这条路由定义在了 `/{frame_id}` 那几条之后；结果 FastAPI 把 `reorder` 当成了一个 `frame_id` 整数去匹配，返回 422："无法把字符串解析成整数" [A3]。这正是契约边界没理清惹的祸——两条路由的"菜单"在 URL 形状上撞了车，框架按定义顺序选错了入口。agent 自己没发现这个问题，是那个独立的 evaluator agent 点出来的，而且报告得足够具体、能直接照着改 [A3]。这给了一个很实在的提示：在 agent 时代，"谁来验收边界对不对"本身要被设计进流程，不能指望干活的 agent 自查。

**长任务里，结构是 agent 能不能干成的前提。** A3 还给了一条贯穿全文的经验：要让 agent 在长时间、多步骤的任务里干成事，靠的是两件结构性的事——把构建拆成一块块能啃下来的小任务，以及用结构化的工件在会话之间交接上下文 [A3]。在它那套 harness 里，generator 是一个 feature 一个 feature 地推进的；每开工一段之前，generator 和 evaluator 还会先谈一份"sprint 契约"——在写任何代码之前，先就"这一块怎么算完成、用哪些可测的行为来验证"达成一致 [A3]。换句话说，连 agent 自己干活，都需要清楚的分解和明确的"完成定义"才跑得稳。结构不是给人看的装饰，是 agent 能不能把活干完的地基。

## 常见反模式

- **"能跑就行"当验收标准。** 一次跑通只证明这次没塌，不证明边界划对了。在 agent 时代尤其危险，因为下一次 agent 会沿着你没划清的那根管线，把不该动的墙凿穿——而那一次它跑得比你拦得快。
- **AI 调用绑死在业务代码里。** 每个用到模型的地方各自直接调供应商 SDK，参数、重试、错误处理各写各的。换模型时要在一堆不相干的模块里挨个改，正好把 agent 的爆炸半径引到核心系统上。
- **为还不存在的扩展性先造抽象。** 单个调用方就先上三层接口、五个配置项。这违背"复杂度按需增加"——A3 那条"每个组件都编码了一条可能过时的假设" [A3] 同样适用于你过早造的抽象，它们多半会变成没人敢动的死结构。
- **给 agent 的工具职责太宽、太多、太重叠。** 一个工具什么都能干，或者十个工具职责互相盖。A8 明说，工具一多、目的一糊，agent 就会选错用错 [A8]。窄职责不是洁癖，是在帮 agent 少犯错。
- **把约束写进 prompt，不写进契约。** 指望一句"请注意调用顺序""别动订单模块"就管住边界。真正的边界要靠类型、schema、模块划分硬卡出来，写在 prompt 里的约束 agent 随时可能忽略。
- **让干活的 agent 自己验收边界。** A3 的经验是 agent 倾向于夸自己的活、测得浮于表面 [A3]；那个路由撞车的 bug 是独立 evaluator 揪出来的，不是干活的 agent 自查出来的 [A3]。验收边界要外置，别内包给作者。

## 检查清单

- 我能不打开一个模块，光看它对外的接口，就说清它是干嘛的吗？
- 模型 / AI 调用是收在**一个**统一接口后面，还是散落在好几个业务模块里？换一个供应商，改动会被关在一个模块内吗？
- 跨模块、跨服务传的数据契约，字段名、类型、单位三处一致吗（ORM ↔ 接口 ↔ 前端）？
- 我的接口契约，把**输入、输出、错误、权限、性能**这五件事都写到代码能检查的地方了吗，还是只活在某个人脑子里？
- 给 agent 用的工具，职责是不是窄而清楚、输出稳定、错误信息能照着改？
- 如果让 agent 改这块代码，它一次会横跨几个模块？爆炸半径有没有被承重墙（清楚的边界）关住？
- "算完成"有没有可测的判据，而不是"这次跑通了"？验收边界对不对的，是不是一个独立于作者的角色？

## 自测

- 你现在系统里那个"AI 调用最散"的地方，如果明天要换一个模型，得动几个文件？如果答案是"散在十几处"，你缺的不是更强的模型，是一道把它收进来的承重墙。
- 上一次你让 agent 做跨多个文件的改动，它一处改动牵连到了你没预料的地方吗？那个牵连，是模型不够聪明，还是你的模块边界本来就缠在一起？
- 给一个新来、完全不懂你家结构的人（或 agent）一份你某个模块的接口描述，他能不看实现就正确调用吗？看不懂的地方，正是你契约没写清的地方。
