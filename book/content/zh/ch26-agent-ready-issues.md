把活派给 agent，issue 就是那张派工单。写得含糊，一个不懂你家规矩的人只能瞎猜；写清楚背景、要达成什么、哪些不能碰、怎么验、跑哪条命令验、风险在哪——它才有机会一次干对。这一章讲两件事：怎么把一个 issue 写成 agent 能直接执行的规格，以及仓库里那几份指令文件（AGENTS.md / CLAUDE.md 一类）怎么收敛成一份让三家 agent 都听得懂的真源。

issue 的质量直接决定产物的质量。这不是空话——同一个模型，喂一句"把那个推荐修一下"和喂一份写明目标、禁区、验收命令的派工单，出来的 PR 完全是两个东西。

## 生活类比

你把一摞活外包出去，每件活附一张派工单。

坏派工单长这样：一句"把那个推荐弄一下"。外包不知道"那个"指哪个页面，不知道你要的是修 bug 还是加功能，不知道哪些文件是祖传别动的，不知道做完拿什么证明做对了。他只能按自己的理解猜一个方向，闷头干两天，交上来很可能不是你要的东西——而且你要到验收那一刻才发现。

好派工单是另一回事：哪个页面的推荐模块，现在的毛病是新用户进来看到空白，目标是给冷启动用户填上默认榜单；后端接口在 `api/recommend.py`，别动 `api/migrations/`；做完跑 `make test` 和 `pnpm -C web test`，两个都得绿；把改了什么、跑了哪些命令贴在交接说明里。拿着这张单子，一个第一次进你家门的人也能照着干，干完你扫一眼证据就知道对不对。

issue 之于 agent，就是这张派工单。GitHub 写过一份给 issue 的指南，第一条就是"当成写给第一次接触这个代码库的人来写" [G3]。agent 比人更需要这个——人会察言观色、会回头问一句，agent 只读你显式写下的东西。

## 工程定义

agent-ready issue 是一份**可执行规格，不是一句愿望**。它把一个任务拆成 agent 能照着动手、并且能自己判断"做完了没"的若干要素：背景、目标、非目标、可能涉及的文件、验收标准、验证命令、预期证据、风险等级、允许的权限、需要人类拍板的点、回滚说明。

GitHub 把"怎么给 coding agent 写 issue"总结成一个叫 WRAP 的框架 [G3]：

- **W — Write effective issues（写清问题）**：当成写给第一次接触代码库的人。用能说明"在哪里干活"的标题，补上新人需要的背景，给一个期望实现的例子。G3 的反例是"把整个仓库都改成用 async/await"，正例是"把 authentication 中间件改成更新的 async/await 写法……并补单测来验证" [G3]。
- **R — Refine your instructions（细化指令）**：把反复要交代的偏好沉淀进仓库级的自定义指令，而不是每个 issue 里重写一遍。G3 区分了三类：仓库级指令、组织级指令、以及处理重复性任务的自定义 agent [G3]。
- **A — Atomic tasks（原子化任务）**：把大问题拆成小而独立的任务，而不是甩一个巨大的范围。G3 的例子是：与其写"把 300 万行 Java 重写成 Golang"，不如按模块拆成一个个 issue，比如 authentication、data validation 各一个 [G3]。
- **P — Pair with the coding agent（与 agent 配对）**：发挥各自所长。G3 的分工是——人擅长理解上下文、处理模糊、跨系统思考；agent 擅长不知疲倦地执行、做重复性任务、探索多种实现路径；人做战略决策，agent 做实现，由人来确认改动是否真的解决了底层问题 [G3]。

这套框架的内核，跟前面讲 AGENTS.md 的那条主线是一样的：你不是在写文档，你是在给一个能干但不懂规矩的新人交代清楚一件具体的活。

## 为什么重要：AI 时代的变化

过去 issue 写得糊一点，问题不大：领活的工程师会在站会上当面问你"这个'那个'到底指啥"，会在写之前先翻一圈代码摸清现状，踩坑了旁边同事会拉一把。这些校正动作，把一张糊涂派工单的损失兜住了。

agent 把这层兜底抽走了。它不会回头问（除非你显式让它问），不会"感觉这里不对劲"，不会在交付前找人复核常识。它读你写下的，然后开干。糊涂的目标、缺失的验收、没划的禁区，会被它原样执行——而且执行得很快。GitHub 的 coding agent 一领到 issue 就开一个草稿 PR、边干边推 commit、把关键步骤记进日志 [G2]；等你发现方向错了，它可能已经把错误铺进了好几个文件。

还有个流程上的变化：issue 成了异步流水线的入口。GitHub 这套是 issue → 草稿 PR → 你 review 的异步流——你派完活就可以走开，agent 在 GitHub Actions 里的一个临时环境里探索代码、改、跑测试和 linter，干完把 PR 标题描述补好、@ 你来 review [G2]。入口写得糊，整条流水线就在错误的前提上空转，等你回来 review 才发现白跑。所以"把 issue 写成规格"这件事的回报，在异步 agent 流里被放大了。

A1 把这点说得很直白：最有用的规格是自包含的——点名涉及哪些文件和接口，写明哪些不在范围内，并以一个端到端的验证步骤收尾，证明这个功能真的能跑；把规格打磨精确所花的时间，比盯着实现过程更值 [A1]。

## 实践方法

下面是一份可直接抄改的 agent-ready issue 模板。不是每个字段每次都要填满——小改动填前四块就够，越是不可逆、跨模块、money 相关的活，后面几块越要写全。

```markdown
## 背景 / Context
（这个任务为什么存在；现在的毛病是什么；相关的历史决策。当成写给
第一次进这个仓库的人。）

## 目标 / Goal
（要达成的一句话结果，可观测、可验收。）

## 非目标 / Non-goals
（明确不在这次范围内的事，防止 agent 顺手"改善"相邻代码。）

## 可能涉及的文件 / Likely files
- api/recommend.py        # 后端接口
- web/src/pages/Home.tsx  # 前端入口

## 禁区 / Do-not-touch
- 不要动 api/migrations/（迁移由人手写并评审）
- 不要读写 .env / secrets/

## 验收标准 / Acceptance criteria
- 新用户进首页看到默认榜单，不再是空白
- 新增行为有对应测试（测真实行为，别迎合实现）

## 验证命令 / Verify with
- make test            # 改后端必须跑
- pnpm -C web test     # 改前端必须跑
- make typecheck

## 预期证据 / Evidence to show
（贴命令 + 输出 / 截图，而不是一句"已完成"。）

## 风险等级 / Risk
低 / 中 / 高 —（涉及钱、迁移、删除、跨服务 → 高）

## 允许的权限 / Allowed actions
（只读？可写文件？可跑测试？可联网？可开 PR？）

## 需要人类拍板 / Needs a human decision
（列出 agent 不该自己定的点：选型、对外契约改动、数据删除……）

## 回滚 / Rollback
（出错怎么撤：分支、快照、回滚脚本。）
```

几个落地要点：

- **验收要可执行，不是"看起来对"。** A1 反复强调给 Claude 一个它自己能跑的检查——测试、构建、把输出和 fixture 比对、拿截图对设计稿——它就能自己跑、自己读结果、自己迭代到通过 [A1]。验收标准里写一句"用户登出时的边界要覆盖"，远不如给出具体测例和一条 `make test`。
- **让证据说话。** A1 的建议是让 agent 拿出证据而非声称成功：测试输出、它跑了什么命令返回了什么、结果截图——审证据比你自己重跑一遍快 [A1]。模板里的"预期证据"就是逼出这个。
- **风险等级决定权限和审查深度。** 低风险（加日志、改文案）可以放手；高风险（动钱、动迁移、删数据、跨服务）要把"需要人类拍板"写满，并收紧"允许的权限"。
- **细化的指令沉淀进 repo，别塞进每个 issue。** 这就是 WRAP 里的 R [G3]，也接上下一节。

## 真实案例

**同一个需求，坏 issue vs 好 issue。**

坏 issue（一句愿望）：

```
标题：修一下推荐
正文：推荐那块有问题，看着改改。
```

agent 拿到这个，只能猜：哪个推荐？什么问题？改后端还是前端？怎么算修好？它很可能挑一个自己看得顺眼的方向，动一片你没想让它动的代码，交一个"跑通了但不是你要的"的 PR。

好 issue（agent-ready）：

```
标题：首页推荐模块——给冷启动新用户补默认榜单
背景：新注册用户进首页，推荐区是空白，因为没有历史行为，
      协同过滤返回空。这是新用户流失的一个点。
目标：无历史行为的用户，推荐区展示默认热门榜单。
非目标：不改协同过滤算法本身；不动榜单的运营后台。
可能涉及：api/recommend.py、web/src/pages/Home.tsx
禁区：不要动 api/migrations/；不要读写 .env。
验收：新用户进首页看到默认榜单；新增分支有单测覆盖。
验证：make test、pnpm -C web test，两个都绿。
证据：贴两条命令的输出。
风险：低。权限：可写文件、可跑测试，不需联网。
```

这就是 G3 那条正反例的精神：从"把整个仓库都改成 async/await"这种范围爆炸的愿望，收敛到"把 authentication 中间件改成 async/await 并补单测" [G3]。区别不在写了多少字，在于 agent 能不能据此采取确定的动作、并自己判断做完没。

**GitHub 的 issue → PR 异步流。** 把上面这个好 issue 在 GitHub 上指派给 Copilot coding agent，会触发一条完整的异步流水线 [G2]：agent 开一个标 `[WIP]` 的草稿 PR 来跟踪进度；在一个由 GitHub Actions 驱动的安全、临时环境里探索代码、改动、跑测试和 linter；边干边推 commit、把关键步骤记进日志，你可以实时看进度；干完把 PR 标题和描述补好、@ 你来 review；你在 PR 上 `@copilot` 留言反馈，它据此迭代 [G2]。这条流的护栏是硬卡在产品层的：agent 只能推到它自己建的 `copilot/*` 分支、不能直接 push 到主干；所有 PR 必须经独立的人工 review；CI/CD 工作流不经批准不会跑；所有 commit 都标记 co-author 以便追溯；组织既有的策略和分支保护自动生效 [G2]。

把这条流和好 issue 摆在一起看就清楚了：issue 是入口规格，产品层的护栏是隔离操作间和验收门，你扫证据是终审。入口写糊，后面这套护栏只能拦住"危险"，拦不住"做错了方向"——方向对不对，只有 issue 里的目标和验收能管。

## 三家的 repo 指令文件：差异与收敛

WRAP 里的 R（细化指令）落到文件上，就是仓库里那几份 agent 指令。三家的情况：

| 维度 | GitHub Copilot coding agent | Claude Code | 通用约定 |
|---|---|---|---|
| 主用文件 | 读根级 `AGENTS.md`，也读嵌套在子目录、只对那部分生效的 `AGENTS.md` [G4] | 主用 `CLAUDE.md`，每次会话开头加载 [A1] | `AGENTS.md` 正成为跨厂商约定 |
| 还兼容 | `.github/copilot-instructions.md`、`.github/instructions/**.instructions.md`、`CLAUDE.md`、`GEMINI.md` [G4] | 通过 `@path` 语法导入别的文件 [A1] | — |
| 声明用途 | 指导 agent 理解项目、以及怎么构建/测试/验证改动 [G4] | 持久化命令、代码风格、工作流规则——代码里推不出来的东西 [A1] | — |
| 分层 | 根级 + 嵌套，嵌套只对那块代码生效 [G4] | 家目录 / 项目根 / 子目录多级加载 [A1] | — |

可以看到一个现实问题：同一个 GitHub agent，会同时认 `AGENTS.md`、`.github/copilot-instructions.md`、`.github/instructions/**.instructions.md`、`CLAUDE.md`、`GEMINI.md` 五种格式 [G4]。能兼容是好事，但一个仓库里散着五份口径可能打架的指令，本身就是新坑——agent 不知道该听哪份，你也不知道是哪份让它跑偏的。

收敛办法和上一章一致：**以 `AGENTS.md` 为单一真源**，其余文件要么删，要么只留一句"规则见 AGENTS.md"。A1 给 CLAUDE.md 的纪律在这里照样适用——只放代码里推不出来的东西（要跑的命令、和默认不同的风格、分支/PR 约定、环境怪癖、踩过的坑），把"写干净代码"这种废话删掉；文件太长，agent 会把一半规则淹没掉、读不到 [A1]。

> 注意 G4 这条来源本身有个教训：原本要核的那条 `2025-10-23` 的 GitHub changelog 链接是 404，根本不存在；AGENTS.md 对 coding agent 的支持实际是 2025-08-28 上线的 [G4]。一个日期格式都规整、看着无懈可击的厂商 URL，可以纯属虚构。无论写教材还是让 agent 引"官方文档"，链接都得自己点开核一遍。

## 常见反模式

- **把 issue 写成愿望清单。** "修一下推荐""让它更好用"——没有可观测的目标，没有验收判据。agent 只能猜，猜错你 review 时才发现。改成"动作 + 验收 + 验证命令"。
- **只写"做什么"，不写"别碰什么"。** 没有禁区，agent 会顺手"改善"相邻代码（改它没被要求改的格式、注释、类型）。非目标和禁区，直接限制它的误伤半径。
- **把验收写成形容词。** "代码要干净""性能要好"不可执行。换成具体测例和一条能跑的命令——A1 整张表都在讲这个：从"实现一个校验邮箱的函数"到"写 validateEmail，例：user@example.com 为真、invalid 为假、user@.com 为假，实现后跑测试" [A1]。
- **把范围写到爆炸。** "把整个仓库改成 async/await""把 300 万行 Java 重写成 Golang"——一个 issue 装不下。按 WRAP 的 A，拆成 authentication、data validation 等模块各一个原子任务 [G3]。
- **每个 issue 里重抄一遍仓库规矩。** 命令、风格、禁区这些反复出现的东西，该沉淀进 AGENTS.md（WRAP 的 R [G3]），不是每张派工单里复制粘贴。
- **五份指令各说各话。** AGENTS.md、copilot-instructions.md、CLAUDE.md……同一仓库散着多份可能矛盾的指令 [G4]。收敛到单一真源。
- **靠 issue 文字当安全边界。** 在 issue 里写"别 push 到主干"不是控制。真正的边界在产品层——GitHub 是 agent 只能推自己的 `copilot/*` 分支、PR 必须人工 review、CI 要批准才跑 [G2]。issue 管"做对方向"，护栏管"出不了大事"，两件事。

## 检查清单

派一个 issue 给 agent 前，过一遍：

- 这个 issue 有没有**可观测、可验收的目标**，而不是一句"修一下"？
- 写了**非目标 / 禁区**吗？agent 知道哪些文件、哪些范围不能碰？
- 验收标准是**可执行的检查**（测例 + 命令），不是"看起来对"？
- 给了**验证命令**和**预期证据**的要求吗？
- **风险等级**标了吗？高风险的活，"需要人类拍板"和"允许的权限"写清了吗？
- 反复要交代的仓库规矩，沉淀进 **AGENTS.md** 了吗，还是每个 issue 重抄？
- 仓库里的多份指令文件，收敛到**单一真源**了吗？
- 范围是不是**原子的**？能不能一个 PR 干完、一眼 review 完？

## 自测

- 把你最近写的一个 issue 拿给一个完全不懂这个项目的人看：他能不能只照这张单子，知道改哪、不碰哪、做完跑什么验证？做不到，agent 也做不到。
- 你上一次让 agent 干活，验收靠的是"它说做完了"，还是一条你自己能跑、能读结果的命令？
- 你仓库里现在有几份 agent 指令文件？它们口径一致吗？如果 agent 跑偏了，你说得清是哪一份带的吗？
