你愿意让一个 agent 自己改文件、自己跑命令、自己开 PR，前提是你信得过它。这份信任从哪来？不是来自它某次给你看的漂亮 demo，而是来自一套能持续告诉你"它干得到底怎么样"的机制。这套机制就是 eval（评估）。没有它，自治就是闭着眼睛踩油门。

eval 和测试不是一回事。传统测试问一个问题：代码按预期工作了吗。agent eval 要多问几个：它真的理解任务了吗、有没有越权乱来、有没有把预算烧光、该验证的地方有没有偷懒跳过、遇到不确定会不会停下来求助。一句话定调：没有 eval，就没有自治。

## 生活类比

招人进来，最不该信的就是面试当天那场表演。面试时人人都把最好的一面端出来——准备过的题、背熟的答案、刻意收着的脾气。真正能告诉你这人可不可靠的，是试用期：给他一摞真实任务，让他干满三个月，然后看几件面试根本看不出来的事。

活干对了没有——交上来的东西是不是真能用，还是看着像那么回事、一上线就崩。

有没有越界乱来——你让他改前台的报价，他会不会顺手把后台的客户数据库也动了；该问的时候问没问。

有没有浪费——同样一件事，他是半天搞定，还是来回折腾、把一周的预算烧在一个本该一小时的活上。

遇到不确定会不会求助——卡住的时候，他是停下来问一句，还是自己瞎猜一个方向闷头跑，最后交上来一堆错的。

试用期考核不是一道是非题，是一张多维表格。一个人可能"活干得对"但"特别费"，也可能"又快又省"但"动了不该动的东西"。把这几栏分开打分，你才知道下次该不该把更大的活、更高的权限交给他。eval 就是给 agent 设计的这张试用期考核表——你要评的，从来不只是它"这次有没有做对"，而是它在真实任务里的整体表现配不配拿到更大的自主权。

## 工程定义

一个 eval，就是给 AI 系统的一次测试：给它一个输入，再用一套打分逻辑去衡量它的输出，看成功没有 [A10b]。这是最朴素的定义，但 agent 把它撑大了。

单轮 eval 很简单：一段 prompt、一个回复、一套打分逻辑，对早期的语言模型够用了 [A10b]。agent 不是单轮的。它跨很多轮调工具、改环境状态、根据中间结果边走边调整——这意味着错误会沿途传播、累积 [A10b]。所以 agent eval 评的不只是最后那个回复，而是整条轨迹加上最终状态。

把几个术语先掰清楚，后面才不会绕（都按 Anthropic 这篇的口径 [A10b]）：

- **task（任务）**，一个有明确输入和成功判据的单一测试，也叫 problem 或 test case。
- **trial（试跑）**，对同一个任务的一次尝试。模型每次输出都不一样，所以同一个任务要跑多次，结果才稳。
- **grader（评分器）**，给 agent 表现的某个侧面打分的那段逻辑。一个任务可以挂多个 grader，每个 grader 里又有多条断言（assertion）。
- **transcript（轨迹）**，一次 trial 的完整记录——输出、工具调用、推理、中间结果，全在里头。也叫 trace 或 trajectory，对应到本书的"监控录像"。
- **outcome（结果状态）**，trial 结束时环境的最终状态。一个订票 agent 在轨迹末尾说"您的航班已订好"，但真正的 outcome，是环境的数据库里到底有没有那条预订记录 [A10b]。这两件事可以不一致——它嘴上说订好了，库里没有，照样是失败。
- **evaluation harness（评估夹具）**，端到端跑 eval 的那套基础设施：发指令、给工具、并发跑任务、记录每一步、打分、汇总。注意它和 agent harness 是两层东西——后者是让模型能像 agent 一样行动的那套脚手架。你评"一个 agent"的时候，其实是把 harness 和模型一起评了 [A10b]。

定义里最容易被忽略的一句，是 eval 和传统测试的分界：传统测试基本只看 outcome 对不对；agent eval 还要看 transcript——它是怎么走到这一步的。因为对一个会自己动手的东西来说，"走对了"和"碰巧蒙对了"是两码事，而后者迟早会在你没盯着的那次崩给你看。

## 为什么重要：AI 时代的变化

团队刚开始做 agent 时，靠手动测、内部试用、一点直觉，能走得出奇地远 [A10b]。这个阶段你甚至会觉得正经做 eval 是种拖慢交付的额外开销 [A10b]。问题出在过了原型期、agent 上了生产、开始放量之后——没有 eval，这套就撑不住了 [A10b]。

崩盘点通常长这样：用户开始反映"改版之后 agent 变差了"，而团队"两眼一抹黑"，除了猜和试没别的办法验证 [A10b]。没有 eval，调试只能是被动的——等投诉、手动复现、修掉这个 bug、再祈祷没把别处弄坏 [A10b]。团队分不清这到底是真的退化还是噪声，没法在上线前拿几百个场景自动跑一遍，也没法量化任何改进 [A10b]。

这件事还卡着你换模型的速度。新的、更强的模型出来时，没有 eval 的团队要花上几周做测试；有 eval 的竞争对手，能很快摸清新模型的长短板、调好 prompt，几天就完成升级 [A10b]。

最该破除的直觉，是"agent 干活和写程序一样，对就是对、错就是错"。前沿模型会找到超出静态 eval 设想的创造性解法 [A10b]。Anthropic 给了个很说明问题的例子：Opus 4.5 在 τ2-bench 一道订机票的题上，通过钻政策的一个空子解出来了——按 eval 写死的判据，它"失败"了，可它实际上给用户找到了一个更好的方案 [A10b]。这正是 agent eval 比传统测试难的地方：当被评的对象比你的判据还聪明，"判它失败"和"它真做错了"就不再等价。

也正因为如此，eval 不该是一次性打分，它是一种可以设计、可以调试、可以复用的能力 [O5]。OpenAI 把这句话说得很具体：一个 eval 拆开看，就是"一段 prompt → 一次被记录下来的运行（trace + 产物）→ 一小组检查 → 一个能跨时间对比的分数" [O5]。把"它感觉变好了吗"换成"分数从 71 涨到 78、回归集没掉"，你才有东西可改。

## 实践方法

把试用期考核表落到工程上，是六个维度。不是每个 agent 都要全上，但你心里得有这张表，知道自己只评了哪几栏、漏了哪几栏。

| 维度 | 评的是什么 | 典型 grader |
|---|---|---|
| 结果质量 | 产物对不对、能不能用 | 单元测试、outcome 状态检查 |
| 过程质量 | 有没有危险动作、有没有跳过该做的步骤 | transcript 分析、工具调用核对 |
| 安全 | 有没有越权、有没有碰不该碰的数据 | 权限/越界检查、产物审查 |
| 效率 | 成本合不合理、有没有反复折腾 | token 用量、turn 数、命令数 |
| 可恢复 | 失败之后，另一个人/agent 接得住吗 | 留痕是否完整、状态是否落盘 |
| 环境噪声 | 这次失败到底是代码错了还是基础设施抽风 | 环境隔离、跨 trial 独立性 |

最后这一栏单拎出来讲一句。每次 trial 都应该从一个干净的环境起步、彼此隔离 [A10b]。run 与 run 之间留下的共享状态——残留文件、缓存数据、资源耗尽——会因为基础设施的抖动造成"相关失败"，而那跟 agent 本身的能力毫无关系 [A10b]。共享状态甚至会反过来虚高分数：Anthropic 在内部 eval 里就观察到，Claude 通过翻看上一轮 trial 留下的 git history，在某些任务上拿到了不公平的优势 [A10b]。环境这一栏没控住，上面五栏的分数都是脏的。

具体到 grader，三种各有各的用处，要搭着用 [A10b]：

- **基于代码的 grader**（快、便宜、客观、可复现、好调试）：字符串匹配、二元通过/失败测试、静态分析、outcome 核对 [A10b]。它的弱点是脆——只要 agent 用了一个合法但不匹配你预设模式的写法，就会被误判 [A10b]。
- **基于模型的 grader**（灵活、可规模化、能抓住细微差别）：rubric 打分、自然语言断言、成对比较 [A10b]。代价是非确定、比代码贵、而且必须拿人工评分来校准才可信 [A10b]。
- **人工 grader**（金标准，但慢且贵）：专家评审、抽样核查、A/B 测试 [A10b]。它的主要用途，是去校准上面那个模型 grader [A10b]。

设计 grader 有一条反直觉但重要的经验：别去死抠 agent 是不是按某个固定步骤序列走的。Anthropic 发现，那种"工具必须按这个顺序调"的检查太死，结果是测试极脆——agent 经常会找到设计者没预想到的、同样合法的路子 [A10b]。更好的做法是评它产出了什么，而不是它走了哪条路 [A10b]。这跟试用期那个直觉一致：你验收的是活，不是他用没用你习惯的那把扳手。

任务集要两边都覆盖：既测"该触发某行为时它触发了"，也测"不该触发时它没乱触发" [A10b]。只测一头，优化也会偏一头——你只奖励"该搜索时去搜索"，最后会养出一个见什么都去搜索的 agent [A10b]。

还要把 capability eval 和 regression eval 分开 [A10b]：

- **capability eval**（能力/质量 eval）问的是"这 agent 能干好什么"，起步通过率就该低，专挑它搞不定的任务，给团队留一座要爬的山 [A10b]。
- **regression eval**（回归 eval）问的是"它以前会的，现在还会吗"，通过率应该接近 100%，分数一掉就说明有东西坏了 [A10b]。一个 capability eval 被优化到高通过率之后，可以"毕业"转成回归集，持续跑、防回退 [A10b]。

最后，eval 不是孤立存在的。OpenAI 描述过这样一圈循环：trace 告诉你发生了什么，人和模型的反馈解释了什么才是重要的，eval 把这些期望固化成可复用的东西，然后 Codex（或别的执行 agent）真的去改 [O4]。trace + human-in-the-loop + eval，三样扣成一圈，才是让 agent 持续变好的那台飞轮——单有 trace 是一堆没人读的录像，单有 eval 是一张没人维护的旧考卷。

## 真实案例

**钻空子反而对用户更好，却被判失败。** 前面提过的那道 τ2-bench 订机票题值得再展开。Opus 4.5 找到了政策里的一个漏洞，用它解出了题——对真实用户来说这是个更好的结果，但按 eval 写死的判据，它算"失败" [A10b]。这不是 agent 的错，是 eval 的判据没跟上一个比它聪明的被测对象。它逼出一条原则：分数不能照单全收，得有人钻进 eval 内部、读几条 transcript，确认失败是"公平的失败"——一眼能看清 agent 到底错在哪、为什么错 [A10b]。

**96.12 被判错，其实期望是 96.124991…。** 同一篇里有个更刺眼的例子。Opus 4.5 在 CORE-Bench 上一开始只拿了 42%，直到一位 Anthropic 研究员钻进去，发现一堆毛病：grader 死板到把 "96.12" 判成错，因为它期望的是 "96.124991…"；还有任务描述本身有歧义、以及根本无法复现的随机性任务 [A10b]。修掉这些 bug、换上一个约束更松的 scaffold 之后，Opus 4.5 的分数从 42% 跳到了 95% [A10b]。同一个模型，分数差出 53 个百分点，差的全在 eval 自己身上。这就是为什么有条铁律：在有人钻进细节、读过几条 transcript 之前，eval 分数一律不照面值采信 [A10b]。

**一年从约 30%–40% 涨到 >80%，逼近饱和。** SWE-bench Verified 给 agent 一批 GitHub issue，跑测试套件来判分——只有在修好失败测试、又不弄坏已有测试时才算通过 [A10b]。这一年里，前沿模型在它上面从 30% 起步、如今逼近 >80% 的饱和（这篇另一处把同一段进步表述为"一年内从 40% 到 >80%"）[A10b]。越接近饱和，进步越慢，因为只剩最难的题没解 [A10b]。这里藏着一个陷阱：越饱和，能力的大提升越只表现成分数上的小变化 [A10b]。代码审查公司 Qodo 一开始对 Opus 4.5 不以为然，正是因为他们那套一次性（one-shot）的编码 eval 抓不住模型在更长、更复杂任务上的提升；后来他们专门做了一套 agentic eval 框架，才看清真实的进步 [A10b]。

**一道用了约 18 个月、筛过 1000+ 候选人的题，被一次模型升级报废。** Anthropic 性能团队从 2023 年底起设计了一道 take-home 题：一个模拟某种加速器（接近 TPU 架构）的 Python 模拟器，要求候选人做并行树遍历优化，初版给 4 小时、后来压到 2 小时 [A19]。这道题好用了约 18 个月，被 1000 多人做过，从中招到的工程师撑起了 Anthropic 的不少基础设施 [A19]。然后模型追了上来：到 Opus 4，它在时限内已经胜过大多数人类申请者；到 Opus 4.5，它在 2 小时窗口里追平了顶尖的人类选手 [A19]。一道筛了一年半、千人验证过的好题，就这么被一次 Opus 4→4.5 的升级直接打穿了 [A19]。它给所有做 eval 的人提了个醒：能力一直在涨，你的考卷会比你想象中更快过期，eval 是要持续维护的活资产，不是一锤子买卖。

## 常见反模式

- **eval theater（评估剧场）。** 拿几个好看的、精心挑过的 demo case 当 eval，跑过了就宣布"agent 没问题"。真正的 eval 要用真实失败样本、边界样本、回归样本来构 [A10b]：从你的 bug tracker 和支持工单里把用户报过的失败转成 test case [A10b]，测两个方向（该触发 / 不该触发）[A10b]，而不是端几道你知道它能过的题上来演一场。
- **分数照单全收。** 看见一个分数就当真，不读 transcript。前面 96.12 和订机票那两个例子说明，分数低可能是 eval 自己的 bug，不是 agent 的错。规矩是：没人钻进细节、读过几条轨迹之前，分数不采信 [A10b]。
- **只评结果，不评过程。** 只看产物对不对，不看它是怎么干出来的。一个会自己动手的东西，"碰巧蒙对"和"真的做对"在分数上没区别，但前者会在你没盯着的那次出事。该测 transcript：动了哪些工具、跳没跳过验证、有没有越界。
- **grader 死抠步骤。** 要求 agent 必须按某个固定的工具调用顺序走，结果测试脆得一碰就碎——agent 老会找到你没预想到、但同样合法的路子 [A10b]。评产出，别评路径 [A10b]。
- **一次性 demo 当永久结论。** 今天过了的 eval，不等于换个模型还过、半年后还过。能力在涨，考卷会过期 [A19]。eval 是要有人 own、要持续维护的活资产 [A10b]。
- **无 eval 的自治。** 让 agent 能写文件、跑命令、开 PR，却没有任何验收和回归检查。这是本书反复敲的那句：没有 eval，就没有自治。

## 检查清单

- 我这套 eval，是用真实失败 / 边界 / 回归样本构的，还是几个我知道它能过的好看 case？
- 我评的只是"结果对不对"，还是也评了过程（越界、跳验证、瞎折腾）？六个维度里我漏了哪几栏？
- grader 是评 agent **产出了什么**，还是死抠它**走了哪条路**？
- capability eval 和 regression eval 分开了吗？回归集是不是接近 100% 通过、一掉就报警？
- 我有没有把环境隔离干净——每次 trial 从干净状态起步，失败能分清是代码还是基础设施？
- 看到分数，我是照单全收，还是会钻进去读几条 transcript 确认"失败是公平的"？
- trace + 反馈 + eval 有没有扣成一圈能持续改进的循环，还是三样各放各的？

## 自测

- 你上一次说"这个 agent 表现不错"，依据是几个跑给你看的 demo，还是一套你能复跑、能跟上次对比出分数的 eval？
- 如果现在有人质疑你某个 agent 的某次失败"到底是模型的锅还是环境的锅"，你能不能凭 trace 当场分清？分不清，说明你这套 eval 的环境隔离还没做。
- 你给 agent 的自主权，和你对它的 eval 覆盖，是匹配的吗？如果它的权限远大于你能验收的范围，那多出来的那部分自治，就是闭着眼睛踩的油门。
