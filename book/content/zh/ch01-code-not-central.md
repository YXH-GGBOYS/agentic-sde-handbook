过去十几年，一个软件工程师交付的东西，最后都收敛成一件：代码。设计文档、白板、聊天记录都是过程，merge 进主干的那几行才是结果。评职级看它，复盘看它，出了事也回去翻它。

agent 进来之后，这条收敛线断了。代码还在，但它从"那件最终产物"退回成"产物之一"。真正决定一次交付成不成的，越来越多地落在代码外面：你给 agent 的指令、它跑活的隔离环境、判定它干完了没有的那套验收、它一路留下的 trace、出事时能不能撤回。这一章讲的就是这个位移——你的核心产物，从一盘菜，变成了一整套能稳定出餐的后厨。

## 生活类比

设想你是个手艺很好的厨师，开店之前，你的全部产出就是端上桌的那盘菜。火候、调味、摆盘，从头到尾你一个人盯，客人满意不满意，直接对应你这盘菜做得好不好。你的产物，就是这盘菜。

现在你不炒了，你开了家餐厅，后厨雇了几个手很快的厨子。这时候你交付的东西，不再是某一盘菜——是整个后厨能不能稳定出餐。这套东西包含一长串你以前根本不用操心的环节：一张**写清楚"我们家这道菜怎么做"的备菜单**（哪样先下、火候多少、摆盘什么样，不能靠厨子自己猜）；**备料区**，开火前把料配好摆齐，免得上灶手忙脚乱；一个**出餐口**，菜做好了从这儿走，谁取、怎么核对；一个**验收的人**，菜出后厨前得有人尝一口、看一眼对不对，不是厨子说好了就直接端上桌；一张**排班表**，三个厨子同时开工，谁管热菜、谁管凉菜、别挤在一个灶台上互相撞；还有一套**出错能撤的约定**，这盘咸了能拦下来重做，不能直接送到客人面前。

你自己那手炒菜的本事还在，但它不再是你交付的全部。一个开餐厅的人，价值不在他比厨子多颠几下勺，而在他把这套后厨系统设计得能稳定出餐——哪怕今天换一批厨子，菜的水准不塌。你从"做那盘菜的人"，变成了"让一整个后厨稳定出菜的人"。

这正是 agent 时代工程师身上发生的事。你还会写代码，但你交付的，是让一群 agent 能稳定产出可靠软件的那套系统。

## 工程定义

先说清楚旧的"中心产物"是什么意思。在一个传统团队里，几乎所有工程价值最后都沉淀进版本库里的代码：评审审它、CI 测它、生产跑它。文档会过期、对话会沉底，唯独代码是那件"权威的、被持续维护的、出了事回去翻的"东西。这是过去的默认设定。

agent 把这个设定改了。当一个 agent 能读整个 repo、改多个文件、跑命令、开 PR、等 review、然后从中断处恢复，它干活就需要代码之外的一整套配套件。这些配套件不再是"过程性的草稿"，它们和代码一样进版本库、一样要认真维护。把它们列出来，就是 agent 时代真正的产物清单：

- **代码与测试**——还在，但只是其中一环。
- **issue / 任务描述**——agent 的活从这里来。GitHub 的 coding agent 就是你把一个 issue 指派给它、它才开工 [G1]。
- **agent 指令文件（AGENTS.md / CLAUDE.md）**——贴在仓库里的家规，告诉 agent 该用哪些命令、哪些目录别碰、怎么算完成。OpenAI 做 Sora 安卓版时，让 Codex 在整个代码库里维护了一批 AGENTS.md 文件，好把同一套规矩和最佳实践跨会话复用 [O6]。
- **sandbox / 隔离执行环境**——agent 跑活的地方。GitHub 的 coding agent 在一个由 GitHub Actions 驱动的临时环境里开机、克隆仓库、跑测试和 linter [G1][G2]。
- **eval / 验收**——判定 agent 干完没干完的那套可执行标准。Anthropic 在长任务 agent 里用一个 JSON 特性清单当验收：claude.ai 克隆版列了两百多条端到端特性，每条初始都标成"未通过"，agent 只有真测过才能把状态改成"通过" [A2]。
- **trace / 会话留痕**——agent 一路读了什么、试了什么、哪步失败。GitHub 的 agent 把每一步推进都写进 session log，让你能回溯它的决策 [G1]。
- **PR 与其中的讨论**——交接和评审都在这儿发生，commit 还会标记 co-author 留痕 [G2]。
- **发布脚本、监控、CI 配置**——在 OpenAI 那个全 agent 生成的内部产品里，这些全是产物的一部分：agent 产出的不只是产品代码和测试，还有 CI 配置、发布工具、文档、eval harness，乃至生产仪表盘的定义文件 [O1]。

所以"代码不再是中心产物"这句话的准确意思是：工程产物从一件（代码）变成了一套——代码 + 测试 + issue + 指令文件 + sandbox + eval + trace + PR 讨论 + 发布脚本 + 监控。代码仍然重要，但它现在是这套系统里的一环，不再是那个唯一被供奉的中心。

## 为什么重要：AI 时代的变化

这个位移会逼着四件事从旧做法迁到新做法，而且基本不可逆。

**一、prompt engineering → harness engineering。** 早期大家以为，用好 AI 的关键是把提示词调得够神。真上手做大东西就会发现，瓶颈不在那句话怎么写，而在 agent 周围那套环境搭得好不好——它能跑哪些工具、在哪执行、出错怎么恢复、谁来验收。OpenAI 那支团队的复盘里，早期进度慢"不是因为 Codex 不行，而是因为环境没规定清楚"：agent 缺的是工具、抽象和内部结构，于是工程师的主要工作变成了"补上缺的能力，让 agent 能干活" [O1]。提示词只是喂给大脑的一句话，环境才是大脑之外那台操作系统。

**二、chatbot → agent runtime。** 聊天机器人是一问一答，你贴代码、它回代码、你自己拷回去。agent 不一样：它要自己读 repo、自己改文件、自己跑命令、自己开 PR、等 review、还要能从中断处接着干。这就要求一个真正的运行时。最难的一块是跨上下文窗口的连续性——Anthropic 打了个很到位的比方：长任务 agent 像一个排了班的工程团队，每个新来的人都对上一班发生的事毫无记忆 [A2]。它们的解法是让上一班留下交接物：一个 `claude-progress.txt` 进度文件、git 提交历史、一个 `init.sh` 启动脚本，下一班靠读这些东西接上 [A2]。这些交接物本身就是产物——没有它们，agent 根本接不上自己上一轮的活。

**三、benchmark → production eval。** 跑分能告诉你模型在精心准备的题目上多强，但真实失败几乎从不长这样。真实的失败来自模型、工具、权限、网络、依赖、环境漂移这些边界：依赖装不上、网络抽风、agent 没权限动某个文件、本地环境和生产对不上。GitHub 之所以默认把 agent 的联网限制在一份可信清单内、把 CI 工作流卡成必须人工批准 [G1]，正是因为这些边界才是真出事的地方。验收因此必须搬到真实环境里去做，而不是看一个静态分数。

**四、个人打字速度 → 组织吞吐。** 当 agent 把写代码这一段大幅加速，你一个人手打多快就不再是瓶颈了。OpenAI 那个全 agent 团队，三个工程师驱动 Codex 开出约一千五百个 PR，平均每人每天三点五个；而且团队涨到七个人之后，人均吞吐不降反升 [O1]。Sora 那边也是同样的转向——团队的瓶颈"从写代码，挪到了做决策、给反馈、整合改动" [O6]。要把吞吐撑起来，靠的不是某个人打得更快，而是把规范、工具、检查、review 都 repo 化、固化进系统，让它们对每一次 agent run 自动生效。

把这四件合起来看，价值的落点整体上移了。人不再把时间花在逐行敲代码上，而是移到定义问题、设边界、定验收、控风险、定架构这几件事上。OpenAI 那篇复盘把它压成四个字：**人来掌舵，agent 执行** [O1]。Sora 团队的说法更直白——AI 辅助开发不是减少了对严谨的需求，是增加了它 [O6]。

## 实践方法

承认这个位移，落到日常就是：别再只盯着代码这一件产物，去把代码外面那几样东西也当成要交付、要维护的工件来对待。

- **派活前先写指令文件。** 把"我们家这套怎么做"写成一份 AGENTS.md：用哪些命令、哪些目录是禁区、怎么算完成。Sora 团队甚至让 Codex 自己跨代码库维护这些文件，好让同一套规矩跨会话复用 [O6]。这是那张备菜单。
- **从 issue 起步，不从聊天框起步。** 把任务写成一个有清楚目标和约束的 issue，再交给 agent，而不是在对话里随口甩需求。GitHub 的整条流程就是这么设计的：issue 进、PR 出 [G1][G2]。
- **先定验收，再让它跑。** 想清楚"怎么算干完"——哪些测试必须绿、哪条标准必须满足——并且尽量做成可执行的，像 Anthropic 那份会被一条条核掉的特性清单 [A2]，而不是事后扫一眼"看着对"。
- **让它在隔离环境里跑。** 别让 agent 直接在你的真仓库、真数据上动手。GitHub 给它的是一个由 Actions 驱动的临时环境 [G2]；你至少也该给它一个独立的工作树或容器。
- **留住 trace。** 确保 agent 干完留下的记录，足够你或者下一个人接手——读了什么、跑了什么、哪步失败。GitHub 的 session log 就是干这个的 [G1]。
- **像 review 队友一样 review 它。** Sora 团队明说，他们 review Codex 的产出，跟 review 一个同事的产出是一样的标准 [O6]。验收不会因为代码是 agent 写的就可以省。

## 真实案例

**OpenAI 用 Codex 在 28 天里做出 Sora 安卓版。** 从 2025 年 10 月 8 日到 11 月 5 日，一支四人工程团队配合 Codex，把 Sora 安卓版从原型推到全球发布：内部版 18 天做出，10 天后公开上线 [O6]。发布当天冲到 Play Store 第一，头 24 小时用户生成了超过一百万条视频；崩溃率保持在 99.9%，整个项目估算约 85% 的代码由 Codex 写出 [O6]。

值得细看的是这 85% 是怎么来的。这支团队**没有**一句"照着 iOS 代码把安卓版建出来，开干"就甩给 Codex——他们实际试过这条路，很快就放弃了，因为单发出来的代码不可靠，产品体验也差 [O6]。真正管用的做法是反过来：人亲手把地基打好——架构、模块化、依赖注入、导航这些结构性决策由团队自己定，认证和基础网络层也是人写的；他们还亲手写了几个有代表性的特性当样板，把全项目要遵循的规矩沉淀下来 [O6]。在这个地基之上，Codex 才在"边界清楚、模式明确"的范围里填进大量代码。人定结构和不变量，agent 在结构里干活。代码占了 85%，但决定这个项目成不成的，是那剩下 15% 里的架构、验收和发布质量把关。这就是价值上移的样子——团队把精力放在架构、用户体验、系统性改动和最终质量上，把"在已知模式里填代码"这件事交给 Codex [O6]。

**GitHub 把 coding agent 做进了从 issue 到监控的全流程。** 它不是一个聊天框，而是一套嵌进 GitHub 的工作流：你把一个 issue 指派给它，它就在后台开一台临时机器、克隆仓库、用 RAG 分析代码，然后一边干一边把 commit 推进一个 draft PR，每一步推进都写进 session log 供你回溯 [G1]。整个流程跑在 GitHub Actions 上，agent 在那个环境里跑测试和 linter [G2]。护栏是硬卡在系统里的：agent 只能推到它自己建的分支、联网被限制在一份可信清单内、CI 工作流必须人工批准、发起任务的人不能批准这个 agent 开的 PR [G1]。这一整条 issue → 分支 → PR → Actions → session log，恰好就是前面那张产物清单的实物版——代码只是从这条流程里流过的其中一样东西。

## 常见反模式

- **只把代码当产物，其余全靠临场。** 指令、验收、隔离、撤回路径都不沉淀进 repo，每次重新口头交代。这等于开餐厅却没有备菜单、没有验收员，全指望厨子今天心情好。
- **把约束全写进提示词。** 以为一句够神的 prompt 能顶替测试、权限和隔离环境。约束该落在工具层和 sandbox 里，不是刷在 agent 脑门上。这是把 harness 塞进 prompt，方向反了。
- **拿 benchmark 分数当生产验收。** 模型在跑分上漂亮，不等于它在你这套依赖、权限、网络、环境里不出事——真实失败几乎都来自这些边界，不来自题目。
- **用 agent 提速，却不动组织那一层。** 把规范、检查、review 还留在某个人脑子里、聊天记录里，光让 agent 跑得快。吞吐很快会撞回人这个瓶颈上——Sora 团队的瓶颈正是这么从写代码挪到决策和整合上的 [O6]。
- **代码是 agent 写的，就免了 review。** 验收不会因为产出方是 agent 就可以省。Sora 团队用 review 同事的同一套标准来 review Codex [O6]。

## 检查清单

- 我交付的，是不是只有代码？指令文件、验收标准、隔离环境、trace、撤回路径，有没有也当成要维护的产物？
- 我的任务是不是从一个目标和约束都清楚的 **issue** 起步，而不是在聊天框里随口甩需求？
- "**算完成**"有没有一条可执行的验收标准，而不是事后"看着对了"？
- agent 是在**隔离环境**里跑，还是直接动我的真仓库、真数据？
- 它跑完留下的 **trace**，够不够另一个人接手？
- 我有没有把规范、检查、review **沉淀进 repo**，让它们对每一次 agent run 自动生效——而不是只活在某个人脑子里？

## 自测

- 如果现在让你把团队的全部工程产物列一张清单，代码占第几位、占多大比重？两年前你会怎么列？
- 你上一次用 agent 干活，约束和验收是写进了仓库、对下一次 run 自动生效，还是这一次口头交代完就没了？
- 把那盘菜端上桌之前——谁尝的那一口？如果答案是"没人，测试过了就直接送"，你缺的是验收员，不是更强的厨子。
