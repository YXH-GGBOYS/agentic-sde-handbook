这本书前面写了三十来章，从"代码不再是中心产物"一路讲到失败分类学。读到这里，最容易犯的错是想把它们一口气全装进脑子——结果是哪一章都记不牢，更别说落地。

这一章不教新东西。它做一件事：把前面所有章节重新排成一条**能上手的路线**。不按天排（"第一周学 harness"这种计划基本都会落空），而是按模块排——每个模块有一个当场能做出来的小目标，做出来了，再进下一个。

## 生活类比

学游泳的人没有一个是先把《流体力学》从头背到尾、再下水的。

你是分模块来的。第一节课，教练只让你做一件事：抱着浮板，脸埋进水里，体会"原来人是浮得起来的"。这一关过了，下一节课才教换气——侧头、吐气、再侧头。换气顺了，才轮到蹬腿、划手。每一关都小到当堂能做出来，做出来了你才有底气进下一关。没人会因为你"还没学完转身和出发台技术"就不让你下水。

这套学法有三个好处，搬到学 agentic SDE 一样成立。**一是每关都有一个能摸到的成品**——你这堂课结束时是真的浮起来了，不是"理解了浮力的概念"。二是顺序有依赖：不会换气就练蹬腿，只会呛水；同理，没把 repo 整理成 agent 能读的样子，就急着上多 agent 编排，只会乱成一团。三是允许分叉：想横渡的人和想学花样游泳的人，前几关一样，后面才分开。学这套东西也分人——个人上手、团队落地、组织治理，前几个模块共用，后面才各走各的。

这一章就是那张"分模块的课程表"，外加三条不同目标的报名路线。

## 工程定义

**学习路线（learning path）** 在这里指一组有依赖关系的学习模块，每个模块用四样东西界定清楚：

- **学习目标**：学完这一关你能"说清"并"做出"什么。
- **核心材料**：对应本书的哪些章节（这是 synthesis 章，材料全部指向书内，不引外部新事实）。
- **练习**：一个具体的、当场能做的动手任务。
- **交付物（artifact）**：一个能直接放进真实项目的工件——一份 AGENTS.md、一张 harness spec、一个 eval rubric。**交付物是这套路线的硬通货**：学完一个模块，你手里多一个能用的东西，而不只是多一段印象。

"模块"和"天数"的区别值得掰开。按天排是按**时间**切（"周一到周三学 X"），它假设每个人学每样东西花的时间一样、且不会被打断——这两个假设在真实工作里几乎从不成立。按模块排是按**能力关卡**切（"做出一份能让 agent 跑通测试的 AGENTS.md"），它不关心你花了三小时还是三天，只关心那个交付物到底做出来没有。前者像健身房贴的"30 天计划表"，后者像"先做到标准俯卧撑 10 个，再加重量"。

一个模块没"通关"的判据很简单：**它的交付物还没做出来，或者做出来了但经不起用**。一份 AGENTS.md 写得漂亮但 agent 照着跑不通测试，这个模块就没过——回去补，别硬往下走。

## 为什么重要：AI 时代的变化

过去学软件工程，路线大体是线性的：语法 → 数据结构 → 框架 → 系统设计。每一层叠在上一层之上，一个人慢慢爬。

agent 时代多出来一批**新的工程对象**——前面第三卷专门讲过：repo 指令文件、context、tool/MCP、state artifact、eval、trace、sandbox。这些东西在传统课程表里根本不存在，没有现成的学习顺序可抄。于是很多人凭直觉乱抓：今天试个新模型，明天调个 prompt，后天上个多 agent 框架，看起来很忙，半年后回头发现手里没有一个能复用的工件，团队也没法照着复制。

按模块学之所以在这个时代更重要，有三个理由。

第一，这些新对象之间有**真实的依赖**，跳级会塌。第四卷反复强调过一条硬规矩：没有 eval 就没有自治。你要是没先学会 eval（怎么判断 agent 干对没干对），就急着把权限放开让它自己合并 PR，等于不会验收就让外包直接发货。同理，repo 没整理成 agent 读得懂的样子（第三卷），就上多 agent 编排（第五卷），等于让一个班组的外包在没贴家规便条的屋里乱凿。依赖关系决定了顺序，不是个人偏好。

第二，交付物导向能挡住"看个热闹"。agent 工具的 demo 都很漂亮，看完容易产生"我懂了"的错觉。但第一卷引过的那条实证提醒过：人对自己"用 AI 变快了"的直觉并不可靠。挡住错觉的办法，是每个模块都逼出一个能进真实项目的工件——工件做不出来，就说明没真懂。

第三，三类人需要的顺序不一样。一个想自己用顺 agent 的资深工程师，和一个要在团队推行委派制的技术负责人，和一个要给整个组织定治理底线的人，起点和重点都不同。线性课程表服务不了这三种人，分模块 + 分路线可以。

## 实践方法

下面是完整的模块图。十三个模块，每个对应本书的一卷或几章。先看全表，再看怎么按路线挑。

### 模块图

| 模块 | 名称 | 对应本书 | 学习目标（能说清+做出） | 交付物 |
|---|---|---|---|---|
| M1 | 2026 SDE 全景 | 第一卷 | 说清 agent 时代 SDE 的产物从"代码"变成了"一套交付系统"；说清什么过时、什么降级、什么更重要 | 一页"我的产能瓶颈在哪"自评（卡在打字还是卡在定义与验证） |
| M2 | Agent-ready repo 与 AGENTS.md | 第三卷 ch10 | 把团队默会规矩写成机器可读、agent 真会照做的便条 | 一份覆盖项目形状/命令/DoD/安全边界/review 期望的 AGENTS.md |
| M3 | Issue 工程 | ch06、ch26 | 把模糊需求翻成可验证的问题定义；按 WRAP 写 agent-ready issue | 同一需求的"坏 issue vs 好 issue"对照 + 一张 issue 模板 |
| M4 | Harness 工程 | 第四卷 | 说清 harness 是什么、不是什么；画出从接活到交活的系统边界 | 一份最小 harness spec（六问：能做什么/谁批准/在哪跑/算完成/怎么撤/留什么 trace） |
| M5 | Context 工程 | ch11 | 把上下文当预算花：该检索的检索、该摘要的摘要，不硬塞全仓库 | 一个任务的 context 清单（喂什么、不喂什么、为什么） |
| M6 | Tool / MCP 工程 | ch13 | 说清"给人好用的 API" ≠ "给 agent 好用的工具"；写一份 tool contract | 一份带错误语义的 tool contract |
| M7 | Eval 工程 | ch18 | 说清 eval ≠ test；用真实失败样本（不是 demo）构 eval | 一个多维 eval rubric + 一个回归集种子 |
| M8 | Sandbox 与安全 | ch17 | 列出威胁模型；按风险/路径/命令/网络/secret 分级设权限 | 一张权限矩阵 + 一份安全底线清单 |
| M9 | 三家对比实战 | 第六卷 | 同一任务在 Claude Code / Codex / Copilot 上各自怎么用；会选型 | 一张三家选型对照表（按任务类型） |
| M10 | 多 agent 编排 | 第五卷 | 说清什么时候该上多 agent、怎么不让它们打架爆预算 | 一个 orchestrator-workers 的小流程设计 + 独立 evaluator |
| M11 | Agentic PM | 第七卷 | 从排人天转到排 agent-ready work；用新指标而非 PR 数报进度 | 一份 agent-ready backlog 标准 + 一套失败分类学 |
| M12 | 发布与运营 | 第七卷 + ch17 | 把 agent 产出的改动安全送上线：灰度、可回滚、留痕可审计 | 一份 release runbook（含回滚路径与审计点） |
| M13 | Capstone | ch31 | 把以上全部用一遍，给一个真实 repo 建一套 harness 并跑通 | 一份结业报告：什么适合委派、什么不适合、下一轮怎么改 harness |

读这张表的方法：**从上往下是依赖顺序**，不是难度顺序。M2（agent-ready repo）必须在 M10（多 agent）之前，因为没整理好的仓库经不起多个 agent 同时改。M7（eval）必须在 M12（发布）之前，因为没有验收就谈不上安全上线。但 M5、M6 这种平行模块，先学哪个无所谓。

### 三条并行路线

不是每个人都要从 M1 走到 M13。按你的目标挑：

**路线 A · 个人上手（你只想自己用顺一个 coding agent）。**
先走 M1 → M2 → M3 → M4 → M5。
目标是让你交给 agent 的活，质量稳定、不用反复返工。这五个模块给你的是：知道自己该交什么（M1），仓库整理好（M2），任务写清楚（M3），给 agent 搭好最小工作环境（M4），把上下文喂对（M5）。M6 之后的可以按需取——比如你常踩工具用错的坑，再补 M6；想引入验收纪律，再补 M7。这条路线**最短能见效**，一两个工件就能让你日常用 agent 顺很多。

**路线 B · 团队落地（你要让一个团队都可靠地用上委派）。**
在路线 A 的基础上，重点加 M6 → M7 → M8 → M9。
个人用顺只是起点；团队落地的瓶颈是**验收和安全**，不是速度。M7（eval）是这条路线的核心——它是"没有 eval 就没有自治"那条硬规矩的落地，团队敢不敢放权全看它。M8（sandbox 与安全）划出团队的安全底线，M9（三家对比）帮团队统一选型、别每人一套工具各说各话。这条路线的交付物要能被全队复用，不是个人手感。

**路线 C · 组织治理（你要给整个组织定底线和度量）。**
重点是 M4 → M7 → M8 → M11 → M12，M1 当背景，M13 当验收。
治理者不必精通每个 agent 的手感，但必须吃透 harness 的系统边界（M4）、验收标准（M7）、安全底线（M8）、新的项目度量与失败分类（M11）、安全发布与回滚（M12）。这条路线的产物是**制度性的**：权限矩阵、安全底线清单、失败分类学、release runbook——这些是组织层面要落成文档、要定期回看的东西。M13 capstone 是验收：能不能给一个真实 repo 把这套跑通。

三条路线在前几个模块上重叠（M1、M4 几乎人人要过），后面才分叉——跟游泳前几关共用、后面才分横渡和花样是一个道理。

### 怎么用这张表

每进一个模块，做三件事：读对应章节、做练习、产出那个交付物。**交付物没做出来，就不算通关，别往下走。** 一份做不出来的 eval rubric，比"我读过 eval 那章了"诚实得多——它会直接告诉你哪里没真懂。

## 真实案例

下面这条路径是**示意性的**，用来说明"按模块、看交付物"具体长什么样，不是某个真实团队的实录。

设想一个五人后端小组，过去半年一直在"试 agent"：有人开着 IDE 里的助手补全，有人偶尔让 cloud agent 开个 PR，但没人说得清"我们到底在多大程度上敢放权"。他们决定按路线 B 走一遍。

**M2 这一关，他们卡了三天。** 第一版 AGENTS.md 写得很顺，但拿去让 agent 跑，agent 还是改了不该动的 migration 目录、还是用了错的测试命令。交付物没过——按模块法的判据，这一关不算通关。回去补：把禁区写死（`migrations/` 不准动）、把命令列全（测试用哪条、类型检查用哪条）。第二版 agent 照着能跑通了，这一关才过。真正的产出不是那份文档，是"agent 照着真能跑通"这件事。

**M7 这一关，他们差点用 demo 糊弄过去。** 一开始他们拿几个"agent 干得漂亮"的成功案例当 eval。写到一半发现这不对——这正是前面反模式章点过的 eval theater：拿挑过的成功 demo 当考核，等于只考会做的题。他们改成翻历史：找出过去 agent 真正搞砸的几次（漏了边界、跳过验证、悄悄扩大范围），用这些失败样本当 eval 的种子。交付物——那个 rubric——这才有了真实的牙齿。

走完路线 B，他们手里多了四样能复用的东西：一份团队 AGENTS.md、一张 issue 模板、一个 eval rubric、一张权限矩阵。半年的"试 agent"第一次沉淀成了别人能照着用的工件。这就是交付物导向和"看热闹"的区别。

## 常见反模式

- **按天死排（"第一周学完 harness"）。** 时间表假设你学每样东西花一样的时间、且不被打断，真实工作里这两条几乎从不成立。按交付物走，做不出来就停在那一关。
- **跳级。** 没学 eval（M7）就放权让 agent 自动合并，没整理 repo（M2）就上多 agent（M10）。依赖关系是真实的，跳过去前面塌的账后面要连本带利还。
- **只读不做。** 把十三个模块当十三篇文章读一遍，一个工件都没产出。读完的错觉最骗人——挡它的唯一办法是交付物。
- **收藏代替学。** 把工具、框架、教程囤一堆，囤本身产生一种"在进步"的错觉，但囤不等于会。学习的判据是手里的工件，不是收藏夹的长度。
- **三条路线全都要。** 想一次把个人、团队、组织三条路线全走完。先认准你当下的角色挑一条，走完再说——同时铺三条，多半三条都半途而废。
- **把模块图当固定教条。** 这张表是参考顺序，不是法律。你的项目如果安全压力特别大，把 M8 提前到 M4 之后完全合理。依赖关系（M2 在 M10 前、M7 在 M12 前）要守，平行模块的次序随你。

## 检查清单

- 我是按**交付物**判断"通关"的，还是按"我读完了"判断的？
- 我当前在哪个模块？它的那个交付物，我**做出来了**吗，**经得起用**吗？
- 我有没有跳级——比如还没碰 eval（M7）就在放权，还没整理 repo（M2）就上多 agent？
- 我认准自己当下是**哪条路线**了吗（个人 / 团队 / 组织），还是三条同时铺？
- 我走过的每个模块，手里是不是真多了一个**能进真实项目**的工件？
- 卡住的那一关，我是回去补，还是硬往下走、把欠账留给后面？

## 自测

- 不看书，你能说出从"agent-ready repo"（M2）到"多 agent 编排"（M10）之间，**为什么不能直接跳过去**吗？中间缺的那几关是什么？
- 你手里现在有几个**能直接放进真实项目**的工件（AGENTS.md / issue 模板 / harness spec / eval rubric / 权限矩阵）？如果一个都没有，你大概率是在"看热闹"，不是在"学落地"。
- 如果只能先走一条路线，你会选哪条？说出你这个选择背后的理由——理由说不清，说明你还没认准自己当下的角色。
