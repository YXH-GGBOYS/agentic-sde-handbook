一个 agent 干活，至少在做两件不一样的事：一边是想——这个任务该怎么拆、下一步调哪个工具、读到的东西意味着什么；一边是动手——真的去读文件、跑命令、改代码。这两件事被绑在一起时，看着自然，扩起来却处处别扭：手上的活一卡（容器崩了、依赖装不上），脑子里攒的整盘思路也跟着没了；想多开几个并行去干，又怕它们互相踩。

把"脑"和"手"拆开，是这一章的主线。脑负责决策和规划，手负责在划清楚的范围里执行；中间用稳定的接口连着。拆开之后，手可以坏了重来、可以按需多开，脑可以独立地扩；难的不在拆，在于把手交回来的结果可靠地拼回去、并验收。这一层省不得。

## 生活类比

工程上的总包和分包，是这套关系的现成样板。

总包不亲自砌墙。他做的是决策：这栋楼怎么排期、水电和土建谁先谁后、这道工序的活算不算合格、尾款放不放。分包不一样——他在一个划得清清楚楚的范围里干具体的活：你这段就负责这层楼的电路，图纸给你、材料给你，干完了我来验。分包不直接跟甲方扯需求（那是总包的事），也不会闲着没事去碰隔壁的水暖工段。

这样安排有两个好处，都不是"为了好看"。一是人能加、活能并行：三个分包同时在三层楼上干，互不打架，因为各自的范围和接口是分开的。二是出了岔子不连累全局：某个分包的活返工，总包重新派一遍就行，整栋楼的排期不会因为他一个人的台面塌掉。

但这套之所以不失控，全靠中间那个常被忽略的环节——**验收**。总包不会因为分包说"我这段干完了"就直接放尾款。他得去量、去通电试、去对图纸。分包的自述只是一个待核对的声明，不是结论。把验收省掉、听见"完工了"就收货，总包这个角色就名存实亡了——剩下的只是一个转交了责任、却没法兜底的中介。脑和手解耦能不能成立，就卡在这道验收上。

## 工程定义

把一个 agent 拆成"脑"和"手"，Anthropic 在 Managed Agents 这套托管服务里给了一个具体的拆法 [A4]。Managed Agents 是面向长周期 agent 任务的托管服务，它的设计起点是一个观察：harness 里写死的，是一堆"关于 Claude 自己做不到什么"的假设，而**这些假设会随着模型变强而过时** [A4]。

A4 给了个很具体的例子。Claude Sonnet 4.5 有过一种叫"context anxiety"的毛病——快到上下文上限时会提前草草收尾；工程师为此在 harness 里加了 context reset 来对付它。可同一套 harness 换到 Claude Opus 4.5 上，这毛病自己没了，那些 reset 反倒成了纯粹的多余开销 [A4]。harness 是会过期的，这就是把它和稳定接口分开的理由。

它的办法借自操作系统：操作系统当年要给"还没被设想出来的程序"建系统，靠的是把硬件虚拟化成进程、文件这类耐用的抽象 [A4]。Managed Agents 照搬这个思路，把 agent 虚拟化成三个组件 [A4]：

- **Session**：一份只追加的日志，把发生过的一切都记下来。
- **Harness**：那个调用 Claude、并把工具调用路由出去的循环。
- **Sandbox**：跑代码和文件操作的执行环境。

三者分开，每一个的实现都能独立变，接口保持稳定 [A4]。落到"脑和手"上：**脑 = Claude 加它的 harness（决策、规划、调度），手 = sandbox 加工具（真正去执行）**，中间还有那份 session 日志当外部状态 [A4]。

这里要把两个概念分清楚，它们容易被当成一回事，其实是不同层面的东西：

- **Managed agent**，说的是上面这种把整个 agent 拆成 brain / hands / session 的架构思路——脑和手在物理上、责任上都解耦。
- **Subagent**，说的是脑这一侧的一种用法：主 agent 把一个子问题派给另一个 agent，后者在自己独立的上下文窗口里、带自己那套允许的工具去干，干完把结论回报上来 [A1]。Claude Code 里的 subagent 就是这么定义的：它"在自己的上下文里运行，有自己那套被允许的工具"，适合那些要读很多文件、或需要专门聚焦、又不想把主对话搞乱的任务 [A1]。

一句话区分：managed agent 是"脑和手的整体解耦"，subagent 是"脑把一块探索外包出去、只收压缩后的结论"。两者都在做同一件事——别让一个上下文、一个执行环境扛下所有，但作用的层面不同。

## 为什么重要：AI 时代的变化

脑和手绑在一起，在单机 demo 里没问题，一上规模就开始处处硌人。A4 把早期那种"所有组件塞进一个容器"的设计叫做养"宠物"（pet）——一台有名字、得人工伺候、丢不起的机器 [A4]。绑在一起的代价，A4 列得很具体 [A4]：

- 容器一挂，整个 session 跟着没了。
- 容器不响应，就得人工去调试。
- harness 默认所有资源都在同一个容器里，假设一旦不成立就处处出错。
- 客户要连进自己的 VPC 时，还得处理网络对等（peering）的麻烦。

解耦之后，这些问题的性质变了。A4 把 harness 挪到容器外面，让它把容器当成一个工具来调：`execute(name, input) → string`；容器要是挂了，harness 就当成一次工具调用出错来处理，Claude 可以换一个新开的环境重试，harness 自己的故障也能靠那份耐用的 session 日志恢复回来 [A4]。手坏了重来，脑不受影响——这正是"宠物变牛群"（cattle）那句老运维口号的意思：单台机器不再金贵，坏了换一台就是。

安全也跟着改善。解耦之前，Claude 生成的不可信代码，是和凭证（credentials）跑在同一个容器里的，注入风险很大 [A4]。解耦的设计保证 token 永远不会进到那个跑 Claude 生成代码的 sandbox 里 [A4]——这正是中心隐喻里"隔离操作间"那一层：不让外包碰你的钥匙，他要用的东西在初始化时按需配给，或经代理从安全保管处取。

性能的提升是能量化的。A4 给的数字：把容器和 harness 解耦后，首字延迟（TTFT）p50 降了大约 60%，p95 降了 90% 以上 [A4]。原因是推理可以立刻开始，不必干等着容器开起来——容器只在真正要用时才开 [A4]。

最后，长周期任务的上下文有了去处。长任务会超出 Claude 的上下文窗口，硬靠"选择性保留"很容易把关键信息丢掉 [A4]。A4 让那份 session 日志当外部上下文对象用：harness 通过 `getEvents()` 去取、去变换事件流里某一段（按位置切片），整个过程不占 Claude 的上下文窗口 [A4]。这跟上一章"长任务靠状态文件接续、不靠聊天历史"是同一条思路，只是落在了托管架构这一层。

## 实践方法

不是每个人都要去建 Managed Agents 那套托管基建。但"把脑和手解耦"这个判断，落到日常用 agent 上，有几个能直接照做的动作。

**把大块探索派给 subagent，只收压缩后的结论。** 这是最值回票价的一招。Anthropic 自己的说法是：context（上下文）是 agent 最根本的约束，subagent 因此是最有力的工具之一 [A1]。当 Claude 去研究一个代码库时，它要读大量文件，每读一份都在烧你的主上下文；subagent 在另一个独立的上下文窗口里探索，回报给你的是一份摘要 [A1]。给它一句明确的派活就行，比如"用 subagent 调查我们的认证系统怎么处理 token 刷新、有没有现成的 OAuth 工具可以复用"——它去翻、去读、回来只给结论，主对话保持干净 [A1]。

**让验收的人和干活的人不是同一个。** A1 把这点说得很直白：让一个跑在全新 subagent 上下文里的评审者，只看到 diff 和你给的判据，看不到产出这份改动的那串推理过程，于是它是在按自己的标准独立评判，而不是替自己刚写的代码背书 [A1]。换句话说，做这份活的 agent，不该是给这份活打分的那个 [A1]。内置的 `/code-review` 就是这么干的：在一个全新 subagent 里审当前 diff、把发现报回来 [A1]。

**把验收判据写进派活里，别只说"审一下"。** A1 给的评审 prompt 是这样的：点名要审的是哪份改动、对照哪份计划、什么才算一个"发现"——"对照 PLAN.md 审 rate limiter 的 diff，核对每条需求是否都实现了、列出的边界情况有没有测试、有没有动到任务范围之外的东西。报告 gap，不要报风格偏好" [A1]。判据给得越具体，评审越不会跑题。

**给 subagent 划清工具范围。** subagent 跑在自己那套被允许的工具里 [A1]——这就是中心隐喻里的"门禁"。一个只该读代码、不该改代码的调查型 subagent，就别给它写文件的工具。批量并行时这点尤其要紧：A1 在 fan-out 的例子里专门用 `--allowedTools` 把每个并行实例能做的事框住，理由是"无人值守地跑时，限制它能做什么这件事很重要" [A1]。

## 真实案例

**TTFT：p50 降约 60%，p95 降 90% 以上。** A4 这个数字值得单拎出来，因为它说明解耦不是纯架构洁癖，是有实打实回报的 [A4]。把容器和 harness 绑在一起时，每开一次活都得先等容器起来才能开始推理；拆开之后，harness 在容器外、推理立刻就能开始，容器只在真要执行时才按需开 [A4]。脑（无状态 harness）可以独立扩、手可以按需供给，这正是"并行而不失控"在基建层的样子 [A4]。

**"反驳我"式的验收，比"看起来对"靠谱。** A1 把一种常见的出错模式命名为"trust-then-verify gap"——Claude 产出一份看着挺像样、却没处理好边界情况的实现 [A1]。它给的对策是：永远配上可执行的验证（测试、脚本、截图），验证不了就别上线 [A1]。再往上一层是对抗式评审：让一个新模型在全新上下文里试着去反驳这个结果，这样打分的就不是干活的那个 [A1]。

但这里有个反直觉的坑，A1 专门提醒了：一个被要求"去找 gap"的评审者，几乎总会报出一些 gap，哪怕这活其实是好的——因为它就是被这么要求的 [A1]。追着每一条发现去改，会滑向过度工程：多余的抽象层、防御性代码、给根本不会发生的情况写测试 [A1]。所以派活时要补一句：只标会影响正确性、或违反明确需求的 gap，其余当成可选 [A1]。验收这道关，本身也得有判断，不是评审报什么就改什么。

**写作者 / 评审者分两个会话。** A1 给的 Writer/Reviewer 模式很朴素：会话 A 实现一个 rate limiter，会话 B 在全新上下文里审同一个文件、专挑边界情况、竞态、和现有中间件风格的不一致 [A1]。它点出了为什么要分开——全新的上下文能改善代码评审，因为这个 Claude 不会偏袒它刚写的代码 [A1]。测试也可以照这个来：一个 Claude 写测试，另一个写代码去过它 [A1]。这是 subagent 思路的最朴素形态，连托管基建都不用，开两个会话就能落地。

## 常见反模式

| 反模式 | 表面看 | 实际危险 |
|---|---|---|
| **无边界授权** | "subagent 嘛，给它全套工具最省事" | 一个本该只读的调查型 subagent 拿到了写文件、跑命令的权，越界改动无人拦。A1 的做法相反：subagent 跑在自己那套被允许的工具里，并行时用 `--allowedTools` 把范围框死 [A1] |
| **无 eval 的自治** | "它说干完了，那就是干完了" | 把活交出去、却没有任何可执行的验收和回归检查。A1 的硬判据：验证不了就别上线 [A1]。这正是"trust-then-verify gap" [A1] |
| **省掉整合 / 验收层** | "子 agent 的自述就是结论，直接拼回去" | 分包说"我这段合格了"不等于合格。把验收省掉、信子 agent 的自述，等于总包不去量墙就放尾款——脑和手解耦的全部价值，就卡在这道没被验的缝上 |
| **同一个 agent 既干又评** | "让它自己检查一遍不就行了" | 干活的 agent 看得见自己的推理，会偏袒自己刚写的东西。A1 要的是全新 subagent 只看 diff 和判据，独立评判 [A1] |
| **把脑和手绑死在一个容器** | "一台机器全装下，简单" | 容器一挂 session 全没、不响应得人工救、连 VPC 还得处理网络对等 [A4]。这就是养"宠物"的代价 [A4] |

## 检查清单

- 我有没有把"想"和"动手"分开看？哪些是脑的决策、哪些是手的执行，我说得清吗？
- 派给 subagent 的活，有没有**划清工具范围**（只读 / 可写 / 能不能跑命令），还是图省事给了全套 [A1]？
- 大块探索是**让 subagent 在独立上下文里压成摘要回传** [A1]，还是把原始过程全倒回主对话？
- "算完成"有没有**可执行的验收**（测试、脚本、截图），而不是听 agent 一句"看起来对了" [A1]？
- 验收的人和干活的人**是不是同一个**？有没有让一个全新 subagent 只看 diff 和判据独立评判 [A1]？
- 评审的派活里，有没有交代**什么才算一个 gap**，免得它为找而找、把你带进过度工程 [A1]？
- 手（执行环境）坏了，脑能不能**当成一次工具出错、换个新环境重试**，而不是整盘思路跟着没 [A4]？

## 自测

- 你上一次用 subagent，是真把一块探索"外包出去只收结论"，还是只是换了个名字、照样把所有原始材料倒回了主上下文？
- 如果现在让一个全新的 agent，只看你这次改动的 diff 和一句"什么算合格"，它能独立判断这活成没成吗？判断不了，说明你缺的不是更强的模型，是那道被省掉的验收。
- 你现在并行多个 agent，靠的是"各自范围和接口划清楚了"，还是"但愿它们别撞上"？答案基本就是你这套能不能再往上扩一倍的上限。
