给人用得顺手的 API，不一定给 agent 用得顺手。一个返回全部联系人、让你自己翻的接口，人类点几下就过去了，agent 却得把每条记录逐字读进有限的上下文里——同一个接口，对两类用户是两种体验。工具工程，就是把"给 agent 用的那一面"当成一个要设计的界面来做，而不是把后端函数原样裸露出去。

Anthropic 把这件事说得很直白：传统软件是在两个确定性系统之间立契约，`getWeather("NYC")` 每次都以同样方式取同一个城市的天气；而工具是确定性系统和**非确定性 agent** 之间的契约——同样问"今天要带伞吗"，agent 可能调天气工具，可能凭常识答，也可能先反问你在哪个城市，偶尔还会用错工具或干脆没看懂怎么用 [A8]。所以给 agent 写工具，不能照着"给别的开发者写函数"的老路走。

## 生活类比

想想厨房里那面挂刀的工具墙。

一面好的工具墙得趁手：常用的菜刀、剥皮刀挂在最顺手的位置，不常用的塞进抽屉，不会一伸手抓到一把根本用不上的。还得贴标签：哪把切肉、哪把切面包、哪把专门去鱼鳞，一眼能认，不用每把都拔出来比划。最危险的那几把要锁起来：剔骨刀、磨得飞快的主厨刀，要么放高处，要么单独收纳，不让没经验的人随手够到。

现在来了个手艺不错、但完全不懂你这间厨房的新帮厨。你不会把整面墙连同抽屉里几十把刀一股脑推到他面前说"自己看着拿"——他会愣在那儿，挨个拔出来端详，端详完还可能抄起那把最锋利的去切番茄。你会做的是：把这次要用的三五把刀拣出来、摆好、说清哪把干哪个活、哪把别碰。

给 agent 设计工具是同一回事。别让那个不懂规矩的新人对着一墙的刀现挑现拼，更别让他拿错刀。

## 工程定义

工具（tool）是给 agent 用的一层界面：它把某个能力——查日志、跑测试、改记录——包装成 agent 能理解、能正确调用、调用完能看懂结果的形式。它的设计对象是 agent，不是另一个后端服务，也不是人类开发者 [A8]。

一个常见的错误，是把工具当成 API 的薄包装：后端有什么 endpoint，就裸露什么工具。Anthropic 专门点了这个反模式——agent 的"可供性"（affordance，它能感知到自己能做哪些动作）和传统软件不同，照搬 API 往往不好用 [A8]。判断一个工具好不好，可以拆成六个要素：

- **名称**：动词加对象、范围清楚，让 agent 一看就知道这把刀干什么活，比如 `run_unit_tests`、`search_logs`。当工具一多，还要做命名空间（namespacing）——按服务或资源加前缀，`asana_search`、`jira_search`、`asana_projects_search`，让 agent 在几十上百个工具里不至于挑错 [A8]。
- **参数**：少而明确，能用枚举就用枚举，能给默认值就给默认值。命名要无歧义——用 `user_id` 而不是 `user` [A8]。别让 agent 去拼一长串复杂 shell 或自由文本，那等于把出错的空间还给它。
- **输出**：结构化、短、只回高信号的东西。Anthropic 的建议是别返回 `uuid`、`256px_image_url`、`mime_type` 这类底层标识符，多返回 `name`、`file_type` 这类能直接指导下一步的字段；把无意义的 UUID 解析成有语义的名字，能显著降低 Claude 的幻觉、提高检索精度 [A8]。
- **错误**：失败时别甩一个不透明的错误码或 traceback。Anthropic 建议把错误响应也当 prompt 来写——说清具体哪里错了、该怎么改 [A8]。一个好的错误信息得回答三件事：为什么失败、能不能重试、下一步建议干什么。
- **权限**：边界由工具自己检查，不靠 prompt 里写一句"请不要删生产库"。这条和上一章的 harness 是一条线——真正的护栏在工具层和 sandbox，不在系统提示里。
- **成本**：把这把刀用一次要花多少 context 显式暴露出来。工具定义本身、每次调用的返回，都在吃 agent 有限的上下文。Claude Code 默认把单次工具响应截到 25,000 token，并配合分页、范围选择、过滤、截断和合理的默认值 [A8]——这不是抠门，是因为上下文是 agent 最稀缺的资源。

这六个要素里，前四个决定 agent 用不用得对，后两个决定它用不用得起。

## 为什么重要：AI 时代的变化

工具的成本，大头不在"调用一次花多少钱"，而在"光是把工具摆在那儿，就先吃掉多少上下文"。

这是给人写 API 时根本不存在的问题。人类不会因为你的 SDK 有 200 个方法就变慢，文档放在那儿，他用到哪个查哪个。agent 不一样：多数 MCP 客户端会把所有工具定义**一次性**全部加载进上下文。Anthropic 给的数字是，这一下可能吃掉 50,000 token 以上 [A18]。换个更具体的例子：一个接了 5 个 MCP server 的系统——GitHub 35 个工具（约 26K token）、Slack 11 个（约 21K）、Sentry 5 个（约 3K）、Grafana 5 个（约 3K）、Splunk 2 个（约 2K）——总共 58 个工具，还没开始干活，就先烧掉约 55K token [A18]。这些 token 是 agent 在读你第一句话之前就已经花掉的。

针对这个问题，Anthropic 做了一个叫 Tool Search Tool 的东西：不再把全部定义塞进上下文，而是让 agent 按需搜索、按需加载。效果是把"干活之前占用的上下文"从约 77K token 降到约 8.7K token [A18]。

更值得资深读者留意的，是同一个功能在不同模型代际上收益不一样。Anthropic 的内部测试里，Tool Search Tool 把工具选择的准确率在 Opus 4 上从 49% 提到 74%，在 Opus 4.5 上从 79.5% 提到 88.1% [A18]。两件事同时成立：一是工具检索这件事确实有用；二是模型越强，它本来就越能在一堆工具里挑对，外挂机制能榨出的增量就越小。这意味着你不能把别人在某个模型上测出的工具方案直接搬来当定论——换个模型代际，账可能要重算。

## 实践方法

**先做减法，再谈检索。** 工具不是越多越好。Anthropic 的经验是：针对几个高价值工作流，精心做少数几个工具，再往外扩 [A8]。能合并的就合并——与其给 `list_users`、`list_events`、`create_event` 三个工具，不如做一个 `schedule_event`，它在底下自己找空档、自己建日程；与其 `read_logs`，不如 `search_logs`，只回相关的几行加上下文 [A8]。一个工具把多步操作收进一次调用，省的不只是 agent 的调用次数，还有那些中间结果本来要占的上下文。

**当工具确实多到几十上百个，再上检索式加载。** Anthropic 给的触发线很具体：工具定义占用超过 10K token、可用工具到 10 个以上、或者已经观察到工具选择不准，就该考虑按需加载而不是全量加载 [A18]。

**数据量大时，让 agent 写代码在执行环境里过滤，而不是把原始结果全倒进上下文。** 这是 MCP 的 code execution 模式要解决的第二个问题。直接工具调用有个隐形开销：每个中间结果都得穿过模型。Anthropic 举的例子是，一段两小时会议的转录稿，如果先从 Google Drive 取出来、再写进 Salesforce，整段文字会流经模型两次，可能多花约 50,000 token [A17]。换成 code execution——agent 写一段代码，在执行环境里取、过滤、再写，模型只看到它显式 log 出来的部分。同样地，一张一万行的表，agent 可以在代码里 filter 完只 log 前 5 行，模型就只看到 5 行而不是一万行 [A17]。

这一招的代价要算清楚：跑 agent 写的代码，需要一个带 sandbox、资源限制和监控的安全执行环境，这本身就是一摊运维和安全开销，是直接工具调用没有的 [A17]。省 context 和加基础设施之间要权衡，不是白捡的便宜。

## 真实案例

**一份 `run_targeted_tests` 的工具契约。** 把上面的六要素落到一个真实工具上，最该想清楚的是错误语义。下面这份契约的关键，在于它区分了"测试失败"和"基础设施坏了"——这两件事 agent 的下一步动作完全不同：

```json
{
  "name": "run_targeted_tests",
  "description": "在隔离工作树里跑指定的测试目标。只回失败用例的名称、文件:行号和断言摘要，不回完整日志。",
  "input_schema": {
    "type": "object",
    "properties": {
      "targets": {
        "type": "array",
        "items": {"type": "string"},
        "description": "测试节点 ID，如 'tests/test_orders.py::test_refund'。留空则跑受改动影响的测试。"
      },
      "verbosity": {
        "type": "string",
        "enum": ["summary", "full"],
        "default": "summary"
      }
    }
  }
}
```

返回结构把成功、断言失败、基础设施错误分成三种 `status`：

```json
{
  "status": "fail",          // pass | fail | infra_error
  "failed": [
    {
      "test": "tests/test_orders.py::test_refund",
      "location": "tests/test_orders.py:88",
      "assertion": "expected refund_status 'done', got 'pending'",
      "retryable": false
    }
  ],
  "next_step": "断言失败，是代码逻辑问题，改实现后重跑这一条。"
}
```

为什么 `fail` 和 `infra_error` 必须分开：`fail` 是 agent 该接着干的活——改代码、重跑；`infra_error`（依赖装不上、容器起不来、网络不通）则不是 agent 改几行代码能解决的，它该停下来报告，而不是对着一个跟自己改动无关的错误反复试。`retryable` 和 `next_step` 这两个字段，就是 A8 说的"把错误响应当 prompt 写"——告诉 agent 失败原因、能否重试、下一步干什么 [A8]。

**GitHub 把 agent 当工具暴露给另一个 agent。** GitHub 的 custom agents 是一个有意思的工件：你可以定义一组 agent persona，每个带自己的 prompt、工具选择和 MCP server，配置放在仓库的 `.github/agents` 里。关键在于——这些 custom agent 是作为工具暴露给 Copilot 的，当需要时，模型会调用某个相关的 custom agent，由它起一个新的 agentic loop [G4]。也就是说，"工具"这个界面不一定包的是一个 API，也可以包的是另一个完整的 agent。从调用方看，它就是工具栏上又多了一把刀；刀背后是一整个会自己读、自己改、自己跑的循环。

## 常见反模式

- **all-tools-in-context（一次把全部工具和文档都丢给 agent）**：把几十上百个工具定义连同文档一股脑加载进上下文。这正是前面那个 58 工具烧掉约 55K token 的场景 [A18]——agent 还没读到你的需求，上下文已经塞满了它这次根本用不到的刀。
- **把 API 原样裸露成工具**：后端有什么 endpoint 就开什么工具，一个不漏。这是 A8 点名的错误——agent 的可供性和传统软件不同，`list_contacts` 返回全部联系人让 agent 逐条读，纯属浪费它的上下文 [A8]。该做的是 `search_contacts`。
- **靠 prompt 做权限**：在系统提示里写一句"不要删生产库"就当防住了。边界得工具自己检查，这条在 harness 那章已经说透。
- **错误甩 traceback**：工具失败时回一整段原始堆栈或一个不透明错误码，agent 看不懂，只能反复试。改成说清原因、能否重试、下一步建议 [A8]。
- **不区分 fail 与 infra_error**：测试失败和环境坏掉用同一个出口，agent 分不清该改代码还是该停下报告，于是对着一个跟自己无关的基础设施错误反复重跑。

## 检查清单

- 每个工具的**名字**，agent 一看就知道它干什么活、范围多大吗（动词+对象+范围）？
- **参数**是不是少而明确、该枚举的枚举、该有默认值的有默认值，没逼 agent 去拼复杂 shell？
- **输出**是结构化、短、带路径/行号/状态的，还是把底层标识符和完整日志一股脑回过去？
- **错误**有没有说清失败原因、是否可重试、下一步建议，而不是甩 traceback？
- **权限**边界是工具自己检查的，还是只写在 prompt 里？
- 我数过**所有工具定义一共吃掉多少 context** 吗？超过 10K token / 10 个工具了吗？需不需要上按需加载？
- 数据量大的工具，有没有让 agent 在执行环境里先过滤，而不是把原始结果全倒进上下文？

## 自测

- 把你给 agent 的工具栏打印出来：有几把刀是它这类任务**根本用不到**、却一直占着上下文的？
- 你最近一次工具调用失败，agent 拿到的是"断言失败、改这行重跑"这种能指导下一步的信息，还是一段它看不懂、只能反复试的 traceback？
- 如果把同一套工具方案从一个模型换到下一代模型，你凭什么相信收益还一样？你有没有在自己的任务上重新量过？
