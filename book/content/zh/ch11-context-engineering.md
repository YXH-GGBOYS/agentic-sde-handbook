派活给 agent 之前，你要决定它眼前摆着哪些材料：哪段代码、哪份文档、哪条规矩、上一轮的什么结论。这份"摆在它面前的材料"就是 context（上下文）。很多人下意识把它当垃圾桶——能塞的都塞进去，反正窗口大。这恰恰是它干不好活的主要原因之一。

context 是 agent 的注意力预算，不是仓库的副本。窗口塞得越满，真正关键的那几行就越容易被淹没。Context 工程要解决的，就是在派活前把该备的背景备齐、摆顺，把不该让它翻的挡在外面——让同一个模型，在一个更干净的范围里，显得更聪明。

## 生活类比

后厨有个词叫 mise en place，字面是"各就各位"。一个像样的厨师，开火之前会先把这道菜要用的东西全部备好：葱切段、蒜拍碎、酱汁量好、火候顺序在脑子里过一遍，所有料按下锅的次序摆在灶台伸手就够的地方。

为什么非得提前？因为火一上来，时间是以秒计的。这时候你才想起葱没切、还得拉开冰箱翻半天找那瓶生抽——等你找回来，锅里的蒜已经焦了，整道菜毁了。备菜的纪律不是讲究，是因为"临场再找"这件事，在高温下根本来不及。

但 mise en place 还有另一半，常被忽略：不该上灶台的东西，别堆上去。你不会把整个冰箱、整个调料架、还有隔壁三道菜的料一起端到手边——灶台就那么大，堆满了反而找不到正在用的那一样。备菜是"备齐这道菜要用的"，不是"把厨房搬过来"。

给 agent 派活是同一回事。该交代的背景提前备好、按用得到的顺序摆好；不该让它翻的，别让它翻。把整个仓库、整份 wiki 倒给它，就像把整个冰箱端上灶台——东西是多了，可它要的那一样反而被埋住了。

## 工程定义

Context 工程，是为一次 agent 任务挑选、组织、维护那组 token 的工程。Anthropic 给的定义很直接：context 指的是"从大语言模型采样时纳入的那组 token"，而 context 工程是"在推理过程中策划并维护那组最优 token（信息）的一套策略，包括所有可能落进窗口、却不在 prompt 里的信息" [A13]。

它和 prompt 工程是两件相关但不同的事。prompt 工程关心的是"那一句指令怎么写"；context 工程关心的是更大的问题——系统指令、工具定义、检索来的外部数据、历史消息，这一整个上下文状态，怎么配置才最可能让模型干出你要的活 [A13]。一次任务里真正喂进窗口的，远不止你打的那段字：还有 agent 读进来的文件、跑命令的回显、上一轮的对话、工具的返回。这些全是 context，全在花你的注意力预算。

要点是把 context 当成一种"有限且珍贵的资源"来管 [A13]。窗口标称能装二十万 token，不等于你该装二十万。一次任务最该追求的，是"能最大化拿到想要结果的、那组最小的高信号 token" [A13]——不是塞得最满，是留得最准。

下面把 agent 实际会用到的上下文拆成五类，每一类都有自己常见的坑。这五类放在一起，差不多就是 agent 一次任务的全部"材料"。

- **repo 指令**（AGENTS.md / CLAUDE.md 这类）。坑是写成给人看的散文。"我们高度重视代码质量"这种话里没有一条 agent 能执行的规则，它读完不知道该跑哪个命令、哪些目录别碰。
- **任务上下文**（你这次到底要它干什么）。坑是只写"修个 bug"。它不知道哪个 bug、怎么复现、改完算什么样、哪些文件不许动——缺的这些，它只能猜。
- **状态 artifact**（一个长任务跨多轮怎么续上）。坑是指望它靠聊天历史恢复进度。历史一旦被压缩或截断，"上一轮做到哪了"就丢了。
- **检索结果**（从代码库、文档里捞回来的相关材料）。坑是把全仓库硬塞进去，指望模型自己大海捞针。
- **历史反馈**（过往的评论、review 意见）。坑是把所有评论无差别拼进去——三个月前一条早已过时的吐槽，和这次真正相关的一条，权重一样，照样稀释信号。

## 为什么重要：AI 时代的变化

最该破除的直觉是"窗口大就该塞满"。

Anthropic 把一个关键现象起名叫 **context rot**（上下文腐烂）：随着上下文窗口里 token 数变多，模型准确召回信息的能力反而下降 [A13]。这里有个容易误读的地方——它不是"到某个硬上限突然断崖"，而是一条逐渐衰减的曲线：模型仍然能用，只是在更长的上下文里，检索和推理的精度持续走低 [A13]。这是一条性能梯度，不是一道悬崖 [A13]。

机制上，这跟 transformer 的注意力有关。每个 token 都要和其它每个 token 发生关系，n 个 token 就有 n² 对两两关系；上下文越长，这张关系网越难维持 [A13]。再加上训练数据多偏短文本，模型见过的超长序列本来就少 [A13]。两件事叠起来，结果就是：你多塞进去的那十万 token，不光没帮上忙，还在主动拖低它对真正关键那几行的注意力。

所以这件事的方向和直觉是反的。窗口变大，不是让你可以更省事地"全塞进去"，而是让"塞什么、不塞什么"这个决定变得更要紧。同一个模型，喂给它一个干净、对准任务的上下文，和喂给它整个仓库，干出来的活可以差很远——差的那部分，不在模型聪不聪明，在你有没有替它把材料备对。

## 实践方法

核心动作只有四个，可以按顺序套用：**检索、摘要、压缩、分阶段**。目标始终是同一个——让窗口里只留下这次任务用得上的高信号 token。

按需检索，不要预加载。与其一上来把所有相关数据灌进窗口，不如让 agent 手里只攥着轻量的"指针"——文件路径、URL、查询语句，到运行时再用工具按需把数据捞回来 [A13]。这其实就是人的做法：我们不会把整本资料背下来，而是知道去哪儿查 [A13]。Claude Code 就是这么干的——它写有针对性的查询，用 `head`、`tail`、`grep` 这类命令去翻大文件、大数据库，而不是把整个对象读进上下文 [A13]。

让 sub-agent 替你把探索压成摘要。这是最值得记住的一招。让一个 sub-agent 在自己干净的上下文里放手去探索一个子问题——读很多文件、跑很多命令——但回传给主协调 agent 的，不是它翻过的所有原始材料，而是一份压缩好的摘要，量级通常在 1,000–2,000 token [A13]。主 agent 的窗口因此始终只装"结论"，不装"它是怎么翻出这个结论的"。一大堆探索过程被挡在主上下文之外，注意力预算留给真正需要协调的部分。

长任务靠状态文件接续，不靠聊天历史。这一点 Anthropic 在长任务 agent 上专门验证过：哪怕是前沿模型，在一个 loop 里跨多个上下文窗口去建一个生产级 web app，光靠压缩历史是不够的——压缩并不总能把清楚的交接说明传给下一轮 [A2]。它给出的办法是把进度落到上下文之外的文件里：一个 `init.sh` 负责把环境起起来，一个 `claude-progress.txt` 记着历轮都干了什么，再配上 git 提交历史 [A2]。每一轮新 agent 开工，先读进度文件和 git log 把状态接上，而不是去翻已经被压没了的聊天记录 [A2]。

## 真实案例

**150,000 → 2,000：把工具和数据挡在窗口外。** Anthropic 给过一个很具体的例子：把一份 Google Drive 文档（一次会议的纪要）转写进 Salesforce 的一条 lead 记录 [A17]。

朴素的做法是直接调工具。问题出在两处。其一，大多数 MCP 客户端开局就把所有工具定义一股脑灌进上下文；连着几十个 server、几百上千个工具时，模型还没读到你的请求，就得先吞下成十万 token 的工具说明 [A17]。其二，中间结果要在模型里过两遍：`getDocument` 把整份纪要读进上下文，写进 Salesforce 时模型又得把整份纪要原样写一遍——一场两小时的会议，光这一进一出就可能多烧掉约 50,000 token [A17]。

换成"用代码调工具"，两个问题一起解决。把工具呈现成文件系统上的代码 API，agent 先列目录发现有哪些 server、只读它这次真正要用的那两个工具文件，按需加载 [A17]。数据则在执行环境里就地处理：纪要从 Google Drive 取出后直接写进 Salesforce，根本不经过模型的上下文 [A17]。同一个任务，token 用量从 150,000 降到 2,000，省了 98.7% [A17]。

**10,000 行只看 5 行。** 同一篇里还有个更直白的例子。要处理一张一万行的表格，朴素做法是把一万行全返回进上下文，让模型自己筛 [A17]。用代码执行，筛选在执行环境里完成，只把头 5 行打印出来给模型过目——模型看到的是 5 行，不是 1 万行 [A17]。聚合、跨源 join、抽特定字段，都能照这个路子在数据进窗口之前先压一遍 [A17]。这正是"备菜"那一半被忽略的纪律：不该上灶台的，先在台下处理掉。

**200 多个功能，靠一个状态文件撑住。** 回到长任务那篇。让 agent 去克隆一个 claude.ai，初始化 agent 会先写一份详尽的功能清单，把用户那句笼统的要求展开成具体条目——在这个例子里超过 200 条，每一条都先标成"未通过" [A2]。这份清单存成 JSON 文件（实测模型改 JSON 比改 Markdown 更克制，不太会乱删），活在上下文之外 [A2]。后面每一轮 coding agent 只挑一条没做的功能来推进，做完把对应条目的 `passes` 翻成通过，再提交 git、更新进度文件 [A2]。整个项目的"做到哪了"，从头到尾扛在文件里，而不是扛在那条早就装不下两百个功能的聊天历史里。

## 常见反模式

- **把整个仓库塞进窗口。** "反正窗口大，全给它最保险"——恰恰相反。多塞的那部分在主动稀释信号，把 context rot 喂大 [A13]。该做的是按需检索，让 agent 手里只攥指针 [A13]。
- **repo 指令写成散文。** 一段"我们追求卓越工程文化"的漂亮话，agent 提取不出任何可执行规则。换成动作、命令、禁区。
- **任务只写一句"修个 bug"。** 缺了复现步骤、完成判据、禁改区，agent 只能猜，猜错的成本由你承担。
- **长任务靠聊天历史接续。** 历史会被压缩、会被截断，"上一轮做到哪"随时可能丢。把状态落到文件里 [A2]。
- **历史反馈无差别全拼进去。** 三个月前一条过时吐槽和这次相关的一条同等权重，照样占预算、稀释信号。该筛就筛。

## 检查清单

- 这次任务，我喂进窗口的是"用得上的那部分"，还是"我能找到的全部"？
- repo 指令里每一条，agent 能据此**采取一个具体动作**吗，还是只能"体会一种精神"？
- 任务交代里，有没有**复现方式、完成判据、禁改区**？还是只有一句"修个 bug"？
- 大块探索有没有交给 **sub-agent 压成摘要**回传（约 1,000–2,000 token），而不是把原始过程全倒回主上下文？
- 大块数据，是在**进窗口之前**先筛/聚合，还是整块灌进去让模型自己捞？
- 长任务的状态，落在**文件**里（进度文件 + git），还是赌它能从聊天历史里恢复？

## 自测

- 你上一次给 agent 派活，喂进去的上下文，有多大比例是这次任务真正用得到的？剩下那部分，是在帮它，还是在稀释它？
- 如果现在把这次任务的聊天历史清空，只留下你落在文件里的东西，下一个 agent（或下一个人）接得住吗？接不住，说明你的状态还活在会消失的地方。
