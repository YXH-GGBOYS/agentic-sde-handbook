上一章把 harness 讲成"给新外包搭的整套工作环境"。这一章把这套环境拆成九层，每层负责挡一类风险——任务进不进得来、谁拆活、它眼前摆什么料、能动哪些工具、哪些动作要批、在哪间屋干、怎么算干完、谁签收、出了事怎么回放和改进。

九层不是某个产品的实现，也不绑某家供应商。它是一份检查表：你现在用 agent 的方式，把这九层对一遍，缺哪层就知道风险从哪儿漏出去。每层都给两档做法——一个人周末就能搭起来的"最低可用"，和一个团队长期演进的"成熟做法"。九层全上是少数高风险场景才需要的；大多数任务，前几层做扎实就够了。

## 生活类比

把一个零件从图纸到出厂，想象成一条完整的车间流水线。

活从**收件台**进来：不是什么单子都接，先看图纸全不全、尺寸标没标、是不是本厂能做的活——不合格的当场退回。接下了，**工艺组**先拆工序：这活分几步、每步的验收标准是什么、动哪几台设备，排成工单。然后是**备料**：把这道工序要用的图纸、料、上一班的交接记录摆到工位伸手够得着的地方，别让师傅满车间找。**工具墙**上挂着这道工序该用的几样工具，贴好标签，别让他随手抄起隔壁工序的家伙。车间有**门禁**：普通工序自己干，要动那台贵设备、要往成品库里写数据，得叫班长来按一下。真正动手是在**隔离工段**里——一间和成品线物理隔开的屋子，砸坏了也波及不到客户的货。干完进**质检**：卡尺量、跑测试台，过不了这道闸不放行。全程头顶有**监控录像**，谁几点动了哪台设备、哪步报警，都录着。最后**车间主任签收**——而且按厂规，下这单的人不能自己给自己签收。出了批量问题，调录像、翻留样，倒查到是哪步、哪台设备、哪个参数错了，把这条教训写进工艺卡，下次收件台就拦得住。

收件台、工艺组、备料、工具墙、门禁、隔离工段、质检、签收、监控加复盘——这九个环节，少哪个都漏风险。少了门禁，新人能直接往成品库写；少了质检，废品流到客户手上；少了监控，出了事谁也说不清是哪步坏的。harness 的九层，就是这条流水线搬到 coding agent 身上。

## 工程定义

**Harness 的 9 层参考架构**，是把一次 agent 任务从"接活"到"交活"拆成九个有先后、各管一类风险的系统边界。它们大致按一次 run 的时间顺序排开：

| 层 | 名字 | 管什么 | 一句话职责 |
|---|---|---|---|
| ① | Intake 任务入口 | 什么任务能进 | 分诊、评风险、拦不合格 |
| ② | Planning 计划 | 谁拆任务、拆成什么 | 出计划、定验收标准和文件范围 |
| ③ | Context 上下文 | 它眼前摆什么料 | 备齐、摆顺、挡掉无关 |
| ④ | Tools 工具 | 它能动哪些工具 | 给顺手的工具，标清楚 |
| ⑤ | Permissions 权限 | 哪些动作要批 | 按风险分级、设审批点 |
| ⑥ | Sandbox 隔离执行 | 在哪间屋干 | 隔离环境，砸坏波及不到生产 |
| ⑦ | Evals 验收 | 怎么算干完 | 可执行的完成判据 |
| ⑧ | Review 接受结果 | 谁签收 | 人审 + 审查 agent + 职责分离 |
| ⑨ | Trace + Learning | 怎么追踪和改进 | 留痕、复盘、把教训喂回前几层 |

每层都有"最低可用 vs 成熟做法"两档。最低可用是周末能搭起来、覆盖大多数低风险任务的版本；成熟做法是在高吞吐、高风险场景里需要的版本。

这九层不是凭空发明的，它是把几家厂商各自讲过的 harness 拆法对齐到一张表上。OpenAI 把 harness 定义成 agent 周围那层"验证、反馈、恢复、工具、上下文和人类判断"的结构 [O1]——这正好覆盖了 ③④⑦⑨ 和人类审批点。Anthropic 讲长任务 harness 时给的零件——initializer、状态文件、artifact、evaluator、checkpoint、人工干预和恢复 [A2][A3]——落在 ②③⑦⑨ 上。说法不同，拆的是同一台机器。

这九层是边界，不是组织结构。你不必为每层配一个人或一个服务。小项目里，③④⑤⑥ 可能就是"在一个 git worktree 里跑、写文件前确认、用项目自带的测试命令"几行约定。层是用来检查"这类风险有没有人管"的，不是用来堆架构的。

## 为什么重要：AI 时代的变化

把约束摊成九层、放在模型外面，原因有三个，都跟"模型变强"这件事的方向相反。

第一，出事的地方大多不在模型，在层与层之间的边界。Anthropic 量化过：在 Terminal-Bench 2.0 上，光是基础设施配置的差异就能让分数差出 6 个百分点，足以盖过排行榜上顶尖模型之间的差距 [A11]。在最严的资源限制下，约 5.8% 的任务是被基础设施错误（而非模型）搞挂的；把资源放开到不设上限，这个比例降到 0.5% [A11]。这些失败没一样是"模型不够聪明"——是依赖装不上、内存被硬杀、子进程起不来。它们出在 ⑥ sandbox 配置和 ⑦ eval 环境里，得在那两层被隔开、被归因，否则你连"是模型错了还是环境错了"都分不清。A11 那句话说得直白：领先几分，可能是真本事，也可能只是配了台更大的虚拟机。

第二，跑得越快，边界越值钱。OpenAI 那个团队，三个工程师驱动 Codex，合并了约 1,500 个 PR，人均一天 3.5 个；团队涨到七人后，人均吞吐不降反升 [O1]。这个速度下，一个模糊的目标、一个缺失的验收，会比手写时代快得多地被复制进更多文件。他们的应对不是"让模型更小心"——是把架构约束、命名规范、文件大小上限做成 lint，一旦编码，到处一起生效 [O1]。约束做成机器能查的闸，才跟得上吞吐。

第三，层是会过期的，得能拆。Anthropic 那篇拆 harness 的文章里有句话值得贴在墙上：harness 里每一个组件，都编码了一条"模型自己做不到"的假设；这些假设值得反复压测，因为它们可能本来就错，也会随模型变强而过期 [A3]。他们就真在 Opus 4.6 出来后，把原来为了治"context anxiety"而加的 context reset 整个删掉了，因为新模型自己就不焦虑了 [A3]。九层不是越多越好——是该有的边界有人管，过期的脚手架敢拆掉。

## 实践方法

按九层顺序过一遍，每层只问两个问题：最低可用怎么做，成熟了往哪儿走。

**① Intake 任务入口。** 最低可用：一个统一的 issue 模板，逼提单人写清"要干什么、怎么算完成、哪些文件别碰"。成熟做法：模板之上加自动分诊——按任务类型和风险打分，高风险的（碰 migration、碰 auth、碰 secret）强制走人审，明显不合格的（没复现步骤、范围无界）自动退回。一条硬边界可以直接抄 GitHub 的：只有对仓库有写权限的人能触发 agent，没写权限的人留的评论根本不会进 agent 的上下文 [G8]。

**② Planning 计划。** 最低可用：让 agent 先出一份计划，人看过再让它动手，别让它读完 issue 就直接改文件。成熟做法：计划成为一等工件，存进仓库、带进度和决策日志 [O1]；每一条都能追到验收标准和文件范围。Anthropic 的三 agent 架构里，planner 专门把一句话的 prompt 扩成完整产品 spec，但只定"要交付什么"、不抠"怎么实现"——因为实现细节定早了、定错了，错误会顺着往下游一路放大 [A3]。

**③ Context 上下文。** 最低可用：一份 AGENTS.md 加这次相关的几个文件。成熟做法：检索/摘要/状态文件/上下文预算（上一章详谈过）。这里只补一个九层视角的点——AGENTS.md 别写成百科全书，写成目录。OpenAI 把它当"目录页"用，约 100 行，指向 `docs/` 里更深的真相来源，让 agent 从一个稳定的小入口开始、被教着往下找，而不是一上来被淹 [O1]。

**④ Tools 工具。** 最低可用：文件读写加几条测试命令。成熟做法：把 CI、日志、ticket、feature flag、dry-run 部署都做成 agent 能直接调的工具，再用 MCP 接外部能力 [G1]。一个反直觉但管用的招：lint 报错信息里直接写上"该怎么修"，把补救指令注进 agent 的上下文 [O1]。工具顺不顺手，直接决定 agent 能不能自己把活干完、不回头找你。

**⑤ Permissions 权限。** 最低可用：写文件前确认。成熟做法：按风险分级。Claude Code 的 auto mode 给了一个能照抄的三档模型——第一档是安全只读操作和用户白名单，直接放行；第二档是项目内的文件编辑，因为有版本控制可回溯，免审；第三档是 shell 命令、外部服务、项目目录外的文件操作，全部送去过分类器 [A5]。背后那条原则很硬：agent 自己擅自决定的一切，在用户点头之前都算未授权 [A5]。分级的轴不止一条——风险、路径、命令、网络、secret，都可以是分档的依据。

**⑥ Sandbox 隔离执行。** 最低可用：一个独立的 git 工作树，别让多个实例在同一个目录里互相覆盖。成熟做法：容器 / VM / 隔离网络 / 不挂生产 secret / 用完即销毁。Anthropic 把它总结成一条排序原则：先在环境层做隔离，再到模型层修行为 [A6]——监督它"能做什么"，比监督它"做了什么"更靠得住。具体到产品，他们给三种场景配了三种强度：claude.ai 用一次性的 gVisor 容器，Claude Code 在开发者机器上用 OS 级沙箱（macOS 的 Seatbelt、Linux 的 bubblewrap），默认姿态是"读放行、写只限工作区、网络默认拒绝"，给非技术用户的 Cowork 则上整机隔离的封闭 VM [A6]。还有一条选型铁律：别太信你自己手搓的隔离层——"最弱的一层，是你自己造的那层" [A6]。

**⑦ Evals 验收。** 最低可用：跑测试和 lint。成熟做法：任务级 eval + 回归集 + 安全检查 + AI judge 配人工 rubric。Anthropic 把验收手段分成三类，各有取舍：代码型 grader（快、客观、可复现），模型型 grader（灵活、能打分、能读自然语言断言），人工 grader（金标准但贵，用来校准模型型）[A10]。建议从 20–50 个真实任务起步，把人工测过的 case 和用户报的故障转成基准 [A10]。这里有个 agent 特有的坑：eval 环境本身得隔离干净，否则 ⑥ 那 6 个百分点的基础设施噪声会混进分数，让你把环境抖动当成模型回归 [A11]。

**⑧ Review 接受结果。** 最低可用：人审 PR。成熟做法：agent 自审 + 专门的审查 agent + code owner + 风险门禁。GitHub 给云端 agent 的硬边界是现成的范本：agent 不能直接 `git push`、只能推自己那条分支；草稿 PR 必须人审，agent 不能把自己的 PR 标成"待评审"、也不能批准或合并；发起任务的人不能批准这个 agent 的 PR（职责分离）[G8]。OpenAI 走得更远——把几乎所有 review 都推成 agent-to-agent，让 Codex 本地自审、再请求别的 agent 审，循环到所有审查 agent 都满意，只有需要人类判断时才升级给人 [O1]。

**⑨ Trace + Learning 追踪与改进。** 最低可用：保存失败 case。成熟做法：让这一圈转起来——review comment 入规则、失败样本入 eval、模板持续迭代。OpenAI 把人的品味持续喂回系统：review 评论、重构 PR、线上 bug，都被收成文档更新或直接编码进工具；文档拦不住的，就把规则提升成代码 [O1]。trace 这层得记够字段才能复盘——GitHub 的 agent 会签名提交、标注共同作者、留 session log 和审计事件，commit 里直接带一条指向 session log 的链接 [G8]。

把这九层各自的答案写下来，就是一份 harness spec。下面给一份骨架。

## 真实案例

**工件：一份 harness spec 骨架。** 把九层的答案落成一份能 review、能版本化的文件，长这样：

```markdown
# Harness Spec: <任务族名，如 "bugfix-web">

## ① Intake 任务入口
accept:    [bug 复现明确, 范围 ≤ N 文件, 有验收标准]
reject:    [无复现步骤, 碰 migration/auth/secret 却没标风险, 范围无界]
risk_tiers: low=自动 | medium=人审计划 | high=人审计划+人审 PR
trigger:   仅对仓库有写权限者可触发        # 抄 G8

## ② Planning 计划
agent_roles:
  - planner:   一句话 prompt → 产品 spec, 只定交付物不抠实现   # A3
  - generator: 按 spec 一次一个 feature 推进
  - evaluator: 独立验收, 不与 generator 合并人格           # A3
plan_artifact: plans/<id>.md, 带进度与决策日志, 入库       # O1
must_link:  每条任务 → 验收标准 + 文件范围

## ③ Context 上下文
sources:    [AGENTS.md(目录页, ~100行), 相关文件, 进度文件, git log]  # O1/A2
state_file: claude-progress.txt + feature_list.json(passes 字段)    # A2
budget:     按需检索, 不预加载全仓库

## ④ Tools 工具
read_write: [文件读写, grep/head/tail]
exec:       [测试命令, lint, dev server via init.sh]            # A2
integrations:[CI, 日志查询, ticket, feature flag, dry-run deploy] # G1
mcp:        [按需挂载, 只列本任务要用的工具]

## ⑤ Permissions 权限矩阵           # 三档模型抄 A5
tier_1_auto:    [文件读, 代码导航, 白名单只读命令]
tier_2_auto:    [项目内文件编辑]            # 版本控制可回溯
tier_3_gate:    [shell 命令, 外部服务, 项目外文件操作]
principle:      agent 擅自决定的一切, 用户点头前都算未授权   # A5

## ⑥ Sandbox 隔离执行
isolation:  独立 git worktree → 容器/VM                      # A6
network:    默认拒绝, 出网走 allowlist                        # A6/G8
secrets:    不挂生产 secret
teardown:   用完即销毁
principle:  先环境层隔离, 再模型层调行为                       # A6

## ⑦ Eval 门
gates:      [单测绿, lint 过, 任务级 eval, 回归集, 安全扫描]
graders:    code-based + model-based + 人工 rubric 校准        # A10
seed_set:   20-50 真实任务, 含历史故障转化的 case             # A10
env:        eval 环境隔离干净, 隔离基础设施噪声                 # A11

## ⑧ 人类审批点 (Review)
self_review:   agent 先本地自审                               # O1
agent_review:  专门审查 agent, 循环到通过                     # O1
human_gate:    high-risk 任务人审 PR
hard_rules:    [不能直接 git push, 只推自己分支,
                requester 不能自批 PR, workflow 跑前要人批]   # G8

## ⑨ Trace + Learning
trace_fields:  [读了哪些文件, 跑了哪些命令, 每步结果,
                session_log 链接, 签名提交, 共同作者]          # G8
recovery:      git revert + 进度文件 + checkpoint             # A2
feedback_loop: review comment → 规则; 失败样本 → eval;
               模板持续迭代                                   # O1

## 成本与失败
cost_budget:   单 run token/时长上限 (e.g. 全 harness ~$200/6hr vs solo ~$9/20min)  # A3
failure_taxonomy: [模型错 | 环境错(infra) | 上下文缺 | 验收漏]   # A11 区分前两类
```

这份骨架不必九层全填。低风险任务族，③④⑥⑦ 填扎实，其余几行带过就行；只有碰生产、碰钱、碰 secret 的任务族，才值得把权限矩阵和审批点写满。

**长任务 harness 的恢复 artifact 长什么样。** Anthropic 让 agent 去克隆一个 claude.ai，初始化 agent 先写一份超过 200 条的功能清单，每条先标成"未通过"，存成 JSON（实测模型改 JSON 比改 Markdown 更克制，不太乱删）[A2]。配一个 `init.sh` 把环境起起来、一个 `claude-progress.txt` 记历轮干了什么、再加 git 历史 [A2]。每一轮新 agent 开工先跑 `pwd`、读进度文件和 git log 接上状态，再挑一条没做的功能推进，做完把对应条目翻成通过、提交 git [A2]。这套东西就是 ⑨ 那层的恢复 artifact——它让"上一班做到哪了"扛在文件里，而不是扛在早被压没的聊天历史里。

**权限层在产品里怎么落。** Claude Code 的 auto mode 把 ⑤ 做成了一个分类器：每个动作执行前先过一道快速的单 token 过滤（yes 拦 / no 放），只有被拦下的才追加一轮带推理的复核 [A5]。这个分类器只看用户消息和 agent 的工具调用，故意把 agent 自己的话和工具输出剥掉——这样 agent 没法靠花言巧语说服分类器，也堵住了 prompt injection 的主要入口 [A5]。被拦下时 agent 收到的是一条"换个更安全的做法"的提示而不是直接终止，只有连续三次或累计二十次被拦才结束会话 [A5]。

## 常见反模式

- **把九层当组织架构来堆。** 给每层配一个微服务、一个 agent、一份 yaml，小项目直接被 harness 自己压垮。层是检查表不是部署图——该有人管这类风险就行，③④⑤ 完全可以是几行约定。
- **只搭前几层就开自治。** 上下文、工具给得漂漂亮亮，⑤⑥⑦⑧ 全省了——能写文件、能跑命令、能开 PR，却没有隔离、没有验收门、没人签收。这就是上一章那句"没有 eval 就没有自治"在九层上的版本。
- **eval 环境不隔离，把基础设施噪声当模型回归。** 同一个模型，资源配置一抖，分数能差 6 个百分点 [A11]。eval 跑在没隔离干净的环境里，你会把"今天集群忙"误读成"模型退步了"，然后去改一个根本没坏的东西。
- **审批点写在 prompt 里，不写在权限层。** "记得别 push 到 main"写进系统提示，等于把交规刷在司机脑门上。真边界在 ⑤⑥——GitHub 那条"agent 根本不能直接 `git push`"是工具层硬卡的，不是 prompt 里的客套 [G8]。
- **永不拆层。** 为旧模型加的脚手架，新模型来了还原封不动留着。Anthropic 在 Opus 4.6 后把 context reset 删了，因为那条假设过期了 [A3]。harness 要定期回看，把不再承重的脚手架拆掉。
- **发起人自批 PR。** ⑧ 缺了职责分离，自治就成了自说自话。GitHub 把"requester 不能批自己的 PR"做成硬规则，正是为了守住"必要评审"这条线 [G8]。

## 检查清单

- 我能把这次任务在九层上各自的答案，写成一份 spec 吗？还是有几层从来没人想过？
- ① 不合格的任务，是被一个模板/分诊**挡在门外**，还是照单全收？
- ② agent 是先**出计划等人确认**，还是读完 issue 直接改文件？计划能追到验收标准和文件范围吗？
- ⑤ 审批点是写在**权限层/工具层**（硬卡），还是只写在 prompt 里（软求）？
- ⑥ 它在**隔离环境**里跑，还是直接动真仓库、挂着生产 secret？出网是 allowlist 还是放任？
- ⑦ "算完成"有**可执行判据**吗？eval 环境隔离干净了吗，还是基础设施噪声混在分数里？
- ⑧ 谁签收？**发起人能不能自批**自己这次 run 的 PR？
- ⑨ 失败 case 有没有**喂回前几层**（入 eval、入规则、改模板），还是存了就忘？
- 我有没有把九层当组织架构在堆？哪几层其实几行约定就够？哪条脚手架已经过期该拆？

## 自测

- 拿你最近一次真实的 agent run，对着九层逐层问"这层谁在管"。卡在第几层答不上来？那一层就是你现在风险漏得最多的地方。
- 如果现在把这次 run 的聊天历史清空，只留你落在文件里的 trace 和恢复 artifact，下一个人（或下一个 agent）接得住吗？接不住，说明你的 ⑨ 还活在会消失的地方。
- 你给某个任务族写的权限矩阵，是按风险/路径/命令分了级，还是一刀切"全部要批"或"全部放行"？前者是 harness，后两者一个累死人、一个迟早出事。
