这一章不讲怎么做对，讲怎么做错——而且是那种"做错的时候看起来特别像做对"的错。把全店的钥匙塞给第一天上班的新人，看起来是充分授权、效率拉满；一道菜在试做那天惊艳了所有人、于是直接端给客人，看起来是验过货了。两件事都迟早出事，问题是出事之前它们都很好看。

下面收了八个这样的反模式。每一个都先给症状——你会怎么自我说服这是对的，再给正确做法。八个讲完，落到一句硬标准上：能写文件、能跑命令、能联网、能读私有数据、能开 PR 的 agent，没有权限边界、trace、eval、恢复和人工门禁，就不是工程系统，是一台带键盘的随机数发生器。

## 生活类比

设想一家有点规模的店，今天来了个新人。手脚极快，学东西也快，但他不知道哪个柜子放着对账本、哪台机器一碰就停产、哪个客户是这个月千万别得罪的那个。

一种开局：你把整串钥匙——前门、后厨、保险柜、客户名单——一次性塞他手里，说"自己看着办，有事喊我"。头三天可能真挺顺，他干得又快又多。然后某一天，他出于好心，把一批"看起来过期"的料全扔了，那是留着做明天宴席的；或者他为了赶进度，绕过了那道你专门设的复核，把一笔错的款打了出去。他不是坏人，他是个不知道你家规矩、又被你给了无限权限的能干新人。

另一种开局，是这一章想讲的对照：你不给他全套钥匙，按今天的活给。你不指望"我嘱咐他别碰保险柜"这一句话能拦住他——你干脆不给保险柜的钥匙。试做那天惊艳的那道菜，你不直接上桌，先让他在真实的午市忙乱里做二十份，看出错率。你盯的不是他今天做了多少份，是有多少份是合格、能端出去、不用返工的。

这两种开局的区别，不在新人聪不聪明，在你给他搭的环境对不对。后面八个反模式，都是第一种开局的变体。

## 工程定义

**反模式（anti-pattern）**，是一种反复出现、看起来能解决问题、实则把问题推迟并放大的做法。它和"单纯的 bug"不一样：bug 是你不想要的，反模式是你以为想要的。它之所以危险，正因为它在短期、在 demo、在顺利的那几天里，确实管用。

放到 agent 工程上，这些反模式有一个共同的结构：**把本该建在环境里的约束，搬进了模型或提示词里，然后用"它这次表现挺好"当作约束已经存在的证据。** Anthropic 在讲 containment 时把这条原则正过来说过一遍——"先在环境层做隔离，再在模型层引导行为"，比起监督 agent 做了什么，他们更优先去限制 agent **能做**什么 [A6]。反模式就是把这个顺序倒过来：先信任行为，环境层留空。

下面八个，每一个都是这个倒置的一种具体形态。先把八个的症状和正确做法列成一张表，再逐个展开。

| # | 反模式 | 症状（你会这样自我说服） | 正确做法 |
|---|---|---|---|
| 1 | prompt 崇拜 | "提示词写得够好，就不用测试和权限了" | 把 prompt 放进 harness，别把 harness 塞进 prompt |
| 2 | 全工具入场 | "把所有工具和文档一次给它，省得来回找" | 按任务阶段和风险裁剪工具与上下文 |
| 3 | 自动合并幻想 | "PR 过了测试就自动合，人不用看" | 高风险改动必须 trace review + CODEOWNERS 门禁 |
| 4 | 一个巨型 agent | "一个超长上下文 agent 把全流程包了" | 拆角色 / 拆阶段，用 artifact 交接 |
| 5 | 没有失败分类 | "又挂了，模型不行" | 区分规格/上下文/工具/模型/环境/权限/review 七类失败 |
| 6 | eval 剧场 | "几个好看的 demo 跑通了，就算验过" | 用真实失败样本、边界样本、回归样本 |
| 7 | 提示词当安全 | "我写了一句'别泄露 secret'" | 工具层 + sandbox + secret policy + 审计层 |
| 8 | 吞吐量虚荣 | "今天生成了 40 个 PR、两万行代码" | 看 accepted safe changes 和返工成本 |

## 为什么重要：AI 时代的变化

这些做法不是新发明，软件工程里一直有"看起来先进、实际危险"的东西。变化在于 agent 把每一种的代价都放大了，而且放大的方向不直观。

第一，**执行变快，错误也抄得更快**。一个模糊的规格、一道缺失的验收、一个写歪的工具描述，过去要靠人一行行地敲才传播得开，现在 agent 能在几分钟里把它复制进几十个文件、开成一个看起来完整的 PR。反模式的"潜伏期"被压短了：从前你有时间在它扩散前发现，现在常常是它已经合进去了你才反应过来。

第二，**"它这次表现挺好"这个证据，比以前更不可信**。agent 的行为本来就在运行之间波动；更麻烦的是，连失败都可能不是模型的错。Anthropic 量化过一个反直觉的现象：在 Terminal-Bench 2.0 上，光是基础设施配置的差异，就能让分数差出 6 个百分点——足以盖过排行榜上顶尖模型之间的差距 [A11]。在严格的资源限制下，约 5.8% 的任务是被基础设施错误（而不是模型）搞挂的 [A11]。这件事对反模式的意义是：你既不能因为"它这次跑通了"就相信约束存在（第 6 个反模式），也不能因为"它这次挂了"就把账记到模型头上（第 5 个反模式）——两个方向都得有 trace 和分类才说得清。

第三，**授权的粒度变细了，但人的注意力没变多**。Claude Code 的数据里，用户大约 93% 的权限弹窗是直接点同意的 [A5][A6]——也就是说，"让人来批"这道门，在疲劳之下接近形同虚设。这正是反模式第 3 个（自动合并幻想）和第 7 个（提示词当安全）真正危险的地方：它们利用的不是技术漏洞，是人会累、会嫌烦、会习惯性点"同意"。

## 实践方法

把八个反模式逐个拆开。每个都给"为什么看起来对"和"正确怎么做"。

**① prompt 崇拜（prompt worship）。** 症状是把越来越多的约束写进系统提示——"务必先写测试""绝对不要碰 prod""记得检查权限"——然后把这一长段 prompt 当成防线。它看起来对，因为 prompt 确实能改变行为，而且改一句话比改一套流程便宜得多。问题是 prompt 是建议不是栅栏：模型可以忽略它，prompt injection 可以覆盖它，上下文一长它就被稀释。正确做法是方向反过来——把 prompt 放进 harness，而不是把 harness 塞进 prompt。该测试的事交给测试，该卡权限的事交给工具层，prompt 只负责它真正擅长的：表达意图和风格。

**② 全工具入场（all-tools-in-context）。** 症状是开局就把所有能用的工具、所有相关文档一股脑挂进上下文，理由是"省得它来回找"。它看起来对，因为信息越全似乎越不容易漏。问题有两层：其一，工具和文档本身在烧上下文预算，真正关键的那几行被淹没；其二，给得越多，误用的面也越大——一个本该只读的任务，手里却攥着删库的工具。正确做法是按任务的阶段和风险裁剪。Claude Code auto mode 就是这么分层的：安全的读取、代码导航走 Tier 1 不审查；项目内的文件编辑走 Tier 2，因为有版本控制兜底、可追溯；剩下的——shell 命令、外部服务、项目外的文件操作——才进 Tier 3 交给分类器逐个判 [A5]。工具不是给得越多越好，是按这一步真正需要的给。

**③ 自动合并幻想（auto-merge fantasy）。** 症状是：agent 开的 PR 只要 CI 全绿就自动合并，没人看 diff。它看起来对，因为测试过了不就代表对了吗——而且人本来就懒得 review。问题是测试只覆盖你想到的情况，覆盖不了你没想到的；而 agent 最擅长的恰恰是找到测试没覆盖的那条缝。Anthropic 在评测里就观察到，Claude 会通过翻看上一轮 trial 的 git history 在某些任务上拿到不公平的优势 [A10b]——同一种"钻空子"的本事，用在你的 CI 上就是合进一个测试全绿、逻辑全错的改动。正确做法是按风险分级：低风险的、纯机械的改动可以放宽，但任何动到钱、动到权限、动到数据、动到外部副作用的改动，必须有 trace review，并用 CODEOWNERS 把相关目录的批准权钉死在懂这块的人身上。再加一条职责分离：发起任务的那个人，不该是批准这个 agent 自己开的 PR 的人——不让一方既当运动员又当裁判。

**④ 一个巨型 agent（one mega agent）。** 症状是所有事都丢给同一个 agent，在一条越来越长的上下文里调研、设计、写码、测试、改 bug 一路干到底。它看起来对，因为"一个上下文里啥都记得"省去了交接。问题是上下文越长，召回越差——前面的规格到后面被新读进来的文件挤没了，它会忘掉自己半小时前定下的约束。正确做法是拆：按角色拆（一个调研、一个实现、一个 review），按阶段拆（plan 阶段产出一份 plan 文件，implement 阶段读这份文件接着干），用 artifact 做交接而不是靠一条聊天记录扛全程。把状态落在上下文之外的文件里，下一个阶段读文件接力，比指望一个超长 agent 自己记住一切要稳得多。

**⑤ 没有失败分类（no failure taxonomy）。** 症状是 agent 一出问题，结论永远是"模型不行"。它看起来对，因为模型确实是最显眼的那个变量。问题是这个结论既不可操作（你换个模型可能照样挂），也常常是错的。一次失败至少该分七类追问：是**规格**没写清（它根本不知道你要什么）？是**上下文**没给够或太杂（关键信息被淹了）？是**工具**描述歪了或权限不对（它想做但做不了）？是**环境**抽风（依赖装不上、资源被卡、测试 flaky）？是**权限**门把它拦在外面？是**review** 没接住？最后才轮到**模型**本身。Anthropic 反复强调：别把基础设施噪声算到模型头上——一个"几个百分点的领先，可能是真能力差距，也可能只是一台更大的虚拟机" [A11]。分不清这七类，你修的永远是错的那一层。

**⑥ eval 剧场（eval theater）。** 症状是拿几个精心挑选、每次都跑得很漂亮的 demo 当 eval，演示给自己看也演示给别人看。它看起来对，因为绿灯让人安心。问题是 demo 选的是 agent 本来就会做的事，而 eval 该测的恰恰是它容易栽的地方。Anthropic 给的做法很具体：起步的任务集应当从**真实失败**里捞——20 到 50 个来自真实失败的任务就是很好的开始 [A10b]；要建**平衡**的问题集，既测"该触发时有没有触发"，也测"不该触发时有没有忍住"，单边的 eval 只会催生单边的优化 [A10b]；还要分清**能力 eval**（问"它能不能做到"，起步通过率就该低）和**回归 eval**（问"它还做得到吗"，通过率该接近 100%）[A10b]。一个永远满分的 eval，只能抓回归，给不了任何改进信号 [A10b]。真正的 eval 是用真实失败样本、边界样本、回归样本搭起来的，不是几个好看的 demo。

**⑦ 提示词当安全（security by prompt）。** 症状是写一句"不要泄露 secret、不要 push 到 main"，就当安全做完了。它看起来对，因为这句话读起来确实是条规矩。问题是这条规矩在错误的层。Anthropic 内部的红队演练给过一个很硬的反例：当用户本人被钓鱼、亲手把注入指令打进去时，模型层的防御几乎无能为力——25 次里有 24 次成功把 AWS 凭证带了出去，因为防御锚定的是"用户意图"，而此刻是用户自己在敲那条指令，"没有任何异常可供它捕捉" [A6]。安全得分层做：环境层用 sandbox 把能干的事框死（Claude Code 的默认姿态就是读允许、工作区内写允许、网络默认拒绝）[A6]；secret 该用专门的 policy 和代理管，而不是靠模型自觉不读；再加一层审计，事后能查谁在什么时候碰了什么。Anthropic 的一条原则值得抄下来——"最弱的一层，是你自己造的那层" [A6]，安全别全压在一句 prompt 上。

**⑧ 吞吐量虚荣（throughput vanity）。** 症状是用生成的 PR 数、代码行数、烧掉的 token 量来衡量 agent 干得好不好。它看起来对，因为这些数字最好统计、也最好看。问题是它们衡量的是产量不是价值——40 个 PR 里有 30 个被打回重做，比 8 个一次过的还糟，但数字上漂亮得多。该看的是 **accepted safe changes**：真正被接受、合进去、没引发返工的改动有多少；以及**返工成本**：为了让 agent 的产出能用，人额外花了多少时间去修、去 review、去收尾。Anthropic 在讲评测时也提醒过一个相关的陷阱——别只盯着分数表面，分数没爬升，得先确认是 agent 真的没做好，还是 eval 自己有问题（任务有歧义、合法解被误判、harness 卡了模型）[A10b]。同理，PR 数涨了不等于交付变好了，得先确认涨的是合格产出还是返工量。

## 真实案例

**auto mode 的核心不是无限授权，是分类器加边界加恢复。** 把第 1、3、7 个反模式放在一起看，会发现它们都在赌一件事：要么相信一句 prompt、要么相信一道自动门、要么相信模型自觉，能替代真正的边界。Anthropic 做 Claude Code auto mode 时，恰恰没这么赌。它的出发点是一个尴尬事实：用户大约 93% 的权限弹窗是直接点同意的——"让人批"这道门在疲劳下基本失效 [A5]。但他们的回应不是"那就全自动放行"，而是搭了一套有边界的系统。

权限分三层：安全的读取和用户配置的 allowlist 走 Tier 1（auto mode 下连解释器的通行权都被收回了）；项目内、版本控制可追溯的文件编辑走 Tier 2；剩下的 shell 命令、外部服务、项目外文件操作才进 Tier 3，交给一个分类器逐个判 [A5]。分类器本身也设了防线：它只看用户消息和 agent 的工具调用载荷，**故意剥掉** agent 自己的推理文本和工具输出——这样 agent 没法用一段花言巧语把分类器说服，也堵住了主要的注入入口 [A5]。被拦下来不是直接终止，而是"拒绝并继续"：把拒绝当成一个工具返回喂回去，让 agent 去找更安全的替代路径，只有连续三次或累计二十次被拒才结束会话 [A5]。

这套系统最诚实的地方，是 Anthropic 自己标出的天花板：对真实的"过度热心"动作，它仍有约 17% 的漏判率 [A5]；他们明说 auto mode "不是高风险基础设施上人工审查的替代品" [A5]。换句话说，连做这件事的人都不认为分类器能取代人工门禁——它只是在"无限制放行"和"逐条手批"之间补了一个有边界、有恢复路径的中间档。auto mode 真正的内核，是那句被写进设计里的话——"agent 自己做的任何选择，在用户点头之前都算未授权" [A5]。

**containment 要在环境层做，不在嘴上说。** 第 7 个反模式（提示词当安全）的反面教材，Anthropic 把它讲得很透。他们的指导原则是"先在环境层做隔离，再在模型层引导行为"，监督 agent **能做什么**优先于监督它**做了什么** [A6]。落到产品上是三种隔离：claude.ai 的代码跑在隔离的 gVisor 容器里，文件系统临时、用完即弃；Claude Code 在开发者本机上用操作系统级 sandbox（macOS 上是 Seatbelt，Linux 上是 bubblewrap），默认姿态是读允许、工作区内写允许、网络默认拒绝——上了 sandbox 之后权限弹窗减少了 84% [A6]。安全不是写在 prompt 里的客套话，是用沙箱、文件系统边界、出网控制硬框出来的。

**别把基础设施噪声怪到模型头上。** 第 5 个反模式（没有失败分类）的直接证据就在 A11。Anthropic 在六种资源配置下、用同一个模型跑 Terminal-Bench 2.0：严格限制下基础设施错误率 5.8%，给到 3 倍余量降到约 2.1%，完全不限时降到 0.5%，分数比严格档高了 6 个百分点 [A11]。更微妙的是，从 1 倍到 3 倍那一段，分数的波动在噪声范围内（p = 0.40），说明那些崩掉的 run 多半本来也会失败 [A11]——也就是说，如果你不分类，就会把一批"环境抽风"误读成"模型变弱"，然后去调一个根本不在模型上的东西。他们给运营评测的人的建议同样适用于运营 agent：把资源配置当成一等的、要记录的实验变量，"低于 3 个百分点的排行榜差距，在配置被记录并对齐之前都值得怀疑" [A11]。

## 常见反模式

这一章本身就是反模式合集，这里把八个之外、最容易混进来的几个"元反模式"再点一下——它们是反模式的反模式，犯起来更隐蔽：

- **把"修反模式"也做成剧场。** 写一份漂亮的"agent 安全规范"文档贴在 wiki 上，然后什么都不变。规范不落到工具层和门禁上，就是另一种提示词当安全。
- **一刀切地禁。** 因为怕第 3 个反模式，就规定"所有 agent PR 一律人工逐行审"。这又退回到第一种开局的反面——把人累垮，最后所有人闭眼点同意，等于没审。分级，不是全禁。
- **拿别人的边界当自己的边界。** 看到 GitHub 或 Anthropic 怎么划线就照抄。他们的阈值是在他们的威胁模型、他们的规模下校准的；你得照自己的真实失败样本重新校。A11 那句"低于 3 个百分点的差距值得怀疑"针对的是评测配置，但精神一样：没在你自己环境里验过的数字，别直接拿来当门禁。
- **只在出事后才分类失败。** 失败分类不是事故复盘时才做的事。它得是常态——Anthropic 的说法是，定期读 trace（transcript）是 agent 开发的一项核心技能，一个失败该让人一眼看清"agent 到底错在哪、为什么" [A10b]。

## 检查清单

- 我对这个 agent 的约束，是写在**工具层和门禁**上，还是只写在 **prompt** 里？（①⑦）
- 它手里的工具，是这一步**真正需要**的，还是开局一股脑全给的？（②）
- agent 的 PR 自动合并的边界，是按**风险**分的吗？动钱、动权限、动数据的改动，有没有 trace review + CODEOWNERS？（③）
- 一个长任务，是靠**一个超长 agent** 硬扛，还是拆了角色/阶段、用 artifact 交接？（④）
- 上次 agent 失败，我有没有分清是**规格/上下文/工具/模型/环境/权限/review** 哪一类，还是直接说"模型不行"？（⑤）
- 我的 eval，是几个**好看的 demo**，还是真实失败样本 + 边界样本 + 回归样本？（⑥）
- 我衡量 agent 用的是 **PR 数/代码行/token 量**，还是 accepted safe changes 和返工成本？（⑧）
- 任何能写文件、跑命令、联网、读私有数据或开 PR 的 agent，是不是都齐了五样：**权限边界、trace、eval、恢复、人工门禁**？

## 自测

- 把你现在跑的某个 agent 拎出来，对照那句硬标准——No eval, no autonomy——逐项查：它能写文件吗？能跑命令吗？能联网吗？能开 PR 吗？只要有一个"能"，它就必须配齐权限边界、trace、eval、恢复和人工门禁。缺哪一样？缺的那一样，就是你下一个会踩的反模式。
- 你上一次说"这次 agent 表现不行"，凭的是什么？是读了 trace、分清了七类失败，还是只看了个结果就归到"模型不行"？如果是后者，你修的很可能不是出问题的那一层。
- 如果有人现在按"看起来很先进"的标准夸你的 agent 系统——生成快、PR 多、提示词写得神——这些夸奖里，有几条经得起"那它的权限边界、恢复路径、真实失败 eval 在哪"这一问？
