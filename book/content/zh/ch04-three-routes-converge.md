GitHub、OpenAI、Anthropic 这三家，今天都在卖"让 agent 帮你写代码、交代码"这件事。但你只要把各家的博客摊开读，会发现它们从三个完全不同的入口切进来：一家从你已经在用的 issue / PR / Actions 切，一家从模型加运行时切，一家从"怎么让 agent 干完一个跨好几天的长任务"切。

这一章想说清两件事：三家的真实侧重到底差在哪（别把营销话术当共识），以及它们正在朝同一个方向收。这个方向不是"模型更聪明"——是把 agent 塞进真实交付流程，再给它配上边界和验收。看懂这点，你才不会在选工具时把工程押死在某一家的某个托管产品上。

## 生活类比

把"从你下单到包裹签收"这条链想象成一次跨城快递。要让它跑顺，得有好几样东西到位：揽件的网点、分拣线、跑长途的转运车、送到你手上的快递员，还有一套出问题能查能赔的规矩。

现在有三家快递公司都说自己能把这单送到，但各自的重心不一样。

第一家把钱砸在**网点和分拣线**上。它的逻辑是：你早就习惯在我这儿下单、查物流、签收了，那我就把"自动分拣、自动派单"直接长在这套你熟的流程里——你不用换 App，包裹在你眼皮底下从一个站点流到下一个站点。

第二家不太谈网点，它谈的是**快递员本身有多能干**，以及给快递员配什么车、什么扫描枪、什么导航。它的逻辑是：网点谁都能建，真正难的是让一个快递员既能看懂地址、又能爬六楼、还能在你不在家时把件妥善放好——所以我把劲儿全使在"人加装备"上，你拿这套装备自己去搭网点也行。

第三家盯的是**长途多段转运**：一件货要换三趟车、过两个分拨中心、跨三天才到，每次换车都是一次"交接"，交接没交代清楚就丢件。它的逻辑是：短途谁都送得了，真正考验是这种横跨多段、谁都记不全前因后果的长单——所以它专门研究"换班的人怎么知道上一班干到哪了"。

这三家听起来在打三场不同的仗。但只要你退一步看，会发现它们最终都得把同一条链跑通：从下单到签收，中间一个件都不能丢，丢了得能查能赔。网点强的那家迟早要操心长途交接，装备强的那家迟早要把网点铺起来，长途强的那家迟早要有个让你下单查件的门面。重心不同，终点是同一个。这就是这章标题的意思——三条路线在汇合。

## 工程定义

先把三家的"重心"翻译成工程语言，不带营销滤镜。

**GitHub 路线 = workflow-native（长在你已有的工作流里）。** 它的核心动作是：把 agent 塞进你本来就在用的 issue、PR、Actions。你把一个 GitHub issue 指派给 Copilot，它就在后台起一个开发环境、克隆仓库、把改动以 commit 形式推到一条草稿 PR 上，整个过程在 session log 里留痕 [G1]。它的边界也是用平台原语硬卡的：agent 只能推它自己建的分支、发起任务的人不能批准这个 agent 的 PR、CI 工作流要人工批准才跑、联网只走白名单 [G1]。后来这套又扩到桌面 App、CLI 和云：每个 agent session 跑在自己独立的 git worktree 里、有本地和云端 sandbox [G5]。再往上是 Agent HQ——GitHub 在 2025 年 Universe 上把它定位成"编排任意 agent 的统一工作流"，Anthropic、OpenAI、Google、Cognition、xAI 等多家的 coding agent 都能通过付费 Copilot 订阅接进来，OpenAI 的 Codex 当时就能在 VS Code Insiders 里直接用 [G6]。一句话：GitHub 卖的是"把各家 agent 编排进你熟悉的那条流水线"。

**OpenAI 路线 = 平台与系统（模型加 harness）。** 它有两层。一层是 Codex 这个 coding agent 本身，能力一直在往外摊：从写代码扩到能操作你电脑、内置浏览器、生成图片、记忆偏好，还能给自己排将来的活、隔几天醒来接着干 [O7]。另一层是给开发者的底座：Agents SDK 提供一个"模型原生的 harness"，让 agent 能跨文件和工具干活，配上原生 sandbox 执行；它把 MCP、skills、AGENTS.md、shell 执行、apply-patch 这些原语都标准化进来 [O2]。OpenAI 自己有篇文章把这件事命名为 harness engineering：把 harness 定义为 agent 周围那层"验证、反馈、恢复、工具、上下文和人类判断"的结构，并讲了一个内部团队"0 行手写代码"造产品的实验——三个工程师驱动 Codex，平均每人每天 3.5 个 PR [O1]。一句话：OpenAI 卖的是"模型加上把模型用好的那套系统工程"。

**Anthropic 路线 = 把 agent 当长期工程伙伴。** 它的重心在工作法和长任务。Claude Code 的主线工作流是 explore → plan → code → commit：先在 plan 模式只读地探查、再让它出实现计划、切出 plan 模式去写、最后提交开 PR [A1]。它反复强调几样东西：给 agent 一个能自己跑的验收检查（测试、构建、截图对比）、用 CLAUDE.md 给持久上下文、用权限和 sandbox 划边界、用 worktree 跑并行 [A1]。再往上是 context 工程——把上下文当稀缺资源管理，应对"context rot"（上下文越长，模型回忆越差），用 compaction、结构化笔记、子 agent 来扛超出单个上下文窗口的长任务 [A13]。最硬核的是长任务 harness：一个 initializer agent 先搭好环境（写 init.sh、写 feature 清单 JSON、写进度文件、初始 git commit），后续每个 coding agent 每轮只推进一个 feature、干完留下结构化交接，下一轮换班的 agent 靠进度文件和 git 历史接上 [A2]。一句话：Anthropic 卖的是"让一个会换班、会失忆的 agent，靠交接物把长活干完"。

三家的说法不一样，但都在围着同一个对象打转：一个能读能写能跑命令的 agent，怎么进到真实交付里、又不闯祸。GitHub 从流程入口切，OpenAI 从模型加系统切，Anthropic 从工作法和长任务切。入口不同，要解决的是同一道题。

## 为什么重要：AI 时代的变化

为什么要专门花一章讲"三家路线在汇合"？因为选错参照系，会让你在两个地方栽跟头。

第一个坑：**把营销话术当行业共识。** 每家博客都在说"agent 进入真实开发流程""人来掌舵、agent 执行""保留人的判断与问责"。这些话单看都对，但它们是各家在给自己的产品找位置，不是一份已经达成的共识。真去抠细节，分歧很具体：GitHub 默认让 agent 跑在云端、用平台权限卡边界 [G1]；OpenAI 那个实验里几乎把 review 都交给 agent 对 agent、人只在需要判断时介入 [O1]；Anthropic 反复强调"给 agent 一个它能自己跑的验收检查"，把人类验收当成不能省的一环 [A1]。同样一句"保留人的判断"，三家放人的位置并不在一处。你要是把这些话糊成一团"业界都这么干"，就会照着一个不存在的平均方案去搭自己的流程。

第二个坑，也是这章真正想钉的：**别把工程押死在某个托管抽象上。** 这是 AI 时代和以往最不一样的地方——厂商的上层产品 churn（快速更替）得非常快，比模型本身还快。下一节有个活教材，这里先把判断给出来：底层的工程能力（隔离执行、验收、留痕、可撤回）是耐用的，越上层、越"帮你把一切都包好"的托管产品，越容易半年就变样甚至下线。你越早把这条想明白，越不会在某个漂亮的可视化工具上把团队的工作流焊死。

还有一个正面的理由：看懂汇合，能帮你做减法。三家的入口看着花，但落到你要管的东西上是同一份清单——任务从哪进来、agent 能做哪些动作、在哪隔离执行、怎么算完成、出错怎么撤、留了什么 trace 给下一个人接手。不管你最后用 GitHub 的流水线、OpenAI 的 SDK 还是 Claude Code，这份清单都得你自己答。工具帮你实现它，但替你答不了。

## 实践方法

面对三条路线，与其纠结"选哪家"，不如先把判断标准立起来。

**按"你已经在哪"选入口，而不是按谁的博客更热闹。** 如果你的团队重度活在 GitHub 上、issue 和 PR 就是你们的协作中枢，那 workflow-native 这条线摩擦最小——agent 长在你熟的流程里，不用换工具 [G1][G6]。如果你要把 agent 能力嵌进自己的产品、需要对 harness 有可见性和控制权，那 OpenAI 给开发者的 SDK 这条线更对路 [O2]。如果你的痛点是"一个任务跨好几个上下文窗口、agent 老是失忆或半途宣布完工"，那 Anthropic 那套长任务 harness 的思路最值得抄 [A2]。

**分清哪些是耐用工程、哪些是会 churn 的托管产品。** 一个粗判据：越靠近"隔离执行 / 验收 / 留痕 / 可撤回"这些底层能力的，越耐用；越靠近"拖拽几下就出一个 agent"这种托管上层的，越要留个退路。对后者，下手前先问一句：万一这个产品半年后下线，我把它产出的东西迁出去要多大代价？

**用一份跨厂商都成立的清单，去衡量任何一家的方案。** 三家入口不同，但都得回答这几问：任务从哪进、agent 能做哪些动作、在哪隔离跑、怎么算完成、出错怎么撤、留了什么 trace。拿这份清单去对照各家产品，缺哪项你心里就有数——这也正是后面 harness 专章要展开的东西。

**把可迁移的资产攥在自己手里。** AGENTS.md / CLAUDE.md 这类 repo 指令文件、你的测试套件、你的验收标准——这些是跨厂商通用的，谁来当 agent 都读得懂、用得上。三家的 SDK 和博客里都把 AGENTS.md 当一等公民 [O1][O2][G6]。把规矩沉淀在仓库里，而不是沉淀在某个厂商的可视化画布里，你换厂商的代价就小。

## 真实案例

**一个活教材：AgentKit 的 Agent Builder，活不到半年。**

OpenAI 在 2025 年 10 月 6 日的 DevDay 上发布了 AgentKit，其中的招牌是 **Agent Builder**——一块可视化画布，拖拽节点就能编排多 agent 工作流、连工具、配 guardrails，还自带预览运行、内联 eval 配置和版本控制 [O10]。博客里的客户证言很亮眼：Ramp 几小时就搭出一个采购 agent，迭代周期缩短 70%；LY Corporation 不到两小时就跑通了第一个多 agent 工作流 [O10]。这是典型的"托管上层产品"——帮你把一切都包好，拖几下就能出活。

然后看日期。2025 年 10 月 6 日发布；2026 年 6 月 3 日，OpenAI 就通知用户 Agent Builder 进入 deprecation（弃用），计划 2026 年 11 月 30 日彻底关停 [O11]。从发布到宣布退役，中间不到八个月。同一批通知里，OpenAI 自家的 Evals 平台也一起被弃用了 [O11]——也就是 AgentKit 当初拿来"衡量 agent 性能"的那块功能。官方给的迁移路径是转到 Agents SDK 或 ChatGPT Workspace Agents [O11]。

这个例子的价值不在嘲笑某家产品，而在它把前一节那条判断变成了可触摸的事实：越是"帮你把一切都包好"的托管上层，churn 越快。如果有团队当初照着 Agent Builder 的可视化画布把核心工作流焊死了，这会儿就得在半年内连搬两次家（先迁 Agent Builder，再迁 Evals）。反过来，那条更底层的线——Agents SDK 提供的 harness 和 sandbox 原语 [O2]——反而是 OpenAI 让大家迁过去的落脚点。耐用的是底层能力，会变的是上层封装。

顺带一个小提醒，跟选型同样重要：这章引的厂商 URL，好几条本身就不老实。G5 那篇 GitHub 博客，任务给的链接直接 404，是从另一个 canonical URL 才找回正文的；OpenAI 好几篇文章对自动抓取直接返回 403，靠归档快照才取到内容。厂商的官方链接会失效、会改版、会挡爬虫。你让 agent（或你自己）引一段"官方文档"时，链接得亲手点开核一遍，别假设它一定还在、一定是你以为的那篇。

## 常见反模式

- **把营销话术当行业共识。** 三家都说"agent 进真实流程、保留人类判断"，听着像已经定下的标准。其实是各家在给自家产品找位置，落到细节分歧很大（谁来 review、在哪执行、人卡在哪一环都不一样）[G1][O1][A1]。别照着一个被你糊出来的"业界平均方案"去搭流程。
- **把工程押死在某个托管抽象上。** 照着一个可视化画布或托管 API 把核心工作流焊死，等它半年后下线，你就得连夜搬家。Agent Builder 就是现成的例子 [O10][O11]。托管产品可以用，但别让它成为你迁不走的唯一真源。
- **"选一家就一劳永逸"。** 以为站好队、绑定一家就不用再操心。可三家都在快速变形、互相抄、还互相接（Agent HQ 直接把 OpenAI、Anthropic、Google 的 agent 编排进 GitHub）[G6]。真正稳的不是押注哪家，是把跨厂商通用的那份清单和资产攥在自己手里。
- **只看上层封装、不问底层能力。** 被"拖几下就出一个 agent"的演示打动，却不问它底下有没有隔离执行、有没有验收、能不能撤回、留不留 trace。演示好看和真能托付，是两码事。

## 检查清单

- 我选某家方案，是因为**我的团队已经在那儿**（GitHub / 自建产品 / 长任务场景），还是因为它的博客写得热闹？
- 我能分清手上用的东西，哪些是**耐用的底层能力**（隔离执行、验收、留痕、可撤回），哪些是**会快速 churn 的托管上层**吗？
- 万一我重度依赖的那个托管产品**半年后下线**，我把产出迁走的代价有多大？我有退路吗？
- 我的核心规矩，是沉淀在**仓库里**（AGENTS.md、测试、验收标准），还是焊死在某个厂商的**可视化画布**里？
- 我有没有把各家博客的"agent 进真实流程、保留人类判断"当成**已达成的共识**，而没去抠它们各自把人卡在哪一环？
- 我让 agent 或自己引的那条"官方文档"链接，**亲手点开核过**吗？

## 自测

- 用你自己的话说清：GitHub、OpenAI、Anthropic 这三条路线，各自的**入口**是什么？它们最终都在朝哪个**同一个方向**收？
- 假如你现在重度用的某个托管 agent 产品，下周宣布半年后退役。你需要迁走哪些东西？哪些迁起来很疼、哪些根本不疼？你的答案，基本就是你"被某家锁死"程度的体温计。
- 下次再看到"业界都在往 agentic 转"这种话，你能不能立刻追问一句：具体是谁、把人留在了哪一环、用的是耐用底层还是会 churn 的托管上层？
