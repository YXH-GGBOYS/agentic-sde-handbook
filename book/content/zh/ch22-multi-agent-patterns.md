一个 agent 干不完的活，很自然会想到再开几个。但"多开几个"不是免费的：多个 agent 之间要分工、要不打架、最后还得有人把各自的活拼成一个能交付的整体。组织得好，几个 agent 并行能把一周的探索压进一个下午；组织得差，它们会同时凿同一面墙，互相覆盖，最后你花在拼装和收拾首尾上的时间，比一个 agent 老老实实干还多。

这一章讲五种已经被验证有用的多 agent 模式，每种的适用场景和注意事项；也讲清楚一个常被忽略的事实——多 agent 是要付代价的，不是 agent 越多越好。

## 生活类比

把一个 agent 当成你前面几章一直在打交道的那个新外包：能力强，但不懂你家规矩。现在活大了，一个人干不完，你要组个班组。

组班组的第一个问题不是"找几个人"，是"怎么分活"。你不能让两个泥瓦匠同时凿同一面墙——他们会互相碍事，凿出来的洞还对不上。你得把墙分段，张三砌东头，李四砌西头，中间留好接缝。第二个问题是"谁统筹"。得有个工长：他不亲自砌墙，而是把整面墙的活拆成段、派下去、盯着进度、最后把各段拼到一起验收。第三个问题是"怎么验"。活交上来不能自己说自己干得好，得有个独立的验收员拿着标准去查。

这里有个反直觉的地方：人多不等于活好。你往工地多塞五个泥瓦匠，如果没人分活、没人统筹，他们大概率会扎堆在好干的那段、撂下难干的那段，材料还都堆在脚下互相绊。一个组织混乱的十人班组，干得可能比一个分工清楚的三人班组还慢、还乱。多 agent 也一样——它的价值全在"组织"两个字上，不在"数量"上。

## 工程定义

多 agent 模式，是把一个任务拆给多个各自带独立上下文的 agent 实例去做，再把它们的产出协调成一个结果的一类工程结构。关键词是"独立上下文"：每个 agent 有自己的注意力预算、自己的工具、自己看到的那部分材料，彼此之间通过明确的接口（派活、回传摘要、提交 artifact）交流，而不是共享一个大窗口。

为什么要拆成独立上下文，而不是把所有事塞进一个 agent？回到上一卷讲 context 工程的那条曲线：上下文越长，模型对关键信息的召回越差。一个 agent 同时背着"探索 A 模块、探索 B 模块、还要综合判断"的全部材料，它的窗口会被三件事的细节挤爆。拆开之后，探索 A 的 agent 只看 A 的材料，把结论压成一份摘要回传；主 agent 的窗口里始终只装结论，不装三路探索的全部原始过程。这是多 agent 在工程上最实在的一个好处：用多个干净的小上下文，换掉一个被塞满的大上下文。

下面五种是实践里反复出现的模式。它们不是互斥的——一个真实系统经常是几种叠在一起（比如一个 orchestrator 底下挂着若干 specialist worker，每个 worker 的产出再过一道 adversarial review）。

| 模式 | 一句话结构 | 典型用途 | 最该盯的注意点 |
|---|---|---|---|
| orchestrator + workers | 一个编排器拆任务、派给若干 worker、再综合 | 大项目拆子任务、复杂研究 | 编排器要控住范围、预算和最终一致性 |
| generator + evaluator | 一个生成方案、一个独立评判 | 方案生成、代码修复 | evaluator 必须独立，别被 generator 的解释带偏 |
| parallel exploration | 多路并行探索同一问题、再合并 | 架构选型、bug 根因定位 | 合并时去重，且禁止多路同时改同一处 |
| adversarial reviewer | 一个专门挑毛病的 reviewer 卡在产出后面 | 安全、兼容、回归审查 | reviewer 要有明确 rubric，不能泛泛挑刺 |
| specialist agents | 按角色分工的若干专才 agent | 前端/后端/数据/测试/文档 | 工具和上下文按角色裁剪，别给全套 |

**orchestrator + workers（编排器 + 工人）。** 一个 lead agent 负责想策略、把大任务拆成子任务派下去，若干 worker 各自在独立上下文里把分到的子任务做完，lead 再把结果综合成最终产出。Anthropic 的多 agent 研究系统就是这个结构：lead agent 制定调查策略、把任务委派给并行的 subagent、再综合它们的发现，另有一个独立的 CitationAgent 专门负责给最终输出标好来源 [A7]。编排器是这个模式的命门——它要在派活前把范围切清楚（别让两个 worker 的活重叠），要给每个 worker 划预算（不然 token 会失控，后面会讲为什么），最后要保证综合出来的结果是一致的，而不是几份互相矛盾的子结论硬拼在一起。

**generator + evaluator（生成器 + 评判器）。** 一个 agent 负责生成（一版方案、一个修复 patch），另一个 agent 负责拿标准去评判好不好、要不要返工。这个模式在方案生成和代码修复里特别常用。要害是 evaluator 必须独立：它评判的是产出本身对不对，不是听 generator 解释一通"我这么改是因为……"就放行。一旦 evaluator 被 generator 的叙述带着走，它就退化成一个橡皮图章，等于没评。让它只看结果（测试过没过、有没有满足验收标准），别看辩护词。

**parallel exploration（并行探索）。** 同一个问题，开几路 agent 同时从不同方向探，最后把结论合并。架构选型时让几路分别评估不同方案，bug 根因定位时让几路分别从不同假设查起，都属于这类。它适合"宽度优先"的问题——要同时铺开好几个方向。两个注意点：一是合并时要去重，几路很可能查出大量重叠的结论，不去重就是把同一句话抄三遍占预算；二是这几路只能"探"不能"改"——如果让并行的几路同时去动同一处代码，它们会互相覆盖，这是多 agent 里最容易翻车的地方之一。

**adversarial reviewer（对抗式审查者）。** 在产出后面卡一个专门唱反调的 agent，它的任务不是夸，是找出这版改动会在哪里出事——安全漏洞、兼容性破坏、可能的回归。这个模式的价值全系于 rubric：你得给 reviewer 一份明确的检查清单（查哪些安全项、哪些兼容点、哪些回归面），它才能落到实处。没有 rubric 的 reviewer 只会泛泛地"提一些建议"，看着像审了，其实什么都没卡住。

**specialist agents（专才 agent）。** 按角色分：前端一个、后端一个、数据一个、测试一个、文档一个，每个 agent 的工具和上下文都按它的角色裁剪——给前端 agent 浏览器自动化工具，给数据 agent 数据库访问，而不是每个都塞一套全家桶。Anthropic 在长任务 agent 的研究里把这件事列为一个开放方向：目前还不清楚一个通用的 coding agent 是否在所有场景下都最好，专门的测试 agent、质量保证 agent、代码清理 agent 很可能在各自的子任务上做得更好 [A2]。注意这是被点名的"未来方向"，不是已下定论的结果。

## 为什么重要：AI 时代的变化

多 agent 不是新概念，但它现在值得认真对待，是因为有了具体数字撑着——既撑着它的好处，也撑着它的代价。

先看好处。Anthropic 报告，在他们的研究评测上，一个以 Claude Opus 4 为 lead、Claude Sonnet 4 为 subagent 的多 agent 系统，比单个 Claude Opus 4 高出 90.2% [A7]。这个提升在"宽度优先"的查询上尤其明显——那种需要同时朝多个方向并行探索的问题 [A7]。并行还直接省时间：他们发现并行工具调用把复杂查询的研究时间最多压掉了 90% [A7]。

但代价是真金白银。同一篇里给了一组很清醒的数字：普通 agent 大约消耗标准聊天 4 倍的 token，而多 agent 系统大约消耗聊天 15 倍的 token [A7]。你开一个多 agent 系统，账单不是翻一倍，是奔着十几倍去。他们由此给的判断很直接：这套只在"任务价值足够高、值得这笔算力开销"时才划算 [A7]。这不是一句客套——15 倍的 token 意味着，把多 agent 用在一个本来一个 agent 就能搞定的小任务上，是纯亏。

还有一个数字值得记住，因为它直接指向"该往哪花预算"。在 BrowseComp 评测上他们做了归因分析：token 用量本身就解释了 80% 的表现差异，模型选择和工具调用加起来只占剩下的 15% [A7]。这条很反直觉——它说的是，在这类任务上，你给系统多少 token 预算去探索，比你换哪个模型更能决定结果。多 agent 之所以强，很大程度上正是因为它通过并行，把更多 token 预算有效地铺到了更多探索方向上。

那么什么时候该上多 agent？A7 给了清楚的边界。适合的场景：需要重度并行、信息量超出单个上下文窗口、涉及复杂工具交互的问题 [A7]。不适合的场景同样明确：要求所有 agent 共享同一份上下文、或者子任务之间紧耦合的问题，对当前的多 agent 架构来说是糟糕的匹配 [A7]。这条边界很实用——如果你的任务里几个部分必须时刻看到彼此的最新状态、谁动一下另一个就得跟着改，那它根本不该被拆成独立上下文的多个 agent，拆了只会让它们互相等、互相覆盖。

## 实践方法

**先问"这任务该不该拆"，再谈怎么拆。** 拿 A7 那条边界当筛子：这任务是宽度优先、能并行、信息超窗、工具交互复杂吗？如果是，多 agent 有戏。如果它要求几路共享同一份实时上下文、或者子任务紧耦合，就别拆——一个 agent 带状态文件做完更省心。15 倍 token 的代价，只在任务价值撑得起时才付。

**让 sub-agent 回传摘要，不回传原始过程。** 这是多 agent 省 context 的核心动作，上一卷讲 context 工程时已经展开过：让 worker 在自己干净的上下文里放手探索，但回给 orchestrator 的是一份压缩好的结论，不是它翻过的所有原始材料。orchestrator 的窗口因此始终只装结论。这一招同时解决了两件事——既给了每个 worker 充足的探索空间，又没让主上下文被几路的过程撑爆。

**给每个 worker 划预算和范围。** 编排器派活时要把两样东西写死：这个 worker 负责哪一段（范围别和别人重叠），它最多能花多少（token、轮数、工具调用次数的上限）。A7 的提示工程经验里专门提到，要给 lead agent 明确的委派框架、而不是模糊的指示，还要嵌入"努力程度匹配查询复杂度"的扩展规则 [A7]——翻译过来就是：别让编排器自由发挥，把"派多少活、给多少预算"这件事规则化。

**evaluator 和 reviewer 只看产出，不看辩护。** generator + evaluator 也好，adversarial reviewer 也好，独立性是它们唯一的价值来源。让评判方拿可执行的判据去卡——测试过没过、rubric 每一项是否满足——而不是读一遍 generator 写的"我为什么这么改"。一旦评判方开始被生成方的叙述说服，这道关就废了。

**用 trace 去 debug，而不是去翻对话。** 多 agent 系统的 debug 比单 agent 难，因为 agent 的决策是非确定性的——同样的 prompt，两次可能走不同的路。A7 的做法是上完整的生产级 tracing 来诊断 agent 为什么失败、再系统性地修，同时只看模式不看对话内容 [A7]。多个 agent 并行时，没有 trace 你根本说不清是哪一路、在哪一步出的岔子。

**长任务的状态落在上下文之外。** A7 也提到，agent 系统会跨较长时间维持状态，所以错误处理很关键；他们上了基于 checkpoint 的恢复机制，让 agent 能从失败点续上，而不是整个从头再来 [A7]。这和上一卷讲长任务时的状态文件是同一个道理——状态要活在文件 / 日志里，不能活在会被压缩、会丢的对话历史里。

## 真实案例

**brain 与 hands 解耦，让并行可扩展。** Anthropic 的 Managed Agents 把一个 agent 拆成三件可独立替换的东西：session（一条只记不改的日志，记下发生过的一切）、harness（那个调用 Claude、转发工具调用的循环）、sandbox（跑代码和文件操作的隔离执行环境）[A4]。关键的一步是把"大脑"（Claude 和它的 harness）和"双手"（sandbox 和工具）以及 session 日志解耦开 [A4]。

这个解耦直接服务于"并行代理可扩展"。harness 被挪到容器外面，把容器当成一个工具来调用，签名就是 `execute(name, input) → string`；容器要是挂了，harness 把它当成一次工具调用错误处理，Claude 可以换一个新开的环境重试 [A4]。这样一来，"大脑"可以通过无状态的 harness 独立扩展，"双手"则按需现开 [A4]——你要并行十路 agent，不必预先开十个重型容器占着，而是用到哪只手才开哪只。解耦还顺带改善了启动延迟：p50 的首 token 时间降了约 60%，p95 降了 90% 以上，因为推理不必再等容器开好才开始 [A4]。

这里还藏着一个对多 agent 尤其重要的安全收益。解耦之前，Claude 生成的不受信代码和凭据跑在同一个容器里，注入风险很大；解耦之后，token 永远不进 Claude 生成代码所在的那个 sandbox [A4]。当你并行跑很多个会自己写代码、自己执行的 agent 时，这种"手上没有钥匙"的隔离就不是锦上添花，而是底线。

**把规划、生成、评估拆给不同角色。** A2 那篇长任务研究里，初始化和推进就是拆开的两个角色：一个 initializer agent 在第一轮把环境搭好（写 `init.sh`、建进度文件、做初始 git 提交），之后每一轮由 coding agent 接力，挑一条没完成的功能往前推、做完留下结构化的交接 [A2]。在克隆 claude.ai 的例子里，initializer 先把用户那句笼统的要求展开成一份超过 200 条的功能清单，每条先标成"未通过"，存成 JSON 文件（实测模型改 JSON 比改 Markdown 更克制）[A2]。

这篇文章自己把"该不该再往多 agent 拆"当成一个明确的开放问题留着：目前还不清楚一个通用 coding agent 是不是在所有场景下都最好，专门的测试 agent、质量保证 agent、代码清理 agent 可能在各自的子任务上做得更好 [A2]。这是个诚实的态度——把规划/生成/评估拆给不同角色，方向上有道理，但"拆得越细越好"还没有被证明，别当成定论照搬。

## 常见反模式

- **多 agent 崇拜。** 以为只要 agent 够多，活就一定干得好。A7 的 15 倍 token 摆在那儿——把多 agent 用在一个 agent 就能搞定的任务上，是纯烧钱 [A7]。先问"这任务该不该拆"，答案是"不该"的时候，一个 agent 带状态文件干完最省。
- **拆了紧耦合的任务。** A7 明确说了，要求所有 agent 共享同一份上下文、或者子任务紧耦合的问题，是当前多 agent 架构的糟糕匹配 [A7]。这种活硬拆成独立上下文，几路只会互相等、互相覆盖。
- **上下文碎片。** 把任务拆给几个 agent，却没设计它们怎么交换信息，结果每个 agent 只看到自己那一小块，谁也没有全局，综合出来的结果互相矛盾。拆上下文的前提是想清楚"谁该看到什么、结论怎么回传"。
- **预算失控。** 不给 worker 划 token / 轮数上限，编排器一高兴就无限派活、无限深挖。15 倍的基数上再失控，账单很快就难看。每个 worker 都要有明确的预算边界。
- **多路同时改同一处。** parallel exploration 里最致命的一种：让并行的几路 agent 同时去动同一处代码，它们会互相覆盖，最后你拿到的是一团没人能解释的乱码。并行只用来"探"，"改"要么收敛到一路、要么靠编排器串行化。

## 检查清单

- 这个任务是宽度优先、能并行、信息超窗、工具交互复杂吗？还是几部分紧耦合、必须共享上下文——后者就别拆？
- 我算过多 agent 的 token 代价吗？这任务的价值，撑得起大约 15 倍于普通聊天的开销吗？
- 每个 worker 有没有明确的**范围**（不和别人重叠）和**预算**（token / 轮数 / 工具调用上限）？
- sub-agent 回传给主 agent 的，是**压缩好的摘要**，还是它翻过的全部原始材料？
- evaluator / reviewer 是**独立**评判产出本身，还是会被 generator 的解释带着走？reviewer 有没有一份明确的 rubric？
- 并行的几路，是只"探"不"改"吗？有没有可能两路在同时动同一处？
- 多 agent 的状态和 trace，落在**文件 / 日志**里了吗，够不够事后定位是哪一路、哪一步出的问题？

## 自测

- 你上一次想"再开几个 agent"，是因为这任务真的宽度优先、能并行，还是只是因为"多开几个总没坏处"？后者的话，这笔 15 倍的 token 你打算怎么跟自己交代？
- 如果你现在并行的几路 agent 里有两路都改了同一个文件，你的系统会发生什么？你能在 trace 里一眼看出是哪两路撞了吗？答不上来，说明你的多 agent 还停在"人多"，没到"组织好"。
