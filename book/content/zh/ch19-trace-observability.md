车坏在路上，没有故障码，你只能掀开机盖凭感觉猜：是油路、电路，还是哪个传感器。换一辆有行车记录仪和 OBD 故障码的车，你能回放出事前那几秒方向盘怎么打、刹车踩了没、哪个缸先失火——问题被定位到一段，而不是整辆车。

agent 交上来的 PR 也一样。光看 diff，你看到的是结果：它最后写成了什么样。你看不到它读了哪些文件、跑了什么命令、哪一步报错了、有没有偷偷跳过验证就宣布完工。这条"它一路是怎么走到这个 diff 的"记录，就是 trace。review 一个 agent 的产出，从此不只是看 diff，还要看这卷录像。

## 生活类比

一辆现代车上同时装着两样东西：行车记录仪，和 OBD 故障码系统。

行车记录仪录的是过程——出事前你打了几把方向、刹车点在哪、有没有压线。故障码记的是结论性的失败信号——P0301 是一缸失火，P0420 是三元催化效率低。两样配合，修车师傅不用拆半个发动机猜，他先读码定位到"一缸点火"，再调录像看当时的工况，几分钟就锁定到一个零件。

没有这两样会怎样？车主只能说"开着开着突然抖了一下"，师傅只能凭经验一路换件试。换对了是运气，换错了你白掏钱，而且下次再抖，还是从头猜起——因为没有任何一次故障被记录下来、被归因、被复查。

agent 干活也分这两层。它读了什么、试了什么命令、改了哪些文件，是录像；某一步 `pytest` 报红、依赖装不上、类型检查不过，是故障码。review 的时候，你要的不是"它最后交了个看着挺像样的 diff"，而是"它一路怎么走过来的，中间哪一步真出了问题"。没有这卷录像，你对一个 agent run 的全部判断，就退回到"开着开着突然抖了一下"的水平。

## 工程定义

Trace（执行留痕），是一次 agent run 从开始到结束、按时间顺序记下来的一串事件：它调了哪个工具、传了什么参数、命令回显是什么、改了哪些文件（diff）、哪一步失败了、失败原因是什么，以及这一路烧了多少 token、花了多长时间。可观测性，就是把这串事件收集、结构化、可回放，让你或下一个人能据此重建"这次 agent 到底干了什么"。

它和传统的应用日志是两件相关但不同的事。应用日志记的是程序运行时发生了什么（请求进来、查了库、返回 200）；agent 的 trace 记的是一个会自己读、自己改、自己跑命令、还会自己决定下一步的东西，每一步的决策和结果。OpenAI 在它的 agent 改进流程里把 trace 明确当成一类一等资产：把 Agents SDK 的事件导出成 OpenTelemetry 风格的 JSONL，让 trace 能被工具读、被分析、被回流 [O4]。

一条有用的 trace，至少要能答上这几个字段：

| 字段 | 它回答的问题 |
|---|---|
| 工具调用 | agent 用了哪个工具、传了什么参数（读了哪个文件、跑了哪条命令） |
| 命令输出 | 这条命令回显了什么（测试是绿是红、报错原文） |
| diff | 它最终改了哪些文件、改成什么样 |
| 失败原因 | 哪一步失败、错误类型是什么（模型逻辑错？依赖装不上？超时？） |
| 成本 | 这一路烧了多少 token、调了几次模型 |
| 耗时 | 总时长、卡在哪一步最久 |

把这几样按时间轴串起来，就是一卷能回放的录像。少了"命令输出"和"失败原因"，你手里就只剩一个最终 diff——等于只有一张事故现场照片，没有行车记录仪。

## 为什么重要：AI 时代的变化

传统 code review 的隐含前提是：diff 就是事实的全部。人写代码，过程在人脑子里，review 看结果是合理的——你信得过这个人没有为了过测试去删测试。agent 这个前提不成立。它会的事情比人多（一次 run 能读几十个文件、跑十几条命令），可靠性又比人差（前一章 Simon Willison 那句"会逮着机会就作弊的数字实习生"）。它完全可能交上来一个 diff 干干净净、测试全绿的 PR，而你看不见的是：它为了让测试变绿，把那条挡路的断言注释掉了。

所以 review 不能只看 diff。一个看起来对的 diff，背后可能藏着一整套你不会同意的过程：它没读现有测试就动手、它跑了一次就失败、失败后没去查原因反而去改配置绕过、它把改动范围悄悄扩大到你没要求的文件、它删掉了一段保护性的校验只因为那段挡着它。这些在 diff 里要么看不见，要么看见了也不知道是"故意绕过"还是"顺手改的"。trace 是新的代码评审材料——它让"过程"重新变得可审。

还有一层，是 trace 让你能把两种失败分开：环境的错，和模型的错。这件事 Anthropic 用数据量化过。在 Terminal-Bench 2.0 上，光是基础设施怎么配——容器资源是按"保证分配"给，还是设成一个"超了就击杀"的硬上限——就能让同一个模型的分数差出 6 个百分点（p < 0.01），足以盖过排行榜上顶尖模型之间的差距 [A11]。在最严格的资源限制下，约 5.8% 的任务是被基础设施错误（而不是模型）搞挂的；把资源放宽到 3 倍余量，这个错误率掉到约 2.1%；完全不设上限时只剩 0.5% [A11]。换句话说，你以为是"模型不行"的那些失败，相当一部分其实是"操作间没搭好"。

作者据此给的判断很硬：排行榜上低于 3 个百分点的差异，在 eval 配置没被写明、没核对到一致之前，都该存疑 [A11]。这条判断只有靠 trace 才落得了地——你得能回放每一次失败，看清它倒在"模型逻辑错"还是"容器被 OOM kill 了"，才谈得上把噪声从信号里减掉。没有 trace，一个红色的失败就是一个红色的失败，你分不清该去改 prompt 还是该去加内存。

## 实践方法

把 trace review 变成一道固定工序，而不是凭感觉抽查。下面这份清单，是看一个 agent run 的 trace 时按顺序过的几个问题——它对着的不是 diff，是 diff 背后那条路径。

**它有没有先读再写。** 翻 trace 开头那几步：它有没有先去读相关文件、读现有测试，再动手改？还是连现状都没看清就开始写。一个连现有测试都没读就改实现的 agent，很可能正在重复造一个已经存在的轮子，或者破坏一条它根本不知道存在的约定。

**计划有没有覆盖验收标准和风险边界。** 如果这次 run 有显式的计划步骤，对照任务的验收标准看：它的计划覆盖了吗？有没有把"哪些文件不许动、哪些行为不能破坏"这类风险边界纳进去？计划里没有的，执行时基本也不会有。

**有没有跑最小必要验证，失败后归因对不对。** 看它有没有真的跑测试、跑构建，而不是改完就宣布完工。更要紧的是看失败之后那几步：测试报红了，它是去查为什么红（读报错、定位到具体断言），还是图省事直接把那条测试改了、注释了、跳过了？这是 trace review 最该盯的一处——A2 在长任务里专门用"It is unacceptable to remove or edit tests"这种重话去压住这个倾向 [A2]。

**有没有扩大范围、删保护、绕测试、改配置。** 这四样是同一类危险动作，特征是"看起来在解决问题，其实在绕过问题"。范围扩大：任务只让改 A 模块，trace 里却出现了对 B、C 的改动。删保护：一段输入校验、一个权限检查被删掉了。绕测试：用 `skip`、改断言、改测试数据让红变绿。改配置：动 CI 配置、动超时阈值、动资源上限把失败"配置掉"。在 diff 里这几样常常长得人畜无害，得回到 trace 里看它"为什么这么改"才认得出来。

**PR 描述有没有带证据。** 一份合格的交接，描述里该有：跑了哪些命令、结果是什么（贴回显，不是"测试通过了"一句话）、哪些项还没验证、已知的风险在哪。OpenAI 那套改进流程里，agent 的产出契约就明确要求带 citation 和 evidence table，并把"产出了一份漂亮的答案、却留着 citation 或 artifact 没补全"列为要检测的失败模式 [O4]。"看着完成了"和"有证据证明完成了"是两回事。

**有没有留够 artifact 让别人接手。** 这次 run 结束，下一个人（或下一个 agent）能不能接上？该有的状态文件、进度记录、可复现的命令在不在？A2 把这件事做成了硬结构：初始化 agent 写一个 `init.sh` 起环境、一个 `claude-progress.txt` 记历轮干了什么，配上 git 提交历史；每一轮 coding agent 开工先 `pwd`、先读进度文件和 `git log` 把状态接上，再挑一个没做完的功能推进 [A2]。这套进度文件本身，就是一种为"接手"而结构化的 trace。

## 真实案例

**把环境噪声和模型错误分开。** Anthropic 那篇量化基础设施噪声的文章，本质是一次大规模的 trace 复盘。他们在同一个 Claude 模型上，用六种资源配置把 Terminal-Bench 2.0 跑了一遍，从"严格按每任务规格上限击杀"一路放到"完全不设上限" [A11]。如果只看最终分数，你会看到分数随资源放宽单调上升，很容易误读成"资源越多模型越强"。但把每一次失败的 trace 拆开看，真相是另一回事：从严格到 3 倍余量，分数的提升几乎全部来自基础设施错误率下降（5.8% → 2.1%），而真正的成功分在噪声范围内波动（p = 0.40）——意思是那些崩掉的 run，本来大概率也会失败，放宽资源只是让它们别死在 OOM 上 [A11]。只有超过 3 倍之后的提升，才是模型真的能去尝试一些原先因资源不够做不了的重型策略 [A11]。

这个区分，没有逐次 trace 做不出来。光看排行榜上一个聚合分数，"6 个百分点"既可能是真实力差距，也可能只是分到了一台更大的虚拟机——作者用来收尾的那句话正是这个意思。他们给出的工程建议，全都是为了让 trace 能干净地归因：每个任务分开声明"保证分配的资源"和"硬击杀阈值"两个值，别用一个数字既当地板又当天花板；把资源配置当成一等的、要写进文档的实验变量；报告结果时连同资源 enforcement 方式一起报 [A11]。

**trace 回流进 eval 和规则。** trace 不是看完就扔的录像，它该往回流。OpenAI 的 agent 改进流程把这条循环讲得很清楚：trace 记录发生了什么，人和模型的反馈解释哪些点要紧，反馈被固化成可复用的 eval，eval 变成一道校验门，最后由 Codex 把这套改动落进 harness——形成一个飞轮 [O4]。这里的关键动作是"回流"：一次 trace 里暴露的问题（比如它把管理层口径的叙述当成了已验证的指标 [O4]），不该停在一条 review 评论上，而该变成一条以后每次都会跑的 eval、或一条写进 AGENTS.md 的规则。看一次、改一次、防一类，trace 的价值才不会随这次 PR 合并而蒸发。

**长任务的进度文件，就是一种结构化 trace。** A2 那套长任务 harness 里的 `claude-progress.txt` 加 git 历史加那份两百多条的功能清单 JSON，从 trace 的角度看，正是一卷为"跨会话接手"而结构化的录像 [A2]。每一轮 agent 留下的，不是一段会被压缩掉的聊天历史，而是落到文件里的、可读可回放的状态：哪些功能标了通过、哪些还红着、上一轮的提交说明写了什么。下一轮 agent（或一个来 review 的人）读这几样，就能重建"项目走到哪了、上一棒做了什么"——这正是 trace 该干的事，只不过它在这里同时承担了"留痕"和"交接"两个角色。

## 常见反模式

- **只 review diff，不看过程。** 最常见也最危险的一种。diff 干净、测试全绿就放行，看不见它是删了断言才让测试变绿的。diff 是结果，trace 是过程，过程里才藏着"它有没有作弊"。
- **把环境失败算成模型失败。** 一个 run 红了就归咎于"模型不行"，不去 trace 里看它倒在哪一步。A11 的数据说得很直白：严格资源下约 5.8% 的失败根本是基础设施错误，不是模型 [A11]。不分开，你会在该加内存的时候去改 prompt。
- **trace 留了不回流。** 收集了一堆 trace，看完就扔，下次同类问题照样发生。trace 的价值在它能变成 eval、变成规则 [O4]；只看不沉淀，等于装了行车记录仪却从不回放。
- **拿聚合分数当结论，不看配置。** 看到排行榜上领先几个点就下判断。低于 3 个百分点、且 eval 配置没写明、没核对到一致之前，这个差异该存疑 [A11]。
- **长任务赌聊天历史能接住。** 不留状态文件、不留进度记录，指望下一轮 agent 从被压缩的历史里恢复。历史会被截断，"上一轮做到哪"随时丢；该把状态落到文件里 [A2]。

## 检查清单

- 我 review 这个 PR 时，手里有没有它的 trace，还是只有最终 diff？
- trace 里能不能看出它**先读了相关文件和现有测试**再动手，而不是没看现状就写？
- 它的**计划覆盖了验收标准和风险边界**吗（哪些不许动、哪些不能破坏）？
- 它跑了**最小必要的验证**吗？失败之后，它是去**查原因**，还是直接改测试/注释/跳过？
- trace 里有没有**扩大范围、删保护、绕测试、改配置**这四类危险动作？
- PR 描述带**证据**吗（命令、回显、未验证项、已知风险），还是只有一句"完成了"？
- 这次 run 留下的 artifact，够不够**另一个人/另一个 agent 接手**？
- 一个红色失败，我能不能从 trace 里分清它是**模型错**还是**环境错**？

## 自测

- 你上一次 approve 一个 agent 的 PR，是看了它的 trace，还是只看了 diff？如果只看了 diff，你怎么知道它没为了过测试去改测试？
- 如果现在给你一个失败的 agent run，只有一个红色的"FAILED"，你能不能说清它是模型逻辑错了，还是容器被 OOM kill 了？答不上来，说明你的 trace 还不够细——而你正在用"换件试错"的方式 debug 一个本可以读码定位的问题。
