面试时一个人能写出漂亮的演示代码，不代表他三个月后能在你的代码库里稳定干活。试用期考核的意义，正在于把"面试当场的表现"换成"接了真实任务、干满一段时间的表现"。

测试和验证就是软件工程里的那个试用期考核。过去它考的是代码：这个函数对不对、这几个模块能不能合作、用户走完一遍流程会不会崩。agent 进来之后，被考的对象悄悄变了——你既要继续考代码，更要开始考那个写代码的 agent：它有没有听懂任务、有没有越权、有没有偷偷跳过验证。这一章先把测试的老基础讲透，再讲这层重定价。

## 生活类比

招人的时候，面试那一两个小时能看出来的东西其实有限。题目是准备好的，环境是干净的，对方状态也绷得最紧。真正决定要不要留人的，是试用期：把他放进真实项目，给真实的活，看他三个月里怎么干——遇到没见过的模块会不会乱来、被打断之后能不能接上、交上来的东西要返工几次。

试用期考核和面试是两件事，差别有三层。

一是**持续**，不是一锤子。面试是一次性的快照；试用期是隔一段时间复盘一次，看的是稳定性，不是某一天的高光。

二是**看真实任务**，不是看演示。演示是他挑好的、练熟的；真实任务是你派的、他没见过的。一个只会做演示题的人，到了真项目里照样抓瞎。

三是**既看结果，也看过程**。活干完了只是一半。另一半是：他走了哪条路？有没有图省事跳过该做的检查？有没有动不该动的东西？一个把活做对了但一路违规操作的人，和一个规规矩矩做对的人，对你的风险完全不同。

把这三层记住。下面讲的所有东西——单元测试、回归测试、边界用例、eval——本质上都是在回答"怎么把这场试用期考核做扎实"。

## 工程定义

先把基础讲清楚，不假设你已经会。

**测试**，就是给一段代码喂一组输入、检查它的输出是不是你期望的。最常见的分法是按"测多大一块"分三层，业内叫它测试金字塔——之所以叫金字塔，是因为越往下的测试应该越多、越往上越少。

- **单元测试（unit test）**：测一个最小的单元，通常是一个函数。比如一个 `calcDiscount(price, coupon)`，你喂进去一个价格、一张券，断言它算出来的折后价正好对。它快、便宜、定位准——挂了你立刻知道是哪个函数错了。金字塔最宽的一层，应该最多。
- **集成测试（integration test）**：测几个模块凑在一起能不能合作。单个函数都对，不代表它们拼起来对——结算模块调用了优惠券模块、又调用了库存模块，三者之间传的字段名、单位、状态对不对得上，这是单元测试照不到的。
- **端到端测试（end-to-end test，E2E）**：模拟一个真实用户，从头到尾走完一条关键流程。比如"用户加购物车 → 输入优惠码 → 下单 → 看到正确的实付金额"，整条路串起来跑一遍。它最贴近真实、也最慢最脆，所以放在金字塔顶端，只留给少数几条最关键的流程。

在这三层之外，还有两个概念是这一章的重点。

**回归测试（regression test）**：新功能上线后，确保那些早就做好、绝不能坏的旧功能没被碰坏。"回归"指的就是"本来好的东西退回到坏了"。它的特征是几乎应该永远全绿——一旦有一条红了，信号很明确：你这次改动碰坏了不该碰的地方。Anthropic 把这件事和"能力考核"分得很清：能力 eval 问的是"这个 agent 还能做到什么"，起点分数应该低，给团队一座山去爬；回归 eval 问的是"它原来会的，现在还会不会"，应该接近满分，掉分就说明出事了 [A10b]。

**边界用例（edge case）**：正常路径之外的那些情况。代码写顺手了，最容易只覆盖"一切正常"的主路径，而真实世界的 bug 几乎都藏在边界上——空数据、权限不足、网络中途失败、用户连点两下造成重复提交、超长或超大的极端输入。把这些列全、各写一条测试，是测试功底里最见水平的地方。

到这里，测试的老定义就完整了：用可执行的断言，把"代码应该怎么表现"固定下来，让任何一次改动都能自动验一遍。下一节讲，当代码改由 agent 来写，这套东西的重心往哪挪。

## 为什么重要：AI 时代的变化

老的测试是为"人写代码"设计的。人会累、会手滑、会忘记某个边界，于是我们用测试网住这些失误。agent 不太会手滑，但它会以新的方式出错，于是测试的重心跟着挪了。

**测试从"事后验收"前移成"规格"。** 过去测试常常是写完代码补的。在 agent 这边，先写清"怎么算对"反而是最划算的起手式——因为那份判据本身就是你交给 agent 的任务规格。Anthropic 观察到一个很具体的好处：两个工程师读同一份初始规格，对"边界情况该怎么处理"完全可以得出不同理解，而一套 eval 把这种歧义直接消解掉了 [A10b]。换句话说，你写边界用例的过程，就是在逼自己把任务讲清楚——过期券到底算不算用过、退款要不要返还优惠，这些你不写进考核，agent 就只能猜。

**eval 是一种能设计、能调试、能复用的能力，不是一次性打分。** 这里要把三个容易混的词掰开。测试（test）是工程里早就有的；benchmark 是一套标准化、用来横向比模型的题（比如 SWE-bench Verified）；eval 则是你为自己这个 agent 攒的一套考核，长期养着。OpenAI 把一次 eval 拆成一串很清楚的步骤：一个 prompt → 一次被记录下来的运行（trace + 产出物）→ 一小组检查 → 一个能跨版本比较的分数 [O5]。重点在"长期养着"和"能跨版本比较"：模型升级、prompt 改动、skill 迭代，你都拿这套考核去量它变好了还是变坏了，而不是凭手感说"这版感觉更快了"。Anthropic 也强调同一件事——eval 的价值是复利型的，前期成本看得见，收益是后面慢慢累出来的 [A10b]。

**权重从"测代码对不对"上移到"验证 agent 这个人靠不靠谱"。** 代码对不对，单元测试照旧管。但 agent 会带来代码之外的问题，而这些恰恰是老测试照不到的：它有没有听懂任务（还是做了个貌似相关但不是你要的东西）、有没有越权（动了不该动的文件、调了不该调的工具）、有没有跳过验证（活看着干完了，可它根本没跑测试）。这就是为什么 OpenAI 把考核目标分成四类，不只看结果：结果（任务完成没有）、过程（有没有调用该调的工具、按该走的步骤走）、风格（产出符不符合约定）、效率（有没有空转、反复重跑命令、烧掉过多 token）[O5]。光看最后那个产出对不对，是验收不了一个 agent 的——你得看它一路是怎么走过来的。

**AI 写的代码爱覆盖主路径、漏边界路径；AI 写的测试爱"迎合实现"写假测试。** 这两条要分开防。前者：让 agent 写实现，它大概率把"一切正常"那条路写得很顺，却把过期、超额、并发这些边界漏掉——所以边界用例这件事，在 agent 时代不是更不重要，是更要由人来兜底地列全。后者更隐蔽：让 agent 既写实现又写测试，它会倾向于写出"恰好能通过它自己那份实现"的测试——实现错了，测试跟着错，两边一起把 bug 盖住，跑起来全绿，你却被骗了。Anthropic 给过一条相关的硬经验：如果一个任务在很多次尝试里 0% 通过（0% pass@100），对前沿模型来说，这通常不是模型不行，而是题目或判分器本身坏了 [A10b]。反过来也一样——全绿也未必是真对。

**别把基础设施的噪声当成模型不行。** 这是 agent 测试里最费钱的误判。agent 跑活要真装依赖、真跑测试、真起子进程，于是运行环境本身成了"问题的一部分"，而不是被动的背景。Anthropic 量化过这件事：在 Terminal-Bench 2.0 上，光是基础设施配置的差异就能让分数差出 6 个百分点（p < 0.01）——足以盖过排行榜上顶尖模型之间的差距；在严格的资源限制下，约 5.8% 的任务是被基础设施错误、而不是模型搞挂的 [A11]。他们给这事配了句很到位的话：榜单上领先几分，"可能是真实力差距，也可能只是对方那台虚拟机更大" [A11]。落到你身上就是：一条 flaky test、一次依赖装失败、一个 runner 抽风，都不该被记成"agent 不行"。先把环境噪声隔开，再谈模型对错。

## 实践方法

把上面的东西落成一套能动手的流程。

**先定"怎么算对"，再让 agent 跑。** 派活之前，把验收标准写下来，而且尽量写成可执行的。OpenAI 的建议很朴素：在写 skill 之前就把成功定义清楚，按结果 / 过程 / 风格 / 效率四类列出必过项，清单要小、要聚焦在最在意的几条上 [O5]。这就是你那份试用期考核表。

**以"优惠券结算"为例，把边界用例列全。** 主路径只有一条："有效券 + 正常订单 → 折后价正确"。值钱的是下面这些边界，每一条都该有一条测试盯着：

- **过期券**：券过了有效期还能不能用？
- **重复使用**：同一张一次性券，同一单里用两次、或下单失败重试时再用一次，会不会被算两遍？
- **订单退款**：用了券的订单退款，优惠那部分怎么处理、库存和券额怎么回滚？
- **优惠超过总价**：50 元的券打到一个 30 元的订单上，实付会不会算成负数？
- **不同币种**：券面额和订单币种对不上时怎么折算？（平台货币若以最小单位存储，这里尤其容易出单位错。）
- **会员叠加**：会员折扣和优惠券能不能叠、按什么顺序算？
- **并发抢券**：限量券在同一瞬间被多个请求同时抢，超发了没有？

这七条里，前沿模型写实现时最容易漏的恰恰是后面几条——退款回滚、负数兜底、并发超发。列清单这件事，人比 agent 更该上心。

**判分用确定性检查打底，模糊的地方再加 rubric。** 能用确定性判分就用——快、便宜、可复现、好调试。OpenAI 的做法是：用 `codex exec --json` 把 agent 这一趟实际干了什么录成结构化事件流，然后写确定性检查去问"它到底跑没跑 `npm install`""有没有真的生成 `package.json`"，而不是只看最后输出像不像 [O5]。真正主观的部分——代码质量、风格是否符合约定——再叫一个模型按一份清楚的 rubric 打分。Anthropic 把判分器分成三类供你搭配：确定性的代码判分（字符串匹配、单元测试、静态分析）、模型判分（按 rubric 打分、自然语言断言）、人工判分（最准但最贵，用来校准模型判分）[A10b]。

**判"它产出了什么"，别死磕"它走了哪条路"。** 这是 Anthropic 反复强调的一条反脆弱经验：有种本能是去检查 agent 是不是严格按某个工具调用顺序走的，但这种检查太死板、太脆——agent 经常找到设计者没预料到的有效路径 [A10b]。为了不惩罚这种创造力，多数时候该判它最终产出对不对，而不是它有没有照你想的步骤走。但这条有例外：当"越权""跳过验证"本身就是你要防的，过程就必须查（比如"有没有在没跑测试的情况下就声称完成"）。

**留一条负样本。** O5 里有个容易被忽略的细节：考核里要专门放一条"这个 skill 不该被触发"的用例，去抓"误触发"——agent 太积极、在用户只想小改一下时却新建了整个项目 [O5]。映射到结算场景：你不光要测"该打折时打了折"，还要测"不该打折时没乱打"，否则你会调出一个见谁都打折的东西。

**每条 trial 从干净环境起跑。** agent 行为本身就有随机性，环境再带噪声，结果就没法读了。Anthropic 的硬规矩是每次 trial 都从一个干净环境开始，不要有遗留文件、缓存这类共享状态——他们甚至遇到过 Claude 通过翻上一轮 trial 留下的 git 历史拿到不该有的优势 [A10b]。

## 真实案例

**一道用了约 18 个月的考题，被一次模型升级直接报废。** Anthropic 性能工程团队从 2023 年底起，用一道 take-home 编程题招人：一个模拟假加速器（接近 TPU 架构）的 Python 题，要求选手做并行树遍历的优化，用上手工管理的 scratchpad 内存、VLIW 指令打包、SIMD 向量化、多核并行这些特性。这道题好用了大约 18 个月，跑过 1000+ 次作答，招进的人里有把 Trainium 集群拉起来、从 Claude 3 Opus 起每一代模型都参与发布的工程师 [A19]。

然后模型追了上来。到 Claude Opus 4，它在时限内已经胜过大多数人类应聘者；到 Opus 4.5，在两小时的窗口里已经能追平顶尖的人类选手 [A19]。一道筛过上千人的题，就这么失效了。

有意思的是作者 Tristan Hume 怎么出下一版：他用"被测的那个 AI"去找它自己的弱点——拿 Claude Opus 4 跑这道题，看它在哪儿卡住，把它卡住的地方当成新题的起点 [A19]。但这条路也不是一劳永逸：他基于真实 kernel 优化工作出的"数据转置"新题，被 Opus 4.5 找到了一个没预料到的解法——它不去转置数据，而是转置计算本身——打了补丁之后它还在继续找新解 [A19]。这件事的教训不是"AI 太强了出不了题"，而是：当考官和被考者是同一个 AI，出题这件事本身变成了一个持续的、对抗性的循环。

**给 agent 设计 eval，比想象的难——两个被判错的例子。** 第一个：Opus 4.5 在一个订票 benchmark（τ2-bench）上解一道订机票的题时，钻了政策的一个空子，给出的方案对用户其实更好；但按题目原本写死的判分，它"失败"了 [A10b]。模型没错，是这道题没料到会有更优解。第二个更隐蔽：Opus 4.5 在 CORE-Bench 上一开始只拿 42%，一位 Anthropic 研究员深挖之后发现是判分本身的毛病——死板的判分把 `96.12` 判成错的（其实正确答案是 `96.124991…`，模型答得没问题），外加一些题目本身说得含糊、还有些题随机得根本没法精确复现；修掉这些 bug、换一个约束更松的 scaffold 之后，它的分数跳到了 95% [A10b]。

这两个例子合起来是同一条警告：当分数不涨，你不能默认是 agent 不行——很可能是你的考题或判分器坏了。Anthropic 因此立了一条规矩：在有人真去翻细节、读几份 trace 之前，eval 的分数不当真 [A10b]。读 trace 不是可选项，是这门手艺的核心技能。

## 常见反模式

- **拿 benchmark 当生产验收。** 模型在 SWE-bench Verified 这种题上从 40% 涨到 80% 多 [A10b]，是好消息，但那是横向比模型的标准题，不是你这套依赖、权限、网络、业务规则下的验收。真实失败几乎都来自这些边界，不来自标准题。
- **让 agent 既写实现又写测试，然后照单全收。** 它会写出"恰好通过自己实现"的迎合式测试，实现错了测试跟着错，两边一起把 bug 盖住。边界用例和关键断言要由人把关，至少人要审。
- **只测主路径。** 优惠券结算只测了"有效券正常下单"，过期、超额、退款、并发全没测。AI 写的实现尤其爱漏这些，主路径绿不代表没事。
- **把环境噪声记成模型不行。** 一条 flaky test、一次依赖装失败、一个 runner 抽风，就下结论"这模型不行"。先隔开基础设施噪声——榜单上那点差距，可能只是对方虚拟机更大 [A11]。
- **死磕工具调用顺序。** 把 eval 写成"必须按这个步骤走"，结果 agent 找到了更好的有效路径反被判错 [A10b]。多数时候判产出，别判路径——除非越权和跳过验证本身就是你要防的。
- **eval 写完就不管了。** 当成一次性资产，不随真实失败往里加用例。它该是个活的清单——每一次线上踩的坑，都补一条进去 [O5][A10b]。

## 检查清单

- 派活之前，我有没有把"怎么算对"写成可执行的判据，而不是事后扫一眼"看着对"？
- 我的边界用例列全了吗？（空数据、权限不足、网络失败、重复提交、极端输入——以结算为例：过期、重复、退款、超额、币种、叠加、并发？）
- 我的 eval 除了看结果，有没有看过程（越权？跳过验证？）和效率（空转、反复重跑）？
- agent 写的测试，关键断言和边界用例是我审过的，还是它自己写自己验、我直接收货？
- 这次失败，我分清了是模型错、还是 flaky test / 依赖 / runner 这类基础设施噪声吗？
- 每条 trial 是从干净环境起跑的吗？有没有遗留状态在偷偷帮它或者坑它？
- 分数不涨的时候，我有没有真去读几份 trace，确认是 agent 的问题、不是判分器的问题？

## 自测

- 你上一次让 agent 干活，是先写好了"怎么算对"再放它跑，还是它跑完你才临场想"这算不算干完了"？
- 把"优惠券结算"这个场景给你，你能当场列出七到八条边界用例吗？哪几条是 AI 写实现时最可能漏的？
- 你最近一次说"这个模型不行"，是真读了 trace 确认的，还是有可能你撞上的只是一台更小的虚拟机、一条 flaky test？
