到这一章为止，前面几章把"工作方式正在变"这件事讲清楚了：你不再主要靠手敲代码交付，而是靠把一个能力很强、但不懂你家规矩的新人安排好、约束好、验收好。这一章不展开任何一个细节，它只做一件事——把后面所有卷会反复用到的工程原则，先列一遍，像一本书的目录。

每条原则一句话说清是什么，再一句说清为什么。具体怎么落地、有哪些坑、给什么样的工件，留到对应的章节去讲。你现在不需要全记住，只需要心里有这张骨架：后面每一章，基本都是在把这十二条里的某一条拆开、做实。

## 生活类比

你要开一家餐厅。手艺好的厨师、漂亮的菜单、网红的装修，这些都重要，但都不是第一位的。开张前真正决定这家店能不能长久活下去的，是一套**店规**——而且是写下来、贴出来、人人照做的店规。

一套像样的店规长这样：

- **食材索证**：每批进来的肉菜，供应商、批次、日期登记在册，出了问题能倒查到是哪一批、哪一家。
- **出餐前必尝**：菜不是炒完直接端走，传菜口有人尝一口咸淡、看一眼卖相，不合格退回去重做。
- **收银对账**：每天打烊，营业额对得上小票，对不上当场查，不拖到月底。
- **退菜规矩**：客人说这道菜不对，怎么退、谁有权退、退了怎么记账，事先写死，不靠当时吵。
- **后厨禁区**：哪些动作绝对不许（地上掉的菜捡起来接着用、过期料"差不多还能用"），白纸黑字，不留模糊地带。
- **试用期考核**：新来的厨师，不是面试当场炒一盘惊艳就留下，是看他试用期里一连串真实出餐稳不稳。

这些规矩没有一条是在教厨师怎么颠勺。它们是在规定**这家店怎么运转**——食材怎么来、活怎么验、错怎么退、人怎么考。规矩立在前头，后面换菜单、换厨师、开分店，骨架都不变。规矩立不住，厨师再好，这家店也是早晚出事。

前沿 SDE 的工程原则，就是 agent 时代的这套店规。不是教模型怎么写代码——那是它的本事；是规定你这套"用 agent 交付软件"的体系怎么运转：活怎么交代、规矩写在哪、错怎么撤、谁来验收、什么交给人判断。这一章先把店规挂上墙，后面各卷再一条条做实。

## 工程定义

这里说的"工程原则"，是一份给前沿 SDE 的操作清单：约十二条，每条对应后面某一卷会展开的主题。它们不是口号，是判断依据——拿到一个"用 agent 干活"的方案，你能照着这十二条逐条问"这条满足了吗"。

下面逐条过。每条的格式是：**一句话是什么 → 一句话为什么**。

**① 没有 eval 就没有自治。** eval 是 agent 的试用期考核——拿真实任务给它一组可打分的检查，看它干得稳不稳。没有这套验收，你就只能靠"看起来对了"放手，而"看起来对了"恰恰是 agent 最会糊弄你的地方 [A1][A10]。

**② issue 是可执行规格，不是愿望清单。** 派给 agent 的任务要像写给"一个完全没碰过这个代码库的新人"看：标题说清在哪改、给出期望实现的例子、写明怎么算完成 [G3]。"把整个仓库改成 async/await"是愿望；"把鉴权中间件改成 async/await，并补单测验证"才是规格 [G3]。规格不清，agent 跑得越快，错得越远。

**③ repo 指令是一等工程资产。** AGENTS.md / CLAUDE.md 这类仓库级指令文件，要当代码一样进版本库、过评审、定期修剪 [A1]。它每次会话都被读进上下文，写得好不好直接决定 agent 范围外乱改多不多——这不是文档摆设，是真正影响产出的源文件。

**④ context 是预算，不是垃圾桶。** 上下文窗口装得下你整段对话、读过的每个文件、跑过的每条命令，但它填满得很快，而且越满，模型越容易"忘"掉早先的指令、越容易出错 [A1]。所以该交代的交代清楚，不该塞的别往里堆。一份什么都"重要"的巨型指令文件，等于没有指令——agent 会淹在里面，开始就近瞎猜 [O1]。

**⑤ 工具要为 agent 设计，不是把 API 直接甩给它。** 给人写的函数和给 agent 写的工具是两码事：agent 上下文有限、不确定，工具得替它把多步操作合并、把返回裁剪到只剩有用信息、把命名改成它认得的自然语言 [A8]。工具越多不一定越好——重叠、含糊的工具反而让它选错刀 [A8]。

**⑥ 先探索，再计划，再执行。** 让 agent 上来就写代码，容易写出"解错了问题"的代码。正确顺序是分三步：先只读地探索（读文件、答问题、不改动），再产出一份明确的实现计划，最后才切到执行、对着计划写 [A1]。三步分开，是为了在动手前就发现"方向就错了"。

**⑦ 自动化必须可撤销。** 凡是让 agent 自动改、自动跑的地方，都要先想好"搞砸了怎么恢复原状"——分支、快照、checkpoint、"撤销刚才那步" [A1]。GitHub 给它的云端 agent 划的硬线就是这个意思：agent 只能推到它自己那条分支、不能直接动主干，CI 必须人批准才跑 [G2]。验不了的东西，不要让它合进去 [A1]。

**⑧ sandbox 是基础设施，不是可选项。** agent 该在隔离的操作间里干活——独立的工作目录、隔离的执行环境、用完即弃的依赖和服务，而不是直接在你的真仓库、真数据上动手 [A1]。GitHub 的 agent 跑在"安全、用完即弃的环境"里 [G2]；OpenAI 那套让每个 worktree 能独立起一个 app 实例、连日志和指标都是这个 worktree 专属、任务做完就拆掉 [O1]。隔离不到位，agent 一出错就是真伤。

**⑨ 多 agent 不是越多越好。** 并行开多个 agent、worktree、agent team 能放大产出 [A1]，但"多"本身不是目的。同一个道理在工具上也成立——工具堆太多反而分散注意力、让 agent 选不对 [A8]。值得参照的是 OpenAI 那个数：三个工程师驱动 agent，人均一天约 3.5 个合并的 PR，而且团队涨到七人后人均吞吐还在涨 [O1]——靠的是协作的结构，不是单纯堆人头。

**⑩ trace 是新的代码评审材料。** 过去评审看 diff，现在还要看 trace——agent 到底读了什么、跑了哪些命令、按什么顺序、哪步失败 [O5]。一个 eval 的本质就是：一个 prompt →抓下来的一次运行（trace + 产物）→一小组检查→一个能跨版本比较的分数 [O5]。定期翻 trace，是你发现"它其实在瞎试瞎绕"的唯一办法 [A10]。

**⑪ 基础设施噪声要量化，别把 flaky 当模型错。** 依赖装不上、测试 flaky、资源被卡、网络抽风——这些都能让一次 run 挂掉，但没一样是"模型不够聪明"。Anthropic 量过这件事：在 Terminal-Bench 2.0 上，光是资源配置的差异就能让分数差出 6 个百分点，足以盖过顶尖模型之间的差距；严格资源限制下约 5.8% 的任务是被基础设施错误（而非模型）搞挂的 [A11]。分不清是环境错还是模型错，你的判断全是噪声。

**⑫ 人类判断要上移。** 人不再主要待在"敲代码"这一层，而是上移到"定优先级、把用户反馈翻译成验收标准、验收结果、在需要判断时兜底" [O1]。人擅长理解上下文、处理模糊、跨系统权衡；agent 擅长不知疲倦地执行、跑重复活、并行试多种方案——分工照这个来 [G3]。

强调一遍：这是**原则总览**，是全书的目录式骨架，这里不展开细节。下面几节给一点点落地的影子，但每条的真正展开都在后面对应的卷里。

## 为什么重要：AI 时代的变化

这十二条里，有相当一部分在 agent 出现之前就是好工程的常识——可撤销、隔离环境、清晰的验收、把规矩写下来。那为什么要专门列一遍？因为 agent 把它们**重新定了价**。

过去手写代码，一个人一天产出有限，错误是慢慢累积的，你有时间回头看、有机会顺手改。现在不一样了：agent 改多个文件、跑命令、开 PR，可以一晚上不停 [O1]。产出快了，错误也抄得一样快——一个模糊的目标、一条缺失的验收、一个坏掉的依赖，会以同样的速度被复制进更多文件。于是"约束、验证、可撤销"这几样东西的价值是**上升**的，不是下降的。

还有一层变化：人类新人会"察言观色"，看同事怎么做、踩了坑会被拉住；agent 不会，它只读你显式给它的东西 [O1]。凡是只活在团队默会知识里的规矩，对 agent 等于不存在。所以"把规矩写下来"（③）、"任务写成规格"（②）、"context 交代清楚"（④）这些原来可做可不做的事，现在变成硬约束——不写，agent 一定踩。

最后，agent 时代多出来一类过去没有的失败：**基础设施噪声**。手写代码时，"环境抽风"和"我写错了"通常分得清；agent 自己装依赖、跑测试、起子进程，运行时变成了解题过程本身的一部分，"模型能力"和"环境行为"的边界变模糊了 [A11]。⑪之所以要单列一条，就在这里——你得能量化噪声，否则会把一次 flaky 当成模型变笨，然后去"修"一个根本不存在的问题。

## 实践方法

这一章不教任何单条原则怎么落地（那是后面各卷的事）。这一节只给一个用法：**把这十二条当 checklist 用**。

拿到任何一个"用 agent 干活"的方案——不管是你自己设计的，还是别人塞给你评审的——逐条问一遍：

- eval 有没有？（①）任务写成规格了吗？（②）规矩进版本库了吗？（③）
- context 是不是该给的给了、不该塞的没塞？（④）工具是为 agent 设计的还是直接甩 API？（⑤）
- 流程是不是先探索后执行？（⑥）出错能不能撤回？（⑦）跑在隔离环境里吗？（⑧）
- agent 数量是按需还是为多而多？（⑨）trace 留够了吗、有人看吗？（⑩）
- 基础设施噪声量化了吗？（⑪）人的判断有没有上移到该上移的层？（⑫）

任何一条答不上来，就是一个你还没设计、迟早会出事的地方。这张表的用法不是"全勾才能动"，而是"哪条没勾，心里得清楚为什么没勾、风险在哪"。

## 真实案例

把这套店规真正跑通的，OpenAI 的内部实验是目前最完整的一份公开样本。

他们用一个由 agent 写代码的团队，从一个空 git 仓库起步，五个月里攒到约一百万行代码、约 1500 个合并的 PR，全程**没有一行人手写的代码**——应用逻辑、测试、CI 配置、文档、内部工具，全是 Codex 写的 [O1]。这件事能成，靠的几乎全是这十二条原则的落地：

- 他们把 AGENTS.md 当"目录"而不是"百科全书"（③④）——一份约 100 行的 AGENTS.md 注进上下文，只当地图，指向更深的真源；因为"context 是稀缺资源""什么都重要就等于什么都不重要" [O1]。
- 他们把"先探索后执行"和"计划是一等工件"做进了系统：复杂的活先落成 execution plan，连决策日志一起 check 进仓库 [O1]（⑥）。
- 他们让每个 worktree 能独立起一个 app 实例、连日志和指标都是这个 worktree 专属、用完即拆——这就是 sandbox 当基础设施（⑧），也让"启动要在 800ms 内完成"这种 prompt 变得可验证 [O1]。
- 他们把人的角色明确上移：人定优先级、把用户反馈翻译成验收标准、验收结果、只在需要判断时兜底 [O1]（⑫）。
- 而且他们诚实地标了边界：这套强烈依赖这个仓库特定的结构和工具，不该假设能直接照搬 [O1]。

另一个角度的样本是 OpenAI 的"agent 改进飞轮"：trace 显示发生了什么，反馈解释什么重要，eval 把期望固化成可复用的检查，然后让 Codex 据此产出具体的改动 [O4]。这正是⑩和①合起来的样子——trace 不是日志垃圾，是驱动下一轮改进的评审材料。

## 常见反模式

- **Prompt 崇拜。** 以为一句足够神的提示词能替掉测试、权限、隔离和验收。方向是反的：把 prompt 放进这套店规里，而不是指望一句话扛下所有约束。
- **把验收等同于"测试过了"。** agent 在"看起来做完了"的地方就停 [A1]——没有可跑的检查，"看起来对了"就是唯一信号，而那恰恰是它最会糊弄的地方。验收要有可执行判据，不是肉眼觉得对。
- **指令写成散文。** 一段"我们高度重视代码质量"的漂亮话，里面没有一条可执行规则，agent 用不上。要动作、命令、禁区。
- **把 flaky 当模型错。** 一次 run 挂了就归咎"模型变笨"，去调 prompt、换模型——结果根因是依赖没装上或资源被卡 [A11]。先量化基础设施噪声，再谈模型。
- **为多而多。** 并行开一堆 agent、堆一堆工具，以为越多越快。重叠含糊的工具反而让 agent 选错 [A8]；没有协作结构的"多"只是更快地制造混乱。
- **人停在原来的层。** agent 都能开 PR 了，人还守在"逐行写代码、逐行 review"那一层，没上移到定优先级、定验收标准——于是人成了瓶颈，而不是杠杆 [O1]。

## 检查清单

- 我能对着这十二条，逐条说清这个方案"满足了吗"吗？答不上来的那几条，是我还没设计的地方。
- 我的 agent 流程里，有没有一条**可执行的验收**，而不是"看起来对了"就放手？（①）
- 我派给 agent 的任务，更像**规格**还是**愿望清单**？（②）
- 我的规矩是**写下来进了版本库**，还是只活在我脑子里？（③）
- 出错时我有没有**撤回的路径**？agent 是在**隔离环境**里跑，还是直接动真仓库？（⑦⑧）
- 上次一次 run 失败，我有没有先分清是**环境错还是模型错**，再动手"修"？（⑪）

## 自测

- 不翻书，你能默写出这十二条里的几条？写不出来的那几条，正是你目前还没当成原则、最容易栽的地方。
- 你现在用 agent 的方式，更像"先把店规挂上墙再开张"，还是"招了个好厨师就直接开门营业、出了事再说"？
- 上一次 agent 干出来的活让你不满意，你的第一反应是"换个更强的模型"，还是"这套店规里哪一条我没立住"？后一种反应，才是前沿 SDE 的反应。
