模型再强，没有 harness 也只是"一台带键盘的随机数发生器"。harness 不是某个工具，而是你给 agent 搭的一整套可控工作环境——它决定了同一个模型，是帮你交付，还是帮你闯祸。

## 生活类比

你雇了个手艺极好、但完全不懂你家规矩的新外包。

你不会把他直接空降到客户正在住的房子里，扔下一句"装吧"就走人。你会先给他安排好几样东西：一个**工位**，一面贴着标签、趁手的**工具墙**，一份**门禁**（哪几间屋能进、哪几间不能），一间**样板操作间**（先在这儿练手，别上来就砸客户家的承重墙），一个**验收员**（活干完得有人查），一个**能撤回的约定**（搞砸了能恢复原状），还有一张**写清"做完怎么交接"的便条**。

这一整套——工位、工具、门禁、隔离操作间、验收、撤回、交接便条——合起来，就是 harness。里头没有一样是"把这个外包换成更贵的外包"。换个更强的人当然管用，可环境一乱，再强的人也会照着错误的墙猛凿。

Simon Willison 给 coding agent 起过一个很贴切的外号：一群"会逮着机会就作弊的怪异数字实习生" [S1]。重点不在"别用实习生"，而在：对付这种又能干又靠不住的人，办法不是更信任他，是给他更紧的环境。harness 就是那个环境。

## 工程定义

**Harness 工程**，是围绕 agent 建造"可控工作环境"的工程。这个环境至少包含这么几层：任务入口、上下文路由、工具、权限、sandbox（隔离执行）、状态与 artifact、eval（验收）、trace（留痕）、恢复机制、人类确认点、反馈循环。

OpenAI 把 harness 定义为 agent 周围那层"验证、反馈、恢复、工具、上下文和人类判断"的结构 [O1]；Anthropic 讲长任务 agent 时，把 harness 拆成 initializer、状态、artifact、evaluator、checkpoint、人工干预和恢复机制 [A2][A3]。说法不同，说的是同一件事：**让一个会犯错的家伙，在一个可控的盒子里干活。**

要避免两个常见误读：

- harness **不是** test harness。测试夹具干的是"喂给被测代码一组输入、断言输出"；agent harness 干的是"给一个会自己读、自己改、自己跑命令的东西搭运行时和护栏"。名字撞了，东西是两码事。
- harness **不是** 一句写得很神的 prompt。prompt 是喂给大脑的那句话；harness 是大脑之外的整个操作系统。把约束全写进 prompt，等于把交规刷在司机脑门上，却不去修红绿灯。

## 为什么重要：AI 时代的变化

模型够强是必要条件，不是充分条件。一个能改多个文件、能跑命令、能联网的 agent，会不会闯祸，取决于权限怎么分、在哪执行、谁来验收——这些全在模型之外。

真出事的地方，大多在系统边界，不在模型本身。依赖装不上、测试 flaky、网络抽风、环境和生产对不上——这些都能让一次 agent run 失败，可没一样是"模型不够聪明"。Anthropic 把这件事量化过：在 Terminal-Bench 2.0 上，光是基础设施配置的差异就能让分数差出 6 个百分点——足以盖过排行榜上顶尖模型之间的差距；在严格的资源限制下，约 6% 的任务是被基础设施错误（而非模型）搞挂的 [A11]。没有 harness 把这些隔开，你连"是模型错了还是环境错了"都分不清。

跑得越快，错误也抄得越快。模糊的目标、缺失的验收、坏掉的依赖，会越快被复制进更多文件。约束、验证、能撤回，这几样东西因此更值钱，而不是更不值钱。

环境对了，好结果才能复现。Boris Cherny 讲过一条经验：plan 一旦做对，实现"几乎每次都能一遍过" [S8]（二手，访谈口径）。可"好 plan 加一遍过"不靠运气——靠的是每个 agent 实例都跑在隔离的 checkout 里、有测试当验收、有 PR 当交接。这套能复现的东西，就是 harness。Thoughtworks 的 Birgitta Böckeler 索性给它起了名，叫 "harness engineering"，当一门新手艺来谈 [S5]（二手）。

## 实践方法

搭 harness 不必一上来就上九层全套。最省事的起步，是派活之前先把六个问题答清楚：

1. 这个 agent **能做哪些动作**？（只读？能写文件？能跑测试？能联网？能开 PR？）
2. 哪些动作**要人批准**？
3. 它**在哪执行**？（直接在你仓库，还是隔离的工作树 / 容器？）
4. **怎么算完成**？（哪些测试必须绿、哪条验收标准必须满足？）
5. **出错怎么撤**？（分支、快照、回滚脚本？）
6. 它**留下了什么 trace**，让你或下一个人能接手？

把这六个答案写下来，就是一份最小的 harness spec。下一章会把它展开成能落地的九层参考架构。一句话：harness 就是把你对这个 agent 的全部约束摆到明面上、放到它外头，而不是闷头指望它自觉。

## 真实案例

**正面：一人一天 20–30 个 PR。** Boris Cherny 讲过一种用法：同时开 5 个终端标签，每个跑一个 Claude 实例，各自在一份独立的代码 checkout 上干活，一个人一天能推 20–30 个 PR [S8]（二手，访谈）。能这么干，靠的不是"模型神"，是 harness：5 个隔离的 checkout 就是 5 间互不干扰的操作间，测试套件是验收员，PR 是交接便条。抽掉任何一环——比如让 5 个实例在同一个工作目录里乱改——立刻就乱成一锅，互相覆盖。

**产品层的 harness 长什么样。** 看 GitHub 给它云端 coding agent 划的硬边界 [G8]：agent **不能直接执行 `git push`**，只能把改动推到它自己那条 PR 分支；CI 工作流**必须人工批准**才会跑；**发起任务的人，不能批准这个 agent 开的 PR**（职责分离）；联网只许走白名单。这些不是 prompt 里的客套话，是工具层、权限层、流程层硬卡出来的护栏——这就是 harness 落到一个真实产品里的样子。

**反面：把外包空降生产车间。** 没有隔离、没有验收门、测试一过就自动合并、发起人自己批自己的 PR。这套在 demo 里跑得飞快，在真实仓库里迟早出事。

## 常见反模式

- **Prompt 崇拜（prompt worship）**：以为一句足够神的提示词，能替代测试、权限和审查。正确的方向是反过来的——把 prompt 放进 harness，而不是把 harness 塞进 prompt。
- **靠提示词做安全（security by prompt）**：写一句"不要泄露 secret、不要 push"就当防住了。真正的边界在工具层和 sandbox，不在系统提示里。
- **无 eval 的自治**：让 agent 能写文件、跑命令、开 PR，却没有任何验收和回归检查。一句硬标准：**没有 eval，就没有自治。**

## 检查清单

- 我能一句话说清这个 agent **能做和不能做**哪些动作吗？
- 哪些动作**要人批准**，是写在工具/权限层，还是只写在 prompt 里？
- 它在**隔离环境**里跑，还是直接动我的真仓库 / 真数据？
- "**算完成**"有没有可执行的判据（测试、验收标准），而不是"看起来对了"？
- 出错了，我有**撤回**的路径吗？
- 它跑完留下的 trace，够不够另一个人**接手**？

## 自测

- 你现在用 agent 的方式，更像"给新外包搭好了工位再交活"，还是"把他空降到生产车间然后祈祷"？
- 如果此刻把网断了、或者换一个人来接手这次 agent run，会发生什么？你的答案，基本就是你 harness 成熟度的体温计。
