团队的规矩如果只活在老员工脑子里，新外包一定会踩。AGENTS.md 就是把这些规矩写成**机器读得懂、agent 真会照做**的一张便条——它是 agent 时代多出来的一类"源文件"，跟代码一样进版本库、一样得认真维护。

## 生活类比

你请了个钟点工，第一次来，不懂你家任何规矩。你不可能全程跟着，于是在**冰箱上贴一张便条**。

一张好便条长这样：碗洗完放左边橱柜（不是随手哪个）；阳台那袋是猫粮，**别给孩子吃**；用完的铁锅擦干别泡水池，会生锈；干完了发我个消息说一声。

这张便条有三个特征，每一个都直接对应 AGENTS.md：

1. 它是写给不懂你家规矩的人的，不是写给你自己的备忘。你自己知道铁锅不能泡，不用写；正因为对方不知道，才要写。
2. 它具体到动作。不写"请爱护厨具"（这等于没说），写"铁锅擦干别泡水"。
3. 它包含"别碰什么"和"做完怎么交"。光说该做什么不够，得划出禁区，得说清交接方式。

AGENTS.md 就是贴给 agent 的这张冰箱便条。Anthropic 在讲 Agent Skills 时用了几乎一样的比喻：写一个 skill"就像给新人准备一份 onboarding 指南" [A15]。同一条主线——你不是在写文档，你是在给一个能干但不懂规矩的新人交代规矩。

## 工程定义

**AGENTS.md / CLAUDE.md** 是仓库级、机器可读的 agent 指令文件，放在代码库里，跟代码一起走版本控制、一起过评审。

它不是某一家的私有格式。GitHub 的 coding agent 会读仓库根目录的 AGENTS.md，**也读嵌套**在子目录里、只对那部分代码生效的 AGENTS.md；同时它还兼容 `.github/copilot-instructions.md`、`.github/instructions/**.instructions.md`、`CLAUDE.md`、`GEMINI.md` 等多种格式 [G4]。Claude Code 这边主用 CLAUDE.md，逻辑一致 [A1]。AGENTS.md 正在变成一个跨厂商的通用约定。

## 为什么重要：AI 时代的变化

人类新人会"察言观色"——看别人怎么做、踩了坑会被同事拉住。agent 不会。**它只读你显式给它的东西。** 凡是只活在团队默会知识里的规矩（"我们从不直接改 migration 文件""上线前必须跑 e2e"），对 agent 来说等于不存在，于是变成它一定会踩的坑。

所以 repo 指令文件是**一等工程资产**，不是文档摆设 [A1]。它写得好不好，直接决定 agent 干出来的活好不好——同一个模型，一份好的 AGENTS.md 能明显减少它的范围外改动。

还有个反直觉的地方：写成给人看的散文，agent 反而用不上。一段"我们非常重视代码质量，希望大家秉持工匠精神"的话，里面没有一条可执行的规则。agent 需要的是"动作 + 命令 + 禁区"：跑什么命令测、哪些目录别动、PR 描述里必须有什么。

## 实践方法

一份能用的 AGENTS.md，至少覆盖五块：**项目形状、命令、完成的定义（DoD）、安全边界、review 期望**。下面是一份可直接抄改的实样（针对一个假想的 Python 后端 + TypeScript 前端的 web 应用）：

```markdown
# AGENTS.md — Acme 订单系统

## 项目形状
- 后端：FastAPI（`api/`），Postgres，SQLAlchemy。
- 前端：React + TypeScript（`web/`），Vite。
- 金额一律以**分（cents）整数**存储；前端展示除以 100。不要硬编码汇率。

## 命令（你必须用这些，别自己拼）
- 安装：`make install`
- 单测：`make test`          # 改后端必须跑
- 前端测试：`pnpm -C web test`
- 类型检查：`make typecheck`  # 改前端必须跑
- 本地起服务：`make dev`

## 完成的定义（Definition of Done）
- 相关测试全绿；新行为有对应测试（测真实行为，不要迎合实现）。
- 改了 API 形状 → 同步更新 `web/src/api/types.ts`。
- PR 描述里必须有：改了什么、跑了哪些验证命令、未验证项、风险。

## 安全边界（禁区）
- 不要修改 `api/migrations/`（迁移由人手写并评审）。
- 永远不要打印或读取 `.env`、`secrets/`、生产连接串。
- 默认禁止联网；需要新依赖先在 PR 里说明来源和理由。
- 不要执行任何形如 `DROP`、`TRUNCATE`、`rm -rf` 的破坏性命令。

## Review 期望
- 解释每个改动文件为什么动。
- 附验证证据（命令 + 结果）。
- 列出"我没验证 / 我不确定"的假设和后续项。
```

要点：**分层**。根目录放全局规矩；某个特殊子系统（比如支付）可以再放一个嵌套 AGENTS.md，只对那块生效——GitHub 的 agent 支持这种嵌套 [G4]。

## 真实案例

**一手 URL 也可能是幻觉。** 写这本教材时，我们去核两份旧教材引用的一条 GitHub changelog——一个形如 `2025-10-23-...agents-md...` 的 URL，本该是 AGENTS.md 支持的官方公告。两轮抓取，两次 404：这个链接根本不存在。真实情况是：AGENTS.md 对 coding agent 的支持其实在 2025-08-28 上线，而 custom agents 是 2025-10-28 在 GitHub Universe 发布的 [G4]。一个长得"很对"、日期格式都规整的厂商 URL，可以纯属虚构。这个教训放到你手上一样成立：无论是写教材，还是让 agent 引一段"官方文档"，链接和出处都得自己点开核一遍——agent（和人）都会一本正经地编一个看起来无懈可击的来源。

**指令文件正在 sprawl。** 同一个 GitHub agent，会同时认 AGENTS.md、`.github/copilot-instructions.md`、`.github/instructions/**.instructions.md`、CLAUDE.md、GEMINI.md 五种格式 [G4]。能兼容是好事，但一个仓库里散着五份口径可能打架的指令，本身就是新坑。收敛办法很简单：以 AGENTS.md 为单一真源，其余文件要么删，要么只留一句"规则见 AGENTS.md"，别让 agent 在五份互相矛盾的便条里挑。

## 常见反模式

- **写成散文**：一段充满"工匠精神""高度重视"的漂亮话，没有一条可执行规则。改成动作、命令、禁区。
- **当 README 堆细节**：把架构长篇大论塞进去，淹掉真正的硬规矩。AGENTS.md 要短、要密、要可执行。
- **只写"该做什么"，不写"别碰什么"**：禁区比待办更重要——它直接限制 agent 的"误伤半径"。
- **五份指令各说各话**：见上。收敛到单一真源。

## 检查清单

- 我的 AGENTS.md 里每一条，agent 能据此**采取一个具体动作**吗（而不是体会一种精神）？
- **命令**齐了吗（测试、类型检查、本地起服务）？agent 不用猜怎么验证？
- **禁区**写清了吗（别动哪些文件、别读什么、能不能联网、禁哪些命令）？
- **完成的定义**和 **PR/review 期望**写了吗？
- 多份指令文件之间，有没有**单一真源**？

## 自测

- 把你的 AGENTS.md 给一个完全不懂这个项目的人看，他能不能只照这张便条，跑通测试、知道哪些地方不能碰？做不到，agent 也做不到。
- 你上一次让 agent 引"官方文档"，那个链接你亲手点开核过吗？
