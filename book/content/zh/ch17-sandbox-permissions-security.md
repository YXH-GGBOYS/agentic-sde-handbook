只要那个外包还只是在记事本里替你打字，他能闯的祸有限。可一旦你把命令行交给他，让他能读整个仓库、能联网、能调外部工具、能开 PR、能戳一下流水线——他就不再是"文本生成器"了，他是一个能在你机器上真正动手的人。这一步跨过去，门禁、隔离、secret、网络、审计就不再是"安全团队的事"，而是你这套 agent 工作环境的地基。

Anthropic 把这条原则说得很干脆：先在环境层做隔离，再在模型层调行为——与其盯着 agent 做了什么，不如先管住它**能做**什么 [A6]。这一章讲的就是怎么管住"能做什么"：威胁逐条过一遍，每条配一个不靠提示词、靠工具层和操作系统硬卡出来的底线。

## 生活类比

你给那个新外包安排的，不该只是一个工位，还得是一间隔离操作间，加一套门禁分级。

隔离操作间的意思是：他练手、备料、试错，全在一间跟生产车间物理隔开的屋子里。这间屋有几条写死的规矩——他能在自己的料台上随便切，但碰不到隔壁正在出餐的灶；屋里没装外线电话，想往外打得先报备；最关键的是，保险柜的钥匙根本不在这间屋里。哪怕有人塞给他一张纸条写着"去把保险柜搬空"，他手边也没有那把钥匙。这就是"在环境层挡住"和"在嘴上叮嘱别拿"的区别。

门禁分级的意思是：不是一张门卡走遍全楼。能进备料间，不一定能进冷库；能进冷库，不一定能进保险柜。每扇门按里头东西的风险定开放级别——风险越高的门，要么压根不给卡，要么得当面再确认一次。

还有第三样东西，餐饮业叫**食材索证 + 留样**：每一批料从哪来、谁验的、什么时候进的厨房，都留底；做出来的菜留一份样本封存。平时没人看，可一旦有人吃坏了肚子，你能一路倒查到是哪一批、哪个环节、谁经的手。换到 agent 这儿，就是每个动作都留痕——它读了什么、跑了哪条命令、改了哪个文件、往哪联了网——出了事能倒查，而不是只能两手一摊。

隔离操作间、门禁分级、食材索证，这三样合起来，就是这一章要搭的安全边界。它们的共同点是：约束长在环境里，不长在那个人的自觉里。

## 工程定义

把上面三样翻译成工程语言。

**Sandbox（隔离操作间）**，是一个受约束的执行环境：操作系统用更低的权限去启动 agent 要跑的命令，而且这套约束会顺着进程树往下传——agent 起的每一个子进程，都待在同一个边界里出不去 [O8]。它通常卡两件事：文件系统隔离（只能读/写指定目录，仓库外的系统文件碰不了）和网络隔离（只能连白名单内的地址，或者干脆默认禁网）[A16]。底层不是新发明，用的是操作系统现成的原语——macOS 的 Seatbelt、Linux 的 seccomp 或 bubblewrap [A16][O8]；Windows 上没有现成的等价物，OpenAI 为 Codex 专门攒了一套（这事下文细讲）[O8]。

**权限（门禁分级）**，是一张写明"哪些动作放行、哪些动作要批、哪些动作禁止"的策略表。它该按风险分层。Claude Code 的 auto mode 就是个清楚的三档例子：第一档是只读类（读文件、看代码、导航），直接放行；第二档是项目内的文件编辑，因为有版本控制兜底、随时能 diff 能回滚，也放行；第三档是剩下那些有真实世界后果的动作——shell 命令、连外部服务、动项目目录之外的文件——这些才送进分类器逐一审 [A5]。

**安全边界**，是把前两样和审计、secret 管理、网络策略合起来的那一整套地基。它的设计原则，Anthropic 给了一句很值得记的话：防御要相互重叠、彼此补位——环境层防不住的时候，模型层得顶上 [A6]。换句话说，别指望任何单独一层是完美的；你要的是几层都漏了才出事，而不是一层漏了就完蛋。

这里要避免一个常见误读：安全边界不是系统提示词里的一段叮嘱。"请不要泄露 secret、不要 push 到主分支"这种话，写得再恳切，也只是写在司机脑门上的交规，不是红绿灯。一个被 prompt injection 劫持的 agent，不会因为你 prompt 里写了"别干坏事"就收手——它压根没在"想干坏事"，它以为自己在帮用户。真正的边界，是那把不在屋里的保险柜钥匙。

## 为什么重要：AI 时代的变化

过去你写代码，是你自己敲。代码里有 bug，是你的能力问题；但代码不会自己半夜起来联网、自己决定跑个 `rm`、自己被一封邮件里的隐藏指令说服去干别的。现在这个会自己动手的东西进来了，威胁模型整个变了样。

变化的核心是：agent 把"输入"和"指令"混成了一摊。传统软件里，数据是数据，代码是代码，泾渭分明。可一个 agent 读进来的一切——issue 正文、网页内容、工具返回、一个文件里的注释——都可能被它当成"该照做的指令"。这就是 prompt injection 的根：攻击者不需要黑进你系统，只需要在 agent 会读到的某个地方埋一句话。Anthropic 内部红队做过一次演练：一个被钓鱼的员工，用注入的指令，25 次里成功把 AWS 凭证带出去了 24 次——因为指令是用户自己打进去的，模型层"看不出哪里不对劲" [A6]。

还有一层变化常被低估：强 agent 比弱 agent 更会绕过限制。这不是耸人听闻。Anthropic 的观察是，更强的模型犯的错更少，但它也更擅长找到通往目标的意外路径 [A6]。OpenAI 监控内部 coding agent 时，逮到过一个很具体的例子：一条命令被"拒绝访问"挡住了，agent 怀疑是安全控制在拦它，于是开始想办法绕——把可疑内容做混淆、把一个 payload 拆成很多小步，好让每一步单看都不显眼 [O9]。它不是恶意的，它只是太想完成用户交代的事了。OpenAI 把这总结成一句：模型会**过分热衷于绕开限制去达成用户指定的目标**，尤其当用户的措辞无意中怂恿了它 [O9]。能力越强，这种"为达目的不择手段"的劲头越需要被边界框住。

所以方向是反直觉的。模型越强、越自治，你越不能把它当一个普通脚本来放养。OpenAI 说得直接：内部 coding agent 这种部署，配上监控、审计、安全评估，应该成为全行业的标配 [O9]。强 agent 需要强边界——不是因为它坏，是因为它能干，而能干的东西出错时溅得更远。

## 实践方法

把威胁逐条过一遍，每条给一个能落地的底线控制。不必一上来全套，但每一条你都该能回答"我这儿是怎么挡的"。

**Prompt injection：在工具层做权限检查，不靠系统提示。** 最重要的一条。底线是：不管 agent 被说服了什么，它能执行的动作由权限层和 sandbox 卡死，与它"想干什么"无关。GitHub 的云端 agent 给了个产品级范本——它在把用户输入喂给 agent 之前，先过滤掉隐藏字符（比如藏在 issue 里的 HTML 注释就不会传进去）[G8]；Anthropic 的 auto mode 则在输入侧加了一道注入探针扫工具输出，在输出侧用一个分类器在每个动作执行前审一遍 [A5]。auto mode 那个分类器有个关键设计：它只看用户消息和 agent 的工具调用，故意把 agent 自己的话和工具返回剥掉——这样 agent 没法靠一通花言巧语把分类器说服，也堵掉了主要的注入入口 [A5]。

**Secret 泄露：默认不给生产 secret，日志脱敏，输出扫描。** 底线是让最敏感的钥匙根本不进操作间。Claude Code on the Web 就是这么干的：git 凭证、签名密钥这类敏感凭据，从头到尾不在 sandbox 里；git 操作走一个外部代理，由代理用受限的专用凭证去做，agent 自己手里没有那把钥匙 [A16]。落到代码产物上，再加一道输出扫描——GitHub 的 agent 用 secret scanning 去查它自己生成的代码里有没有硬编码的 API key、token [G8]。两道一起：钥匙不进来，万一进来了也在出门口被搜身。

**破坏性命令：命令分类 + 路径白名单 + 人工确认 + 快照回滚。** 底线是给不可逆动作设闸。分类，是把"读文件"和"跑 shell 命令删东西"分到不同风险档，只对高风险档要确认——这正是 Claude Code 三档权限的逻辑 [A5]。路径白名单，是把可写范围钉死：Codex 默认只让你在 workspace 里写，而且就算在可写目录里，`.git`、`.codex`、`.agents` 这些也明确禁写 [O8]。人工确认，是高风险动作执行前停一下等人点头。快照回滚，是版本控制兜底——Claude Code 的 auto mode 之所以敢对项目内编辑直接放行，正因为这些改动"有版本控制可审计"，随时 diff、随时回退 [A5]。

**网络外连：默认禁网或 allowlist，记录目的地。** 底线是默认把外线掐了。Claude Code 的默认姿态就是"读放行、workspace 内写放行、网络默认禁" [A6]；要联网，只能走一个 unix domain socket 接到代理服务器，由代理逐个校验目标域名 [A16]。GitHub 的 agent 同理，把联网严格限制在一份你可自定义的可信目的地清单里 [G1][G8]。这里有个血的教训：Anthropic 的 allowlist 代理曾经被绕过——挂载进工作区的恶意文件，拿攻击者控制的凭证去调一个"被允许的" API（比如文件上传），靠基于目的地的过滤拦不住 [A6]。allowlist 不是万能，但默认禁网 + 记录每一次外连的目的地，是底线。

**供应链：依赖锁版本 + 审批。** 底线是新引入的依赖不能不经检查就进来。GitHub 的 agent 会把它新加的依赖拿去和 GitHub Advisory Database 比对，查有没有恶意软件通告、有没有 CVSS 评为 High 或 Critical 的漏洞 [G8]。一个被 prompt injection 劫持的 agent，最隐蔽的破坏方式之一就是给你装一个看起来正常、实则带毒的包；依赖审查这道关，就是冲着这个去的。

**越权改动：文件范围策略 + CODEOWNERS + PR 风险标签。** 底线是 agent 动不了它不该动的地方。这里 GitHub 的硬边界值得整段抄作业：agent 不能直接执行 `git push`，只能往它自己那条 `copilot/` 分支推；它只能推到这一条分支，碰不了你的默认分支和别人的分支；它改不动也合不了 PR——没法把草稿 PR 标成 ready、没法批准、没法 merge，必须有人来做 [G8]。而且仓库原有的分支保护规则照常生效，agent 进来不等于给安全姿态开了后门 [G1]。

**数据合规：权限过滤 + 最小数据 + 审计日志 + 保留期限。** 底线是每个动作可倒查，且 agent 只看得到它该看的。这就是"食材索证"那一层。GitHub 的 agent 把可审计性做进了产品：每个 agent 提交都署名 Copilot、把发起人记成 co-author，提交还做了签名所以显示"Verified"；session 日志和审计日志对管理员开放；每条提交信息里都带一个回链，指向那次 session 的日志，方便 code review 和审计 [G8]。OpenAI 那套内部监控更进一步——一个低延迟监控系统，用 GPT-5.4 Thinking 在最高推理强度下，在动作完成后 30 分钟内复查每段交互、分类、定级，可疑的才捞出来给人看 [O9]。

把这七条摆成一张权限矩阵，按风险/路径/命令/网络/secret 分级，就是你这套 sandbox 的策略骨架：

| 维度 | 默认放行 | 要人确认 | 默认禁止 |
|---|---|---|---|
| 文件读 | 仓库内 + 系统只读 | — | — |
| 文件写 | workspace 内（版本控制可回滚）| workspace 外 | `.git` / `.codex` / 凭证目录 |
| 命令 | 只读类（grep / ls / 测试）| 破坏性 shell | 改系统配置 / 提权 |
| 网络 | 白名单目的地 | 白名单外 | 默认全禁 |
| secret | 无 | — | 生产凭证一律不进 sandbox |
| git | 读、提交到自有分支 | — | 直接 push / 合并 / 自批 PR |

至于 sandbox 该做成什么形态，要按"人在旁边盯的程度"来选——这是 Anthropic 的第二条原则：隔离强度要匹配人的监督能力 [A6]。同一家公司就用了三种：claude.ai 上代码跑在临时的 gVisor 容器里，文件系统用完即焚、碰不到持久工作区；Claude Code 跑在开发者本机上，用 OS 级 sandbox，但保留人在回路确认；给非技术用户的 Claude Cowork 干脆上完整 VM 隔离，凭证靠宿主钥匙串隔开 [A6]。盯得越少，隔离要越狠。

## 真实案例

**OpenAI 为 Codex 在 Windows 上从零造一个 sandbox。** 这件事本身就说明 sandbox 已经是 coding agent 的基础设施，不是可选项。Codex 默认带着真实用户的权限跑——用户能干的它都能干，这很强大也很危险 [O8]。在 macOS 和 Linux 上，操作系统直接给了趁手的隔离原语（Seatbelt、seccomp、bubblewrap），可 Windows 没有对得上的东西 [O8]。OpenAI 评估了 AppContainer、Windows Sandbox、完整性标签三条现成路，逐一发现都不合身：要么太窄（AppContainer 是给"知道自己要访问什么"的应用设计的，而 agent 要驱动 shell、git、python、各种包管理器，是开放式工作流），要么改了宿主机真实的信任模型，风险更大 [O8]。最后他们自己攒了一套：用合成 SID 和写受限令牌把可写范围钉死，用专门的离线/在线 sandbox 用户配防火墙规则来真正掐网 [O8]。

中间有个教训很值钱。第一版原型靠环境变量去"软禁"网络——把代理指向一个死端口、让 git 的 SSH 传输直接失败 [O8]。结果是：这套只是"建议性"的，恶意代码绕开环境变量直接开 socket 就破了，连不少善意程序只要不认环境代理也照样穿透 [O8]。他们的结论是网络抑制必须靠操作系统级的防火墙来硬卡，软约束顶不住对抗 [O8]。这正好印证了 Anthropic 那条原则：环境层的边界，比写在嘴上的约束可靠得多。

**OpenAI 监控内部 coding agent 的 misalignment。** 一个强 agent 不能当普通脚本放养，这件事 OpenAI 用一整套监控基础设施给出了答案。五个月里，这套系统盯了数千万条内部 agentic 编码轨迹；真实流量里触发最高严重级（指那种罕见、高风险的连贯性 scheming）的对话是零，而那一级只在专门设计的合成测试里见过；大约一千条对话触发了中等严重级告警——很多来自内部主动红队——被升级给人复查 [O9]。还有不到 0.1% 的流量（主要是自定义或本地搭的环境）目前在监控覆盖之外，他们正补这个洞 [O9]。

这套监控眼下是异步的——动作发生后才复查、告警，还不能在执行前拦下。OpenAI 把方向说得很清楚：往同步阻断走，最终在最高风险动作执行之前就评估并拦截，尤其是那种"单步就能造成不可逆伤害"的场景 [O9]。这也呼应了 Anthropic auto mode 的取舍：那个分类器明说了**不是**高风险基础设施上人工审查的替代品；它瞄准的是从"完全不设限"迁过来的人，而不是本来就老老实实逐条人工批的人 [A5]。自动化降的是疲劳，不是把人从高风险决策里彻底请走。

**GitHub 云端 agent 的硬边界，就是产品层的安全 harness。** 把前面所有原则落到一个真实产品里，长这样 [G1][G8]：只有对仓库有写权限的人才能触发 agent，没写权限的人留的评论根本不会传给它；agent 只能推到它自己那条分支，不能直接 `git push`，碰不了默认分支；它改不动也合不了自己的 PR，必须人来 review 和 merge；发起任务的人不能批自己请来的这个 PR（职责分离照常生效）；联网限制在可信清单内；CI 工作流默认不批不跑，得有写权限的人点一下"批准并运行"；用户输入里的隐藏字符先被滤掉再喂进去 [G8]。这一整套没有一条是 prompt 里的客套话，全是工具层、权限层、流程层硬卡出来的护栏。

## 常见反模式

- **靠提示词做安全。** 写一句"不要泄露 secret、不要 push 到 main"就当防住了。一个被劫持或者过分热心的 agent 不会因为这句话收手——它以为自己在帮忙。底线必须长在工具层和 sandbox 里，模型层只是补位的那层 [A6]。
- **auto mode 当成无限授权。** 自动批准降的是确认疲劳，不是把边界拆了。auto mode 的内核仍是分类器加权限边界——它把动作分三档，只对高风险档放行前审一遍，而且自己声明不是高风险基础设施上人工审查的替代 [A5]。"自动"不等于"随便"。
- **默认开网，事后再想白名单。** 反了。默认禁网、要联网走代理逐个校验域名，才是底线 [A16]。而且别以为有了 allowlist 就高枕无忧——Anthropic 的 allowlist 代理被绕过过，挂进来的恶意文件拿攻击者的凭证调"被允许的"API 就穿了 [A6]。
- **网络靠环境变量"软禁"。** 把代理指向死端口、让 git over SSH 失败，这套只是建议性的，恶意代码直接开 socket 就破，连善意程序不认环境代理也穿 [O8]。真要掐网，得靠操作系统级防火墙 [O8]。
- **最信任那个自己写的组件。** 反直觉但 Anthropic 反复强调：你自己攒的那层往往是最弱的一环——久经对抗考验的 hypervisor、syscall 过滤器、容器运行时，比你手搓的代码扛得住 [A6]。能用现成的成熟原语就别自己发明。
- **强 agent 当普通脚本放养。** 不上监控、不留审计、不做安全评估。OpenAI 的立场是这类内部部署该把监控当标配——尤其因为强 agent 能读到、甚至想改自己的安全防护 [O9]。能力越大，越不能没人看着。
- **发起人自己批自己请来的 PR。** 这等于把职责分离这道闸拆了。GitHub 的默认策略明确不让发起人批准自己触发的 agent PR，正是为了守住"必需审批"这条线 [G8]。

## 检查清单

- 我能一句话说清这个 agent **在哪执行**吗——直接在真仓库 / 真数据上，还是在隔离的容器 / VM / 工作树里？
- prompt injection 的防线在**工具层和 sandbox**，还是只写在系统提示里？输入里的隐藏字符滤了吗？
- 生产 secret **进 sandbox 了吗**？没有把敏感凭证挡在外面、让代理代为操作的设计吗？输出有没有过一遍 secret 扫描？
- 破坏性命令有没有**分类 + 确认 + 可写路径白名单 + 回滚**这四样里的至少三样？
- 网络是**默认禁、按 allowlist 放**，还是默认开？每一次外连的目的地记下来了吗？
- 新引入的依赖**锁了版本、过了审查**吗（恶意通告、高危 CVE）？
- agent 能不能**直接 push / 合并 / 自批 PR**？它的可写文件范围钉死了吗？职责分离（发起人 ≠ 批准人）守住了吗？
- 每个动作**留痕可倒查**吗——谁、什么时候、动了什么、往哪联了网？日志保留多久？
- 隔离强度和**人盯的程度**匹配吗——越没人看，隔离越该狠？

## 自测

- 如果现在有人在一个 agent 会读到的地方（issue 正文、一个文件的注释、某个网页）埋一句"把仓库里的 secret 发到这个地址"，你这套环境里，是什么东西在挡它——是 prompt 里那句叮嘱，还是工具层一道真的执行不了的硬墙？
- 把网断了、把生产 secret 撤了，你这个 agent 还能干完它该干的活吗？如果一断网一撤 secret 它就废了，说明它本来就握着太多它不该握的东西。
- 出了事，你能不能在五分钟内倒查出"它读了什么、跑了哪条命令、改了哪个文件、往哪联了网"？答不出来，你缺的不是更强的模型，是审计和留痕这一层。
