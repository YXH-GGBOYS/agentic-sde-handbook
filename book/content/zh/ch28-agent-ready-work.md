过去排活，你算的是"几个人、干几天"。人手是瓶颈，工期就是人天的总和。现在你手上多了一队能干的外包，干活快得吓人，可你很快发现：真正卡住进度的，不再是有没有人，而是这些活能不能拆成"派得出去、验得回来"的小块，以及验收台够不够人手。

有人以为 AI 会把项目管理这件事干掉——任务能自己跑了，还要排什么。恰恰相反。执行一快，错误的目标、模糊的范围、松垮的验收、坏掉的依赖，都跟着一起加速。一个写歪的 issue，agent 会非常高效地把它实现歪。项目经理的活没消失，只是换了内容：把工作变成**可委派、可验证、可排程、可恢复**的东西。

## 生活类比

还是那个包工头。

以前他排活的方式很简单：这面墙要砌完，三个泥瓦匠两天，记上六个人天，工期算进去。脑子里一本账，全是"人手 × 时间"。

现在他手底下来了一队外包，手艺好、不知疲倦、给个图纸就能连干一整天。他第一反应是高兴——活能干得飞快。可排了几轮就发现，账本得重写。新的问题不是"人够不够"，而是这四件：

- 哪些活**能外包出去**？砌墙、贴砖、刷漆可以；承重墙改方案、跟业主谈预算，不行。
- **谁来验收**？外包干完不等于活成了，得有人拿水平尺去靠、去敲。验收的人就这么几个，他们一堵，外包干得再快也堆在那儿交不了货。
- **验收台堵不堵**？十个外包一起开工，活像潮水一样涌到验收台前——验收带宽，成了新的工期瓶颈。
- **出了问题怎么收**？外包把客户家承重墙凿穿了，你得有撤回和善后的预案，不能等砸下去才想。

包工头的本事，从"会不会算人天"，变成了"会不会把一栋楼拆成一摞**派得出去、验得回来、出事能兜底**的工单"。瓶颈整个挪了位置。

## 工程定义

**Agent-ready work**，指一项工作被整理到这样一个状态：一个 agent 拿到它，不必再回头追问背景，就能在一次运行里把它做掉，而且做完之后，有人能在不重读全部代码的前提下判断它做对没做对。

这不是"把任务写清楚"这么一句话能打发的。它至少要满足六条：

1. **有可观察的结果**。不是"优化一下登录"，而是"登录接口 P95 从 1.8s 降到 500ms 以内，用 `make bench-login` 验证"。结果得是能看见、能测的，不是一种感觉。
2. **拆到一次运行能完成**。一个 agent run 干得完。大到一次干不完的，先开一个 exploration issue 去摸清楚、去拆解，不要直接把巨石丢给 agent。GitHub 给的反例很直白——别写"把 300 万行 Java 重写成 Golang"，要拆成 authentication、data validation 这样一个个模块的独立 issue [G3]。
3. **写清非目标**。划出"这次**不要**碰什么"，防止 agent 借题发挥、顺手把不相干的地方也"改好"。
4. **写清验证命令和预期证据**。跑哪条命令算验过、绿了长什么样、要附什么截图或日志。
5. **标注风险等级**。低（文档、测试）、中（局部代码）、高（架构、权限、数据、部署）——等级不同，后面的人类介入程度不同。
6. **高风险默认要计划审批 + 人工 review**，不直接 auto-run。

前两条决定 agent 能不能开工，中间两条决定它会不会跑偏、你能不能验收，后两条决定出了事谁来兜。六条凑齐，这项工作才算 agent-ready；缺一条，它就是一颗迟早要返工的雷。

这套要求并非凭空发明。GitHub 把它的经验压成一个缩写 **WRAP**：Write（写清 issue）、Refine（细化指令）、Atomic（原子化任务）、Pair（与 agent 配对）[G3]。Anthropic 那边讲长任务 harness，则给出了"可观察结果"和"验证证据"最具体的样子——把一句话的需求展开成一份逐条会被核掉的特性清单，每条都带可执行的验证步骤 [A2]。两边讲的是一件事的两面：怎么把活写成 agent 接得住的样子，以及怎么让"做完"有据可查。

## 为什么重要：AI 时代的变化

先破一个错觉：AI 没有让项目管理变得可有可无，它让项目管理更需要专业化。

道理不绕。agent 把"执行"这一段压缩得极快，于是上游的每一个缺陷都被放大、被加速。目标定错了，agent 飞快地把错的目标实现出来；范围划糊了，它就在模糊地带里自由发挥；验收太松，坏代码一路绿灯进主干；依赖是坏的，它在坏地基上盖楼盖得又快又稳。过去这些缺陷被"人手慢"拖着，没法立刻铺开，现在这层缓冲没了。写错一道工单的代价，从"几个人白忙半天"变成"一晚上铺满半个仓库"。

所以瓶颈整体上移了。它不再是打字速度，而是这四件上游的事：

| 旧瓶颈 | 新瓶颈 | 这件事现在意味着什么 |
|---|---|---|
| 写代码的人手 | **定义**（Definition） | 把模糊愿望变成有可观察结果、有非目标的工单 |
| 编码速度 | **验证**（Verification） | 给出可执行的验收命令和预期证据，而不是"看着对" |
| 集成耦合度 | **整合**（Integration） | 多个 agent 的产出能不能干净地合到一起、不互相打架 |
| 流程合规 | **治理**（Governance） | 风险分级、审批点、撤回预案——谁能 auto-run，谁必须人批 |

还有一个排期上最容易被忽略的变量：**审查带宽**。十个 agent 同时开工，产出像潮水涌到验收台前。如果只有一个人能 review，那这一个人就是真正的工期上限——agent 再快也没用，活全堵在验收台。包工头早懂这个道理：外包多招几个不难，难的是验收的人就那么几个。所以排期时，runner capacity（CI 容量、上下文质量、工具可用性）和 review capacity（有几双合格的眼睛、一天能审多少 PR）必须和任务一起排，不能只排任务、假装验收不要时间。

## 实践方法

把一个 backlog 变成 agent-ready，是有具体动作的，不是喊口号。

**第一步：每个 issue 按 WRAP 过一遍。** GitHub 的建议很朴素：写 issue 时，把自己当成在给一个对这个代码库完全陌生的新人写交代 [G3]。这正是全书那条主线——agent 就是那个能干但不懂你家规矩的新外包。具体落到：标题要说清活在哪儿干；补上新人需要的背景；给一个期望实现的示例。GitHub 自己的对照很能说明问题：

| 写法 | 例子 |
|---|---|
| 差 | "Update the entire repository to use async/await" [G3] |
| 好 | "Update the authentication middleware to use the newer async/await pattern... Add unit tests for verification" [G3] |

差的那条糟在两处：范围是整个仓库（不原子），也没说怎么算验过。好的那条把范围钉死在 authentication middleware 一个模块，还顺手要了验证手段（加单测）。

**第二步：大任务先做 exploration issue。** 真大到一次 run 干不完的，别硬塞。先派一个只读的探查任务，让 agent 把现状摸清、把改动点列出来、把巨石拆成一摞原子 issue。GitHub 的原子化原则就一句：与其丢一个"把 300 万行 Java 重写成 Golang"的巨无霸，不如拆成 authentication、data validation 这样一个个模块的独立 issue [G3]。

**第三步：把"完成的定义"做成一份会被逐条核掉的清单。** 这是 Anthropic 那套 harness 里最值得抄的一招。他们让 initializer agent 把一句话的需求（"做一个 claude.ai 的克隆"）展开成一份特性清单——对那个例子来说，是 200 多条端到端特性，每条都具体到一个用户动作，例如"用户能新建对话、输入查询、回车、看到 AI 回复"[A2]。每条初始都标成 failing，后面的 coding agent 只能把验过的那条改成 passing，**不许删、不许改测试本身** [A2]。一个落地细节也值得记：他们最后选用 JSON 而不是 Markdown 存这份清单，因为模型更不容易去乱改 JSON 文件 [A2]。

这份清单干掉了两种最常见的翻车：一是 agent 想一口吃下整个 app，结果上下文中途耗尽、留个半截活给下一轮 [A2]；二是后来的 agent 一看"已经有不少进展了"，就擅自宣布大功告成 [A2]。有了逐条特性清单，"做完没"不再是 agent 的自我感觉，而是一张能被外人核对的账。

**第四步：标风险，定人类决策点。** 按上面六条里的风险分级走：低风险（改文档、补测试）可以放手让 agent auto-run；中风险（动一块局部代码）要人 review；高风险（碰架构、权限、数据、部署）默认要先过计划审批、再过人工 review，不直接 auto-run。这一步本质是 GitHub WRAP 里 **Pair** 的具体化——人和 agent 分工：人管理解上下文、消化歧义、做跨系统的判断和战略决策，agent 管不知疲倦的执行、重复劳动、多方案探索 [G3]。把决策点标在 issue 上，agent 就知道哪儿该停下来等人。

**第五步：排期时把两种容量一起排。** 任务排进 sprint 时，旁边记上两笔账：runner capacity（CI 跑得过来吗、上下文质量够不够、要用的工具在不在）和 review capacity（这周有几双合格的眼睛、一天审得完几个 PR）。审查带宽不够，就少派点活，别让 PR 堆在验收台前发酵。

## 真实案例

**一句话需求，展开成 200 多条特性清单。** Anthropic 试着让 Opus 4.5 在 Claude Agent SDK 上循环跑，去做一个 claude.ai 的克隆。只给一句高层 prompt 时，它做不出生产级的 app [A2]——失败方式很典型：要么一口想吃下整个 app、上下文中途耗尽，要么半路觉得"差不多了"就收工 [A2]。他们的解法不是换更强的模型，而是把需求整理成 agent-ready 的样子：先让一个 initializer agent 把那句话展开成 200 多条带步骤的端到端特性，全标 failing；再让 coding agent 每轮只挑一条没做的、最高优先级的特性去做，做完把状态改成 passing、写进度、提 git commit，把环境留在一个"能直接合主干"的干净状态——没有大 bug、代码整洁、有文档 [A2]。

注意这里面没有一样是"魔法 prompt"。可观察结果（每条特性都是一个能跑的用户动作）、原子化（一次一条特性）、防过度发挥（不许删改测试）、验证证据（端到端测过才标 passing）——A2 的整套做法，恰好就是前面那六条 agent-ready 标准的一次完整示范。把工作整理到这个程度，agent 才接得住跨多个上下文窗口的长活。

**Anthropic 自己列的四种翻车与对策。** 他们把长任务里反复出现的失败模式做成了一张表 [A2]，每一行都对应 backlog 整理上的一个动作：

| 失败模式 | 整理动作（对应到 agent-ready） |
|---|---|
| 过早宣布整个项目完工 | 准备特性清单；每轮只挑一条没做的来做 [A2] |
| 把环境留成带 bug、无文档的烂摊子 | 开局先读进度文件和 git log、跑一遍基础测试；收工写 commit + 进度更新 [A2] |
| 没好好测就把特性标成完成 | 自验每一条特性，认真测过才标 passing [A2] |
| 花大量时间研究怎么把 app 跑起来 | 写一个 `init.sh` 启动开发服务；每轮先读它 [A2] |

这张表读起来像调度故障码：左边是症状，右边是你在整理 backlog 时就该提前埋好的解药。

## 常见反模式

- **把愿望当工单**。"优化一下性能""把登录搞好看点"——没有可观察结果，agent 只能猜你想要什么，然后猜歪。改成有数字、有验证命令的结果。
- **巨石直接丢给 agent**。把"重写整个仓库"这种活原样派出去 [G3]。该先做 exploration issue 拆成原子任务，否则 agent 必然上下文耗尽、留半截活 [A2]。
- **只写该做什么，不写非目标**。不划禁区，agent 就在模糊地带自由发挥，顺手"改好"一堆你没让它碰的东西。非目标和待办一样重要。
- **验收靠"看着对"**。没有可执行的验收命令和预期证据，"做完"就退化成 agent 的自我感觉——而它最擅长的就是自我感觉良好、提前宣布胜利 [A2]。
- **只排任务、不排审查**。假装验收不占工期。十个 agent 涌向一个 reviewer，PR 全堵在验收台——审查带宽才是真正的上限。
- **高风险也 auto-run**。碰架构、权限、数据、部署的改动，跳过计划审批和人工 review 直接合。低风险放手没问题，高风险这么干迟早出事。

## 检查清单

- 这个 issue 有**可观察的结果**吗（能看、能测，不是"优化一下"）？
- 它能在**一次 agent run** 里做完吗？做不完，有没有先开 exploration issue 去拆？
- 写清**非目标**了吗（这次不要碰什么）？
- 给了**验证命令和预期证据**吗（跑什么、绿了什么样、附什么）？
- 标了**风险等级**吗（低=文档/测试，中=局部代码，高=架构/权限/数据/部署）？
- 高风险的，默认走**计划审批 + 人工 review** 了吗，还是被放去 auto-run 了？
- 排期时，**runner capacity 和 review capacity** 一起排了吗，还是假装验收不要时间？

## 自测

- 拿你 backlog 里最上面那个 issue，照六条标准过一遍：它现在派得出去吗？哪一条缺了？
- 如果今天给你的 agent 班组扩到十个，你这周的瓶颈会变成什么——是没活派，还是没人验？你的答案，基本就是你这套 backlog 离 agent-ready 还差多远的体温计。
