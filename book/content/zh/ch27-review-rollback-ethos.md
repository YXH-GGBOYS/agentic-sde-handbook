agent 把活交上来那一刻，最危险的念头是"测试过了就合吧"。测试绿，只说明它没碰坏它自己想到的那些东西；它有没有真听懂需求、改动有没有越界、它声称跑过的命令是真跑了还是顺嘴说的——这些测试一概不管。

这一章讲三件事，正好对应装修队交活时你要做的三件事：怎么验收（看 diff、看 trace、看证据，不只看 diff），怎么留退路（谁能批 PR、谁不能、出了事怎么退回原状），以及把这一卷三家工具串起来看——形态各异，工程精神趋同。

## 生活类比

装修队说"做完了"，你不会当场掏尾款。一个稳妥的流程是这样的：

你不自己验自己的活——找个跟施工队没利害关系的验收员来看，最好不是当初签合同那个人，免得"自己交活自己签收"。验收员不光看墙面平不平（这是 diff），还要翻施工记录：水管打压试了没、试压表读数多少、电路通断逐路测过没（这是 trace 和证据）。尾款永远押在验收通过之后，不是材料进场就付。最要紧的是，合同里得留一条退货退款、能把屋子恢复成动工前样子的路——万一承重墙被凿了，你得有得退、有得修。

agent 的活一模一样。验收要独立（让另一个干净上下文来审，不是让写代码那个 agent 自己给自己打分），尾款（合并、上线）要押在验收之后，并且永远留一条随时能回滚、能把代码退回原状的路。review、门禁、回滚，缺一不可。

## 工程定义

**Review（验收）** 是有人独立检查 agent 的交付物，判断能不能合、能不能上。对 AI 生成的改动，验收的对象有三层，缺一层都不算验过：

- **diff**：改了哪些文件、加了哪些行——这是看得见的成品。
- **trace**：agent 一路读了什么、试了什么、哪步失败、为什么这么改——相当于施工记录。GitHub 的 coding agent 会把推理和验证步骤写进 session logs，让你能倒查它的决策、发现问题 [G1]。
- **证据**：它声称"测试过了"，有没有把测试输出、跑的命令和返回结果贴出来。Anthropic 的建议是让 Claude 拿证据说话，而不是口头断言成功——把测试输出、跑过的命令、截图摆出来，审证据比你自己重跑一遍快 [A1]。

**Rollback（回滚）** 是出问题后把系统退回改动前状态的能力。它分两层：会话级的撤销（Claude Code 的 `/rewind` 和 checkpoint，能把对话和代码恢复到之前任一节点），和工程级的回退（git revert、分支保护、不让 agent 直接 push 到主干）。注意一个边界：Claude Code 的 checkpoint 只追踪 **Claude 自己做的改动**，不追踪外部进程，官方明说这不能替代 git [A1]。会话级撤销是练手草稿纸，工程级回退才是真正的退货政策。

**门禁（gate）** 是合并和上线之前必须过的人工关卡：谁能批、谁不能批、哪些动作必须人点头。这是把"独立验收"从口头约定变成流程硬约束。

## 为什么重要：AI 时代的变化

人写代码时，验收和回滚也重要，但有个天然的缓冲：人写得慢，一天就那么几个 PR，reviewer 跟得上，出了错半路上自己也会嘀咕一句"这块我没把握"。

agent 把这两个缓冲都抽走了。它写得快——按访谈口径，一个工程师同时开 5 个 Claude 实例、各自一份独立 checkout，一天能推 20–30 个 PR（二手，见下节）。量一上来，"reviewer 跟得上"就不再自动成立，潦草验收的诱惑陡增。它还不会自我怀疑：agent 不懂的地方照样一本正经地往下写，trace 里看着步步有理，结果可能整段跑偏。

更微妙的是**自己给自己打分的盲区**。让写代码的 agent 顺手评审自己刚写的代码，它会偏袒——这跟人不愿推翻自己刚下的判断是一个道理。Anthropic 给出的解法是结构性的：让评审在一个**全新的 subagent 上下文**里跑，它只看到 diff 和你给的验收标准，看不到产出这段代码的推理过程，于是按结果本身来评，而不是顺着原来的思路点头 [A1]。这就是"验收员不能是交活那个人"在 agent 上的对应物。

回滚同理。生成越快，错误被复制进更多文件也越快；一条随时能退回原状的路，因此更值钱，不是更不值钱。

## 实践方法

### 给 AI 改动的一份 PR review 清单

看 diff 只是第一层。一份完整的验收，逐项过下面这些——任何一项答不上来或答得心虚，就别合：

```text
[ ] 需求理解
    agent 是否真听懂了 issue 要解决的问题，而不是解决了一个相邻的、更好做的问题？

[ ] 改动范围
    每个被改的文件，是否都能追溯到 issue？有没有"顺手"改了范围外的东西？

[ ] 测试质量
    新增测试测的是真实行为，还是在迎合实现（把当前输出钉死成"期望值"）？
    删掉实现、只看测试，能反推出需求吗？

[ ] 风险面
    有没有触及：安全 / 权限校验 / 数据迁移 / 并发 / 破坏性命令 / 向后兼容？
    这些是高风险区，必须逐条人审，不能 auto-merge。

[ ] 验证证据
    agent 是否真的跑了验证命令？输出贴出来了吗？证据可信吗，
    还是只有一句"测试通过"？

[ ] 失败与未验证项
    失败的步骤、跳过的检查、"我没验证 / 我不确定"的假设，有没有如实说明？
    （没坦白比有失败更危险。）

[ ] 可维护性
    换个人接手，看得懂为什么这么改吗？还是只有 agent 当时的上下文里才说得通？

[ ] 回滚影响
    这个改动要是出事，回滚的影响面清楚吗？有没有不可逆的部分
    （已发出的通知、已写脏的数据、已跑的迁移）？
```

这份清单和 Anthropic 的几条建议是咬合的：让 agent 拿验证证据说话 [A1]；用一个对抗式评审 subagent 在新上下文里只比对 diff 和计划、报告 gap [A1]。但有个反向的坑要提防——Anthropic 自己也点了：一个被要求"挑毛病"的评审，几乎总能挑出点什么，哪怕活本身没问题，因为它的任务就是挑刺；照单全收会导致过度工程（多余的抽象层、防御式代码、为不可能发生的情况写测试）。所以清单里"风险面"那条要收紧到"真影响正确性或既定需求的 gap"，其余当可选项 [A1]。

### 回滚与门禁怎么设计

把"独立验收"和"可回滚"落成流程硬约束，看 GitHub 给它云端 coding agent 划的默认边界 [G1]：

- agent **只能 push 到它自己创建的分支**，碰不到默认分支和团队分支。
- **发起任务（让 agent 开 PR）的人，不能去批准这个 PR**——你仓库里设的"required reviews"规则照常生效。这是职责分离：自己交的活不能自己签收。
- GitHub Actions 工作流**不经你批准不会跑**，给你一个抽查 agent 代码的机会。
- agent 的联网被限制在一份你可自定义的可信目的地白名单里。
- 仓库已有的 branch protection、ruleset、组织策略一并生效 [G1]。

把这几条翻译成你自己团队的规矩：高风险代码（安全、迁移、权限）必须走 trace review 加 CODEOWNERS 指定的人审，不能 auto-merge；agent 开的 PR 默认进 draft，人批准后才触发 CI；合并前确认回滚路径存在（git revert 干净、没有不可逆副作用）。会话级的 `/rewind` 和 checkpoint 用来在干活当下快速试错、撤销 [A1]，但别拿它当上线后的退货政策——那得靠 git 和分支保护。

## 真实案例

### Claude Code 的工程精神（二手，来自 Boris Cherny 访谈）

下面这组说法来自 Gergely Orosz（The Pragmatic Engineer）对 Anthropic Claude Code 负责人 Boris Cherny 的一期访谈 [S8]。**这是二手来源——访谈口径的观点和经验，不是 Anthropic 的一手产品文档或官方基准**。引用时照此理解。需要说明的是，Boris Cherny 还有别的访谈（Latent Space、Lenny's Newsletter、YC、YouTube 等），本书只抓取了这一篇，所以下面的引述限于这一篇里能核到的内容。

- **先探索、再计划、再执行；好计划几乎一遍过。** 访谈里 Boris 的原话是："一旦有了好计划，实现几乎每次都能一遍过"（once there is a good plan, it will one-shot the implementation almost every time）[S8]。这条和 Anthropic 一手文档里"先探索、再计划、再编码"的四阶段工作流是同一个意思 [A1]——一手文档把它写成了可操作的步骤，二手访谈给了它一句经验背书。注意因果方向：不是"模型神所以一遍过"，是"计划做对了所以一遍过"。

- **一人一天 20–30 个 PR。** 访谈称 Boris 通过同时跑 5 个并行 Claude 实例（5 个终端标签，各自独立 checkout）、采用计划先行的方式，一天推 20–30 个 PR [S8]。能这么干靠的是隔离：5 份独立 checkout 就是 5 间互不干扰的操作间，谁也覆盖不了谁。

- **glob + grep 故意不用向量库。** 访谈提到，Claude Code 的代码检索用的是基础的 glob 和 grep，而不是 RAG 或向量数据库——这套被证明更有效，还绕开了索引过期的问题 [S8]。这是个值得记住的反直觉判断：简单胜过复杂。有意思的对照是 GitHub 那边——它的 coding agent 恰恰用 GitHub 代码搜索驱动的 RAG 来分析代码库 [G1]。同一类问题，两家选了相反的路；不存在唯一正解。

- **质量不是道德要求，是吞吐量要求。** 访谈引 Boris 在 Meta 的经验：代码质量对工程生产力有"可测量的、两位数百分比的影响"[S8]。干净的代码库对人和模型都更友好——这正是为什么前面那份 review 清单里"可维护性"是一条硬指标，不是软建议。

这些都是访谈里的口径，当经验参考，别当成厂商承诺的硬数字。

### 三家工具的形态对比

| 维度 | GitHub Copilot coding agent [G1] | Claude Code [A1] | Boris 的并行用法（二手）[S8] |
|---|---|---|---|
| 隔离单元 | GitHub Actions 起的虚拟机 | worktree / 沙箱 / 云端隔离 VM | 5 份独立 checkout |
| 代码检索 | GitHub 代码搜索驱动的 RAG | 基础 glob + grep（二手口径）[S8] | 同 Claude Code |
| 交付物 | 推到自建分支的 draft PR | PR / commit | 一天 20–30 个 PR |
| 谁来验收 | 不能是发起人；CI 需人工批 | 新上下文的对抗式评审 subagent | 计划先行 + 测试当验收 |
| 回滚 | branch protection / ruleset / 不能直接 push 主干 | `/rewind` + checkpoint（非 git 替代）+ git | 隔离 checkout 互不污染 |

形态不同，骨架一致：隔离的操作间、独立的验收、可回滚的边界。

## 常见反模式

- **只看 diff 就合。** diff 漂亮不等于需求被听懂、命令被真跑过。少看了 trace 和证据，等于只验墙面平不平，不翻施工记录。

- **让写代码的 agent 自己评审自己。** 它会偏袒自己刚写的东西。要换一个干净上下文来审——这是结构问题，不是它"态度"问题 [A1]。

- **发起人批准自己 agent 开的 PR。** 自己交活自己签收，职责分离形同虚设。GitHub 把这条做成了默认硬约束 [G1]，你的团队也该有等价规则。

- **高风险代码 auto-merge。** 安全、迁移、权限、并发这些区，测试绿也不能自动合，必须人审加 CODEOWNERS。

- **把会话级撤销当回滚保险。** `/rewind` 和 checkpoint 只追踪 Claude 自己的改动、不替代 git [A1]。上线后真出事，靠的是 git revert 和分支保护，不是 `/rewind`。

- **照单全收对抗式评审的每一条。** 评审被要求挑刺就一定挑得出，全改会导致过度工程 [A1]。只追"影响正确性或既定需求"的，其余当可选。

- **迁移开了头不收尾。** Boris 的一条经验（二手）："开了 migration，一定要把它做完"[S8]。半截的迁移是回滚噩梦——数据卡在中间态，退也退不干净。

## 检查清单

- 我验收时，是不是 diff、trace、证据**三件套**都看了，而不是只看 diff？
- 评审是不是由**独立的上下文 / 人**来做，而不是让写代码那个 agent 给自己打分？
- 高风险改动（安全 / 迁移 / 权限 / 并发）有没有走 **trace review + CODEOWNERS**，而不是 auto-merge？
- **发起任务的人，是不是被挡在"批准这个 PR"之外**？
- CI / 上线前有没有**人工批准**这一关？
- agent 声称的验证，有没有**可信证据**（命令 + 输出），还是只有一句"测试通过"？
- 失败项和"我没验证"的假设，agent 有没有**如实说明**？
- 这个改动出事了，**回滚影响面清楚吗**？有没有不可逆的部分？
- 我有没有把**会话级撤销当成了工程级回滚**（git 才是真退路）？

## 自测

- 把你上一个 agent 开的 PR 拉出来：你当时是看了 trace 和证据，还是 diff 一过就合了？如果只看了 diff，你怎么知道它声称跑过的命令真跑了？
- 你团队里现在，发起任务的人能不能批准自己 agent 开的 PR？如果能，你的"独立验收"是写在流程里，还是只活在大家的自觉里？
- 上一个 agent 改动如果此刻要回滚，你能一句话说清影响面吗？有没有已经不可逆的部分（发出的通知、写脏的数据、跑了一半的迁移）？答不上来，就是你回滚设计的体温计。
