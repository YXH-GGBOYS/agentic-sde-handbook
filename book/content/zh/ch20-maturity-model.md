把 agent 用进生产，不是一个开关，是一段台阶。最底下是"遇到问题去聊天框问一句、把答案复制回来"，最顶上是"一支能并行交付、失败会自动变成下次的考题的外包班组"。中间隔着好几级，每一级要补的东西都不一样——补的多半不是更强的模型，是模型外面那套环境。

这一章给一个 L0–L5 的成熟度模型，每级讲清两件事：这一级长什么样，以及要上下一级该补哪一样。它的用法不是"赶紧冲到 L5"，而是先诚实地认出自己在哪一级，再只补下一级要补的那一样。跳级才是最容易出事的地方。

## 生活类比

一家小作坊用外包，也是分阶段的。

最开始，你遇到一个搞不定的活，临时打电话问一个懂行的朋友："这个该怎么弄？"他口头给你支个招，你挂了电话自己照着做。这管用，但每次都得你转述、你动手，他从没进过你的车间，不知道你的料放哪、规矩是什么。

再往上，你开始让一个临时工进门帮你做一两件具体的事，但你得全程盯着——他每拧一颗螺丝你都得看一眼对不对。再往上，你给他配了工位、贴了工具墙、立了几条规矩，他能照着规矩独立做完一件交回来给你验。再往上，是好几个外包同时在不同操作间干活，有人统筹、有人验收，你管的是"谁先做哪件、做完谁查"，而不是每一颗螺丝。

最成熟的那一档，是这套班组会自己学：哪种活老出错，就把它编成一条新规矩贴上墙、编进验收单，下次同类活进来自动被卡住。从"打电话问朋友"到"一支会自我改进的班组"，中间这几级，每一级缺的东西都很具体——缺工位、缺规矩、缺验收员、缺统筹、缺复盘机制。哪一级没补齐就硬往上跳，结果都是同一个：活越干越快，错也越抄越快。

## 工程定义

成熟度模型，是一把按"harness 完备程度"排序的尺子。它衡量的不是模型多强，而是你给 agent 搭的那套环境完备到哪一步：能不能独立执行、在哪执行、谁来验收、出错怎么撤、失败会不会被记下来变成下次的防线。

下面是六级。每一级的"特征"是你现在就能照镜子的样子，"下一步"是上一级要补的那一样东西。

| 级别 | 名称 | 特征 | 上一级要补的那一样 |
|---|---|---|---|
| L0 | Chatbot 辅助 | 在聊天框里问答、手动复制粘贴；repo 里没有给 agent 的指令；没有任何自动验证 | 把"怎么跑测试、怎么 lint、怎么起服务"写进 repo；建一份问题模板 |
| L1 | 个人 AI 编码 | 在 IDE / CLI 里让模型生成代码，人逐行肉眼检查 | 把测试、lint、运行方式固化进 repo，让"算不算对"有可执行判据 |
| L2 | Agent-ready repo | 有 AGENTS.md、issue 写得清楚、CI 能在干净环境跑起来 | 加 sandbox（隔离执行）、trace（留痕）、任务按风险分级 |
| L3 | 委派式 PR | agent 从 issue 自己开 PR，人来 review | 加 eval（自动验收）、自动风险扫描、可回滚 |
| L4 | 受管 agent | 多个 agent 并行，有调度、有审查 | 量化成本、质量、吞吐、失败类型，让规模可被管理 |
| L5 | 会学习的 harness | 失败样本自动进 eval、规则持续更新 | 把 agentic 交付纳入组织级的工程治理 |

几点要先说清，免得把这把尺子用歪。

一是，级别是给**某条工作流**打分，不是给整个公司打分。同一家公司，前端可能在 L1，后端的某条 CI 流水线已经在 L3。别问"我们公司是几级"，问"这条工作流是几级"。

二是，每一级的门槛是**环境**，不是模型。从 L2 到 L3，差的不是换个更聪明的模型，是补上 eval 和回滚。模型再强，没有验收和撤回，你也不敢让它自己开 PR 合进去。

三是，台阶要**一级一级上**。L2 的标志是有 sandbox 和 trace；L3 在它之上加自动验收和回滚。你不可能跳过 sandbox 直接做"多 agent 并行"——没有隔离操作间，几个 agent 在同一个工作目录里互相覆盖，这一章后面会讲到这个具体的坑。

## 为什么重要：AI 时代的变化

这把尺子的意义，是逼你诚实回答"我到底在哪一级"，从而不去做超出当前 harness 能兜住的事。

跳级的代价，是自动化跑在了护栏前面。你在 L1（人逐行检查）时，模型抄一个坏模式进来，你当场就拦下了；可一旦你跳到"agent 自己开 PR 自己合"（L3 的动作）却没补上 L3 的 eval 和回滚，同一个坏模式会在你没看的时候被复制进几十个文件。生成越快，错也抄得越快——这不是新道理，但在 agent 把生成速度抬高一个量级之后，它从"偶尔踩一脚"变成"系统性风险"。

OpenAI 那个全 agent 生成的内部产品，给这件事提供了一个反面教训的注脚：团队曾经每周五（一周的 20%）专门手动清理 agent 留下的"slop"——重复的、次优的模式，agent 会照着 repo 里已有的样子继续复制 [O1]。手动清理扛不住规模，他们最后是把"什么是好模式"编成机械规则、配上一套定期自动清理才解决的 [O1]。这恰恰是 L4→L5 那一级的形态：失败和坏模式不再靠人一次次擦，而是被编码成持续生效的防线。

反过来，没到那一级却硬装成到了，代价是看不见的债。L0 的人以为自己"已经在用 AI 了"，其实每次都在重新转述、重新动手，agent 从没真正进过车间——这种用法不会随时间自动长成 L3，它会一直停在 L0，只是用的人误以为自己在前进。这把尺子的第一个用处，就是把这种错觉戳破。

## 实践方法

落地的原则只有一条：认准当前级别，**只补下一级要补的那一样**，补完再往上看。下面按级别给出具体动作和卡点。

**L0 → L1：让 repo 能自己说清楚怎么跑。** 最小动作是在 repo 里写下三件事——测试怎么跑、lint 怎么跑、服务怎么起。再加一份问题模板（这个 issue 要描述什么、复现步骤放哪）。卡点：很多人停在 L0 是因为"我问我的、它答它的"已经够用了，没意识到这条路通不到自治——agent 始终没有可执行的"算对"判据，你就永远得在每一步当那个判据。

**L1 → L2：让陌生人也能在干净环境里把活跑起来。** 加三样：一份 AGENTS.md（贴在冰箱上的家规便条，写动作、命令、禁区，不是写"我们追求卓越"那种散文）；让 CI 能在一个干净环境里从零跑通；加 sandbox（隔离执行）和 trace（留痕）。这一步的判据很硬：一个新来的 agent（或新来的人）只读 repo，不问任何人，能不能把环境起起来、把测试跑绿？起不来，你就还在 L1。

**L2 → L3：补上验收和撤回，才敢放手开 PR。** 这一步加 eval（自动验收）、自动风险扫描、可回滚。GitHub 给它云端 coding agent 划的边界，就是 L2–L3 这条线的标准形态 [G2]：你把一个 issue 指派给 agent，它自己开一个 draft PR、一边干一边把关键步骤记进 PR（这是 trace）；它只能 push 到自己建的分支（copilot/* 这类），不能直接动主干；所有 PR 都必须有人独立 review；CI 工作流不经批准不会跑；联网走防火墙白名单 [G2]。这些不是 prompt 里的客套话，是工具层和流程层硬卡出来的。卡点：很多团队跳过 eval 直接开自治——让 agent 能开 PR、测试一过就自动合。一句硬规则：没有 eval，就没有自治。

**L3 → L4：能并行，前提是先有隔离和调度。** 这一步是让多个 agent 并行干活，有人统筹、有人审查。技术前提是 sandbox 必须真隔离——每个 agent 一份独立 checkout / 一个独立工作树，否则几个实例在同一个目录里互相覆盖。Anthropic 把多 agent 这件事拆成了 planner、generator、evaluator 三个角色：planner 把一句话的需求展开成完整 spec，generator 一次只挑一个功能来实现，evaluator 用浏览器自动化（Playwright）像真用户一样点一遍、按几条硬标准打分，任何一条低于阈值这一轮就算失败、把具体反馈打回 generator [A3]。值得记的一点是：把"干活的"和"验收的"拆成两个 agent，比让同一个 agent 既干又自评要可靠得多——LLM 评自己的活几乎总是偏松，单独调一个"挑刺的"评审 agent 反而好调 [A3]。

**L4 → L5：把失败编进 eval，让 harness 自己变好。** 最高一级的标志，是失败样本不再靠人记，而是自动进 eval、规则持续更新，并且整件事被纳入组织级的工程治理。OpenAI 那个内部产品给了一个具体形态：他们把"什么是好模式"编成一组机械规则（golden principles），配一套定期跑的后台 Codex 任务去扫偏差、更新质量评级、开针对性的重构 PR——大多数这种 PR 一分钟内能 review 完、自动合并 [O1]；还有一个常驻的"doc-gardening" agent 专门扫和真实代码对不上的过时文档，自动开修正 PR [O1]。人的判断被捕捉一次，然后持续作用在每一行代码上 [O1]。

## 真实案例

**三个人，约 1,500 个 PR：受管 agent 怎么扩。** OpenAI 一个内部产品做了一件极端的事：五个月里，0 行手写代码——应用逻辑、测试、CI 配置、文档、可观测性、内部工具，每一行都由 Codex 写，由人来掌舵 [O1]。规模上，这个 repo 累计约一百万行代码，约 1,500 个 PR 被开出并合并，背后只有三个工程师在驱动 Codex，人均每天约 3.5 个 PR [O1]。最反直觉的一个数字：当团队从 3 人涨到 7 人，人均吞吐不降反升 [O1]。这不是"模型神"能单独解释的——能这么干，靠的是他们把 AGENTS.md 当目录而非百科（约 100 行，只做地图、指向更深的真相源），把 review 几乎全推成 agent 对 agent，把人的精力留给最稀缺的那一项：人的时间和注意力 [O1]。

**brain 和 hands 解耦：受管 agent 怎么扩得更稳。** Anthropic 讲它的 Managed Agents 时，给出了扩展的另一半答案：把 brain（决策——Claude 加它的 harness）和 hands（执行——sandbox 和工具）以及会话日志解耦开 [A4]。最早的设计把三样东西塞进同一个容器，结果成了运维口中的"宠物"——一台有名字、得人工伺候、丢不起的机器：容器一挂，会话就没了；harness 还假设所有资源都在同一个容器里 [A4]。解耦之后，harness 变成无状态的，把容器当成一个工具来调（`execute(name, input) → string`）；容器挂了就当一次工具调用错误，Claude 换一个新环境重试就行 [A4]。安全上还多了一层好处：Claude 生成的不可信代码跑在 sandbox 里，凭据（token）根本不进那个 sandbox [A4]。这个解耦带来的性能改善是实打实的：首字延迟（TTFT）p50 降了约 60%，p95 降了 90% 以上——因为推理可以立刻开始，不必等容器开好 [A4]。

**issue → agent → PR → 人 review：L2–L3 的典型形态。** GitHub Copilot 的 coding agent 把这条流水线做成了产品：把一个 GitHub issue 指派给它，它在一个安全、用完即弃的环境（由 GitHub Actions 驱动）里探索代码、改动、跑测试和 lint，开一个 draft PR、把关键步骤一路记进去 [G2]。边界前面讲过——只能 push 自己的分支、PR 必须人独立 review、CI 不批不跑、联网走白名单 [G2]。这是一个"委派 + 验收"被产品化的样本：委派不是甩手，是带验收门的授权。

**组织级的使用面在变宽。** Codex 现在每周有三百多万开发者在用，覆盖软件开发全生命周期——写代码只是入口，越来越多的人用它去读懂系统、收集上下文、review 工作、调 bug、推进长跑任务 [O7]。这说明 coding agent 的使用早已不只是"个人在 IDE 里补全"，而是组织级地铺到了多种工程工作流上——这正是成熟度往 L3、L4 走的外部信号。

## 常见反模式

- **跳级。** 没有 sandbox 就上多 agent 并行；没有 eval 就开自治。最常见也最贵的一种。判据很简单：你要做的那个动作（自动合并、并行、自治），它依赖的那一层护栏建了没有？没建就是跳级。
- **以为换更强的模型能跨级。** 级别的门槛是环境不是模型。从 L2 到 L3 缺的是 eval 和回滚，换个更聪明的模型补不上这个缺口——它只会更快地把没人验收的改动塞进来。
- **靠 prompt 当安全边界。** 在 system prompt 里写"不要 push、不要泄露 secret"就当防住了。真正的边界在工具层和 sandbox：agent 只能 push 自己的分支、凭据不进 sandbox，这些是工具和权限层硬卡的，不是嘴上说的 [G2][A4]。
- **把 AGENTS.md 写成百科全书。** 一份巨大的指令文件会挤掉任务、代码和真正相关的文档，而且会瞬间腐烂成一堆没人维护的过时规则——"什么都重要就等于什么都不重要" [O1]。把它当目录，指向更深的真相源 [O1]。
- **L4 不量化就扩。** 多 agent 并行跑起来了，却说不清单位成本、失败类型、质量和吞吐。规模没有度量就是失控——你连"这套到底比人快还是只是更吵"都答不上来。
- **以为 harness 一次搭好就一劳永逸。** harness 里每个部件都编码了一条"模型自己做不到"的假设，这些假设会随模型变强而过期 [A4][A3]。Sonnet 4.5 需要 context reset 来对付"上下文焦虑"，换到 Opus 4.5 这毛病基本没了，那些 reset 就成了纯开销 [A4]。新模型一落地，回头把不再吃劲的部件拆掉 [A3]。

## 检查清单

- 我能说清我**这条工作流**现在在 L 几吗（不是"我们公司几级"）？
- 我下一步要做的那个动作，**它依赖的那层护栏建了没有**？（要自动合并 → 有 eval 吗？要并行 → 有真隔离吗？）
- 一个新来的 agent 只读 repo、不问人，能把环境起起来、把测试跑绿吗？起不来就还没到 L2。
- "算完成"有没有**可执行的判据**（测试、验收标准），还是只能"看起来对了"？
- 要批准的动作，是卡在**工具/权限层**，还是只写在 prompt 里？
- 出错了有没有**撤回路径**（分支、快照、回滚）？强自动化必须配可撤销。
- 到了 L4，我能**量化**成本、质量、吞吐、失败类型吗？还是只看得见"跑了很多"？
- 失败样本是**进了 eval、变成下次的防线**，还是每次靠人重新擦一遍？

## 自测

- 把你最常用的那条 agent 工作流摆出来，对照六级特征，它落在哪一级？你最想做的下一个动作，是不是跳了级？
- 如果现在让一个 agent 自己开 PR 自己合，你睡一觉醒来最担心什么？你的答案，基本就指向你还没补上的那一层护栏——多半是 eval 或回滚。
- 上一次 agent 干砸一件活，那次失败被记下来了吗？它有没有变成一条规则、一道 eval，让下一个 agent 同样的坑不会再踩？如果没有，你的 harness 还不会学习——还在 L5 之下。
