很多人对 agent 的理解还停在"一个模型加一句话"：写一段足够聪明的 prompt，剩下的交给模型。这套想法能跑通玩具任务，一碰到真实工作就漏。真实工作里你要替它决定的，远不止那句话：它能动哪些东西、动之前要不要先问、在哪儿干、干完谁来验、搞砸了怎么收场。这些都不在 prompt 里。

把 agent 当成一个工作单元，而不是一句指令，是这一卷所有内容的起点。一个工作单元有职责边界、有交付物、有验收人、有出问题时的交代方式。一句 prompt 给不了这些——它只是这个工作单元的其中一格。

## 生活类比

你要雇一个新外包。他手艺可能比你强，但他第一天进门，对你这摊活一无所知：你家代码怎么组织、哪些目录是雷区、上线走什么流程、什么事必须先问你一声。

你不会只甩给他一句"把登录这块儿做了"就走人。真要让他干活，你得替他把一整套都定下来：他**能干哪些活**（只是看看，还是能改文件、能跑脚本、能提交）；给他**哪些工具**（哪台机器、哪个仓库、哪些账号）；给**多大权限**（能不能碰生产库、能不能往外发请求）；**什么时候必须把活交回来**（做到哪一步得停下来等你点头）；**谁来验收**（他说"做完了"不算数，得有人查）；**出了岔子他怎么交代**（你得能看到他到底改了什么、为什么这么改）。

这一整套交代下来，才叫"派了一个活给他"。少了任何一格都会出事：不定能动什么，他可能直接改了你不让碰的目录；不定验收人，他测试一过就当交付了；不定交代方式，出了问题你连他干了啥都不知道。一句话给不了这一整套——它顶多是你贴在他工位上的便条里的一行。

## 工程定义

一个 agent，是**目标 + 工具 + 上下文 + 权限 + 评估方式**捆在一起的一个带职责边界的工作单元。拆开看：

- **目标**：这次要它交付什么，以及什么算交付完成。
- **工具**：它能调用的那组动作——读文件、改文件、跑命令、查数据库、开 PR。
- **上下文**：它眼前摆着的材料——哪段代码、哪条规矩、上一轮的什么结论（这是上一卷 context 工程的题目）。
- **权限**：上面那些动作里，哪些它能自己来，哪些必须先问人。
- **评估方式**：怎么判断它干的活合不合格，由谁来判断。

作为搭这个工作单元的 SDE，你要替它拍板的，正是 prompt 之外的那几件事：它**能不能自行动手**（哪些动作放开、哪些卡住）、**什么时候把活交回来**（哪一步必须停下等人）、**如何向人解释**（留下什么样的痕迹让人能接手、能复盘）、**如何被评估**（用什么判据、谁来验）、**如何和别的 agent 协作**（多个 agent 同时干活时怎么分工、怎么不打架）。

这里有一个好记的框架，把一个 agent 的工作方式说清楚了。Anthropic 描述 Claude Code 时给出的工作回路是四个阶段，循环往复：**收集上下文 → 采取行动 → 验证成果 → 重复**（gather context → take action → verify work → repeat）[A14]。这四步对应着工作单元的几格：收集上下文用到它的上下文和检索工具，采取行动用到它的工具和权限，验证成果用到它的评估方式。同一篇里说得很直白——给 agent 派活，你要给它的"不只是一句 prompt：它得能自己去取上下文、更新上下文"[A14]。这句话本身就是"agent 不是一句 prompt"的官方注脚。

把这四步落到工程里，每一步都有要你决定的事。收集上下文：让它读哪些、不让它翻哪些。采取行动：哪些动作放开、哪些要批准。验证成果：A14 给了三种具体的验证手法——定义明确的规则（比如让它跑 lint，告诉它哪条规则没过、为什么）、视觉反馈（UI 任务截图回传让模型自己比对）、用另一个模型当裁判（LLM as a judge，但这一种不太稳、还有延迟代价）[A14]。重复：把上一轮的结果接上，继续推进，直到验证通过。

## 为什么重要：AI 时代的变化

把 agent 当成"一句 prompt"，在 AI 时代会越来越危险，原因有三。

第一，agent 现在真的能动手。它能改多个文件、能跑命令、能联网、能开 PR。能动手就意味着能闯祸。一个只会聊天的模型，最坏不过给你一段错代码；一个能动手的 agent，最坏会把错代码改进十个文件、跑掉一条危险命令、往生产库写一笔脏数据。能力越大，"它能动哪些东西、动之前要不要问"这个决定就越要紧——而这个决定根本不在 prompt 里，在权限层和 sandbox 里。

第二，生成越快，错误抄得越快。一个模糊的目标、一个缺失的验收，会被 agent 飞快地复制进更多文件。这正是为什么"验证成果"必须是工作单元里固定的一格，而不是你事后偶尔抽查一下。A14 把这点说得很清楚：能自查自纠的 agent"从根本上更可靠——它们在错误滚雪球之前就抓住、在跑偏时自我纠正、在迭代中越做越好"[A14]。反过来，没有验证这一格，跑得越快越是在加速犯错。

第三，工作单元的边界，决定了它能不能被复用和编排。一个定义清楚的工作单元——有明确的目标、固定的工具集、确定的验收——你可以把它当一块积木，交给别的 agent 去调度。Codex 就是这么用的：把 Codex CLI 跑成一个 MCP server，它就成了 Agents SDK 工具链里的一员，专门负责生成、修改、测试和审查代码这一类活[O3]。一句散漫的 prompt 没法被这样调度，因为它的边界是糊的——别人不知道把活交给它会得到什么、它会动到哪儿。

## 实践方法

把"派一个 agent"当成"搭一个工作单元"来做，落地有几个固定动作。

**先探索、再计划、再执行，别让它一上来就改代码。** 这是 Claude Code 最反复强调的一条。A1 说得直接：让 Claude 直接跳进编码，"可能产出解决错问题的代码"[A1]。它给的工作流分四步：先 explore（进 plan 模式，只读文件、回答问题，不做改动），再 plan（让它产出一份详细的实现计划），再 implement（切出 plan 模式，按计划写代码并对照验证），最后 commit（写清楚的提交信息、开 PR）[A1]。这四步不是仪式，是把"采取行动"这一格往后推，先让"收集上下文"和把计划摆到台面上这两步发生。当然不是每个活都要走全套——A1 也提醒，plan 模式有它的开销，作用域清楚、改动很小的活（改个 typo、加行日志、重命名变量）直接让它做就行；能用一句话描述出 diff 的，就跳过 plan[A1]。

**明确这个 agent 的自治分级。** 也就是上面那五格里的"权限"和"什么时候交回来"。在 Claude Code 里，这落到几样具体工具上：auto 模式（一个分类器模型先审命令，只拦看起来危险的——越权、碰陌生基础设施、被恶意内容驱动的动作）、权限白名单（放行你确定安全的命令，比如 `npm run lint`、`git commit`）、sandbox（操作系统级隔离，限制文件和网络访问，让它在划定的边界内更自由地干）[A1]。在 Codex 里，自治分级是几个显式参数：`approval-policy` 控制命令要不要批准（`untrusted` / `on-request` / `never`），`sandbox` 控制隔离级别（`read-only` / `workspace-write` / `danger-full-access`）[O3]。无论哪套，要点是一样的：自治不是一个开关，是一组按动作分级的旋钮。

**给它一个能自己跑的验收，而不是靠你盯。** 这是"验证成果"那一格落地的样子。A14 给的三种手法（规则、视觉反馈、LLM 裁判）按可靠度从高到低排：能用确定性规则（测试、lint、构建退出码）就别用模糊判断[A14]。一个独立的验收尤其值钱：A1 建议在算"做完"之前，加一道对抗式评审——让一个 sub-agent 在干净的上下文里复查 diff，它只看到 diff 和你给的判据，看不到产生这段改动的推理过程，所以是在按结果本身打分，而不是给自己批改[A1]。

**需要多个 agent 时，给它们一份共享的任务契约，别让它们靠聊天对齐。** O3 给的多 agent 例子里，一个 Project Manager 协调 Designer、前端开发、后端开发、测试四个角色；协调的办法不是让它们互相喊话，而是 PM 先建好几份共享文件——`REQUIREMENTS.md`、`TEST.md`、`AGENT_TASKS.md`——再带着对交付物的"有门禁的验证"去编排各角色的交接[O3]。每个 agent 仍是一个边界清楚的工作单元，契约写在文件里，不写在转瞬即逝的对话里。

## 真实案例

**一个工作单元的两种粒度：单 agent 和多 agent。** O3 给了一对很能说明问题的例子。单 agent 版本：建一个游戏，用 Designer 和 Developer 两个 agent，Designer 出三句话的创意简报，再 handoff 给挂了 Codex MCP server 的 Developer 去用 HTML/CSS/JS 实现[O3]。多 agent 版本：在前面基础上加一个 Project Manager，由它建好共享的需求和测试文件，再去协调 Designer、前后端开发、测试这几个角色的交接，每一步交付都带门禁验证[O3]。两版的差别不在"模型更强"，在工作单元的拆法和它们之间的契约——单 agent 是一个工位，多 agent 是一个班组加一张排班表。

**trace 让一个工作单元可被复盘。** 还是 O3 这套：Codex 会自动记录 trace，把 prompt、工具调用、各角色之间的 hand-off 都录下来，可以在 Traces 面板里回看，用来调试工作流的卡点、审计 agent 行为[O3]。这正对应工作单元那一格——"如何向人解释"。一句孤立的 prompt 跑完就没了，你不知道它中间读了什么、试了什么、在哪步交了班；一个有 trace 的工作单元，出了问题你能倒回去看每一步。这就是"工作单元"和"一句 prompt"在出事之后的差别。

**对抗式评审的一个真实陷阱。** A1 给了一条很诚实的提醒，值得记住：一个被要求"找问题"的评审 agent，几乎总会报出一些问题，哪怕活本身没毛病——因为找问题正是你叫它干的事[A1]。如果你对它报的每一条都照单全收，结果就是过度工程：多出一层抽象、一堆防御式代码、给不可能发生的情况写的测试[A1]。所以验收这一格本身也要有边界：告诉评审 agent 只标"影响正确性或明确需求"的问题，其余当可选[A1]。验收人也是工作单元里的一个角色，它的职责边界同样得划清楚。

## 常见反模式

| 反模式 | 表面上看 | 实际危险 |
|---|---|---|
| One mega agent（一个超级 agent 包打天下） | 省事，一个上下文装下所有事 | 上下文越堆越满、互不相关的任务互相干扰，模型开始"忘掉"早先的指令、出错变多。该拆成多个边界清楚的工作单元，各自干净上下文 |
| 把约束全写进 prompt | "我写一句'别碰生产库'就行了" | 权限不在 prompt 里，在权限层和 sandbox 里。一句话拦不住一个能跑命令的 agent |
| 让它一上来就改代码 | 快，省掉探索和计划 | 容易产出"解决错问题"的代码[A1]。先探索、再计划、再执行 |
| 没有验收就放它自治 | 测试过了就当交付 | 缺了独立验收，"看起来对"成了唯一信号，错误会滚雪球[A14]。没有 eval，就没有自治 |
| 对抗式评审照单全收 | 评审报的都修，看起来更稳 | 评审被要求找问题就一定找得到，全修会导致过度工程[A1]。只修影响正确性和明确需求的 |
| 多 agent 靠聊天对齐 | 让它们互相喊话就行 | 上下文碎片、协调成本失控。把任务契约写进共享文件，别写进对话[O3] |

这里单独把 **one mega agent** 拎出来，因为它最有迷惑性。一个超长上下文的 agent 看上去无所不能——什么都丢给它，它什么都接。但 A1 把这种"厨房水槽式会话"列为头号失败模式：你先让它干一件事，又问个不相干的，再绕回第一件，上下文里塞满了无关信息，性能随之下降[A1]。正确的方向是反过来的：把一个大目标拆成几个边界清楚的工作单元，每个有自己干净的上下文、自己的工具集、自己的验收，而不是把所有事压进一个永远在膨胀的会话里。

不同 agent 角色有各自的主要风险，派活前心里要有数：

| 角色 | 主要风险 |
|---|---|
| 代码实现 | 改动范围悄悄扩张、误解业务意图、测试覆盖不足 |
| 评审 | 误报漏报、只看表面 diff 不看上下文 |
| 研究 | 来源混乱、结论不可复现 |
| 规划 | 计划写得漂亮但脱离 repo 现实 |
| 运维 | 权限过大、误操作、报警疲劳 |
| 多 agent 编排 | 协调成本高、上下文碎片、预算失控 |

## 检查清单

派一个 agent 之前，把它当工作单元过一遍：

- 这个 agent 的**目标**，有没有写清"什么算交付完成"，而不是一句模糊的"做一下 X"？
- 它**能动哪些东西**？哪些动作放开、哪些必须先问人——这写在权限层/sandbox 里，还是只写在 prompt 里？
- 它**在哪儿干**？直接动真仓库，还是隔离的工作树/容器？
- **谁来验收**？有没有一个它自己能跑的判据（测试、lint、构建），最好再加一道独立的对抗式评审？
- 它**留下什么 trace**？出了问题，另一个人能不能据此接手、复盘？
- 这个活该让**一个 agent** 干，还是该拆成几个边界清楚的工作单元各自干？有没有不小心做成了 one mega agent？
- 如果是多 agent，它们的**任务契约**写在共享文件里，还是赌它们能靠聊天对齐？

## 自测

- 把你最近一次"派给 agent 的活"摊开看：你给的是一个完整的工作单元（目标、工具、权限、验收都定了），还是只给了一句 prompt 然后祈祷？缺的那几格，是你后来手动补上的，还是根本没人补？
- 如果现在把这次 agent run 的聊天记录清空，只留下它留下的 trace 和工件，下一个人接得住吗？接不住，说明你派的还是"一句 prompt"，不是"一个工作单元"。
- 你手头有没有一个 agent，正在悄悄长成 one mega agent——什么活都往它身上压、上下文越来越长？如果有，它该被拆成哪几个边界清楚的工作单元？
