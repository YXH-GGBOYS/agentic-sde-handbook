一项技术大幅降低某件事的成本，真正洗牌的不是这件事本身，而是它周围那一圈技能的价钱。打字快曾经值钱，文字处理软件普及后不值了；但"想清楚要写什么"反而更值钱，因为写得快意味着废话也产得快。

AI 把"把想法变成能跑的代码"这一步的成本压了下来。于是这一章要回答一个很实际的问题：你过去引以为傲的本事，哪些一夜贬值了，哪些只是权重下降，哪些反而升值了。把这三档分清楚，比笼统地喊一句"AI 时代来了"有用得多。

## 生活类比

想想买菜这件事，几十年的变化。

早些年在自由市场买菜，真正的硬本事是两样：**会砍价**，和**会挑**。摊主开价八毛，你能磨到五毛，这是钱；一堆西红柿里，你一眼挑出沙瓤熟透、不挑那些表皮光鲜里头发僵的，这是经验。两样本事里，砍价那一项往往更显眼——谁嘴皮子利索、脸皮厚、肯磨，谁当场就省下真金白银，旁人都看得见。

后来超市来了，明码标价。砍价这门手艺一夜贬值——标签上写五块就是五块，你磨破嘴皮也没用，议价能力直接清零。可同一时间，另外两样东西反而更值钱了。

一是**会挑、会验货**。超市货架上同一种番茄摆了七八个品种，进口的、本地的、有机的、催熟的，价签五花八门，便宜的那堆里混着发硬的、打蜡的、催熟没味的。选择多了，便宜的坑也多了——挑得准的人省的钱和吃的亏，比当年砍那三毛五毛大得多。二是**知道这顿饭要做成什么**。你要炖罗宋汤还是凉拌，决定了你该买哪种番茄；想不清楚这顿饭的目标，站在琳琅满目的货架前反而更容易乱买、多买、买错。

这就是这一章要讲的三档：砍价是**过时**了的本事（议价能力被明码标价清零）；眼疾手快地翻找当然还有用，但**降级**了（货比三家的体力活，没有"会挑"重要）；而**会挑会验、知道要做成什么**，反而**更重要**了——因为选项多了、廉价的坑也多了。AI 之于写代码，就是超市之于买菜：它把某些当场显眼的本事清了零，把另一些不那么显眼、却决定成败的判断力，顶到了台前。

## 工程定义

先把三档说清楚，免得"过时/降级/重要"变成口号。

**过时**，指的是某项做法或观念，在新的成本结构下不再成立，继续照做会主动产生损失，应当替换掉。不是"用得少了"，是"再用就错"。

**降级**，指的是某项技能仍然有用、偶尔还很关键，但它在你能力组合里的权重下降了——以前值得花大量时间专门练，现在不值得把它当核心竞争力来投资。它没消失，只是不再是你和别人拉开差距的地方。

**升值（反而更重要）**，指的是某项能力的边际价值上升了：因为别的环节变快、变便宜，这项能力成了新的瓶颈或新的稀缺点。判断一项技能属于哪一档，问一个问题就够了——**当"把想法变成代码"这一步接近免费时，这项技能是被它替代了，还是因它而更紧缺了？**

这套框架里有个反直觉的支点，值得单独点明：生成变快，错误也被更快地放大。一个模糊的需求、一处缺失的验收、一个搞错的契约，过去要靠人慢慢敲进代码，传播得慢；现在 agent 几分钟就能把它复制进十几个文件、铺满整个 PR。所以约束、验证、治理这些"踩刹车"的能力，边际价值在**上升**，不是下降——速度越高，刹车越值钱。

## 为什么重要：AI 时代的变化

把常见的几条旧观念逐个过一遍，每条给"为什么过时"和"现在怎么做"。

**"需求一次写完，照着做就行。"** 过时。它假设你在动手前就能把需求想全——可一旦实现成本逼近于零，更划算的做法是先让 agent 快速搭一版能跑的，拿它当探针照出你没想到的边界，再回头改需求。Anthropic 给 Claude Code 的官方建议把这步固化成了流程：先 explore、再 plan、最后才 code，明确把"调研和规划"和"实现"分开，理由正是"直接上手写，容易解决错的问题" [A1]（一手）。现在怎么做：把需求当成会随探索而改的草稿，而不是一次签字、照单施工的合同。

**"程序员只负责写代码。"** 过时。当写代码这一步接近免费，价值就从"产出代码"移到了它前后——把问题定义清楚、定架构、写规格、设验收标准、做 QA、审 diff。Simon Willison 数了一遍真正在用 agent 高效工作的资深工程师每天在干什么：研究方案、定高层架构、写规格、定成功标准、设计 agent 循环、规划 QA，"以及花大量时间做 code review" [S1]（二手）。他的结论是，这些几乎全是资深工程师本就该有的能力。现在怎么做：把自己的岗位重新定义为"对产出负责的人"，而不是"敲键盘的人"。

**"测试放到最后。"** 过时，而且现在代价更大。没有测试，agent 会信誓旦旦说"我测过了"其实根本没跑，一处改动还可能悄悄弄坏别的功能而你毫不知情。Willison 把自动化测试列为 agent 最能受益的实践之首，并指出测试先行（test-first）配上能在循环里自我迭代的 agent 尤其有效 [S1]。Anthropic 的说法更直白：给 Claude 一个它能自己跑的检查（测试、构建、截图比对），它就能自己跑完整圈——做、跑、读结果、再改，直到通过 [A1]。现在怎么做：把"可执行的验收"当成派活的前置条件，而不是收尾动作。

**"文档越厚越专业。"** 降级中带反转。一份给人看的、充满"我们高度重视代码质量"的漂亮散文，对 agent 几乎没用——它需要的是能直接照做的具体规则。但同时，**简洁、准确、机器能读的文档反而更值钱了**：Willison 指出，把相关文档喂给模型，它就能在不先读源码的情况下用上别处的 API；文档写得好，模型甚至能只凭这份输入就搭出匹配的实现 [S1]。现在怎么做：不是写得更厚，是写得更准、更短、更可执行。

**"用了 AI 就先进。"** 过时的自我安慰。工具不等于结果。S4 那篇文章里提到一个被反复观察到的"两步退"模式：用 AI 修一个 bug，常常引出好几个新问题 [S4]（二手）；它还点出"演示质量陷阱"——happy path 在 demo 里跑得漂亮，真实使用一上就露馅 [S4]。用了 AI 只是起点，活好不好得另说。现在怎么做：用结果（线上是否更稳、维护成本是否更低）判断，而不是用"我们上了 agent"自我安慰。

接下来是**降级**那一档——还有用，但别再当核心竞争力投资：

- **手写样板代码的熟练度。** 闭着眼睛敲出一个 CRUD、一套表单校验、一段 boilerplate，曾经是熟练度的标志。现在这正是 agent 最擅长、最先替你做掉的部分。它没用了吗？没有——你还得读懂、还得审。但把大量时间花在"我手写得比别人快"上，回报率掉了。
- **靠记忆背 API 细节。** 记得住一个库几十个函数签名、参数顺序、那几个反直觉的默认值，过去能让你少翻文档、显得熟。现在 agent 张口就来。这类记忆没坏处，但它不再是你和别人拉开差距的地方。
- **本地 IDE 个人效率崇拜。** 把自动补全、快捷键、代码片段调到极致，追求一个人单位时间敲更多行——这套"个人手速"叙事的权重在下降。瓶颈正从"我一个人能敲多快"，移到"我能并行委派、验收多少个 agent，并对它们的产出负责"。

最后是**升值**那一档——因为别的环节变快变便宜，这些成了新瓶颈：问题定义、架构的可逆性、契约的清晰度、可维护性、验证与回滚、判断力。下一节展开怎么练。

## 实践方法

把上面的框架落成几个可操作的动作：

1. **派活前先写验收，而不是收尾才补测试。** 把"怎么算完成"写成 agent 能自己跑的检查——一套测试、一个构建退出码、一段把输出和基线比对的脚本。Anthropic 的原话：给 Claude 一个能产出"通过/失败"信号的检查，循环就能自己闭合 [A1]。
2. **先 explore、再 plan、最后 code。** 不确定方案、要改多个文件、或对这块代码不熟时，先让 agent 只读不改地调研，再做计划、你过一遍，最后才放它写 [A1]。如果一句话就能描述这个 diff，跳过计划直接做 [A1]。
3. **把约束外置，别全塞进 prompt。** 速度上来后，刹车要硬。哪些目录不能动、哪些命令禁用、联网白名单——尽量落到工具层、权限层、CI 门禁，而不是只在提示词里好言相劝。
4. **保留撤回路径。** 强自动化必须配可撤销。Claude Code 每次改动前自动快照，可以 rewind 回任意检查点，也可以让它"撤销刚才那步" [A1]——但官方明确提醒：检查点只追踪 Claude 自己的改动，不能替代 git [A1]。
5. **练"会挑"而不是"会砍价"。** 把省下来的时间投到选项判断上：这个方案可逆吗？这个契约清楚吗？这块半年后还维护得动吗？这些就是超市时代的"会挑会验货"。

## 真实案例

**METR 的随机对照实验：自我感觉和实测对不上。** 这是目前关于"AI 到底让资深开发者快了还是慢了"最值得认真对待的一手测量，由 METR 做（我们这里二手转述）。设计很硬：16 位经验丰富的开源开发者，在他们贡献多年、平均两万二千多 star、上百万行代码的成熟仓库里，处理 246 个真实 issue（修 bug、加功能、重构），每个任务平均约两小时；每个 issue 被随机分到"允许用 AI"或"不允许用 AI"，全程录屏 [S7]。用的工具主要是 Cursor Pro 配 Claude 3.5/3.7 Sonnet——在当时是前沿模型 [S7]。

结果反直觉：允许用 AI 时，他们完成 issue 反而**慢了 19%** [S7]。更值得玩味的是感知与现实的裂缝：开发者事前预测 AI 会让自己**快约 24%**；亲身经历了变慢之后，他们**仍然估计** AI 让自己快了约 20% [S7]。也就是说，人在被拖慢的同时，主观上确信自己被加速了。

这个案例对本章有两层意思。其一，"生成更快"不等于"交付更快"——在熟手、熟代码、复杂仓库这种场景里，AI 引入的来回核对、纠正、不可靠输出，可能把省下的打字时间又吃回去。这正好印证：当生成变快，验证和约束的边际价值会上升。其二，自我感觉不可信，必须靠实测——这本身就是"验证能力升值"的一个注脚。

务必看清这个结果的边界，别外推过头。METR 自己写明：这是"early-2025 AI 能力的一张快照"，研究对象是资深开发者、在他们深度熟悉的成熟代码库上；作者明确声明**不主张**这些开发者或仓库能代表大多数软件开发工作，也不主张结论能推广到经验较浅的开发者、不熟悉的代码库或软件开发以外的领域 [S7]。该页后来还挂了更新提示，说 2026 年 2 月他们发布了关于 2025 年底 AI 工具的新数据 [S7]。所以正确的读法是：这条证据是在提醒我们，"快"是个需要验证的假设，而**不是**"AI 永远让人变慢"的结论——那是把一张早期快照、特定人群的样本，错当成了定律。

**一个温和但有力的对照视角。** S4 那篇文章里，Gergely Orosz 引了一组很老的数据做对照：Fred Brooks 在 1975 年估计，软件开发的时间大致是三分之一规划、六分之一编码、四分之一组件测试、四分之一系统测试；他给的现代估计是约 20% 规划、40% 编码与测试、20% 代码审查、20% 上线就绪 [S4]（二手）。两个时代相隔半个世纪，编码本身从来都只占一小块。文章那句话点得很准：软件质量从来主要就不受编码速度限制 [S4]。这正是本章的底色——AI 把占比本就不大的那一块压了下去，于是另外几块的相对权重升了上来。

## 常见反模式

- **把生成速度当生产力。** 看到 agent 几分钟铺出几百行就以为赚了。S4 的"两步退"提醒：用 AI 修一个 bug 常引出几个新 bug [S4]；速度要按净值算，不是按行数算。
- **把"过时"当"无用"，把"降级"当"该扔"。** 降级不是清零。手写 boilerplate 的熟练度权重降了，但你仍得读懂、审得动 agent 写的样板——看不懂就没法验收。把基本功彻底荒废，恰恰废掉了你"会挑会验"的底子。
- **拿"我们上了 AI"当成绩。** 工具不是结果。"用了就先进"是过时的自我安慰，真正的判据是线上是否更稳、维护成本是否更低。
- **把一张早期快照当永久定律。** 反过来也是坑：拿 METR 那个 19% 到处说"AI 让人变慢"，无视它"early-2025、资深人群、熟代码库"的明确边界 [S7]。两个方向的过度外推都得避开。
- **速度起来了，刹车没跟上。** 生成变快却不同步加强验收、约束、回滚，等于油门越踩越深、刹车还是原来那套——这恰恰是错误被放大得最快的配置。

## 检查清单

- 我能把自己的技能分进三档（过时 / 降级 / 升值）吗？我最近花时间最多的，是哪一档？
- 派活时，"怎么算完成"是不是一条 agent 能自己跑的检查，而不是"看起来对了"？
- 我的约束（禁区、禁用命令、联网白名单）是落在工具/权限层，还是只写在 prompt 里好言相劝？
- 出了错，我有没有可执行的撤回路径（分支、快照、回滚）？
- 我判断一个 AI 应用成不成，用的是结果（更稳 / 更省维护），还是"我们上了 agent"这种自我安慰？
- 引用"AI 让人变慢/变快"这类结论时，我有没有交代它的边界（什么工具、什么人群、什么代码库）？

## 自测

- 列出你三项最拿手的本事。逐项问：当"把想法变成代码"接近免费，这项是被替代了、降级了，还是更紧缺了？答得出，你就知道接下来该把时间投到哪。
- 你上一次用 AI"很快"做完一件事，事后回头量过净收益吗——把来回核对、返工、踩坑的时间都算进去之后，到底是快了还是慢了？如果从没量过，你对自己速度的判断，可能正落在 METR 那条感知裂缝里。
