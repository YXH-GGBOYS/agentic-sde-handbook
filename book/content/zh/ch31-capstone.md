毕业设计不是再考一遍概念。前面三十章讲的那些——AGENTS.md 怎么写、harness 九层是什么、eval 和 trace 各管什么——你能复述，不等于你会用。会用的唯一证据，是把整套东西在一个真实仓库上跑通一遍：建好家规便条，挑出能交出去的活，搭好工位和门禁，让 agent 真的干几次，然后把它干砸的地方一条条记下来。

所以这一章没有新概念。它是一张施工图：六个步骤，把全书的零件按顺序装到一个你选定的仓库上。能装上、能跑通、能讲清楚哪里跑不通——到这一步，你才算真学会了。这是一个综合章，不引入任何新事实；所有例子都是为了讲清做法而设的示意，不是对某个真实数字的引用。

## 生活类比

想象你要从"偶尔叫个临时工帮忙"，正式升级到"管一支外包班组"。升级那天，你不会先去面试更贵的人，你会先把工地本身收拾好。

你做的第一件事，是把这家工地的规矩写到冰箱便条上：料堆在哪、哪面是承重墙不能动、干完怎么交。第二件事，是把手头积压的活拿出来分一遍——哪些是"换个灯泡"级别可以放手让人干，哪些是"动燃气管线"级别必须老师傅在场。第三件事，是搭工位：给一间隔离的操作间（别让新人直接上客户家的承重墙练手）、一面贴好标签的工具墙、一道门禁（动贵设备得叫你按一下）、一个验收台、一台全程录像的摄像头。第四件事，才是真的派三个活下去试：一个让新人从头做、一个让他补上老活漏掉的检查、一个让他去验别人的活。第五件事，是把这三次的录像、改了什么、验收过没过、为什么砸了，全存下来，归成一张"这班人最容易栽在哪"的表。最后，你坐下来写一份小结：什么活适合交出去、什么活打死得自己来、下次怎么把工位改得更顺手。

这一整套——便条、分活、工位、试派、归档、小结——就是这一章要你做的事。注意没有一步是"换个更聪明的 agent"。整本书的主线在这里收口：你不是在驯服一个模型，你是在设计一个让会犯错的人也能交付的环境。

## 工程定义

**Capstone** 是一次端到端的 harness 设计与验证：选一个真实仓库，把全书的工程资产和 harness 各层在它上面实际搭起来，跑通至少三次 agent run，并把结果归档成可复盘的工件。它的产物不是一段漂亮的演示，而是一套别人能接手、能审计、能改进的东西。

它由六个步骤构成，每一步对应书里已经讲过的一卷或几章：

| 步骤 | 做什么 | 呼应的书卷/章 |
|---|---|---|
| ① 建指令与文档 | 给真实 repo 写 AGENTS.md / CLAUDE.md / README / runbook | 第三卷·新工程资产 |
| ② 分活 | 挑 10 个真实任务，按风险分级，写成 agent-ready issue | 第二卷·需求重定价；第六卷·写 agent-ready issue |
| ③ 搭 harness | 定权限、工具、sandbox、trace、eval、交接工件 | 第四卷·Harness 工程 |
| ④ 跑 run | 至少三次：一个实现、一个测试补全、一个 review | 第五卷·Agents 工程；第六卷·三大 agent 实战 |
| ⑤ 归档 | 存 trace、diff、验证结果、失败原因，做成 failure taxonomy | 第七卷·新度量与失败分类学 |
| ⑥ 写报告 | 什么适合委派、什么不适合、下一轮怎么改 harness | 第四卷·成熟度模型；第七卷·治理 |

这里要区分清楚两件容易混的事：Capstone 不是"用 agent 做完一个 feature"。做完一个 feature 只证明这次 agent 跑对了；Capstone 要证明的是你**搭的环境**能反复产出可接受的结果，并且在 agent 跑错时你有办法发现、追踪、收拾。前者是一盘菜，后者是一套能稳定出餐的后厨——这正是全书第一章的那个区分。

## 为什么重要：AI 时代的变化

把全书"用一遍"这件事本身，对应了一个真实变化：在 agent 时代，工程能力的瓶颈从"能不能写出这段代码"移到了"能不能设计、约束、验证一套交付系统"。前者现在很大程度上能外包出去，后者不能。Capstone 之所以是这本书的终点，是因为它考的恰好是那部分没法外包的能力。

读和做之间有一道很深的沟，这道沟在 agent 时代比以前更深。原因是 harness 的很多失败不在任何单层内部，而在层与层的缝隙里——你读"权限要分级"和"sandbox 要隔离"时都点头，可只有真的搭起来、真的让 agent 在里面跑，你才会发现：你以为配好了的禁区，agent 绕了个道还是读到了 secret；你以为测试当了验收员，agent 写的测试其实在迎合自己的实现。这些缝隙里的问题，光读读不出来，必须跑一遍才暴露。

还有一层更朴素的理由。前面的章节给的是分立的零件——指令文件、context、tool、sandbox、eval、trace。零件各自看都好懂，难的是把它们**装到一起还能转**。装配本身是一项技能，而且是 agent 时代新增的核心技能。一个能把这六步在真实仓库上走通的人，和一个只会复述六步定义的人，差的不是知识量，是判断力：哪一层在这个仓库上可以薄、哪一层必须厚、哪个任务根本不该交出去。这种判断只能从一次真实的装配里长出来。

## 实践方法

下面把六步展开成可执行的施工顺序。每一步都给"最低可用"的做法——一个人花几天能做到的版本，不是一支团队半年才能搭起来的版本。

**① 选 repo，建指令与文档。** 选仓库的判据很实际：你足够熟（能判断 agent 改得对不对）、有可跑的测试（否则没有验收员）、风险可控（别拿生产支付系统当练习场）。一个中等规模、有测试、你熟的内部工具或开源项目最理想。选定后，按第三卷的做法写四份文档：

- **AGENTS.md / CLAUDE.md**：项目形状、命令、完成的定义、安全边界（禁区）、review 期望。写成动作和命令，不是散文。
- **README**：人读的总览，让接手的人知道这是什么、怎么起。
- **runbook**：怎么起服务、怎么跑全套测试、常见故障怎么排——这同时是给 agent 的 context 和给下一个人的交接。

检验这一步的方法：把这几份文档给一个完全不懂这个项目的人，他能不能只照文档跑通测试、知道哪里不能碰。做不到，agent 也做不到。

**② 挑 10 个任务，按风险分级，写成 agent-ready issue。** 从仓库真实的 backlog 里挑，不要为了好交差专挑简单的。按风险分三档，像医院分诊台那样分流：

| 风险档 | 特征（示意） | 处置 |
|---|---|---|
| 低 | 局部、可逆、有测试覆盖：补文档、加一个工具函数、改文案 | 可以让 agent 较自主地做 |
| 中 | 跨几个文件、改公开接口、动测试 | agent 做，但每步人审，禁止自动合并 |
| 高 | 碰 migration、auth、secret、计费、并发 | 默认不交给 agent，或只让它出方案由人实现 |

每个 issue 按第六卷的写法补齐：背景、可验证的验收标准、要跑的验证命令、可动文件范围、非目标、已知风险。一个含糊的"修一下登录"和一份写清"复现步骤 + 期望行为 + 验证命令 + 别碰 session 存储"的 issue，交给 agent 产出的质量是两个量级。

**③ 设计 harness。** 把第四卷九层落到这个仓库上，每层只答"在这个 repo 上最低怎么做"：

- **权限**：哪些动作 agent 自己能做（读、改非禁区文件、跑测试），哪些要你点头（联网装新依赖、改 CI、写 migration）。把禁区写进工具/权限层，不要只写在 prompt 里——靠提示词做安全是反模式。
- **工具**：给它项目自带的测试、lint、typecheck 命令，命名清楚、错误语义清楚。别让它自己拼命令。
- **sandbox**：每次 run 在一个独立的 git worktree 或容器里跑，别让它直接动你的主工作区。这是"隔离操作间"，砸坏了波及不到生产。
- **eval**：哪些测试必须绿、哪条验收标准必须满足，写成可执行判据，不是"看起来对了"。
- **trace**：保证你能拿到这次 run 的完整留痕——它读了什么、跑了哪些命令、diff 是什么、哪步失败。
- **交接工件**：约定 run 结束要留下什么（计划、状态、决策、验证记录、交接说明），让你或下一个人能接手。

**④ 跑至少三次 agent run。** 三次要覆盖三种不同的活，因为它们考的是 harness 的不同层：

- **一次实现**：从 issue 到 PR，考的是 context、工具、sandbox、eval 全链。
- **一次测试补全**：给一段已有的、测试不全的代码补边界用例，考的是 eval 设计和"防迎合实现"。
- **一次 review**：让一个 agent 去审另一次 run 产出的 diff，考的是 trace 怎么当评审材料、evaluator 要不要独立。

每次都全程记录，别只存最后的 PR。

**⑤ 归档，形成 failure taxonomy。** 把三次 run 的 trace、diff、验证结果、失败原因全存下来，然后按第七卷的分类学把失败归类。哪怕三次都"成功"了，也要记下过程中的小偏差——它们是下一轮 harness 要堵的洞：

| 失败类别 | 它长什么样（示意） | 该改哪一层 |
|---|---|---|
| spec 规格 | agent 做了，但做的不是你要的——issue 没说清 | ② issue / ① 指令 |
| context 上下文 | 它没读到关键文件，或被无关材料带偏 | ③ context |
| tool 工具 | 命令拼错、误判工具输出、把环境噪声当成功 | ④ 工具 |
| model 模型 | 真的是推理/边界考虑错了 | 换模型 / 拆任务 |
| infra 基础设施 | 依赖装不上、测试 flaky、环境和生产对不上 | ⑥ sandbox |
| permission 权限 | 它绕过禁区、读到不该读的、动了不该动的 | ⑤ 权限 |
| review 评审 | 错误漏过了验收、自己批了自己的活 | ⑧ review |

**⑥ 写结业报告。** 不长，但必须回答三件事：在这个仓库上，什么活适合交给 agent（用三次 run 的实际结果说话，不是凭感觉）、什么活不适合（哪几类任务你试过或判断不该交）、下一轮怎么改 harness（failure taxonomy 直接指向要加厚的层）。这份报告就是你这次 Capstone 的真正产物——它证明的不是"agent 能干活"，是"你会设计让 agent 能干活的环境"。

## 真实案例

下面走一遍一个示意性的 Capstone，让六步落到具体。设定是一个内部 CLI 工具仓库——一个 Python 写的、把 CSV 报表转成几种格式的命令行工具，有一套 pytest、单人维护、风险可控。这是为讲清做法虚构的例子，数字都是示意。

**① 文档。** 作者写了一份 AGENTS.md：命令一栏列了 `make test`、`make lint`、`make typecheck`；禁区一栏写了"不要改 `parsers/legacy/`（有外部用户依赖其行为）、不要碰 `release.sh`、默认禁止联网装新依赖"；DoD 一栏写了"新行为必须有 pytest 覆盖，测真实行为不迎合实现"。runbook 里补了"本地起一个示例转换"和"测试 flaky 时先重跑 CSV fixture 生成"。

**② 分活。** 从 backlog 挑了 10 个，分诊结果大致是：6 个低风险（补 `--help` 文案、加一个日期格式、补三个解析器的单测……），3 个中风险（重构输出层、给 CLI 加一个新子命令、改一个公开的返回结构），1 个高风险（改 `parsers/legacy/` 的一处行为）。高风险那个直接标了"不交给 agent，只让它出方案"。

**③ harness。** 每次 run 在 `git worktree` 开的独立目录里跑；agent 能自由跑测试和 lint，但装新依赖、改 CI、碰禁区文件需要人点头；约定每次 run 在 worktree 根下留一个 `HANDOFF.md`（这次做了什么、跑了哪些验证、哪里没把握）。

**④ 三次 run。** 实现选了"加新子命令"那个中风险任务；测试补全选了"补三个解析器单测"；review 让第二个 agent 去审实现那次的 diff。

**⑤ failure taxonomy（示意）。** 三次里两次一遍过，一次出了状况，归档后这张表长这样：

| run | 结果 | 失败类别 | 一句话 |
|---|---|---|---|
| 实现·加子命令 | 一遍过 | — | issue 写得细，验收命令明确，worktree 隔离干净 |
| 测试补全 | 返工一次 | spec | issue 没说"边界用例要包含空文件和超大文件"，第一版只覆盖了正常输入 |
| review | 抓到一个真问题 | tool | 被审的那次 run 把一条 lint 警告当成了无关噪声，其实是改坏了类型注解 |

返工那次的教训直接回流：AGENTS.md 的 DoD 加了一句"补测试时必须覆盖空输入、超大输入、畸形输入三类边界"。trace 暴露问题、问题变成教训、教训写回规则——这正是 harness 第九层"把教训喂回前几层"的实样。

**⑥ 报告（节选）。** "适合交给 agent：有清晰验收命令、范围局部、有测试垫底的任务（本轮 6 个低风险 + 部分中风险都干得不错）。不适合：`parsers/legacy/` 这类'行为即契约、没有显式 spec'的改动——agent 没法从代码反推出外部用户依赖的是哪条边界行为，这种必须人来定。下一轮改 harness：① 给 issue 模板强制加'边界用例清单'一栏；② 把 lint 警告纳入 eval 的必过项，别让 agent 自行判断哪条是噪声。"

这份报告短，但它把全书用了一遍：第一章的"产物变了"，第六章的"模糊需求代价更高"，第九章的"防迎合实现"，第十九章的"trace 当评审材料"，第七卷的"失败分类学"——都在里面落了地。

## 常见反模式

- **挑简单任务交差**：专挑十个"改文案"级别的活，全绿，报告写得漂亮。这骗的是自己——你没碰到中高风险任务，等于没测过 harness 真正要挡的那些风险。十个任务必须覆盖低中高三档。
- **跑通了就当成功**：三次 run 都过了 PR，于是宣布毕业。可你没存 trace、没归档失败、没写报告——你证明的只是"这三次运气好"，不是"环境能复现"。Capstone 的产物是那套可复盘的工件，不是三个绿 PR。
- **把约束全写进 prompt**：禁区、权限、验收标准全堆在给 agent 的提示词里，工具层和 sandbox 一片空白。提示词崇拜在 demo 里看着没事，真有个 agent 绕过去读了 secret，你才发现那根本不是边界。
- **没有失败也不记**：三次都成功就跳过第五步。但"成功"里几乎总藏着小偏差——它绕了个远路、它差点动了禁区、它的测试其实没测到点子上。不把这些记成 taxonomy，下一轮 harness 就没有改进的依据。
- **报告只夸不判**：报告写成"agent 很强，效率很高"。它该回答的是"什么不适合交、下一轮改哪层"——没有这两条，这份报告就没有工程价值。

## 检查清单

- 我选的 repo，自己**够熟**、有**可跑的测试**、风险**可控**吗？
- 四份文档（AGENTS.md、README、runbook、外加 CLAUDE.md 若用 Claude）齐了吗？给一个生人能照着跑通吗？
- 10 个任务**覆盖了低/中/高三档风险**吗，还是全是简单活？
- 每个 issue 有**可验证的验收标准 + 验证命令 + 可动范围 + 禁区**吗？
- harness 六件事（权限/工具/sandbox/trace/eval/交接工件）都**落到这个仓库**上了吗，还是只停在概念？
- 禁区和权限**写在工具/sandbox 层**，还是只写在 prompt 里？
- 三次 run **覆盖了实现/测试补全/review 三种活**吗？
- 三次的 **trace、diff、验证结果、失败原因**都存下来了吗？
- failure taxonomy 里每一条，都**指向了 harness 要改的某一层**吗？
- 结业报告回答了**什么适合委派 / 什么不适合 / 下一轮怎么改**这三件事吗？

## 自测

这一章的自测，也是全书的收尾。三个问题，对着你刚搭好的那套 harness 诚实地答：

- 如果 agent 今天突然不可用了——服务挂了、额度用光了、被禁了——你这套系统还能给用户一个**体面的退路**吗？还是说所有交付都卡死、没人接得上？
- 如果 agent 答错了、改坏了，你**知道怎么发现、怎么追踪、怎么改进**吗？你有没有 eval 在出错时拦住它、有没有 trace 让你回放是哪一步坏的、有没有把这条教训写回规则？
- 如果半年后有人接手这个仓库，他能不能**只通过文档、代码、trace**就理解这个系统怎么运转、agent 在里头干了什么、哪些活是它做的、为什么这么做？

这三个问题答得越干脆，你的 harness 越成熟。答不上来的那一个，就是你下一轮该动手的地方——这也正是这本书想留给你的最后一件工具：不是某个具体的做法，是**对着这三个问题持续改进环境**的习惯。
