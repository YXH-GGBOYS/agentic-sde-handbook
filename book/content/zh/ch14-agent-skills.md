同一套"怎么做某类活"的专业流程，你不会希望每次派活都重新口述一遍，也不希望它一直占着 agent 的脑容量。Agent Skill 就是把这套流程打包成一个文件夹——平时只露个名字和简介，真要用到那类活时才整份调出来照着做。它是 agent 时代多出来的又一类工程资产，跟代码、跟 AGENTS.md 一样进版本库、一样要维护。

Anthropic 自己讲 Skill 时给的比喻很直白：写一个 skill，就像"给新人准备一份 onboarding 指南" [A15]。新人能干，但不懂你们这行这套招牌活怎么做；你不会把全公司的流程一股脑塞进他脑子，而是把"这道菜"的做法写成一份册子，要做时翻出来。

## 生活类比

给新外包配一本活页菜谱。

店里有道招牌菜，工序十几步、火候有讲究、还有一张祖传的调料配比表。你不会指望新人凭空会做，也不会逼他把全店每道菜的菜谱先背进脑子——背不下，也没必要。你的做法是：把这道菜单独写成一张活页，平时收在抽屉里，活页的封面上只写一行字"招牌红烧肉（宴席压轴菜，需提前两小时备）"。

新人接单时，先扫一眼抽屉里这排封面。今天有客人点红烧肉，他才把那张活页抽出来，照着十几步做；做到调料那步，活页里还夹着一张"配比表"的附页，需要时再翻到那一页。其余的菜谱，他根本不用碰，脑子里也不用装。

这本活页菜谱有三个讲究，每一个都对应 Agent Skill 的设计：

1. **封面只放一行**。封面写的是"这是什么菜、什么时候做"，不是整套工序。让新人一眼能判断"今天用不用得上它"，而不必把整本册子先读一遍。
2. **整份工序收在抽屉里，按需取用**。判断用得上，才把那张活页抽出来照做。用不上的菜谱，不占他当下的注意力。
3. **难的步骤配一张确定的附页**。调料配比这种"错一克就毁一锅"的环节，不靠他临场估摸，直接给一张算好的表照抄——又准又省心。

Agent Skill 就是给 agent 配的这本活页菜谱。封面是 metadata，活页正文是 SKILL.md，附页是额外的文件和脚本。

## 工程定义

**Agent Skill** 是把"做某一类活的专业流程"打包成 agent 可复用的能力，物理形态是一个有组织的文件夹：里面有指令、脚本、和资源文件 [A15]。最核心的是一个 `SKILL.md` 文件，开头是一段 YAML frontmatter，必填两个字段：`name`（名字）和 `description`（简介）[A15]。

它真正的设计点，在于**渐进式披露（progressive disclosure）**——信息分三层加载，不是一上来全塞进 context [A15]：

1. **metadata 层**：只有 skill 的 name 和 description，在 agent 启动时就装进系统提示，一直常驻。这一层的作用是让 agent "知道有这个能力存在"。
2. **核心文档层**：当 agent 判断这个 skill 跟当前任务相关，才把完整的 `SKILL.md` 内容加载进来。
3. **附加资源层**：SKILL.md 里引用到的其他文件、脚本，等真用到那一步时再加载 [A15]。

这三层对应活页菜谱的三个讲究：常驻的只有封面（name + description），整套工序（SKILL.md）按需取，难步骤的附页（额外文件/脚本）用到才翻。这样 agent 既知道"我有哪些招牌活能接"，又不必让每一套完整流程一直占着上下文窗口 [A15]。

要把这个概念跟相邻几个概念分清楚，否则很容易乱用：

- **Skill vs tool**：tool 是一个动作原语（`fetchInbox`、`searchEmails` 这种），常驻在 context 里，是 agent 决定"下一步做什么"时直接能调的手 [A14]。skill 是一整套流程的封装，平时不常驻、按需展开。一个 skill 内部可以调用多个 tool。
- **Skill vs AGENTS.md**：AGENTS.md 是这个仓库的"家规"，对所有任务一直生效（第 10 章）；skill 是"某类活的专门做法"，只在做那类活时才加载。家规是天天遵守的便条，菜谱是按单翻出来的活页。
- **Skill vs subagent**：subagent 解决的是上下文隔离——开一个有独立 context 窗口的子 agent 去筛大量信息，只把相关结论送回主 agent [A14]。skill 解决的是专业流程复用。两者正交：一个 subagent 干活时照样可以用 skill。

## 为什么重要：AI 时代的变化

第一层价值是**上下文经济**。agent 的上下文窗口有限，什么都常驻就什么都装不下、还互相干扰。把专业流程做成按需加载的 skill，等于只在桌上摆当下这道菜的菜谱，其余的收进抽屉。Anthropic 把这件事说得很朴素：渐进式披露让上下文窗口保持高效，agent 是逐层发现信息，而不是一开始就把整份 skill 内容全读进来 [A15]。

第二层价值是**固化能固化的**：该确定的地方别让模型现编。有些环节本质是确定性计算——从一份 PDF 表单里抽字段、按固定配比算调料——让模型一个 token 一个 token 地"生成"答案，又慢又可能错。skill 允许打包一段可执行代码（比如 Python 脚本），agent 直接确定性地跑它，连代码本身都不必读进 context [A15]。这一层是 skill 区别于"再写一段提示词"的关键：提示词是让模型更会现编，skill 是把能固化的部分固化掉。

第三层是**复用与共享**。同一套专业流程，写一次打成包，多个 agent、多个项目都能装上。Anthropic 说 Skill 现在已经在 Claude.ai、Claude Code、Claude Agent SDK 和 Claude Developer Platform 上都能用，下一步要做的是 skill 的完整生命周期——创建、发现、分享、使用 [A15]。换句话说，它被当成一类可以流通的资产来设计，而不只是某个项目里私藏的一小段。

顺带交代一下底下这套东西。Claude Code 背后那套 agent harness（即 Claude Code SDK）在 2025-09-29 改名为 **Claude Agent SDK**，因为同一套东西已经不止用于写代码 [A14]。这套 SDK 里 agent 的典型工作循环是四个阶段：收集上下文 → 采取行动 → 验证工作 → 重复（gather context → take action → verify work → repeat）[A14]。skill 主要喂的是头两个阶段——"收集上下文"时让 agent 知道有这套流程、"采取行动"时给它现成的脚本。

## 实践方法

Anthropic 给的几条开发建议，可以直接当操作步骤用 [A15]：

1. **从找能力缺口起步**。先拿一组有代表性的真实任务去试，看 agent 在哪类活上反复掉链子——那就是该写 skill 的地方。别凭想象提前造一堆用不上的 skill。
2. **name 和 description 站在 agent 的角度写**。这两个字段是常驻系统提示里唯一的线索，agent 全靠它判断"这个 skill 跟当前任务相不相关"。写得含糊（比如只写"文档处理"），agent 就该用时不用、不该用时乱用。写清楚"处理什么、什么时候用"。
3. **按规模拆文件**。SKILL.md 别越写越长。内容膨胀了，就把细节拆进单独的文件，由 SKILL.md 在需要时引用——这正好顺着渐进式披露的三层结构。
4. **跟着真实使用迭代**。不要一次性把 skill 设计到"完美"，而是上线后看真实使用模式，再回头改。

还有一条安全硬规则，Anthropic 专门强调：**只从可信来源安装 skill**，部署前要审计它捆绑的代码、依赖和资源文件，尤其留意会不会对外发起网络连接 [A15]。一个 skill 里能塞可执行脚本，这是它强的地方，也是它的风险面——装一个来历不明的 skill，跟在生产机上跑一段来路不明的脚本，是同一件事。

## 真实案例

**PDF 表单抽取：把确定的活交给确定的代码。** Anthropic 给的样例是一个处理 PDF 的 skill：它捆绑了一段脚本，专门从 PDF 表单里抽取字段，不需要让整份 PDF 走一遍 token 生成 [A15]。

这个例子值得拆开看，因为它正好对照两条路。

走"纯模型"那条路：把整份 PDF 喂进模型的上下文，让它逐字读、再逐字"理解"出每个字段的值。文档一长，token 烧得多；字段一多，模型还可能把字段读串、抄错值——这正是"让模型现编确定性答案"的典型出错点。

走"skill + 脚本"那条路：agent 先从抽屉里看到这个 PDF skill 的封面，判断当前任务相关，加载 SKILL.md，照里面的指引调用那段抽取脚本。脚本确定性地把表单字段读出来，结果回到 agent 手里。整个过程中，那段脚本的源码本身都不必读进 context [A15]——agent 只管"用它"，不必"理解它"。又准（确定性抽取，不靠模型估）又省（不烧 PDF 全文的 token）。

这就是 skill 跟"再加一段提示词"的根本区别落到一个具体场景里的样子：能固化成代码的步骤，固化掉；留给模型的，是判断"这个活该用哪本菜谱"。

## 常见反模式

- **把 skill 当 AGENTS.md 用**：把"每个任务都该遵守的家规"写进某个 skill。家规要常驻、要对所有任务生效，写进按需加载的 skill 里，agent 在不相关的任务上根本不会把它调出来。常驻规矩进 AGENTS.md，专项流程才进 skill。
- **description 写得含糊**：name 和 description 是常驻系统提示里唯一的判断依据。写成"文档工具"这种，agent 没法判断什么时候该用它——该用时不用，不该用时乱用。description 要写清"处理什么、什么时候用"。
- **SKILL.md 一直堆**：把所有细节、所有边角情况全塞进一个越来越长的 SKILL.md，渐进式披露的省 context 优势就没了。膨胀了就拆文件，让 SKILL.md 按需引用。
- **该写脚本的地方写成提示词**：确定性的活（抽字段、算配比、格式转换）用自然语言指挥模型一步步做，又慢又不稳。能写成脚本确定性执行的，就别让模型现编。
- **装来路不明的 skill**：skill 能捆绑可执行代码，一个不可信来源的 skill 等于一段会自动跑的陌生脚本。安装前审代码、审依赖、审有没有对外联网 [A15]。

## 检查清单

- 这个能力**该不该做成 skill**？（是"某类专门活的复用流程"，而不是"所有任务都遵守的家规"——后者进 AGENTS.md。）
- skill 的 **name 和 description** 够不够让 agent 一眼判断"当前任务用不用得上它"？
- SKILL.md 是不是**短而密**，长出来的细节拆进了**单独文件**按需引用？
- 里面有没有**确定性的步骤**，本该交给打包脚本、却让模型现编的？
- 这个 skill **来源可信**吗？捆绑的代码、依赖、对外网络连接，部署前**审过**吗？

## 自测

- 给你一个 agent 反复做不好的活，你能说清它该被写成一个 skill，还是该补进 AGENTS.md，还是只是缺一个 tool 吗？三者各解决什么问题，你分得清吗？
- 渐进式披露三层——哪一层常驻在系统提示里、哪一层判断相关时才加载、哪一层用到才取？为什么这么分能同时做到"agent 知道有这能力"又"不一直占 context"？
